# 전국횡단보도표준데이터 API 분석

## 1. 개요

이 데이터는 `도로교통법`에 따라 설치된 횡단보도 정보를 전국 단위로 모은 표준데이터다. 설명상 횡단보도의 위치, 종류, 차로 수, 폭, 연장, 보행자 신호등 유무, 음향신호기 설치 여부, 점자블록 유무, 보도턱 낮춤 여부 같은 보행 안전 관련 항목을 포함한다. 소관기관은 국토교통부와 경찰청이며, 실제 제공기관은 지방자치단체다. 갱신 주기는 반기다. 

이 데이터는 “전국” 이름을 달고 있지만, 각 지자체가 개별적으로 등록한 자료를 포털이 병합해 제공하는 구조다. 포털 안내에 따르면 개별 기관 데이터는 매월 초 병합되므로, 실제 현장과 데이터 사이에 시차가 발생할 수 있다. 따라서 실시간 신호 상태 판단용보다는 **기본 위치 정보와 시설 존재 여부를 파악하는 기준 데이터**로 보는 것이 맞다. 

---

## 2. 제공 방식

이 데이터는 파일 다운로드와 Open API 두 방식으로 제공된다. 화면에는 XLS, XML, JSON, RDF, CSV 다운로드가 보이며, Open API 탭에서 별도 조회도 가능하다. 그리드 다운로드는 5만 건 제한이 있고, 전체 데이터가 필요하면 API 활용을 권장한다고 안내되어 있다. 

Open API의 요청 주소는 다음으로 안내된다.

```text
http://api.data.go.kr/openapi/tn_pubr_public_crosswalk_api
```

활용승인은 개발단계와 운영단계 모두 자동승인이며, 신청 가능 트래픽은 개발계정 1,000으로 표시되어 있고 운영계정은 활용사례 등록 시 증설 신청이 가능하다고 되어 있다. 

---

## 3. 이 API로 조회할 수 있는 내용

요청 변수와 응답 항목을 보면, 이 API는 횡단보도 자체의 **정적 속성 정보**를 조회하는 API다. 즉 “이 위치에 어떤 형태의 횡단보도가 있고, 주변 편의시설이 무엇이 있는가”를 확인하는 데 적합하다. 반면 **현재 신호가 녹색인지 적색인지 같은 실시간 상태**는 제공하지 않는다. 응답 항목에는 녹색신호시간, 적색신호시간은 있으나 이는 타이밍 정보일 뿐 실시간 점등 상태 값은 아니다. 

특히 보행 보조 서비스 관점에서 중요한 항목은 다음과 같다. 

- 위치: 위도, 경도
    
- 식별: 횡단보도관리번호
    
- 구조: 횡단보도종류, 차로수, 폭, 연장
    
- 접근성: 보도턱낮춤여부, 점자블록유무
    
- 안전 보조: 보행자신호등유무, 보행자작동신호기유무, 음향신호기설치여부, 집중조명시설유무
    
- 주변 맥락: 시도명, 시군구명, 도로명, 주소
    
- 관리 정보: 관리기관명, 전화번호, 데이터기준일자
    

---

## 4. 요청 변수 분석

기본 제어용 변수는 아래와 같다. 

- `pageNo`: 페이지 번호
    
- `numOfRows`: 한 페이지 결과 수
    
- `type`: XML/JSON 여부
    

조건 검색용 변수는 거의 모든 컬럼을 그대로 필터로 걸 수 있게 되어 있다. 대표적으로 다음이 있다. 

- `ctprvnNm`: 시도명
    
- `signguNm`: 시군구명
    
- `roadNm`: 도로명
    
- `rdnmadr`: 도로명주소
    
- `lnmadr`: 지번주소
    
- `crslkManageNo`: 횡단보도관리번호
    
- `latitude`, `longitude`: 위도, 경도
    
- `tfclghtYn`: 보행자신호등 유무
    
- `sondSgngnrYn`: 음향신호기 설치 여부
    
- `brllBlckYn`: 점자블록 유무
    
- `referenceDate`: 데이터기준일자
    
- `instt_code`: 제공기관코드
    

### 해석 포인트

이 API는 좌표 기반 반경 검색이나 거리순 정렬 전용 API가 아니다. 위도와 경도 항목이 있긴 하지만, 문서상으로는 단순 필드 조회용 변수로 제시되어 있다. 따라서 실서비스에서는 보통 다음 순서가 더 현실적이다. 

1. 지역 단위로 API 데이터를 수집
    
2. 서버나 앱 내부에서 좌표 거리 계산
    
3. 사용자 현재 위치 기준으로 가까운 횡단보도 선별
    
4. 카메라 인식 결과와 결합해 최종 안내

## 5. 응답 데이터가 의미하는 것

응답 요소는 요청 변수와 거의 같은 구조다. 즉 이 API는 복잡한 중첩 구조보다는 “횡단보도 1건 = 속성 묶음 1개” 형태로 해석하면 된다. 

### 보행 보조에 특히 중요한 항목 해석

#### `tfclghtYn` 보행자신호등유무

횡단보도에 보행자용 신호등이 있는지 여부다. 단순 존재 여부이지, 지금 신호가 바뀌었는지는 알 수 없다. 

#### `fnctngSgngnrYn` 보행자작동신호기유무

버튼을 눌러 작동시키는 신호기가 있는지 여부다. 시각장애인 서비스에서는 “이 횡단보도는 버튼 조작이 필요할 수 있음” 같은 사전 안내에 쓸 수 있다. 

#### `sondSgngnrYn` 음향신호기설치여부

음향신호기 설치 여부다. 시각장애인 보행 보조에서 매우 중요한 속성이다. 다만 이것도 “설치 여부”이지 “현재 정상 작동 여부”는 아니다. 

#### `brllBlckYn` 점자블록유무

점자블록 존재 여부다. 보행 기준선을 잡는 데 중요한 참고 데이터가 된다. 다만 실제 연결 상태나 파손 상태는 별도 비전 인식이나 현장 검증이 필요하다. 

#### `ftpthLowerYn` 보도턱낮춤여부

보도 턱이 낮춰져 있는지 여부다. 휠체어 사용자나 고령자뿐 아니라, 보행약자의 횡단 접근성을 판단하는 데 유용하다. 

#### `greenSgngnrTime`, `redSgngnrTime`

녹색/적색 신호 시간 정보다. 이 값이 있다고 해서 실시간 신호 상태를 판별할 수 있는 것은 아니지만, “보행 가능 시간이 짧은 횡단보도”를 분류하거나 경고하는 데는 사용할 수 있다.

## 6. 장점

### 6-1. 시각장애인 보행 서비스와 잘 맞는 속성이 많다

단순 위치 정보만 있는 것이 아니라 음향신호기, 점자블록, 보도턱 낮춤, 보행자작동신호기 같은 **접근성 중심 필드**가 포함되어 있어서, 일반 지도 데이터보다 보행약자 서비스에 더 직접적으로 연결된다. 

### 6-2. 전국 단위 기준 데이터로 쓸 수 있다

지자체 단위 데이터를 전국 표준 형식으로 모아두었기 때문에, 지역별 포맷 차이를 줄이면서 서버 수집 파이프라인을 만들기 좋다. 

### 6-3. 자동승인이라 초기 개발 진입이 쉽다

개발단계와 운영단계가 자동승인이므로, 프로토타입 제작과 테스트에 빠르게 들어갈 수 있다. 

---

## 7. 한계와 주의점

### 7-1. 실시간 신호 상태는 제공하지 않는다

이 API만으로는 “지금 건너도 되는가”를 판단할 수 없다. 현재 시각장애인 보행 안내에서 필요한 실시간 신호 판별은 카메라 기반 신호등 인식이나 별도 신호제어 데이터와 결합해야 한다. 

### 7-2. 병합 시차가 있다

포털 안내상 지자체 등록 데이터를 매월 초 병합하므로, 현장에 이미 바뀐 횡단보도가 바로 반영되지 않을 수 있다. 따라서 안내 문구는 “근처 횡단보도 정보” 수준으로 두고, 최종 횡단 판단은 실시간 인식에 맡기는 것이 안전하다. 

### 7-3. 필드 값만으로 품질을 완전히 신뢰하면 안 된다

점자블록유무가 `Y`라고 해도 실제 현장에서는 일부 구간만 설치되었거나 연결이 끊겨 있을 수 있다. 음향신호기 역시 설치는 되어 있으나 고장일 수 있다. 이 데이터는 **존재 여부의 기준 데이터**로 보고, 현장 상태는 별도 센서나 사용자 제보로 보완하는 구조가 좋다. 이는 포털이 개별 지자체 자료를 병합하는 구조라는 점에서도 자연스러운 해석이다. 

### 7-4. 거리 기반 검색은 직접 구현해야 한다

문서상 요청 변수는 컬럼 기반 조회이고, 반경 탐색 전용 파라미터는 보이지 않는다. 따라서 “내 주변 30m 횡단보도” 같은 기능은 API 호출 뒤 서버 측 거리 계산으로 처리하는 편이 현실적이다.

## 8. 네 프로젝트에 맞춘 활용 방향

시각장애인 안전 보행 보조 플랫폼 관점에서는 이 API를 **사전 지식 데이터베이스**로 쓰는 것이 가장 적절하다. 

### 추천 활용 구조

#### 1단계. 서버에 지역별 횡단보도 데이터 적재

시도명, 시군구명 단위로 주기적으로 수집해서 DB에 저장한다. 반기 갱신 데이터지만, 포털 병합 시차를 고려해 정기 동기화 전략을 두는 것이 좋다. 

#### 2단계. 사용자 현재 위치 근처 후보 횡단보도 탐색

앱에서 GPS를 받고, 서버에서 가까운 횡단보도 후보를 거리순으로 뽑는다. 이때 `위도`, `경도`, `차로수`, `연장` 값은 난이도 추정에도 쓸 수 있다. 

#### 3단계. 카메라 인식과 결합

API가 알려주는 후보 위치 근처에서만 횡단보도, 신호등, 점자블록을 비전 모델로 집중 탐지하면 탐지 범위를 줄이고 정확도를 높이기 쉽다. 이 API는 위치 후보를 주고, 비전 모델은 실시간 상태를 판별하는 역할로 나누면 된다. 

#### 4단계. 안내 문장 생성

예를 들어 다음과 같은 안내가 가능하다.

- “전방에 횡단보도가 있습니다.”
    
- “음향신호기가 설치된 횡단보도입니다.”
    
- “보행자 버튼 신호기가 있는 횡단보도일 수 있습니다.”
    
- “점자블록이 있는 횡단보도 구간입니다.”
    

이런 문장은 `sondSgngnrYn`, `fnctngSgngnrYn`, `brllBlckYn`, `tfclghtYn` 기반으로 만들 수 있다.

## 9. 실무적으로 중요한 결론

이 API는 **횡단보도 존재 여부와 접근성 속성을 사전에 파악하는 데 매우 유용한 정적 데이터 API**다. 하지만 **실시간 보행 결정 API는 아니다**. 따라서 네 서비스에서는 이 데이터를 “지도 위 지식 레이어”로 쓰고, 실제 건너기 판단은 카메라 기반 횡단보도/신호등 인식과 결합하는 방식이 가장 적합하다.

**1) 요청/응답 예시**, **2) 파이썬 호출 코드**, **3) Spring Boot 연동 방식**, **4) 우리 서비스용 DB 스키마 설계**까지 바로 정리해줄게.
