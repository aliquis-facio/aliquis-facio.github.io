# 1. API 개요

이 API는 **전국 횡단보도 표준데이터 조회 API**다. 요청 주소는 `https://api.data.go.kr/openapi/tn_pubr_public_crosswalk_api`이며, 횡단보도의 **위치, 종류, 차로수, 폭, 연장, 보행자신호등 유무, 음향신호기 설치 여부, 점자블록 유무, 보도턱낮춤 여부** 같은 정적 시설 정보를 조회할 수 있다. 개발계정 기본 트래픽은 **1,000**이고, 개발·운영 단계 모두 **자동승인**으로 표시되어 있다.

도메인으로 보면 이 API는 **실시간 신호 제어 API**가 아니라 **보행 인프라 시설 데이터 API**에 가깝다. 즉, “지금 신호가 파란불인가”를 알려주는 API가 아니라, “여기에 어떤 횡단보도가 있고 어떤 보행 보조 시설이 붙어 있는가”를 알려주는 API다.

우리 서비스에서 중요한 이유는 분명하다.

- 사용자의 현재 위치 주변에 **횡단보도가 존재하는지**
    
- 그 횡단보도에 **보행자신호등이 있는지**
    
- **음향신호기, 점자블록, 보도턱낮춤** 같은 시각장애인 친화 요소가 있는지
    
- **차로수, 횡단 길이** 기준으로 횡단 난도를 가늠할 수 있는지
    

를 확인할 수 있기 때문이다. 즉, 이 API는 시각장애인 보행에서 중요한 **랜드마크**와 **횡단 판단 단서**를 제공한다. 다만 실제 이동 안전을 완성하려면 여전히 GPS, 방향 센서, 카메라 AI, 경로 API가 함께 필요하다.

비슷한 다른 API와 비교하면 역할은 다음처럼 정리된다.

- **지도 경로 API**: 어디로 가야 하는지 알려줌
    
- **신호 제어/실시간 교통 API**: 지금 건너도 되는지 알려줌
    
- **이 API**: 그 횡단보도가 어떤 시설 조건을 가진 횡단보도인지 알려줌
    

즉, 이 API는 **횡단보도 자체의 속성 데이터**에 가장 적합하다.

# 2. 지원 가능한 기능 분석

## 경로 안내

- 사용 가능 여부: **직접은 불가**
    
- 활용 방식: 경로 생성은 못 하지만, 경로 API가 만든 이동선 위에 있는 횡단보도 후보를 보강하는 용도로 사용 가능
    
- 필요한 입력값: 지역명, 도로명, 주소, 관리번호 등
    
- 핵심 응답값: `latitude`, `longitude`, `crslkManageNo`, `roadNm`
    
- 앱 실제 시나리오: 경로 API가 “횡단 필요”를 알려주면, 이 API로 근처 횡단보도 시설 정보를 조회해 어떤 횡단 지점을 우선 안내할지 판단
    

정리하면 이 API는 **길을 만드는 API가 아니라, 길 위의 횡단 지점을 해석하는 API**다.

## 횡단보도 탐색

- 사용 가능 여부: **가능**
    
- 활용 방식: 시도명, 시군구명, 도로명, 주소 등으로 횡단보도 목록 조회
    
- 필요한 입력값: `ctprvnNm`, `signguNm`, `roadNm`, `rdnmadr`, `lnmadr`, `crslkManageNo`
    
- 핵심 응답값: `latitude`, `longitude`, `crslkKnd`, `cartrkCo`, `bt`, `et`
    
- 앱 실제 시나리오: 사용자가 특정 도로 근처에 있을 때 주변 횡단보도 후보를 조회하고, 가장 가까운 횡단보도를 안내
    

이 기능은 너희 서비스에서 바로 쓸 수 있다. 특히 “지금 근처에 횡단 가능한 지점이 있는가”를 판단하는 기본 데이터로 적합하다.

## 신호등 상태 확인

- 사용 가능 여부: **부분 가능**
    
- 활용 방식: **신호등 존재 여부**만 확인 가능
    
- 필요한 입력값: 지역/도로/주소 기반 검색
    
- 핵심 응답값: `tfclghtYn`, `fnctngSgngnrYn`, `greenSgngnrTime`, `redSgngnrTime`
    
- 앱 실제 시나리오: “이 횡단보도는 보행자 신호등이 있는 횡단보도입니다” 같은 사전 안내
    

중요한 한계가 있다. 이 API는 문서상 **보행자신호등유무**, **녹색신호시간**, **적색신호시간** 필드는 있지만, **현재 시점의 실제 신호 상태**를 주는 것은 아니다. 따라서 “지금 빨간불입니다” 같은 실시간 안내는 이 API만으로는 불가능하다. 그 부분은 별도 신호 데이터나 비전 AI가 필요하다.

## 보행자 안전 경고

- 사용 가능 여부: **부분 가능**
    
- 활용 방식: 시설 속성을 바탕으로 위험도 판단
    
- 필요한 입력값: 횡단보도 조회 조건
    
- 핵심 응답값: `cartrkCo`, `et`, `tfcilndYn`, `ftpthLowerYn`, `brllBlckYn`, `sondSgngnrYn`
    
- 앱 실제 시나리오: 차로수가 많고 횡단 길이가 길며 음향신호기가 없는 횡단보도는 주의 등급을 높여 안내
    

이 API는 실시간 위험을 주지는 않지만, **정적 위험도 점수화**에는 꽤 유용하다.

예를 들면 이런 식이다.

- 차로수 많음 + 횡단 길이 길음 + 교통섬 없음 → 위험도 높음
    
- 보도턱낮춤 있음 + 점자블록 있음 + 음향신호기 있음 → 보행약자 친화도 높음
    

## 주변 시설 탐색

- 사용 가능 여부: **제한적**
    
- 활용 방식: 횡단보도 자체만 탐색 가능
    
- 필요한 입력값: 지역/도로/주소
    
- 핵심 응답값: 위치, 주소, 도로명
    
- 앱 실제 시나리오: 일반 주변 시설 탐색은 어렵고, 횡단보도 시설만 탐색 가능
    

즉, 편의점·버스정류장·건물 입구 같은 일반 랜드마크 탐색용 API는 아니다.

## 위치 기반 음성 안내

- 사용 가능 여부: **가능**
    
- 활용 방식: 횡단보도 속성 데이터를 음성 문장으로 변환
    
- 필요한 입력값: 조회 결과 + 현재 위치
    
- 핵심 응답값: `latitude`, `longitude`, `tfclghtYn`, `sondSgngnrYn`, `brllBlckYn`, `ftpthLowerYn`, `cartrkCo`, `et`
    
- 앱 실제 시나리오: “10미터 앞 횡단보도에 음향신호기가 있습니다” 같은 안내 생성
    

이 API는 음성 안내 원재료로 매우 좋다. 다만 원문 데이터를 그대로 읽는 것이 아니라, 사용자 관점으로 다시 해석해야 한다.

## 실시간성 필요한 기능

- 사용 가능 여부: **제한적**
    
- 활용 방식: 사용자가 이동할 때 주변 횡단보도 목록을 재조회하는 정도
    
- 필요한 입력값: 위치를 주소 또는 지역 기준으로 변환한 값
    
- 핵심 응답값: 위치와 시설 속성
    
- 앱 실제 시나리오: 이동 중 새로운 횡단보도 후보 재검색
    

하지만 이 API는 정적 시설 데이터 성격이 강하고, 문서에도 데이터 병합 시점 차이 가능성이 명시되어 있다. 따라서 완전한 실시간 API로 보면 안 된다.

## 사전 데이터 조회 기능

- 사용 가능 여부: **가능**
    
- 활용 방식: 특정 지역의 횡단보도 인프라를 미리 수집해 캐시하거나 위험도 맵 생성
    
- 필요한 입력값: 페이지, 행 수, 지역/도로 조건
    
- 핵심 응답값: 전체 시설 속성 필드
    
- 앱 실제 시나리오: 특정 구역의 “안전한 횡단보도 후보 목록”을 미리 만들어 두기
    

이 용도는 MVP 이후 꽤 중요하다. 사전 적재를 하면 앱 반응성이 좋아지고, 안전 점수 계산도 쉬워진다.

# 3. 엔드포인트별 상세 분석

## `https://api.data.go.kr/openapi/tn_pubr_public_crosswalk_api`

### 1) 역할

- 전국 횡단보도 표준데이터를 조회하는 Open API
    
- 횡단보도의 위치와 시설 속성을 조회
    
- 시각장애인 보행에서 중요한 횡단 관련 보조 시설 존재 여부를 확인 가능
    

### 2) 요청 방식

- HTTP Method: **문서에 명시 없음**
    
    - 공공데이터포털 Open API 관례상 **GET일 가능성이 높다**
        
    - 하지만 이 부분은 현재 자료 기준 **추정**
        
- URL 구조: `https://api.data.go.kr/openapi/tn_pubr_public_crosswalk_api`
    
- Path Variable: **없음**
    
- Query Parameter:
    
    - `pageNo`: 페이지 번호
        
    - `numOfRows`: 한 페이지 결과 수, 최대 1000
        
    - `type`: XML/JSON 여부
        
    - `ctprvnNm`: 시도명
        
    - `signguNm`: 시군구명
        
    - `roadNm`: 도로명
        
    - `rdnmadr`: 소재지도로명주소
        
    - `lnmadr`: 소재지지번주소
        
    - `crslkManageNo`: 횡단보도관리번호
        
    - `crslkKnd`: 횡단보도종류
        
    - `bcyclCrslkCmbnatYn`: 자전거횡단도겸용여부
        
    - `highlandYn`: 고원식적용여부
        
    - `latitude`: 위도
        
    - `longitude`: 경도
        
    - `cartrkCo`: 차로수
        
    - `bt`: 횡단보도폭
        
    - `et`: 횡단보도연장
        
    - `tfclghtYn`: 보행자신호등유무
        
    - `fnctngSgngnrYn`: 보행자작동신호기유무
        
    - `sondSgngnrYn`: 음향신호기설치여부
        
    - `greenSgngnrTime`: 녹색신호시간
        
    - `redSgngnrTime`: 적색신호시간
        
    - `tfcilndYn`: 교통섬유무
        
    - `ftpthLowerYn`: 보도턱낮춤여부
        
    - `brllBlckYn`: 점자블록유무
        
    - `cnctrLghtFcltyYn`: 집중조명시설유무
        
    - `institutionNm`: 관리기관명
        
    - `phoneNumber`: 관리기관전화번호
        
    - `referenceDate`: 데이터기준일자
        
    - `instt_code`: 제공기관코드
        
- Header:
    
    - 현재 자료에는 명시 없음
        
    - 공공데이터포털 API 특성상 **서비스키가 필요할 가능성이 높음**
        
    - 이 부분은 현재 자료 기준 **추정**
        
- Body: 없음으로 보는 것이 자연스럽지만, 이 역시 **추정**
    

### 3) 응답 데이터

주요 필드는 요청 필드와 거의 같은 구조로 내려오는 것으로 보인다.

- `ctprvnNm`, `signguNm`: 행정구역
    
- `roadNm`, `rdnmadr`, `lnmadr`: 위치 설명
    
- `crslkManageNo`: 횡단보도 식별용 관리번호
    
- `crslkKnd`: 횡단보도 종류
    
- `latitude`, `longitude`: 위치 좌표
    
- `cartrkCo`: 차로수
    
- `bt`: 횡단보도 폭
    
- `et`: 횡단보도 연장
    
- `tfclghtYn`: 보행자신호등 유무
    
- `fnctngSgngnrYn`: 보행자작동신호기 유무
    
- `sondSgngnrYn`: 음향신호기 설치 여부
    
- `greenSgngnrTime`, `redSgngnrTime`: 신호 시간
    
- `tfcilndYn`: 교통섬 유무
    
- `ftpthLowerYn`: 보도턱낮춤 여부
    
- `brllBlckYn`: 점자블록 유무
    
- `cnctrLghtFcltyYn`: 집중조명시설 유무
    
- `institutionNm`, `phoneNumber`, `referenceDate`, `instt_code`: 관리 및 메타 정보
    

실제 서비스에서 중요한 필드는 `latitude`, `longitude`, `tfclghtYn`, `sondSgngnrYn`, `brllBlckYn`, `ftpthLowerYn`, `cartrkCo`, `et`, `tfcilndYn` 쪽이다.

의미가 불명확한 필드도 있다.

- `crslkKnd`: 종류 코드가 숫자인지 명칭인지, 각 값 매핑표는 현재 자료에 없음
    
- `bt`, `et`: 단위는 횡단보도폭/연장이므로 보통 미터로 추정되지만, 현재 자료에는 단위 명시가 없어 **추정**
    
- `cnctrLghtFcltyYn`: 집중조명시설의 실제 현장 의미와 설치 기준은 추가 문서 확인 필요
    

### 4) 서비스 활용도

우리 앱에서 붙일 수 있는 위치는 분명하다.

- **음성 안내**: 매우 적합
    
- **진동 안내**: 횡단 시작 지점 접근 시 적합
    
- **지도 표시**: 적합
    
- **위험 감지**: 단독으로는 제한적이지만 위험도 산정용으로 적합
    

단독 사용 가능 범위는 다음 정도다.

- 횡단보도 존재 확인
    
- 보행 친화 속성 확인
    
- 사전 위험도 판단
    
- 음성 안내 문장 생성
    

단독으로 불가능한 것은 다음이다.

- 실시간 신호 상태 판단
    
- 실시간 장애물 탐지
    
- 실제 경로 생성
    
- 사용자의 진행 방향 정합성 판단
    

그래서 경로 API, GPS/나침반, 카메라 AI와 조합하는 구조가 맞다.

### 5) 한계 및 주의점

- 실시간성 한계: 정적 시설 데이터 성격이 강함
    
- 정확도 한계: 데이터 병합 시차가 있을 수 있음
    
- 좌표계 문제 가능성: 위도/경도 제공이므로 WGS84 계열로 보이지만 문서에 좌표계 명시는 없음
    
- 데이터 누락 가능성: 일부 필드 공란 가능, 특히 신호시간 값이 비어 있는 사례가 자료에 보임
    
- 지역별 편차 가능성: 개별 기관 등록 데이터를 병합하므로 지역별 품질 차이 가능
    
- 호출량 제한 가능성: 개발계정 1,000 트래픽 표시
    
- 인증 필요 여부: 공공데이터포털 API 특성상 서비스키 필요 가능성이 높음, 현재 자료에서는 명시 부족
    
- 모바일 환경 주의점: 위치 이동마다 전량 조회하면 비효율적이므로 서버 캐시 또는 지역 단위 사전 적재 필요

# 4. 핵심 필드 정리

- `latitude`
    
    - 의미: 횡단보도 위도
        
    - 왜 중요한지: 사용자 현재 위치와 거리 계산의 기준
        
    - 앱에서 어떻게 쓰는지: 주변 횡단보도 탐색, 가장 가까운 횡단보도 선택
        
- `longitude`
    
    - 의미: 횡단보도 경도
        
    - 왜 중요한지: 위치 좌표 완성

# 5. 사용자 안내 관점 해석

이 API를 기술 값 그대로 쓰면 안 된다. 시각장애인 사용자는 **시설의 존재**, **안전 수준**, **다음 행동**이 중요하다.

예를 들면:

- `tfclghtYn=Y`
    
    - “앞 횡단보도에는 보행자 신호등이 있습니다.”
        
- `tfclghtYn=N`
    
    - “앞 횡단보도는 신호등이 없는 횡단보도입니다. 차량 흐름을 더 주의하세요.”
        
- `sondSgngnrYn=Y`
    
    - “이 횡단보도에는 음향신호기가 설치되어 있습니다.”
        
- `brllBlckYn=Y`
    
    - “횡단보도 진입부에 점자블록이 있습니다.”
        
- `ftpthLowerYn=Y`
    
    - “보도 턱이 낮아진 진입 구간이 있습니다.”
        
- `cartrkCo`가 큼, `et`가 김
    
    - “넓은 도로를 건너야 합니다. 천천히 접근하고 신호 확인이 필요합니다.”
        
- `tfcilndYn=Y`
    
    - “중간 대기 지점이 있는 횡단보도입니다.”
        
- `fnctngSgngnrYn=Y`
    
    - “신호 요청 장치가 있을 수 있으니 횡단 전 확인하세요.”
        

즉, 기술 데이터는 다음 구조로 바꾸는 것이 좋다.

1. 앞에 무엇이 있는지
    
2. 그 시설이 얼마나 안전한지
    
3. 사용자가 무엇을 해야 하는지
    

예시 문장:

- “15미터 앞에 횡단보도가 있습니다.”
    
- “이 횡단보도에는 음향신호기가 있습니다.”
    
- “차로가 많은 큰 도로입니다. 횡단 전 충분히 대기하세요.”
    
- “점자블록이 있는 진입부로 안내합니다.”
    
- “신호등 없는 횡단보도입니다. 차량 소리를 주의해 주세요.”

# 7. 부족한 점과 추가 구현 필요 사항

이 API만으로 부족한 부분은 명확하다.

첫째, **실시간 장애물 탐지 불가**다.  
킥보드, 공사 바리케이드, 적치물, 불법 주정차 같은 현장 장애물은 알 수 없다.  
보완 방법은 **비전 AI 모델**이다.

둘째, **현재 신호 상태 제공 불가**다.  
`greenSgngnrTime`, `redSgngnrTime`은 있어도 현재 빨간불/초록불 상태는 없다.  
보완 방법은 **실시간 신호 데이터 API** 또는 **카메라 기반 신호등 인식**이다.

셋째, **경로 생성 기능 없음**이다.  
횡단보도 자체 정보는 있지만, 사용자를 목적지까지 어떻게 안내할지는 알 수 없다.  
보완 방법은 **보행 경로 API** 추가다.

넷째, **현재 사용자 방향 정보 없음**이다.  
횡단보도가 가까워도 사용자가 어디를 바라보는지 알 수 없다.  
보완 방법은 **자이로/나침반 센서** 활용이다.

다섯째, **좌표 기반 주변 탐색 구조가 불명확하다**.  
문서상 위도·경도 필드가 요청 변수에도 있지만, 반경 검색 전용 구조는 현재 자료에 없다.  
즉, 직접적인 “내 주변 50미터 횡단보도 조회”가 되는지 추가 확인이 필요하다.  
보완 방법은 서버에서 **지역 조건 조회 + 좌표 후처리 필터링**을 하는 것이다.

여섯째, **횡단보도 종류 코드 해석표 부족**이다.  
`crslkKnd`는 중요한 필드지만 코드 의미가 현재 자료에 명확히 없다.  
보완 방법은 원본 데이터셋 설명서 또는 샘플 데이터 분석이 필요하다.

일곱째, **데이터 최신성 한계**가 있다.  
공공데이터 병합 시차가 있을 수 있으므로, 실제 현장과 다를 수 있다.  
보완 방법은 `referenceDate` 기준 품질 관리와 현장 피드백 수집이다.

# 9. 최종 결론

- 이 API는 **횡단보도 시설 속성 조회**에 가장 적합하다.
    
- 우리 서비스에서는 **횡단보도 존재 확인, 보행 친화 요소 확인, 횡단 위험도 보조 판단**까지 활용 가능하다.
    
- 단독으로는 충분하지 않다. **경로 API, 실시간 신호 인식, 장애물 탐지 AI**가 반드시 보완되어야 한다.
    
- 도입 우선순위는 **높은 편**이다. 이유는 시각장애인 보행에서 중요한 횡단보도 관련 핵심 정적 데이터를 바로 얻을 수 있기 때문이다.
    
- 실제 구현 시 가장 중요한 주의점은 **이 API를 실시간 신호 API로 오해하지 말 것**, **좌표·지역 검색 구조를 서버에서 후처리할 것**, **시설 정보와 실제 현장 상황을 구분할 것**이다.
