## 횡단보도 API 분석

제공된 명세를 기준으로 보면, 이 API는 **공공데이터포털의 전국 횡단보도 정보 조회 API**다. 요청 주소는 `http://api.data.go.kr/openapi/tn_pubr_public_crosswalk_api`이며, 개발계정은 **자동승인**, 개발 트래픽은 **1,000건**으로 표시되어 있다. 요청 시 페이지, 출력 형식(XML/JSON), 지역명, 도로명, 주소, 횡단보도 관리번호, 위도/경도, 신호등 유무 등으로 조회할 수 있다. 응답에는 횡단보도 위치, 구조, 신호 관련 정보, 보행 약자 편의 요소가 포함된다. 

### 1. 이 API가 하는 일

이 API는 횡단보도 하나하나에 대해 아래 정보를 내려주는 **정적 시설 정보 조회 API**다.

* 어디에 있는 횡단보도인지
* 어떤 종류의 횡단보도인지
* 자전거 횡단도 겸용인지
* 보행자 신호등이 있는지
* 음향신호기가 있는지
* 점자블록이 있는지
* 보도턱 낮춤이 있는지
* 녹색/적색 신호 시간이 얼마인지

즉, 이 API는 **“현재 신호가 파란불인지 빨간불인지”를 실시간으로 알려주는 API가 아니라**,
**“이 위치의 횡단보도는 어떤 시설을 갖추고 있는지”를 알려주는 API**라고 보는 것이 맞다. 응답 항목에 `greenSgngnrTime`, `redSgngnrTime`, `tfclghtYn`, `sondSgngnrYn`, `brllBlckYn` 등이 포함되어 있다. 

---

## 2. 요청 구조 분석

### 기본 요청 파라미터

명세상 기본 파라미터는 아래와 같다. 

* `pageNo`: 페이지 번호
* `numOfRows`: 한 페이지 결과 수
* `type`: `xml` 또는 `json`

즉, 다건 조회를 전제로 하는 **목록형 API**다.

### 검색 조건 파라미터

이 API는 다양한 조건으로 필터링할 수 있다. 

* 행정구역 기반
	  * `ctprvnNm`: 시도명
	  * `signguNm`: 시군구명
* 도로/주소 기반
	  * `roadNm`: 도로명
	  * `rdnmadr`: 도로명주소
	  * `lnmadr`: 지번주소
* 개별 시설 기반
	  * `crslkManageNo`: 횡단보도관리번호
	  * `crslkKnd`: 횡단보도종류
* 위치 기반
	  * `latitude`: 위도
	  * `longitude`: 경도
* 시설 속성 기반
	  * `tfclghtYn`: 보행자신호등유무
	  * `fnctngSgngnrYn`: 보행자작동신호기유무
	  * `sondSgngnrYn`: 음향신호기설치여부
	  * `ftpthLowerYn`: 보도턱낮춤여부
	  * `brllBlckYn`: 점자블록유무

### 해석 포인트

이 명세를 보면 `latitude`, `longitude`가 있긴 하지만, 보통 이런 공공데이터 API는 **반경 검색**이나 **근접 정렬**을 직접 지원하는 구조가 아니다.
즉, 위도/경도를 넣는다고 해서 “내 주변 50m 횡단보도”를 바로 주는 형태라고 단정하기는 어렵다. 명세에는 **거리 기준 검색 방식**이나 **반경 파라미터**가 없다. 그래서 실제 서비스에서는 다음 방식으로 쓰는 것이 일반적이다.

1. 시군구/도로명 등으로 후보를 조회
2. 응답으로 받은 위도/경도를 앱 서버에서 계산
3. 사용자 현재 위치와 가장 가까운 횡단보도를 직접 판별

따라서 이 API의 위경도는 **검색 보조 조건 또는 결과 필드**로 보는 편이 안전하다.

## 3. 응답 데이터 분석

응답 필드는 요청 필드와 거의 같은 수준으로 내려온다. 

### 위치/식별 정보

* `ctprvnNm`, `signguNm`
* `roadNm`
* `rdnmadr`, `lnmadr`
* `crslkManageNo`

이 값들은 **횡단보도 식별과 지도 매핑**에 중요하다.

### 구조 정보

* `crslkKnd`: 횡단보도 종류
* `bcyclCrslkCmbnatYn`: 자전거횡단도 겸용 여부
* `highlandYn`: 고원식 적용 여부
* `cartrkCo`: 차로 수
* `bt`: 횡단보도 폭
* `et`: 횡단보도 연장

이 값들은 **보행 난이도 판단**에 활용 가능하다. 예를 들어 차로 수가 많고 연장이 긴 횡단보도는 시각장애인에게 부담이 더 크다고 볼 수 있다.

### 신호/보행 보조 정보

* `tfclghtYn`: 보행자 신호등 유무
* `fnctngSgngnrYn`: 보행자 작동 신호기 유무
* `sondSgngnrYn`: 음향신호기 설치 여부
* `greenSgngnrTime`: 녹색신호시간
* `redSgngnrTime`: 적색신호시간

이 부분이 시각장애인 보행 보조 서비스에서 가장 중요하다.
특히 `sondSgngnrYn`, `fnctngSgngnrYn`은 **전자기기 이용 보행 사용자**에게 매우 의미 있는 정보다.

### 보행약자 편의 정보

* `tfcilndYn`: 교통섬 유무
* `ftpthLowerYn`: 보도턱낮춤 여부
* `brllBlckYn`: 점자블록 유무
* `cnctrLghtFcltyYn`: 집중조명시설 유무

이 값들은 **안전성 점수**나 **횡단보도 추천 우선순위**를 계산하는 데 유용하다.
점자블록, 보도턱낮춤, 음향신호기, 보행자 작동 신호기 등의 조합을 기반으로 더 안전한 횡단보도를 고를 수 있다.

## 4. 이 API의 장점

### 4-1. 시각장애인 보행 서비스와 잘 맞는 필드가 많다

일반 지도 API에서는 잘 안 주는 정보인

* 음향신호기
* 점자블록
* 보도턱낮춤
* 보행자작동신호기

같은 값이 포함되어 있다.
이건 시각장애인 안전 보행 보조 플랫폼에 매우 유리하다. 

### 4-2. 단순 위치 조회를 넘어서 “안전도 판단”에 쓸 수 있다

예를 들어 서비스에서 아래 같은 규칙을 만들 수 있다.

* 점자블록 O
* 보도턱낮춤 O
* 음향신호기 O
* 보행자 신호등 O

이면 “보행약자 친화 횡단보도”로 높은 점수를 줄 수 있다.

### 4-3. 사전 안내 데이터로 적합하다

실시간 영상 인식과 결합하면 좋다.

* 공공 API: 이 근처 횡단보도 후보 목록 제공
* 카메라 AI: 지금 실제로 횡단보도가 보이는지 확인
* 신호등 인식 AI: 현재 신호 상태 확인

즉, **정적 데이터 + 실시간 인식 데이터**의 조합 구조에 잘 맞는다.

---

## 5. 이 API의 한계

### 5-1. 실시간 신호 상태는 알 수 없다

`greenSgngnrTime`, `redSgngnrTime`은 **주기 정보**이지 현재 시점의 신호 상태가 아니다.
따라서 이 API만으로는 “지금 건너도 되는지”를 판단하면 안 된다. 

### 5-2. 주변 검색 기능이 약할 가능성이 크다

명세에 반경, 거리순 정렬, 범위 검색 조건이 없다.
그래서 앱에서 “내 주변 횡단보도 찾기”를 하려면 서버나 앱에서 후처리가 필요하다. 

### 5-3. 데이터 최신성 이슈 가능

응답에 `referenceDate`가 있으므로, 데이터는 **기준일자 기반의 정적 공공데이터**다.
현장 변경 사항이 즉시 반영되지 않을 수 있다. 예를 들면:

* 공사로 임시 폐쇄
* 점자블록 파손
* 신호기 고장
* 음향신호기 실제 미작동

같은 문제는 별도 검증이 필요하다. 

### 5-4. 시설 존재 여부만 있고 방향/배치 정보는 없다

예를 들어 `brllBlckYn = Y`라고 해도,

* 점자블록이 정확히 어디에 있는지
* 횡단보도 진입부 좌측인지 우측인지
* 음향신호기가 어느 기둥에 있는지

같은 **공간 배치 정보**는 없다.
그래서 실제 안내 문장을 만들 때는 이 API만으로는 부족하고, 카메라 기반 인식이나 지도 데이터 보강이 필요하다.

## 6. 시각장애인 보행 보조 플랫폼 관점에서의 활용 방법

사용자 이해를 반영하면, 시각장애인 보행은 **랜드마크, 기준선, 직선 보행, Orientation과 Mobility의 결합**이 핵심이다.
그 관점에서 이 API는 다음 역할에 적합하다.

### 6-1. 횡단보도 접근 전 사전 안내

사용자가 횡단보도 근처에 접근하면:

* 전방 20m에 횡단보도 있습니다
* 보행자 신호등이 있습니다
* 음향신호기가 설치되어 있습니다
* 점자블록이 있습니다

같은 **사전 음성 안내**를 할 수 있다.

### 6-2. 횡단 우선순위 추천

근처에 횡단보도가 여러 개 있다면:

* 점자블록 있음
* 보도턱낮춤 있음
* 음향신호기 있음
* 차로 수 적음

인 곳을 우선 추천할 수 있다.

### 6-3. 카메라 AI와 결합한 검증

API가 “이 주변에 횡단보도가 있다”고 알려주면,
카메라 AI가 실제 화면에서

* 횡단보도 줄무늬
* 신호등
* 보도 경계
* 점자블록

을 인식해 최종 안내 정확도를 높일 수 있다.

## 7. 서비스 설계에 바로 쓸 수 있는 판단

### 이 API만으로 가능한 것

* 근처 횡단보도 후보 조회
* 횡단보도 시설 정보 안내
* 점자블록/보도턱/음향신호기 여부 안내
* 더 안전한 횡단보도 추천
* 관리기관 및 기준일자 확인

### 이 API만으로 어려운 것

* 현재 신호가 초록불인지 판별
* 실제 현장 점유 상태 확인
* 공사물, 킥보드, 장애물 여부 확인
* 점자블록의 실제 위치와 방향 안내
* 사용자가 서 있는 방향 기준 좌/우 안내

## 8. 구현 시 권장 구조

### 권장 흐름

1. 사용자 현재 위치 확보
2. 행정구역 또는 주소 기반으로 API 조회
3. 응답의 위도/경도로 가까운 횡단보도 추림
4. 각 횡단보도에 안전 점수 계산
5. 카메라 AI로 실제 횡단보도/신호등 존재 검증
6. 음성/진동 안내 생성

### 안전 점수 예시

아래처럼 가중치를 둘 수 있다.

* 보행자 신호등 있음: +2
* 음향신호기 있음: +3
* 점자블록 있음: +3
* 보도턱낮춤 있음: +2
* 보행자작동신호기 있음: +1
* 차로 수 많음: -2
* 횡단보도 연장 길음: -1

이런 방식으로 “더 안전한 횡단보도”를 고를 수 있다.

## 9. 에러 코드 분석

명세에는 아래 오류들이 포함되어 있다. 

* `00`: 정상
* `03`: 데이터 없음
* `10`: 잘못된 요청 파라미터
* `11`: 필수 요청 파라미터 없음
* `20`: 서비스 접근 거부
* `22`: 요청 제한 초과
* `30`: 등록되지 않은 서비스키
* `31`: 기한 만료 서비스키
* `32`: 등록되지 않은 IP
* `99`: 기타 오류

### 구현 시 주의점

* `03`은 실패가 아니라 **검색 결과 없음**으로 처리
* `22`는 **재시도 제한**이나 **캐시 사용** 필요
* `10`, `11`은 요청 파라미터 검증 로직 필요
* `30`, `31`, `32`는 운영 환경 배포 시 키/IP 설정 점검 필요

## 10. 결론

이 횡단보도 API는 **시각장애인 보행 보조 서비스에서 매우 유용한 “정적 횡단보도 시설 정보 API”**다.
특히 음향신호기, 점자블록, 보도턱낮춤, 보행자 신호등 같은 값이 있어 **보행약자 친화적 횡단보도 추천**에 적합하다. 하지만 **실시간 신호 상태 제공 API는 아니며**, 공간 배치 정보도 부족하므로, 실제 서비스에서는 **카메라 기반 AI 인식, GPS, 지도 후처리**와 함께 써야 한다. 

원하면 다음 단계로 이어서
**1. 이 API를 쓰는 Python 예제 코드** 또는 **2. 우리 서비스용 백엔드 API 설계안** 형태로 바로 정리해주겠다.
