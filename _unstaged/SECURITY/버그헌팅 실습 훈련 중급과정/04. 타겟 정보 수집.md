## 4. 타겟 정보 수집

타겟 정보 수집은 공격 표면을 찾는 과정이다.

정보 수집은 크게 **수동 탐색**, **검색 엔진 활용**, **자동화 도구 활용**, **공개 저장소 분석**, **도메인·서브도메인 수집**으로 나눌 수 있다.

---

## 4.1. 수동 탐색

**수동 탐색**은 브라우저를 통해 직접 사이트를 확인하면서 구조, 기능, 입력값, API 호출 등을 분석하는 방식이다.

### 확인할 요소

* 메인 페이지
* 로그인 / 회원가입
* 비밀번호 재설정
* 마이페이지
* 검색 기능
* 파일 업로드
* 게시글 작성
* 관리자 페이지 경로
* 결제 또는 주문 기능
* API 요청 URL
* JavaScript 파일
* robots.txt
* sitemap.xml

### 수동 탐색에서 볼 것

| 항목         | 확인 내용                                   |
| ---------- | --------------------------------------- |
| URL 구조     | `/user/profile`, `/admin`, `/api/v1` 등  |
| 파라미터       | `id`, `page`, `redirect`, `file`, `url` |
| 인증 방식      | 쿠키, 세션, JWT, OAuth                      |
| 권한 검증      | 일반 사용자가 관리자 기능 접근 가능한지                  |
| 입력값 처리     | XSS, SQL Injection 가능성                  |
| 파일 업로드     | 확장자, MIME 타입, 저장 경로                     |
| 에러 메시지     | 서버 정보, DB 정보 노출 여부                      |
| JavaScript | API endpoint, key, 내부 경로 포함 여부          |

---

## 4.2. 구글 탐색

구글 탐색은 검색 연산자를 이용해 공개된 정보를 찾는 기법이다.

이를 **Google Dorking** 또는 **Google Hacking**이라고 한다.

단, 실제 서비스에 대해 사용할 때는 반드시 허가된 대상에 한정해야 한다.

### 주요 검색 연산자

| 연산자         | 의미                         |
| ----------- | -------------------------- |
| `site:`     | 특정 사이트 또는 도메인에서만 검색        |
| `inurl:`    | URL에 특정 문자열이 포함된 페이지 검색    |
| `intitle:`  | 페이지 제목에 특정 문자열이 포함된 페이지 검색 |
| `link:`     | 특정 링크를 포함하는 페이지 검색         |
| `filetype:` | 특정 파일 형식 검색                |
| `ext:`      | 특정 확장자 검색                  |
| `*`         | 와일드카드, 임의 문자열              |
| `" "`       | 정확히 일치하는 문구 검색             |
| `-`         | 특정 단어 제외                   |

### 예시

#### 특정 도메인의 페이지 검색

```text
site:*.target-system.com
```

#### 특정 경로가 포함된 페이지 검색

```text
site:*.target-system.com inurl:app/kibana
```

#### 특정 회사의 Amazon S3 버킷 검색

```text
site:s3.amazonaws.com target_name
```

```text
site:amazonaws.com target_name
```

#### 타겟 도메인의 text 파일 중 password 단어 검색

```text
site:*.target-system.com ext:txt password
```

#### LFI 취약점 가능성이 있는 페이지 검색

```text
inurl:/etc/passwd root:x:0:0:root:/root:/bin/bash
```

#### 디렉토리 리스팅 취약점 검색

```text
intitle:index of
```

### Google Hacking DB

Google Hacking DB는 다양한 Google Dork 예시를 모아둔 데이터베이스이다.

```text
https://www.exploit-db.com/google-hacking-database
```

### 주의점

* 허가받지 않은 대상에 공격적 검색을 수행하면 문제가 될 수 있다.
* 검색 결과에 민감 정보가 보여도 다운로드하거나 저장하지 않는 것이 안전하다.
* 실제 취약점 보고 시에는 최소한의 증거만 수집한다.

## 4.3. Spidering / Crawling

**Web Spidering** 또는 **Web Crawling**은 웹사이트에서 접근 가능한 페이지와 링크를 자동으로 수집하는 과정이다.

목적은 사이트의 구조와 숨겨진 경로를 식별하는 것이다.

### 개념

```text
Web Spidering / Web Crawling
→ 사이트에서 확인할 수 있는 모든 페이지와 링크를 식별
```

### 대표 도구

| 도구                 | 설명                        |
| ------------------ | ------------------------- |
| OWASP ZAP          | 오픈소스 웹 보안 프록시 도구          |
| Burp Suite Crawler | 요청 기반 크롤링 기능 제공           |
| Scrapy             | Python 기반 크롤링 프레임워크       |
| hakrawler          | URL 수집 도구                 |
| gau                | 과거 URL 수집 도구              |
| waybackurls        | Wayback Machine 기반 URL 수집 |

### OWASP ZAP

```text
ZAP = OWASP Zed Attack Proxy
```

ZAP은 웹 애플리케이션 테스트에 많이 사용되는 오픈소스 프록시 도구이다.

주요 기능:

* Proxy
* Spider
* Active Scan
* Passive Scan
* History 분석
* Request / Response 확인

### Spidering 시 확인할 항목

* 숨겨진 페이지
* 오래된 API 경로
* 관리자 페이지 후보
* 테스트 페이지
* 백업 파일
* 디버그 페이지
* JavaScript에서 호출하는 API
* 인증 없이 접근 가능한 리소스

## 4.4. S3 Bucket 정보 수집

Amazon S3 Bucket은 클라우드 객체 저장소이다.

잘못 설정된 S3 Bucket은 파일 목록 노출, 민감 정보 노출, 권한 오설정 문제로 이어질 수 있다.

### S3 Bucket 탐색 방법

#### 1. Google 검색

```text
site:s3.amazonaws.com [target]
site:amazonaws.com [target]
```

#### 2. GrayhatWarfare

공개된 S3 Bucket과 파일을 검색할 수 있는 서비스이다.

#### 3. Bucket Stream

Certificate Transparency 로그, DNS, 공개 데이터 등을 기반으로 버킷 이름을 탐색하는 방식이다.

### 확인할 수 있는 정보

* 공개 파일 목록
* 설정 파일
* 로그 파일
* 백업 파일
* 이미지, 문서, 업로드 파일
* API key 또는 credential 포함 파일

### 주의점

S3 Bucket이 공개되어 있더라도 무단 다운로드, 대량 접근, 삭제, 업로드 시도는 위험하다.

버그헌팅에서는 다음 정도로만 검증하는 것이 안전하다.

```text
1. 버킷이 공개적으로 접근 가능한지 확인
2. 민감 파일명이 노출되는지 확인
3. 최소한의 파일만 확인
4. 실제 데이터 다운로드는 지양
5. 보고서에 URL과 위험도 중심으로 작성
```

## 4.5. GitHub Recon

**GitHub Recon**은 기업 또는 프로젝트의 공개 저장소에서 민감 정보, 내부 구조, 취약한 코드 패턴을 찾는 과정이다.

### 조사 대상

* 회사 이름
* 프로젝트 이름
* 서비스 이름
* 도메인 이름
* 개발자 이름
* 개발자 닉네임
* 조직 계정
* 과거 저장소
* fork된 저장소
* issue / pull request / commit history

### 확인할 위치

| 위치            | 확인 내용                  |
| ------------- | ---------------------- |
| Repository    | 소스 코드, 설정 파일           |
| Issues        | 버그, 내부 경로, 에러 로그       |
| Pull Requests | 코드 변경 내용               |
| Commits       | 과거에 제거된 민감 정보          |
| Blame         | 특정 코드 작성자와 변경 시점       |
| Releases      | 배포 파일, changelog       |
| Wiki          | 운영 정보, API 문서          |
| Actions       | CI/CD 설정, secret 사용 흔적 |

### 민감 정보 키워드

```text
key
secret
password
passwd
pwd
token
credential
cred
api_key
access_key
private_key
client_secret
db_password
aws_access_key_id
aws_secret_access_key
```

### 대표 도구

| 도구                     | 설명                        |
| ---------------------- | ------------------------- |
| gitrob                 | GitHub 조직과 저장소에서 민감 정보 탐색 |
| trufflehog             | Git history에서 secret 탐지   |
| gitleaks               | Git 저장소 secret scanning   |
| GitHub Code Search     | GitHub 자체 코드 검색           |
| GitHub Secret Scanning | GitHub의 secret 탐지 기능      |

### 도구 URL

```text
https://github.com/michenriksen/gitrob
https://github.com/trufflesecurity/trufflehog
```

### GitHub Recon에서 중요한 점

* 현재 코드뿐 아니라 과거 커밋도 확인해야 한다.
* 삭제된 secret도 이미 노출된 것으로 간주해야 한다.
* `.env`, `.pem`, `.key`, `.config`, `.yml`, `.json` 파일에 민감 정보가 들어있을 수 있다.
* fork된 저장소에 과거 코드가 남아 있을 수 있다.
* issue나 로그에 내부 URL, 에러 메시지, DB 구조가 노출될 수 있다.

## 4.6. Domain Enumeration

**Domain Enumeration**은 타겟의 하위 도메인과 관련 인프라를 수집하는 과정이다.

예를 들어 `target.com`이 있다면 다음과 같은 서브도메인을 찾는 것이다.

```text
www.target.com
api.target.com
admin.target.com
dev.target.com
staging.target.com
test.target.com
vpn.target.com
mail.target.com
```

### 하위 도메인 수집 이유

* 숨겨진 관리자 페이지 발견
* 개발·테스트 서버 발견
* 오래된 서비스 발견
* 취약한 서브도메인 발견
* API 서버 발견
* 서브도메인 takeover 가능성 확인

### 수집 방법

| 방법                       | 설명              |
| ------------------------ | --------------- |
| Certificate Transparency | 인증서 로그에서 도메인 수집 |
| DNS Query                | DNS 레코드 확인      |
| Search Engine            | 검색 엔진 결과 활용     |
| Public Dataset           | 공개 데이터셋 활용      |
| Brute Force              | 단어 리스트 기반 추측    |
| Passive DNS              | 과거 DNS 기록 확인    |

### Certificate Transparency

TLS 인증서 발급 기록에는 SAN 정보가 포함될 수 있다.

```text
SAN = Subject Alternative Name
```

SAN에는 하나의 인증서가 보호하는 여러 도메인이 들어갈 수 있다.

예시:

```text
www.target.com
api.target.com
dev.target.com
```

### 대표 서비스 및 도구

| 도구          | 설명                  |
| ----------- | ------------------- |
| crt.sh      | 인증서 투명성 로그 검색       |
| amass       | 서브도메인 수집 및 OSINT 도구 |
| subfinder   | 빠른 서브도메인 수집 도구      |
| assetfinder | 도메인 관련 자산 수집        |
| dnsx        | DNS 검증 도구           |
| httpx       | 살아있는 HTTP 서비스 확인    |

### Amass

```text
amass
→ subdomain enumeration 가능
```

Amass는 다양한 공개 정보와 DNS 데이터를 활용하여 서브도메인을 수집한다.

## 4.7. 추가로 알아두면 좋은 정보 수집 기법

### 4.7.1. WHOIS 조회

WHOIS는 도메인 등록자, 등록 기관, 네임서버, 등록일 등을 확인하는 방법이다.

확인 항목:

* 등록 기관
* 네임서버
* 생성일
* 만료일
* 관리자 이메일
* 조직명

### 4.7.2. DNS 레코드 조회

DNS 레코드를 보면 서비스 구조를 추측할 수 있다.

| 레코드   | 의미               |
| ----- | ---------------- |
| A     | 도메인 → IPv4 주소    |
| AAAA  | 도메인 → IPv6 주소    |
| CNAME | 별칭 도메인           |
| MX    | 메일 서버            |
| TXT   | SPF, DKIM, 검증 정보 |
| NS    | 네임서버             |
| SOA   | DNS Zone 정보      |

### 4.7.3. Wayback Machine

Wayback Machine은 과거 웹 페이지 기록을 확인할 수 있는 서비스이다.

확인 가능한 것:

* 과거 URL
* 삭제된 페이지
* 오래된 API 경로
* 과거 JavaScript 파일
* 예전 관리자 페이지
* deprecated endpoint

### 4.7.4. JavaScript 파일 분석

프론트엔드 JavaScript에는 API endpoint, feature flag, 내부 경로, 테스트 URL 등이 포함될 수 있다.

확인 항목:

```text
/api/
/graphql
/admin
/internal
/dev
/staging
token
key
secret
client_id
redirect_uri
```
