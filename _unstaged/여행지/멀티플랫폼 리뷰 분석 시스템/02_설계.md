# 멀티플랫폼 장소 리뷰 비교 분석 서비스 설계 문서

## 1. 문서 개요

### 1-1. 문서 목적

본 문서는 특정 장소 1개에 대해 구글맵, 카카오맵, 네이버맵 리뷰를 수집·정규화·분석하여 플랫폼별 선호도와 플랫폼 간 차이를 비교하고, 최종 통합 리포트를 반환하는 서비스의 구현 설계를 정의한다. 프로젝트의 목표는 플랫폼별 종합 점수, 속성별 점수, 공통 강점/약점, 플랫폼 간 인식 차이, 통합 선호도 리포트를 산출하는 것이다. 

### 1-2. 현재 설계 기준

현재 설계는 다음 조건을 전제로 한다.

* 플랫폼별 `place id`를 저장하지 않는다.
* DB에 원본 리뷰나 분석 결과를 저장하지 않는다.
* 사용자의 요청 1건마다 즉시 수집·분석·비교·응답한다.
* 분석 대상 카테고리는 음식점, 카페, 관광지로 한정한다.
* 분석 구조는 공통 프레임과 카테고리별 속성 사전을 함께 사용한다.

### 1-3. 설계 방향 요약

기존 문서가 저장형 파이프라인이었다면, 현재 설계는 **Stateless Runtime Pipeline**이다.
즉, “저장 가능한 개체 설계”보다 “한 번의 실행에서 정확하고 안정적으로 결과를 반환하는 흐름 설계”가 중심이다.

## 2. 시스템 목표

시스템은 한 장소를 기준으로 세 플랫폼 리뷰를 동일 기준으로 분석하고 다음을 반환해야 한다.

* 플랫폼별 종합 선호도 점수
* 플랫폼별 속성별 감성 점수
* 공통 강점과 공통 약점
* 플랫폼별 차이 해석
* 최종 통합 선호도 점수 및 요약 리포트

## 4. 상위 아키텍처

### 4-1. 아키텍처 개요

시스템은 FastAPI 기반 단일 실행 API로 구성한다.
요청이 들어오면 내부 파이프라인이 순차적으로 다음 단계를 수행한다.

```text
[Client]
   ↓
[FastAPI Endpoint]
   ↓
[Pipeline Orchestrator]
   ├─ Place Search Layer
   ├─ Place Matching Layer
   ├─ Review Collectors
   ├─ Normalizer
   ├─ Preprocessor
   ├─ Analyzer
   ├─ Scorer
   ├─ Comparator
   └─ Report Builder
   ↓
[JSON Response]
```

이 구조는 기존 기획 문서의 전체 파이프라인을 유지하되 저장 계층만 제거한 형태다. 기획 문서의 기본 흐름도 장소 입력 → 플랫폼별 매핑 → 리뷰 수집 → 정규화 → 전처리 → 분석 → 점수 산출 → 플랫폼 비교 → 통합 리포트 생성으로 정의되어 있다.

### 4-2. 아키텍처 원칙

1. 상태 비저장
   요청 종료 후 서버는 분석 결과를 보관하지 않는다.

2. 모듈 분리
   수집, 정규화, 분석, 점수화, 비교, 리포트 생성을 별도 서비스로 분리한다.

3. 플랫폼 독립성
   수집기만 교체하면 나머지 분석 로직은 유지되도록 설계한다.

4. 카테고리 확장성
   공통 분석 구조 위에 카테고리별 속성 사전을 얹는다.

## 5. 처리 흐름 설계

## 5-1. 전체 시퀀스

```text
1. 사용자가 장소명과 카테고리를 입력한다.
2. 서버가 각 플랫폼에서 후보 장소를 조회한다.
3. 서버가 이름/주소/카테고리 기준으로 동일 장소 후보를 선택한다.
4. 각 플랫폼 선택 결과를 기준으로 리뷰를 수집한다.
5. 리뷰를 공통 스키마로 정규화한다.
6. 정규화 리뷰를 전처리한다.
7. 전처리 리뷰를 분석한다.
8. 플랫폼별 점수와 속성 요약을 계산한다.
9. 플랫폼 간 비교 문장을 생성한다.
10. 최종 리포트를 JSON으로 반환한다.
```

## 5-2. 장소 매칭 단계 설계

기존 기획 문서는 `master_place_id`와 플랫폼별 place id 연결을 권장했지만, 현재는 저장하지 않으므로 이 개념을 **요청 시점 후보 결정 로직**으로만 사용한다. 동일 장소 판정 기준은 다음과 같다. 

* 장소명 유사도
* 주소 유사도
* 카테고리 일치 여부
* 좌표 근접성 가능 시 활용
* 전화번호 가능 시 활용

자동 선택 규칙 예시:

* 이름 점수 0.45
* 주소 점수 0.35
* 카테고리 점수 0.10
* 좌표/전화번호 보너스 0.10

최종 `matched_confidence`가 임계치 이상이면 자동 선택하고, 미만이면 사용자 확인 대상으로 표시한다.

## 7. 모듈 설계

## 7-1. `api/routes.py`

역할:

* 외부 요청 수신
* 요청 검증
* pipeline 호출
* 응답 반환
* 예외를 공통 에러 포맷으로 변환

핵심 엔드포인트:

* `POST /api/review-compare/run`

## 7-2. `services/pipeline.py`

역할:

* 전체 파이프라인 오케스트레이션
* 단계별 출력 연결
* 실패 플랫폼 처리
* 경고 메시지 누적
* 최종 응답 조립

실행 순서:

1. place search
2. place match
3. collect reviews
4. normalize
5. preprocess
6. analyze
7. score
8. compare
9. build report

## 7-3. `collectors/base.py`

역할:

* 플랫폼 수집기 공통 인터페이스 제공

예상 인터페이스:

```python
class BaseCollector:
    platform: str

    def search_places(self, query: str, category: str | None = None) -> list[dict]:
        ...

    def collect_reviews(self, place_ref: dict, limit: int) -> list[dict]:
        ...
```

기획 문서도 수집 로직은 공통 인터페이스를 두고 플랫폼별 구현체를 분리하는 것이 적절하다고 제안했다. 

## 7-4. `services/place_matcher.py`

역할:

* 플랫폼별 후보 중 동일 장소 선택
* 매칭 신뢰도 산출
* 자동 선택 / 사용자 확인 필요 상태 결정

출력 예:

```python
{
  "google": {"selected": {...}, "confidence": 0.92, "status": "matched"},
  "kakao": {"selected": {...}, "confidence": 0.88, "status": "matched"},
  "naver": {"selected": None, "confidence": 0.31, "status": "needs_user_selection"}
}
```

## 7-5. `services/review_normalizer.py`

역할:

* 플랫폼별 원본 리뷰를 공통 구조로 통일
* 필드명, 타입, 날짜 형식, 메타데이터 표준화

## 7-6. `services/preprocess.py`

역할:

* 텍스트 정리
* 문장 분리
* 언어 식별
* 중복/유사 리뷰 처리
* 짧은 리뷰/저정보 리뷰 태깅

## 7-7. `services/aspect_extractor.py`

역할:

* 카테고리별 속성 사전 기반 속성 추출
* 문장별 언급 속성 태깅

기획 문서는 분석 대상 카테고리를 음식점/카페/관광지로 두고, 공통 프레임 위에 카테고리별 속성 사전을 쓰는 것이 적절하다고 정의한다.

## 7-8. `services/sentiment.py`

역할:

* 전체 감성 분석
* 속성별 감성 분석
* 재방문/추천 의사 탐지
* 신뢰도 점수 계산

기획 문서는 전체 감성, 속성 추출, 속성별 감성, 재방문/추천 의사 탐지를 핵심 분석 단계로 정의하고 있다. 

## 7-9. `services/scorer.py`

역할:

* 리뷰 단위 점수 계산
* 리뷰 가중치 계산
* 플랫폼별 종합 점수 계산
* 속성별 점수 집계
* 최종 통합 점수 계산

기획 문서는 플랫폼별 종합 점수를
`Σ(review_score * review_weight) / Σ(review_weight)`
형태로 계산하고, 이후 플랫폼 비교와 최종 통합 점수로 확장하도록 정의한다.

## 7-10. `services/comparator.py`

역할:

* 플랫폼 간 공통 강점/약점 계산
* 플랫폼별 차이 문장 생성
* 불일치 패널티 계산

기획 문서도 이 프로젝트의 차별점은 점수 나열이 아니라 “왜 플랫폼별 인식이 다른지”를 설명하는 비교 로직에 있다고 본다. 

## 7-11. `services/report_builder.py`

역할:

* 최종 사용자 응답 조립
* 요약 카드
* 점수 테이블
* 비교 문장
* 경고 리스트 생성

## 8. 데이터 구조 설계

현재는 DB를 쓰지 않으므로, 내부 모델은 **Pydantic/Dict 기반 런타임 객체**로 설계한다.

## 8-1. 요청 스키마

```python
class ReviewCompareRequest(BaseModel):
    place_name: str
    category_main: Literal["restaurant", "cafe", "attraction"]
    category_sub: str | None = None
    max_reviews_per_platform: int = Field(default=100, ge=1, le=500)
    enable_user_selection_hint: bool = True
```

## 8-2. 플랫폼 후보 스키마

```python
class PlaceCandidate(BaseModel):
    platform: Literal["google", "kakao", "naver"]
    place_name: str
    address: str | None
    road_address: str | None = None
    lat: float | None = None
    lng: float | None = None
    phone: str | None = None
    category: str | None = None
    rating: float | None = None
    review_count: int | None = None
    source_url: str | None = None
    raw_ref: dict = Field(default_factory=dict)
```

## 8-3. 정규화 리뷰 스키마

```python
class NormalizedReview(BaseModel):
    platform: Literal["google", "kakao", "naver"]
    review_id: str
    reviewer_name_raw: str | None = None
    reviewer_id_hash: str | None = None
    rating: float | None = None
    review_text: str
    review_date: date | None = None
    like_count: int = 0
    has_photo: bool = False
    visit_verified: bool = False
    language: str | None = None
    review_type: str | None = None
    source_url: str | None = None
    meta: dict = Field(default_factory=dict)
```

여기서 `review_type`은 특히 네이버에서 중요하다. 기존 비교 문서는 네이버 방문자 리뷰와 블로그 리뷰를 반드시 분리해야 하며, 이를 섞으면 문장 길이와 감성 강도 해석이 왜곡될 수 있다고 명시한다.

## 8-4. 분석 결과 스키마

```python
class ReviewAnalysisResult(BaseModel):
    sentiment_label: Literal["positive", "neutral", "negative"]
    sentiment_score: float
    revisit_intent_score: float
    recommend_intent_score: float
    confidence_score: float
    informativeness_score: float
    freshness_score: float
    aspects: list[dict]
    keywords: list[str]
```

## 8-5. 최종 응답 스키마

```python
class ReviewCompareResponse(BaseModel):
    input: dict
    matched_places: dict[str, dict | None]
    collection_summary: dict[str, int]
    platform_scores: dict[str, float]
    aspect_summary_by_platform: dict[str, dict[str, float]]
    common_strengths: list[str]
    common_weaknesses: list[str]
    platform_differences: list[str]
    final_preference_score: float
    warnings: list[str]
```

## 10. 분석 설계

## 10-1. 전체 감성 분석

리뷰 전체의 정서 방향을 판정한다.

출력:

* positive / neutral / negative
* 연속 점수 `[-1, 1]`

## 10-2. 속성 추출

리뷰 문장에서 어떤 속성이 언급되었는지 추출한다.
초기 구현은 규칙 기반 + 사전 기반이 실용적이다. 기획 문서도 초기 단계에서는 규칙 기반 + 사전 기반을 추천한다. 

## 10-3. 속성별 감성 분석

속성이 언급되면 그 속성에 대해 긍/부정을 판정한다.
이 단계가 최종 품질에 가장 큰 영향을 준다. 

## 10-4. 재방문/추천 의사 탐지

다음 표현을 별도 신호로 잡는다.

* 재방문 긍정: 또 올게요, 재방문 의사 있음, 단골될 듯
* 재방문 부정: 다시는 안 감, 한 번이면 충분
* 추천 긍정: 추천함, 꼭 가보세요
* 추천 부정: 비추천

## 10-5. 키워드 추출

플랫폼별로 긍정/부정/정보성 키워드를 따로 뽑는다.

## 11. 점수 산출 설계

## 11-1. 리뷰 단위 점수

```text
review_score =
  w1 * overall_sentiment_score
+ w2 * aspect_sentiment_avg
+ w3 * revisit_intent_score
+ w4 * recommend_intent_score
```

## 11-2. 리뷰 가중치

리뷰별 가중치는 다음 요소를 반영한다.

* 최신성
* 텍스트 길이
* 속성 언급 개수
* 사진 포함 여부
* 리뷰 유형
* 언어 신뢰도
* 중복 가능성

## 11-3. 플랫폼별 점수

```text
platform_preference_score =
Σ(review_score * review_weight) / Σ(review_weight)
```

이 수식은 기존 기획 문서의 기본 산식을 그대로 따른다. 

## 11-4. 최종 통합 점수

기획 문서는 플랫폼 종합 감성, 핵심 속성 점수, 재방문/추천 점수, 최신 리뷰 보정, 플랫폼 간 불일치 패널티를 조합하는 방식을 제시한다. 현재 설계도 이를 따른다. 

예시:

```text
Final Score =
0.35 * 플랫폼 종합 감성 평균
+ 0.40 * 핵심 속성 점수 평균
+ 0.15 * 재방문/추천 점수
+ 0.10 * 최신 리뷰 보정
- 0.10 * 플랫폼 간 불일치 패널티
```

## 11-5. 플랫폼 가중치

카테고리에 따라 플랫폼 가중치는 다르게 둘 수 있다.

예:

* 관광지: Google 비중 상향
* 음식점/카페: Naver, Kakao 비중 상향

이 또한 기존 기획 문서의 확장 방향과 일치한다.

## 12. 플랫폼 특성 반영 설계

기존 비교 문서는 플랫폼별 리뷰 생태계가 다르므로 동일 점수라도 같은 의미로 해석하면 안 된다고 본다.

## 12-1. 네이버

* 생활형 실사용 정보 + 장문 맥락 정보
* 방문자 리뷰와 블로그 리뷰를 반드시 구분
* 기능 키워드와 서사형 문장을 분리 분석

## 12-2. 카카오

* 짧고 직접적인 압축형 평가
* 불만 직접성 탐지 중요
* 짧은 판정문 비중 고려

## 12-3. 구글

* 탐색형, 관광형, 상태형 정보 비중 큼
* 다국어 혼합 가능성 높음
* 추천형/가이드형 문장 비중 큼

이 특성 차이는 비교 문장 생성과 경고 메시지 설계에 반영한다.

## 14. 예외 처리 설계

## 14-1. 에러 코드 체계

### `PLACE_SEARCH_FAILED`

* 의미: 특정 플랫폼에서 장소 후보 조회 실패
* 처리: 해당 플랫폼만 제외하고 부분 분석 가능

### `PLACE_MATCH_AMBIGUOUS`

* 의미: 동일 장소 자동 선택 실패
* 처리: 후보 목록 반환 또는 경고 포함

### `REVIEW_COLLECTION_FAILED`

* 의미: 특정 플랫폼 리뷰 수집 실패
* 처리: 나머지 플랫폼 기준으로 분석 진행

### `INSUFFICIENT_REVIEWS`

* 의미: 최소 분석 가능 리뷰 수 미만
* 처리: 부분 분석 또는 실패 반환

### `ANALYSIS_FAILED`

* 의미: 분석 단계 내부 예외
* 처리: 플랫폼별 스킵 후 경고 포함

### `UNSUPPORTED_CATEGORY`

* 의미: 허용되지 않은 카테고리
* 처리: 400 반환

## 14-2. 부분 성공 정책

이 서비스는 3개 플랫폼이 모두 성공하지 않아도 결과를 반환할 수 있어야 한다.
예를 들어 Naver 수집 실패 시 Google + Kakao만으로 리포트를 만들고, 경고에 명시한다.

## 15. 성능 설계

## 15-1. 병렬 처리

* 플랫폼별 장소 검색은 병렬 실행

* 리뷰 수집 타임아웃

* 리뷰 수 과다 시 최신순 우선

기획 문서도 실제 시작 단계에서 리뷰 100~300개 수준으로 시작하는 것을 권장한다.

## 19. 기존 설계와의 차이

기존 문서에서는 `Place`, `PlatformPlace`, `Review`, `PlatformSummary`, `FinalPlaceReport` 같은 저장형 모델과 CRUD 레이어를 두고 `master_place_id`로 조회하는 구조를 채택했다. 
현재 설계는 이를 다음과 같이 변경한다.

* `master_place_id` 제거
* `platform_place_id` 영속 저장 제거
* SQLAlchemy 모델 제거
* CRUD/Repository 제거
* GET 조회 API 제거
* 분석 결과를 응답 JSON으로 직접 반환

즉, 기존은 “데이터 저장 후 조회 가능한 분석 시스템”, 현재는 “실행 즉시 결과를 반환하는 요청 단위 분석 시스템”이다.

## 20. 최종 설계 정의

현재 기준에서 본 시스템은 다음과 같이 정의할 수 있다.

**사용자가 입력한 특정 장소에 대해 구글맵·카카오맵·네이버맵의 후보 장소를 요청 시점에 매칭하고, 선택된 장소의 리뷰를 공통 기준으로 수집·정규화·분석하여 플랫폼별 선호도와 플랫폼 간 인식 차이를 비교한 뒤 통합 선호도 리포트를 즉시 반환하는 멀티플랫폼 리뷰 비교 분석 시스템**이다. 이 정의는 기존 기획의 목표와 분석 축을 유지하면서, 저장형 구조를 제거한 현재 범위에 맞는 구현 설계다.
