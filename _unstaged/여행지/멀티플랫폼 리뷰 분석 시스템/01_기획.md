# 멀티플랫폼 장소 리뷰 비교 분석 서비스 기획문서

## 1. 프로젝트 개요

### 1-1. 프로젝트명

멀티플랫폼 장소 리뷰 비교 분석 서비스

### 1-2. 프로젝트 한 줄 정의

사용자가 입력한 특정 장소에 대해 구글맵, 카카오맵, 네이버맵의 리뷰를 수집하고, 이를 공통 기준으로 정규화·분석하여 플랫폼별 선호도와 플랫폼 간 인식 차이를 비교한 뒤 통합 선호도 리포트를 생성하는 서비스이다.

### 1-3. 프로젝트 배경

같은 장소라도 리뷰 플랫폼에 따라 평가 양상이 다르게 나타날 수 있다.
예를 들어 어떤 장소는 네이버에서는 생활밀착형 후기와 재방문 의사가 많이 드러나고, 카카오에서는 짧고 직접적인 판단형 후기가 많으며, 구글에서는 관광·탐색 중심의 정보가 더 많이 나타날 수 있다. 이런 차이는 단순 평균 별점만으로는 설명되지 않기 때문에, 플랫폼별 리뷰를 각각 분석한 후 비교하는 구조가 필요하다. 

### 1-4. 프로젝트 목적

본 프로젝트의 목적은 특정 장소 1개에 대해 3개 플랫폼의 리뷰를 비교 분석하여 다음을 산출하는 것이다.

* 플랫폼별 종합 선호도 점수
* 플랫폼별 속성별 선호도 점수
* 공통 강점과 공통 약점
* 플랫폼 간 인식 차이
* 통합 최종 장소 선호도 리포트

## 2. 문제 정의

### 2-1. 기존 리뷰 확인 방식의 한계

일반 사용자는 보통 하나의 플랫폼만 보고 장소를 판단한다. 하지만 플랫폼마다 사용자층, 리뷰 작성 방식, 문장 길이, 정보 밀도, 방문 맥락이 다르기 때문에 한 플랫폼의 리뷰만 보면 장소에 대한 전체적인 인식을 놓칠 수 있다. 

### 2-2. 평균 별점 중심 비교의 한계

별점은 빠르게 비교하기에는 편하지만, 실제로는 다음과 같은 한계가 있다.

* 분위기와 가격처럼 상반된 평가가 섞인 복합 감정을 표현하기 어렵다
* 별점과 본문 내용이 불일치할 수 있다
* 플랫폼마다 리뷰 문화가 달라 단순 평균 비교가 왜곡될 수 있다

### 2-3. 동일 장소 비교의 필요성

이 프로젝트의 핵심은 “장소 1개”를 기준으로 3개 플랫폼 리뷰를 비교하는 것이다. 따라서 먼저 각 플랫폼에서 같은 장소를 최대한 정확히 찾아야 하며, 이후에는 리뷰를 공통 구조로 바꾸고 동일한 분석 기준을 적용해야 한다. 기존 기획은 이를 위해 장소 매핑을 핵심 원칙으로 두고 있었다. 현재는 place id를 저장하지 않지만, 요청 시점 후보 선택 로직으로 그 원칙은 그대로 유지해야 한다.

## 3. 현재 기준 서비스 범위

### 3-1. 포함 범위

현재 단계에서 서비스는 다음 범위를 포함한다.

* 사용자 장소명 입력
* 플랫폼별 장소 후보 검색
* 동일 장소 후보 선택
* 플랫폼별 리뷰 수집
* 리뷰 정규화
* 전처리
* 감성 및 속성 기반 분석
* 플랫폼별 점수 계산
* 플랫폼 간 비교 해석
* 최종 리포트 반환 

### 3-2. 제외 범위

현재 단계에서는 다음은 제외한다.

* 플랫폼별 `place id` 영구 저장
* DB 기반 장소/리뷰/리포트 저장
* 과거 분석 이력 조회
* 동일 장소 재사용 캐시
* 사용자 계정별 저장 기능

즉, 현재 서비스는 **저장형 시스템**이 아니라 **즉시 실행형 분석 시스템**이다.

### 3-3. 서비스 형태

사용자가 장소명을 입력하면, 서버는 그 요청 안에서만 후보 장소를 찾고, 리뷰를 수집하고, 분석한 뒤 결과를 반환한다. 분석 결과는 응답으로만 제공되고 별도 영속 저장은 하지 않는다.

## 4. 목표 사용자

### 4-1. 일반 사용자

맛집, 카페, 관광지를 방문하기 전에 여러 플랫폼의 평가를 한 번에 비교하고 싶은 사용자

### 4-2. 콘텐츠 기획자 및 마케터

같은 장소가 플랫폼별로 어떻게 다르게 인식되는지 보고 싶은 사용자

### 4-3. 프로젝트/연구 목적 사용자

리뷰 데이터의 플랫폼 편향과 서술 스타일 차이를 분석하고 싶은 사용자

## 5. 핵심 가치 제안

### 5-1. 멀티플랫폼 통합 비교

하나의 플랫폼이 아니라 구글맵, 카카오맵, 네이버맵을 동시에 본다.

### 5-2. 단순 별점이 아닌 텍스트 기반 분석

전체 감성뿐 아니라 속성별 감성, 재방문/추천 의사, 키워드를 함께 분석한다. 

### 5-3. 플랫폼별 차이 해석

단순히 “어디가 점수가 높다”가 아니라, “왜 플랫폼마다 다르게 평가되는가”를 설명한다. 기존 기획도 플랫폼 비교의 핵심을 바로 이 해석 가능성에 두고 있다.

## 6. 분석 대상 카테고리

현재 분석 대상은 다음 3개 카테고리로 제한한다.

* 음식점
* 카페
* 관광지

이 세 카테고리는 리뷰에서 중요하게 보는 요소가 다르기 때문에, 공통 분석 프레임 위에 카테고리별 속성 사전을 얹는 구조가 적절하다.

## 7. 서비스 동작 흐름

전체 파이프라인은 아래와 같다.

```text
[사용자 장소 입력]
   ↓
[플랫폼별 장소 검색]
   ↓
[동일 장소 후보 선택]
   ├─ 이름 유사도
   ├─ 주소 유사도
   ├─ 카테고리 일치
   └─ 필요 시 사용자 확인
   ↓
[플랫폼별 리뷰 수집]
   ↓
[리뷰 정규화]
   ↓
[전처리]
   ↓
[리뷰 분석]
   ├─ 감성 분석
   ├─ 속성 추출
   ├─ 속성별 감성 분석
   ├─ 키워드 추출
```

이 파이프라인 자체는 기존 기획의 핵심 구조를 유지한 것이다. 다만 현재는 저장 단계를 제거하고 요청 단위 처리로 단순화한다.

## 8. 핵심 기획 원칙

### 8-1. 같은 장소를 정확히 선택해야 한다

플랫폼마다 장소명이 조금씩 다를 수 있다. 예를 들어 같은 매장이라도 지점명 표기가 다르거나, 리저브/본점/별관 같은 표현이 다르게 붙을 수 있다. 따라서 리뷰 수집 전에 동일 장소 후보를 먼저 고르는 단계가 반드시 필요하다. 기존 기획은 장소명 유사도, 주소 유사도, 좌표, 전화번호, 카테고리 일치를 기준으로 삼도록 제안했다. 현재도 저장 없이 이 기준을 요청 시점 선택 로직에 적용해야 한다. 

### 8-2. 플랫폼마다 리뷰 생태계가 다르다

네이버는 방문 인증 기반의 생활형 리뷰와 블로그 기반의 장문 맥락 정보가 결합될 가능성이 높고, 카카오는 장소 평가 중심의 압축형 후기 비중이 높으며, 구글은 관광·탐색·상태 정보 비중이 상대적으로 클 수 있다. 이 차이는 단순 데이터 양이 아니라 플랫폼 구조와 사용자 문화에서 나온다. 따라서 플랫폼별 점수를 곧바로 단순 평균 내면 안 되고, 해석 단계에서 플랫폼 성격을 함께 반영해야 한다. 

### 8-3. 장소 선호도는 속성 단위로 봐야 한다

“분위기는 좋지만 가격이 비싸다”처럼 하나의 리뷰에 여러 감정이 함께 들어갈 수 있으므로, 전체 감성만 보는 분석은 충분하지 않다. 따라서 본 프로젝트는 Aspect-Based Sentiment Analysis, 즉 속성 기반 감성 분석 구조를 핵심 분석 방식으로 사용한다.

## 9. 상세 기능 기획

### 9-1. 장소 입력

사용자는 분석하고 싶은 장소명을 입력한다.
필요 시 카테고리도 함께 지정할 수 있다.

입력 예시:

* 스타벅스 강남R점
* 성수동 카페 어니언
* 경복궁

### 9-2. 플랫폼별 장소 검색

시스템은 입력된 장소명을 바탕으로 각 플랫폼에서 후보 장소를 조회한다.

출력 예시:

* google 후보 3개
* kakao 후보 3개
* naver 후보 3개

### 9-3. 동일 장소 후보 선택

후보들 중 실제로 같은 장소라고 판단되는 항목을 선택한다.

판단 기준:

* 장소명 유사도
* 주소 유사도
* 카테고리 일치
* 좌표 근접성 가능 시 활용
* 전화번호 가능 시 활용

정책:

* 충분히 유사하면 자동 선택
* 애매하면 후보를 사용자에게 보여주고 선택하게 함

### 9-4. 플랫폼별 리뷰 수집

선택된 장소 페이지를 기준으로 각 플랫폼에서 리뷰를 수집한다.

정책:

* 최신순 우선
* 플랫폼별 최대 수집 개수 제한
* 텍스트 없는 리뷰는 별도 분리 가능
* 수집 실패 플랫폼은 경고 포함 후 부분 분석 진행

### 9-5. 리뷰 정규화

플랫폼마다 리뷰 구조가 다르므로 내부 공통 포맷으로 변환한다.

공통 필드 예시:

* platform
* place_name
* address
* rating
* review_text
* review_date
* has_photo
* language
* source_url

### 9-6. 전처리

정규화된 리뷰를 분석 가능한 형태로 정제한다.

포함 작업:

* 공백/특수문자 정리
* 문장 분리
* 언어 태깅
* 너무 짧은 리뷰 표시
* 중복/유사 리뷰 탐지

### 9-7. 리뷰 분석

핵심 분석 단계에서는 다음을 수행한다.

* 전체 감성 분석
* 속성 추출
* 속성별 감성 분석
* 키워드 추출
* 재방문 의사 탐지
* 추천 의사 탐지
* 리뷰 신뢰도 계산 

### 9-8. 플랫폼별 점수 산출

리뷰 단위 결과를 플랫폼 단위로 집계하여 플랫폼별 종합 점수와 속성별 점수를 계산한다. 기획 문서는 리뷰 점수와 리뷰 가중치를 결합한 방식으로 플랫폼 점수를 산출하도록 제안한다. 

### 9-9. 플랫폼 간 비교 분석

플랫폼별로 다음을 비교한다.

* 평균 별점
* 텍스트 감성 점수
* 속성별 감성 점수
* 재방문/추천 의사 비율
* 상위 긍정/부정 키워드
* 리뷰 수와 최신 리뷰 비중 

### 9-10. 리포트 생성

최종적으로 사용자가 읽기 쉬운 형태의 종합 리포트를 생성한다.

리포트 포함 요소:

* 플랫폼별 종합 점수
* 속성별 비교 요약
* 공통 강점
* 공통 약점
* 플랫폼별 차이
* 통합 최종 점수
* 해석상 주의사항

## 10. 카테고리별 분석 체계

### 10-1. 공통 속성

모든 카테고리에 공통으로 적용할 수 있는 속성은 다음과 같다.

* 접근성
* 청결
* 혼잡도
* 직원/응대
* 가격/가성비
* 대기시간
* 재방문 의사
* 추천 의사

### 10-2. 음식점

주요 속성:

* 맛
* 양
* 가격
* 가성비
* 서비스
* 분위기
* 청결
* 대기시간
* 메뉴 다양성
* 주차
* 접근성

핵심 질문:

* 맛 때문에 좋은 평가를 받는가
* 가격이나 대기시간 때문에 손해를 보는가
* 서비스와 분위기가 재방문 의사에 영향을 주는가

### 10-3. 카페

주요 속성:

* 음료 맛
* 디저트
* 분위기
* 인테리어
* 좌석 편안함
* 작업 적합성
* 소음
* 가격
* 가성비
* 청결
* 콘센트/와이파이
* 주차

핵심 질문:

* 음료와 공간 중 무엇이 만족도를 더 좌우하는가
* 작업형 카페로 인식되는가
* 가격 대비 체감이 좋은가

### 10-4. 관광지

주요 속성:

* 경관
* 접근성
* 편의시설
* 혼잡도
* 안전성
* 체험성
* 주차

기획 문서도 관광지의 핵심 질문을 경관, 접근성, 편의시설, 혼잡도, 안전성, 체험성 중심으로 잡고 있다. 

---

## 11. 분석 모델 기획

### 11-1. 전체 감성 분석

리뷰 전체를 긍정, 중립, 부정 또는 연속 점수로 판별한다.

목적:

* 전체적인 만족도 추정
* 플랫폼별 기본 정서 방향 비교

### 11-2. 속성 추출

리뷰에서 어떤 평가 대상이 언급되는지 찾는다.

예:

* “커피는 맛있는데 자리가 불편해요”
* 추출 속성: 음료 맛, 좌석 편안함

### 11-3. 속성별 감성 분석

각 속성에 대해 긍정/부정을 따로 판정한다.

예:

* 음료 맛: 긍정
* 좌석 편안함: 부정

### 11-4. 재방문 의사 탐지

“또 가고 싶다”, “다시 안 갈 듯”, “재방문 의사 있음” 같은 표현을 탐지한다.

### 11-5. 추천 의사 탐지

“추천함”, “worth visiting”, “비추천” 같은 추천/비추천 의사를 탐지한다. 플랫폼별 추천 표현 양식이 다를 수 있다는 점은 기존 비교 문서에서도 강조된다. 

### 11-6. 키워드 추출

플랫폼별 상위 긍정/부정 키워드를 뽑아 사용자가 빠르게 이해할 수 있도록 한다.

---

## 12. 플랫폼별 특성 반영 기획

### 12-1. 네이버

네이버는 방문자 리뷰와 블로그 리뷰가 공존하기 때문에 생활형 정보와 장문 맥락 정보가 함께 나타날 가능성이 높다. 따라서 가격, 대기, 메뉴, 재방문 의사, 분위기 같은 실사용 정보와 서사 정보가 함께 잡힐 수 있다. 

해석 포인트:

* 생활밀착형 평가에 강함
* 맥락 설명이 풍부할 수 있음
* 리뷰 길이 편차가 클 수 있음

### 12-2. 카카오

카카오는 짧고 직접적인 판단형 리뷰가 많을 가능성이 높다. 따라서 핵심 인상은 빠르게 잡히지만, 긴 서사나 방문 맥락 정보는 상대적으로 적을 수 있다. 

해석 포인트:

* 핵심 평가 포인트가 명확함
* 문장이 짧아 분석 단서가 적을 수 있음
* 압축형 긍정/부정 표현 비중이 높을 수 있음

### 12-3. 구글

구글은 관광·탐색형 정보, 상태 정보, 외국어 혼합 가능성이 상대적으로 크다. 따라서 관광객 관점, 정보 제공형 문장, 추천형 문장 비중이 더 클 수 있다. 

해석 포인트:

* 관광/탐색 목적 리뷰가 많을 수 있음
* 다국어 처리 고려 필요
* 장소 상태 정보가 풍부할 수 있음

---

## 13. 점수 산출 기획

### 13-1. 리뷰 단위 점수

기획 문서 기준 리뷰 하나의 점수는 다음 요소를 조합한다.

* 전체 감성 점수
* 속성 평균 감성 점수
* 재방문 의사 점수
* 추천 의사 점수 

개념식:

```text
review_score =
  w1 * overall_sentiment_score
+ w2 * aspect_sentiment_avg
+ w3 * revisit_intent_score
+ w4 * recommend_intent_score
```

### 13-2. 리뷰 가중치

모든 리뷰를 동일하게 보지 않고 다음 요소를 반영한다.

* 최신성
* 텍스트 길이
* 속성 수
* 사진 포함 여부
* 좋아요 수 가능 시
* 중복 가능성
* 지나치게 짧은 리뷰 여부 

### 13-3. 플랫폼별 종합 점수

플랫폼별 선호도는 리뷰 점수와 리뷰 가중치를 종합한 가중 평균으로 계산한다. 기획 문서의 기본 산식은 아래와 같다. 

```text
platform_preference_score =
Σ(review_score * review_weight) / Σ(review_weight)
```

### 13-4. 속성별 플랫폼 점수

플랫폼별로 맛, 분위기, 가격, 접근성 같은 속성 점수를 따로 계산한다. 기존 기획은 실제 비교에서 이 표가 가장 유용하다고 본다. 

### 13-5. 최종 통합 점수

최종 점수는 플랫폼 종합 점수, 핵심 속성 점수, 재방문/추천 의사, 최신성 등을 종합해 계산한다.
플랫폼 간 편차가 지나치게 크면 패널티를 줄 수 있다. 기존 기획도 균등 가중치 방식과 리뷰 품질 반영 방식 등 보정 가능성을 제시한다. 

---

## 14. 비교 결과 생성 규칙

플랫폼 간 비교는 숫자만 나열하면 안 되고 해석 문장을 생성해야 한다.

예시:

* 구글은 관광객 관점에서 경관 평가가 높음
* 네이버는 가격과 대기시간에 더 민감함
* 카카오는 접근성과 주차에 대한 평가가 자주 언급됨 

또한 기존 개선 코드에는 속성별 점수 차이가 일정 기준 이상이면 “어느 플랫폼에서 가장 높고 낮은가”를 문장으로 만들어 주는 비교 로직이 포함되어 있다.

## 15. 품질 경고 및 예외 처리

### 15-1. 리뷰 수 부족 경고

플랫폼마다 리뷰 수가 크게 차이나면 비교가 불안정해진다. 예를 들어 한 플랫폼만 리뷰 수가 압도적으로 많으면, 비교 결과에 주의가 필요하다는 경고를 리포트에 넣어야 한다. 

### 15-2. 언어 혼합 경고

특히 구글은 영어, 일본어, 중국어 등 다국어 리뷰가 섞일 수 있다. 초기 MVP는 한국어/영어 중심으로 처리하고, 그 외 언어는 제외하거나 별도 태깅하는 것이 현실적이다. 

### 15-3. 별점-텍스트 불일치

별점이 5점인데 불만이 많거나, 별점은 낮은데 본문은 생각보다 긍정적인 경우가 있을 수 있다. 따라서 별점은 보조 신호로만 사용하고, 핵심 점수는 텍스트 기반 분석에 두는 것이 적절하다. 

### 15-4. 특정 플랫폼 미수집

어떤 플랫폼에서 후보를 찾지 못하거나 리뷰를 수집하지 못한 경우, 남은 플랫폼만으로 부분 분석을 진행하되 리포트에 명확히 표시한다.

### 15-5. 짧은 리뷰와 정보 부족 리뷰

“굿”, “별로”, “괜찮음”처럼 지나치게 짧은 리뷰는 감성 신호는 줄 수 있어도 속성 분석 근거는 약하다. 따라서 낮은 정보량 가중치를 적용한다.

## 16. 출력 결과 기획

최종 결과는 DB에 저장되는 레코드가 아니라 **즉시 반환되는 분석 리포트 JSON**이다.

예상 응답 구조:

```json
{
  "input": {
    "place_name": "장소명",
    "category_main": "cafe"
  },
  "matched_places": {
    "google": {},
    "kakao": {},
    "naver": {}
  },
  "collection_summary": {
    "google": 120,
    "kakao": 84,
    "naver": 0
  },
  "platform_scores": {
    "google": 81.2,
    "kakao": 77.5
  },
  "aspect_summary_by_platform": {},
  "common_strengths": [],
  "common_weaknesses": [],
  "platform_differences": [],
  "final_preference_score": 79.4,
  "warnings": []
}
```

기존 설계는 저장형 `final_report` 구조를 상정했지만, 현재는 같은 내용을 응답 본문으로 직접 반환하는 방식이 더 맞다. 저장을 제거해도 리포트 구성 요소 자체는 유지할 수 있다.

## 17. 사용자 시나리오

### 시나리오 1. 일반 사용자

사용자가 “성수동 카페 어니언”을 입력한다.
시스템은 각 플랫폼 후보를 찾고 같은 장소를 선택한 후 리뷰를 수집한다.
이후 “분위기는 전 플랫폼에서 높고, 가격은 네이버와 카카오에서 낮게 평가됨, 구글은 관광형 추천 리뷰 비중이 높음” 같은 요약을 반환한다.

### 시나리오 2. 관광지 비교

사용자가 특정 관광지를 입력한다.
시스템은 경관, 접근성, 혼잡도, 편의시설을 중심으로 분석하여 “경관은 전반적으로 높지만, 네이버와 카카오에서는 주차와 혼잡도 불만이 상대적으로 많다”는 식으로 결과를 제공한다.

## 18. 기대 효과

### 18-1. 사용자 관점

* 여러 플랫폼을 일일이 비교하는 시간을 줄일 수 있다
* 단순 별점이 아니라 실제 평가 이유를 이해할 수 있다
* 방문 전 의사결정의 정확도를 높일 수 있다

### 18-2. 분석 관점

* 플랫폼별 리뷰 문화 차이를 파악할 수 있다
* 같은 장소가 플랫폼마다 어떻게 다르게 인식되는지 확인할 수 있다
* 향후 고도화 시 추천 시스템, 상권 분석, 장소 브랜드 분석으로 확장 가능하다

## 19. 현재 기준 MVP 정의

현재 기준 MVP는 다음 수준으로 정의한다.

1. 장소명 1개 입력
2. 3개 플랫폼 후보 검색
3. 동일 장소 자동 선택 또는 후보 표시
4. 플랫폼별 리뷰 수집
5. 한국어 중심 전처리
6. 속성 사전 기반 속성 추출
7. 규칙 기반 또는 경량 모델 기반 감성 분석
8. 플랫폼별 점수표와 비교 리포트 반환

즉, 지금 단계의 핵심은 “저장”이 아니라 “한 번의 요청으로 비교 분석 결과를 안정적으로 내는 것”이다.

## 21. 최종 정리 문장

본 프로젝트는 사용자가 입력한 특정 장소에 대해 구글맵, 카카오맵, 네이버맵의 장소 후보를 요청 시점에 매칭하고, 선택된 장소의 리뷰를 공통 기준으로 수집·정규화·분석하여 플랫폼별 선호도와 플랫폼 간 인식 차이를 비교한 뒤 통합 선호도 리포트를 즉시 반환하는 멀티플랫폼 리뷰 비교 분석 서비스이다. 이는 기존 기획의 목표를 유지하면서도, 현재 범위에 맞게 저장형 구조를 제거하고 요청 단위 실행형 구조로 단순화한 형태다.
