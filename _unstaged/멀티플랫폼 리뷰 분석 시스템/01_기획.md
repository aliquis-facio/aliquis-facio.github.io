즉, 목표는:

**한 장소(place) 기준으로**

* 구글맵 리뷰
* 카카오맵 리뷰
* 네이버맵 리뷰

를 각각 수집하고,
각 플랫폼의 리뷰를 **동일 기준으로 분석**한 뒤,
마지막에

1. 플랫폼별 선호도 차이
2. 공통적으로 좋아하는 요소
3. 플랫폼마다 다르게 평가되는 요소
4. 최종 통합 장소 선호도

를 도출하는 모델을 만드는 거지.

아래처럼 설계하면 된다.

---

# 1. 프로젝트 목표 정의

## 최종 목표

특정 장소 1개에 대해 3개 플랫폼의 리뷰를 비교 분석하여 다음을 산출한다.

* 플랫폼별 종합 선호도 점수
* 플랫폼별 속성별 선호도 점수
* 공통 강점 / 공통 약점
* 플랫폼 간 인식 차이
* 통합 최종 장소 선호도 리포트

## 분석 대상 카테고리

* 음식점
* 카페
* 관광지

이 3개는 리뷰에서 중요하게 보는 요소가 다르므로
**공통 분석 프레임 + 카테고리별 속성 사전** 구조로 설계하는 것이 좋다.

# 2. 전체 시스템 구조

전체 파이프라인은 아래처럼 보면 된다.

```text
[장소 입력]
   ↓
[플랫폼별 장소 매핑]
   ↓
[구글맵 / 카카오맵 / 네이버맵 리뷰 수집]
   ↓
[리뷰 정규화]
   ↓
[전처리]
   ↓
[리뷰 분석]
   ├─ 감성 분석
   ├─ 속성 추출
   ├─ 속성별 감성 분석
   ├─ 키워드 추출
   ├─ 재방문/추천 의사 탐지
   └─ 신뢰도 보정
   ↓
[플랫폼별 점수 산출]
   ↓
[플랫폼 간 비교 분석]
   ↓
[통합 선호도 점수 및 리포트 생성]
```

# 3. 핵심 설계 원칙

이 프로젝트에서 제일 중요한 건 세 가지다.

## 3-1. 같은 장소를 정확히 매칭해야 함

플랫폼마다 장소명이 조금씩 다를 수 있다.

예:

* 스타벅스 강남R점
* 스타벅스 리저브 강남점
* 스타벅스 강남점

그래서 리뷰 분석 전에 먼저 **동일 장소 매핑**이 필요하다.

매칭 기준 예시:

* 장소명 유사도
* 주소 유사도
* 위도/경도
* 전화번호
* 카테고리 일치 여부

즉, 내부적으로는 하나의 `master_place_id`를 만들고,
그 아래에 플랫폼별 place id를 연결하는 구조가 좋다.

```text
master_place_id: P0001
- google_place_id: ...
- kakao_place_id: ...
- naver_place_id: ...
```

---

## 3-2. 플랫폼마다 리뷰 성격이 다름

구글맵, 카카오맵, 네이버맵은 사용자층과 리뷰 스타일이 조금씩 다르다.

예:

* 구글맵: 외국인/관광객/짧은 후기/별점 중심일 수 있음
* 카카오맵: 지역 사용자 중심, 간단 후기 많음
* 네이버맵: 사진 리뷰, 상세 후기, 방문 인증 성격이 강할 수 있음

그래서 “플랫폼별 점수”를 바로 평균 내면 안 되고,
먼저 **플랫폼 특성 보정**을 해야 한다.

---

## 3-3. 장소 선호도는 속성 기반으로 봐야 함

단순히 리뷰 전체가 긍정적인지 보는 것만으로는 부족하다.

예:

* “분위기는 좋은데 가격이 비싸다”
* “맛은 괜찮지만 대기가 길다”

이런 문장은 하나의 리뷰 안에 긍정/부정이 동시에 존재한다.

그래서 반드시
**Aspect-Based Sentiment Analysis (ABSA)** 구조로 가야 한다.

# 6. 플랫폼별 리뷰 수집 설계

여기서는 기술적으로 중요한 부분이 있다.

## 6-1. 수집 대상 필드

플랫폼마다 가능한 범위 내에서 다음 항목을 최대한 맞춘다.

* 장소명
* 플랫폼 place id
* 별점
* 리뷰 본문
* 작성일
* 리뷰어 식별값
* 사진 여부
* 좋아요 수
* 방문 인증 여부
* 리뷰 개수 총합

---

## 6-2. 정규화가 필요함

플랫폼마다 필드명이 다를 수 있으므로 내부 공통 스키마로 통일한다.

예:

* `star`, `score`, `rating` → `rating`
* `content`, `review`, `comment` → `review_text`

---

## 6-3. 주의할 점

실제 구현 단계에서는 플랫폼별 공식 API 제공 여부, 데이터 접근 범위, 크롤링 허용 범위가 다르다.
그래서 설계상으로는 `Collector Interface`를 만들고, 실제 수집기는 플랫폼별로 분리하는 게 좋다.

```python
class BaseReviewCollector:
    def search_place(self, query): ...
    def get_place_detail(self, place_id): ...
    def get_reviews(self, place_id): ...
```

```python
class GoogleCollector(BaseReviewCollector): ...
class KakaoCollector(BaseReviewCollector): ...
class NaverCollector(BaseReviewCollector): ...
```

이렇게 해두면 수집 방식이 바뀌어도 분석 로직은 그대로 유지된다.

# 7. 리뷰 전처리 설계

## 7-1. 기본 전처리

* HTML/특수문자 제거
* 이모지 처리
* 반복 문자 정리
* 띄어쓰기/오탈자 보정
* 언어 감지
* 중복 리뷰 제거

예:

* “맛있어요!!!!!!!” → “맛있어요!”
* “굿굿굿” → 강조 표현으로 따로 태깅 가능

---

## 7-2. 중복 리뷰 제거

같은 사용자가 여러 플랫폼에 비슷한 문장을 남길 수 있다.
또는 복붙 리뷰가 있을 수 있다.

방법:

* 텍스트 유사도
* 리뷰 날짜
* 별점
* 리뷰어명 유사도

중복이 의심되면

* 완전 제거
  또는
* 가중치만 낮춤

---

## 7-3. 짧은 리뷰 처리

예:

* “굿”
* “별로”
* “최고”
* “쏘쏘”

이런 리뷰는 정보량은 적지만 감성은 강하다.
따라서 버리지 말고:

* 전체 감성 분석에는 포함
* 속성 분석에서는 제외 또는 낮은 신뢰도로 반영

---

# 8. 분석 모델 설계

이 부분이 핵심이다.

## 8-1. 1단계: 전체 감성 분석

리뷰 단위로 긍정/중립/부정 분류 + 연속 점수화

출력 예:

```text
리뷰: "분위기는 좋았는데 가격이 좀 비싸요"
전체 감성: 중립
sentiment_score = 0.12
```

---

## 8-2. 2단계: 속성 추출

리뷰에서 어떤 속성이 언급되었는지 찾는다.

출력 예:

```text
언급 속성:
- 분위기
- 가격
```

방법:

* 규칙 기반 키워드 사전
* 문장 분류 모델
* NER 유사 태깅 방식

초기에는 규칙 기반 + 사전 기반이 실용적이다.

예:

* “맛”, “음식”, “간”, “풍미” → 맛
* “친절”, “불친절”, “응대” → 서비스
* “웨이팅”, “줄”, “기다림” → 대기시간

---

## 8-3. 3단계: 속성별 감성 분석

속성이 언급되었으면, 그 속성에 대해 긍정/중립/부정을 판정한다.

예:

* “커피는 맛있는데 좌석이 불편해요”

  * 커피/음료 맛: 긍정
  * 좌석 편안함: 부정

이 단계가 최종적으로 가장 중요하다.

---

## 8-4. 4단계: 재방문/추천 의사 탐지

다음 표현을 별도로 잡는다.

### 재방문 긍정

* 또 올게요
* 재방문 의사 있음
* 다음에 또 방문
* 단골될 듯

### 재방문 부정

* 다시는 안 감
* 한 번이면 충분
* 재방문 의사 없음

### 추천 긍정

* 추천합니다
* 꼭 가보세요
* 친구에게 추천하고 싶음

### 추천 부정

* 추천 안 함
* 비추천

이 값은 선호도 판단에 강한 시그널이다.

---

## 8-5. 5단계: 키워드/토픽 추출

플랫폼별로 많이 나오는 키워드를 분리해서 본다.

* 긍정 키워드
* 부정 키워드
* 중립/정보성 키워드

예:

* 구글맵: “view”, “tourist”, “beautiful”
* 네이버맵: “웨이팅”, “사진”, “인테리어”
* 카카오맵: “동네”, “주차”, “조용함”

이 차이는 플랫폼별 사용자층 차이를 보여준다.

---

# 9. 점수 산출 모델

이제 리뷰 결과를 숫자로 만들어야 한다.

## 9-1. 리뷰 단위 점수

리뷰 하나에 대해:

```text
review_score =
  w1 * overall_sentiment_score
+ w2 * aspect_sentiment_avg
+ w3 * revisit_intent_score
+ w4 * recommend_intent_score
```

예시 가중치:

* 전체 감성: 0.25
* 속성 평균 감성: 0.40
* 재방문 의사: 0.20
* 추천 의사: 0.15

---

## 9-2. 리뷰 신뢰도 가중치

모든 리뷰를 똑같이 보지 않는다.

가중치 요소:

* 최신성
* 텍스트 길이
* 속성 수
* 사진 포함 여부
* 좋아요 수
* 중복 가능성
* 지나치게 짧은 리뷰 여부

예:

```text
review_weight =
  freshness_weight
* informativeness_weight
* credibility_weight
```

예시:

* 최근 리뷰: 1.2
* 사진 포함 리뷰: 1.1
* 내용 거의 없는 리뷰: 0.6

---

## 9-3. 플랫폼별 종합 점수

```text
platform_preference_score =
Σ(review_score * review_weight) / Σ(review_weight)
```

이렇게 하면 플랫폼별 선호도 점수가 나온다.

예:

* Google: 82.4
* Kakao: 76.1
* Naver: 84.7

---

## 9-4. 속성별 플랫폼 점수

플랫폼별로 속성 점수를 분리한다.

예:

| 속성   | Google | Kakao | Naver |
| ---- | -----: | ----: | ----: |
| 맛    |     88 |    84 |    91 |
| 분위기  |     90 |    86 |    93 |
| 가격   |     62 |    58 |    55 |
| 대기시간 |     49 |    53 |    45 |

이 표가 실제 비교에서 가장 유용하다.

---

# 10. 플랫폼 간 비교 로직

이 프로젝트의 차별점은 여기다.

## 10-1. 비교 항목

플랫폼별로 다음을 비교한다.

* 평균 별점
* 텍스트 감성 점수
* 속성별 감성 점수
* 재방문/추천 의사 비율
* 상위 긍정/부정 키워드
* 리뷰 수 / 최신 리뷰 비중

---

## 10-2. 해석 가능한 비교 결과 만들기

예:

* 구글맵은 관광객 관점에서 경관 평가가 높음
* 네이버맵은 웨이팅과 가격에 더 민감함
* 카카오맵은 접근성과 주차에 대한 평가가 자주 언급됨

즉 단순히 점수 차이가 아니라
**왜 플랫폼별 인식이 다른지**를 설명해줘야 한다.

---

## 10-3. 플랫폼 편향 보정

예를 들어 구글맵은 짧고 극단적인 리뷰가 많고, 네이버맵은 상세 리뷰가 많을 수 있다.

그래서 통합 점수 산출 전 아래 보정을 넣을 수 있다.

### 방법 A: 플랫폼 균등 가중치

```text
final_score = (Google + Kakao + Naver) / 3
```

### 방법 B: 리뷰 품질 반영 가중치

```text
final_score = 
  0.3 * Google +
  0.3 * Kakao +
  0.4 * Naver
```

### 방법 C: 카테고리별 플랫폼 가중치

예:

* 관광지: 구글맵 비중 높임
* 음식점/카페: 네이버·카카오 비중 높임

예시:

```text
음식점:
- Google 0.25
- Kakao 0.35
- Naver 0.40
```

이런 식으로 설계할 수 있다.

---

# 11. 통합 장소 선호도 점수 설계

최종적으로는 하나의 장소에 대해 통합 점수를 내야 한다.

## 11-1. 공식 예시

```text
final_place_preference =
  platform_weighted_sentiment
+ platform_weighted_aspect_score
+ revisit_recommend_bonus
- inconsistency_penalty
```

조금 더 풀면:

```text
Final Score =
0.35 * 플랫폼 종합 감성 평균
+ 0.40 * 핵심 속성 점수 평균
+ 0.15 * 재방문/추천 점수
+ 0.10 * 최신 리뷰 보정
- 0.10 * 플랫폼 간 불일치 패널티
```

---

## 11-2. 불일치 패널티

예:

* 구글 90, 네이버 88, 카카오 87 → 안정적
* 구글 95, 네이버 60, 카카오 58 → 플랫폼 간 인식 차이 큼

이 경우 단순 평균만 내면 오해가 생긴다.
그래서 플랫폼 간 분산이 큰 경우 패널티를 준다.

```text
inconsistency_penalty = platform_score_std * alpha
```

# 15. 음식점/카페/관광지별 최종 분석 프레임

## 음식점

중심 질문:

* 사람들이 이곳을 맛 때문에 좋아하나?
* 가격/웨이팅이 발목을 잡나?
* 플랫폼별로 맛 평가가 일관적인가?

핵심 속성:

* 맛, 양, 가격, 서비스, 대기시간, 청결, 분위기

---

## 카페

중심 질문:

* 맛보다 분위기/좌석/작업성이 더 중요하게 작동하나?
* 사진/인테리어 중심 평가가 강한가?
* 플랫폼별 사용자군 차이가 큰가?

핵심 속성:

* 음료 맛, 디저트, 분위기, 인테리어, 좌석, 소음, 작업 적합성, 가격

---

## 관광지

중심 질문:

* 실제 만족도가 경관 때문인지 접근성 때문인지?
* 관광객 관점과 지역 사용자 관점이 다른가?
* 편의시설/혼잡도 이슈가 큰가?

핵심 속성:

* 경관, 접근성, 편의시설, 혼잡도, 안전성, 체험성, 주차
