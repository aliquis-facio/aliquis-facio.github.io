# 1. ERD 관점 설계

전체 엔티티는 크게 7개로 보면 된다.

```text
Place
 ├─ PlatformPlace
 ├─ Review
 │   ├─ ReviewSentence
 │   ├─ ReviewAnalysis
 │   └─ AspectAnalysis
 ├─ PlatformSummary
 └─ FinalPlaceReport
```

관계를 풀어 쓰면:

- `Place` 1 : N `PlatformPlace`
    
- `Place` 1 : N `Review`
    
- `Review` 1 : N `ReviewSentence`
    
- `Review` 1 : 1 `ReviewAnalysis`
    
- `ReviewSentence` 1 : N `AspectAnalysis`
    
- `Place` 1 : N `PlatformSummary`
    
- `Place` 1 : 1 `FinalPlaceReport`

# 2. DB 스키마 상세

DB는 처음에는 SQLite/PostgreSQL 둘 다 가능하게 설계하면 좋다.  
실무적으로는 PostgreSQL 기준 타입으로 잡는 게 깔끔하다.

---

## 2-1. places

```sql
CREATE TABLE places (
    master_place_id        VARCHAR(40) PRIMARY KEY,
    place_name             VARCHAR(255) NOT NULL,
    category_main          VARCHAR(50) NOT NULL,   -- restaurant / cafe / attraction
    category_sub           VARCHAR(100),
    address                TEXT,
    road_address           TEXT,
    lat                    DECIMAL(10, 7),
    lng                    DECIMAL(10, 7),
    phone                  VARCHAR(50),
    created_at             TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at             TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

설명:

- 내부 기준 장소 테이블
    
- 플랫폼별 장소를 하나로 묶는 기준점
    
- `category_main`은 이후 속성 사전 선택 로직에 직접 연결됨
    

---

## 2-2. platform_places

```sql
CREATE TABLE platform_places (
    id                     BIGSERIAL PRIMARY KEY,
    master_place_id        VARCHAR(40) NOT NULL REFERENCES places(master_place_id),
    platform               VARCHAR(20) NOT NULL,   -- google / kakao / naver
    platform_place_id      VARCHAR(255) NOT NULL,
    platform_place_name    VARCHAR(255),
    platform_category      VARCHAR(100),
    platform_address       TEXT,
    platform_rating        DECIMAL(3,2),
    platform_review_count  INT,
    source_url             TEXT,
    matched_confidence     DECIMAL(4,3),
    created_at             TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(platform, platform_place_id)
);
```

설명:

- 플랫폼 원본 장소 정보 보관
    
- `matched_confidence`로 동일 장소 매칭 신뢰도 저장
    
- 나중에 장소 매칭 오류 분석할 때 유용함

설명:

## 2-4. review_sentences

문장 단위 분석을 위해 이 테이블이 있는 게 좋다.

```sql
CREATE TABLE review_sentences (
    sentence_id            BIGSERIAL PRIMARY KEY,
    review_id              VARCHAR(60) NOT NULL REFERENCES reviews(review_id),
    sentence_order         INT NOT NULL,
    sentence_text          TEXT NOT NULL,
    normalized_sentence    TEXT,
    sentence_length        INT,
    created_at             TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

설명:

- ABSA를 문장 단위로 돌리기 위해 필요
    
- 리뷰 전체 단위보다 문장 단위가 속성-감성 연결이 정확함
    

예:

- “분위기는 좋은데 가격이 비싸요”  
    이런 문장은 하나의 문장에서 두 속성이 동시에 나오므로,  
    문장 단위 + 속성 span 기록이 중요하다.

# 5. 점수 산식 구체안

이 부분은 네 프로젝트에서 가장 중요한 핵심 로직이다.

---

## 5-1. 리뷰 단위 원점수

먼저 각 값을 범위 통일한다.

- `overall_sentiment_score`: -1 ~ 1
    
- `aspect_sentiment_avg`: -1 ~ 1
    
- `revisit_intent_score`: 0 ~ 1
    
- `recommend_intent_score`: 0 ~ 1
    

이를 0~100 스케일로 바꿔도 되고, 내부 계산은 그대로 둬도 된다.  
여기서는 내부 계산은 정규화값으로 유지하는 게 낫다.

### 리뷰 점수 공식

```text
review_raw_score =
0.25 * overall_sentiment_score
+ 0.40 * aspect_sentiment_avg
+ 0.20 * (2 * revisit_intent_score - 1)
+ 0.15 * (2 * recommend_intent_score - 1)
```

여기서 `2x - 1`을 쓰는 이유는  
0~1 값을 -1~1 범위로 맞추기 위해서다.

---

## 5-2. 리뷰 점수 100점 환산

```text
review_score_100 = (review_raw_score + 1) * 50
```

예:

- raw = 0.64 → 82점
    
- raw = 0.00 → 50점
    
- raw = -0.50 → 25점
    

---

## 5-3. 리뷰 가중치

### 최신성 가중치

예를 들어 최근 90일 중심으로 가중치를 줄 수 있다.

```text
freshness_weight = exp(-days_since_review / 180)
```

대략 해석:

- 아주 최근 리뷰: 1에 가까움
    
- 오래된 리뷰: 점점 낮아짐
    

---

### 정보량 가중치

```text
informativeness_weight =
min(1.0, 0.4 + 0.02 * token_count + 0.08 * mentioned_aspect_count)
```

예:

- 짧고 속성 없는 리뷰 → 낮음
    
- 길고 다양한 속성 언급 → 높음
    

---

### 신뢰도 가중치

```text
credibility_weight =
1.0
+ 0.10 * has_photo
+ 0.05 * visit_verified
+ 0.03 * log(1 + like_count)
- 0.25 * duplicate_risk
```

---

### 최종 리뷰 가중치

```text
review_weight =
freshness_weight
* informativeness_weight
* credibility_weight
```

권장 범위는 너무 튀지 않게 clamp:

```text
review_weight = min(1.5, max(0.3, review_weight))
```

---

## 5-4. 플랫폼 종합 점수

```text
platform_preference_score =
Σ(review_score_100 * review_weight) / Σ(review_weight)
```

---

## 5-5. 플랫폼 속성 점수

예를 들어 `맛` 속성의 플랫폼 점수는:

```text
platform_aspect_score(aspect) =
Σ(aspect_score_100 * review_weight * aspect_confidence)
 /
Σ(review_weight * aspect_confidence)
```

이렇게 하면

- 속성 언급이 많은 리뷰
    
- 신뢰도 높은 리뷰
    
- 추출 확신도가 높은 경우
    

가 더 반영된다.

---

## 5-6. 플랫폼별 핵심 속성 평균

카테고리마다 핵심 속성이 다르므로 다르게 적용한다.

예를 들어 카페라면:

```text
cafe_core_aspect_score =
0.20 * 음료맛
+ 0.15 * 디저트
+ 0.20 * 분위기
+ 0.15 * 좌석편안함
+ 0.15 * 작업적합성
+ 0.15 * 가격
```

음식점이라면:

```text
restaurant_core_aspect_score =
0.30 * 맛
+ 0.15 * 양
+ 0.15 * 가격
+ 0.15 * 서비스
+ 0.10 * 청결
+ 0.15 * 대기시간
```

관광지라면:

```text
attraction_core_aspect_score =
0.25 * 경관
+ 0.15 * 접근성
+ 0.15 * 편의시설
+ 0.15 * 혼잡도
+ 0.10 * 안전성
+ 0.10 * 체험성
+ 0.10 * 주차
```

---

## 5-7. 플랫폼 최종 점수 재구성

플랫폼별 최종 점수는 리뷰 평균만 쓰지 말고, 핵심 속성/재방문/추천도 같이 반영하는 게 더 좋다.

```text
platform_final_score =
0.35 * overall_sentiment_score_100
+ 0.40 * core_aspect_score_100
+ 0.15 * revisit_recommend_score_100
+ 0.10 * freshness_score_100
```

여기서

```text
revisit_recommend_score_100 =
0.5 * revisit_score_100 + 0.5 * recommend_score_100
```

---

## 5-8. 플랫폼 통합 점수

카테고리별 플랫폼 가중치를 다르게 두는 게 현실적이다.

### 음식점 추천 가중치

```text
Google 0.25
Kakao  0.35
Naver  0.40
```

### 카페 추천 가중치

```text
Google 0.20
Kakao  0.35
Naver  0.45
```

### 관광지 추천 가중치

```text
Google 0.45
Kakao  0.25
Naver  0.30
```

---

## 5-9. 플랫폼 간 불일치 패널티

```text
platform_mean = weighted_mean(platform_scores)
platform_std  = standard_deviation(platform_scores)

inconsistency_penalty = min(15, platform_std * 0.8)
```

예:

- 80, 82, 84 → 표준편차 낮음 → 패널티 작음
    
- 92, 61, 58 → 표준편차 큼 → 패널티 큼
    

---

## 5-10. 최종 장소 점수

```text
final_place_preference =
weighted_platform_score
- inconsistency_penalty
```

조금 더 풀면:

```text
Final Score =
0.35 * 플랫폼 종합 감성 평균
+ 0.40 * 핵심 속성 평균
+ 0.15 * 재방문/추천 평균
+ 0.10 * 최신성 평균
- 불일치 패널티
```

# 6. 해석 로직

## 6-1. 공통 강점 도출 규칙

공통 강점은:

- 3개 플랫폼 중 2개 이상에서
    
- 해당 속성 점수가 75 이상
    
- 언급량이 충분함
    

이면 강점으로 본다.

```text
if aspect_score >= 75 and mention_count >= threshold:
    strength_candidate
```

2개 이상 플랫폼에서 strength면 `공통 강점`

---

## 6-2. 공통 약점 도출 규칙

- 2개 이상 플랫폼에서
    
- 속성 점수 45 이하
    
- 부정 언급량 충분
    

이면 `공통 약점`

---

## 6-3. 플랫폼 차이 도출 규칙

예:

- 한 플랫폼의 속성 점수가 다른 플랫폼 평균보다 15점 이상 차이
    
- 또는 언급 비율 차이가 큼
    

그러면 차이 포인트로 잡는다.

예시 문장 생성 규칙:

```text
if Google.view_score - avg(other_platforms.view_score) >= 15:
    "구글맵에서는 경관/뷰 평가가 다른 플랫폼보다 뚜렷하게 높게 나타남"
```

# 10. 실제 개발 시작용 함수 흐름 예시

```python
def run_place_preference_pipeline(place_query: str, category: str):
    # 1. 플랫폼별 후보 검색
    google_candidates = google_collector.search_place(place_query)
    kakao_candidates = kakao_collector.search_place(place_query)
    naver_candidates = naver_collector.search_place(place_query)

    # 2. 동일 장소 매핑
    matched = place_matcher.match_candidates(
        place_query,
        google_candidates,
        kakao_candidates,
        naver_candidates
    )

    # 3. 플랫폼별 리뷰 수집
    reviews = []
    for platform, place_id in matched["matched_places"].items():
        collector = get_collector(platform)
        reviews.extend(collector.get_reviews(place_id, max_reviews=200))

    # 4. 리뷰 저장 + 전처리
    normalized_reviews = preprocess_reviews(reviews, category)

    # 5. 분석
    analysis_results = analyze_reviews(normalized_reviews, category)

    # 6. 플랫폼 집계
    platform_summary = summarize_by_platform(analysis_results, category)

    # 7. 최종 점수
    final_report = build_final_report(platform_summary, category)

    return final_report
```

- 한국어만 우선 분석

- 한국어/영어만 지원
