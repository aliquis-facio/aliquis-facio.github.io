# OO Concepts
## 1. OO 방법론 개요(Hype)

- OO는 **객체(및 클래스)** 중심으로 시스템을 분석·분해·모듈화하는 기법 묶음. 기능보다 **안정적인 개념 단위(객체)** 에 구조를 맞춘다. 대형·변화가 잦은 시스템에 유리.

## 2. OO Design vs OO Programming

- **OO Design**: 문제 영역의 **객체를 찾아 설계**하는 방법(언어 독립).
- **OO Programming**: ADT, **상속**, **다형성**을 활용해 **구현**하는 활동(언어·구현 이슈).

## 3. OO 설계 접근

- **클래스 식별**: CRC(Card-Responsibility-Collaborator) 등.
- **클래스/객체 관계**: **디자인 패턴**으로 반복 문제 해결.

## 4. 기능 분해 vs 객체지향 분해

- **POP(절차적)**: 상향식이 아닌 **상위→하위 기능(top-down)**, **함수 중심**, 데이터 은닉 어려움, 상속·연산자 오버로딩 없음.
- **OOP**: **하향식보다 하향/상향 혼합이지만 실질은 bottom-up**, **데이터(객체) 중심**, **메시지 전달**, **데이터 은닉**, **상속/오버로딩 가능**.
- Booch의 도식: 기능 분해는 **프로시저→데이터** 흐름, OO 분해는 **ADT(데이터 구조+연산) 단위**로 캡슐화.

## 5. 기본 OO 개념

- **추상화(Abstraction)**: 본질적 속성만 남기고 불필요한 세부 제거.
- **캡슐화/정보은닉(Encapsulation)**: 데이터와 연산을 묶고 **외부에서 내부 표현을 숨김**.
- **상속(Inheritance)**: 상위의 속성/행위를 하위가 재사용·확장(장점: 재사용·일관성 / 단점: 결합도 상승, 과도한 상속 계층 위험). **Coad’s Rules**로 상속 사용 기준 제안.
- **다형성(Polymorphism)**: 상위 타입의 **동일 인터페이스**로 **하위의 구체 행위**를 호출 → **유연한 아키텍처**(상위는 추상, 하위는 구체 구현; 하위 클래스 추가가 상위 로직 변경 없이 가능).

## 6. 인터페이스 vs 추상 클래스 (Java 관점)

- **인터페이스**: 다중 구현 가능, **타입 공유**를 통한 **결합도 낮은 설계**에 핵심.
- **추상 클래스**: **단일 상속**, **필드/생성자/구현 메서드 보유 가능**, `Object` 상속.
- 공통적으로 **패턴 설계에 빈번히 활용**(상위는 계약, 하위는 구체화).
