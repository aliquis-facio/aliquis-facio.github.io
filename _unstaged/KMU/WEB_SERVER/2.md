## 앱 생성하기

장고 앱(app)은 특정 기능을 담당하는 모듈이다. 하나의 장고 프로젝트는 여러 개의 앱으로 구성될 수 있다.

pybo 앱 생성
`django-admin startapp pybo`

## 1. URL과 뷰

### 핵심 개념

장고에서 사용자가 브라우저로 특정 주소를 요청하면, `urls.py`가 URL을 해석해 적절한 뷰 함수로 연결하고, 뷰는 응답을 반환한다.
초기에는 URL 매핑이 없어서 `Not Found: /pybo/` 오류가 발생하고, 이후 `config/urls.py`에 경로를 추가하고 `pybo/views.py`에 `index` 함수를 작성하면서 첫 장고 프로그램이 완성된다. `HttpResponse`는 페이지 요청에 대한 응답을 반환할 때 사용하는 클래스라고 설명한다. 

### 개발 흐름

```md
브라우저 요청 → urls.py가 URL 해석 → views.py의 함수 실행 → 응답 반환
```

즉, 브라우저가 `localhost:8000/pybo`를 요청하면 장고 서버가 이를 받아 URL 패턴을 찾고, 연결된 뷰를 실행한 뒤 결과를 브라우저에 돌려준다. 

### URL 분리

처음에는 `config/urls.py`에서 직접 `pybo/`를 처리하지만, 이후에는 `include('pybo.urls')`를 사용해 앱별로 URL을 분리한다. 이렇게 하면 `pybo/`로 시작하는 요청은 `config/urls.py`가 아니라 `pybo/urls.py`에서 처리하게 된다. 이 구조는 프로젝트가 커질수록 관리성을 높여 준다. `pybo/urls.py`에서는 이미 상위에서 `pybo/`를 처리했기 때문에 빈 문자열 `''`을 경로로 사용한다.

## 2. 모델과 데이터베이스

### migrate와 기본 테이블

장고 프로젝트를 처음 실행하면 migration이 적용되지 않았다는 경고가 나타날 수 있다. 이는 기본 앱들이 필요로 하는 테이블이 아직 DB에 생성되지 않았기 때문이다. `config/settings.py`에는 설치된 앱 목록(`INSTALLED_APPS`)과 데이터베이스 설정이 포함되어 있으며, 기본 DB로 SQLite가 사용된다. `python manage.py migrate`를 실행하면 장고가 필요한 기본 테이블들을 생성한다. 

### DB 확인 도구

자료에서는 DB 상태를 확인하기 위해 `DB Browser for SQLite` 또는 PyCharm의 `Database Navigator` 플러그인을 사용하는 방법을 소개한다. 이를 통해 `user` 테이블 등 장고가 생성한 실제 테이블과 컬럼 구조를 확인할 수 있다.

## 3. 장고 모델 만들기

### Question / Answer 모델

게시판을 만들기 위해 `Question`과 `Answer` 모델을 정의한다.

* `Question`

  * `subject`: 질문 제목
  * `content`: 질문 내용
  * `create_date`: 작성 일시

* `Answer`

  * `question`: 어떤 질문에 달린 답변인지 가리키는 외래키
  * `content`: 답변 내용
  * `create_date`: 작성 일시

자료에서는 `CharField`를 글자 수 제한이 있는 데이터, `TextField`를 제한이 없는 긴 텍스트, `DateTimeField`를 날짜/시간 데이터라고 설명한다. 또한 다른 모델을 참조할 때는 `ForeignKey`를 사용한다고 정리한다. 

### SQL과 ORM 비교
[SQL vs ORM](SQL%20vs%20ORM.md)

자료는 같은 구조를 SQL로 직접 만드는 예와 Django 모델로 정의하는 예를 나란히 보여 주며, ORM이 SQL 작성 없이도 테이블 구조를 정의하고 다룰 수 있게 해 준다고 설명한다. 특히 `Question` 모델의 `id` 필드는 직접 선언하지 않아도 자동 생성된다. 

### 앱 등록과 마이그레이션

모델을 실제 테이블로 만들려면 `pybo` 앱을 `INSTALLED_APPS`에 등록해야 한다. 등록 후 `makemigrations`를 실행하면 `0001_initial.py` 같은 migration 파일이 생성되고, 이후 `migrate`를 실행하면 실제 DB 테이블이 만들어진다. 자료는 `makemigrations`는 작업 파일 생성, `migrate`는 실제 테이블 생성이라는 차이를 강조한다. 생성된 테이블명은 보통 `pybo_question`, `pybo_answer` 형태가 된다.

## 4. ORM으로 데이터 다루기

### 데이터 생성(Create)

장고 셸에서 `Question(...)` 객체를 만든 뒤 `save()`를 호출하면 데이터베이스에 저장된다. `create_date`에는 `timezone.now()`를 사용한다. 저장 후에는 `id` 값이 자동 부여된다. 이는 CRUD 중 **Create**에 해당한다. 

예시 형태:

```python
from django.utils import timezone
q = Question(subject='pybo가 무엇인가요?', content='pybo에 대해 알고 싶습니다.', create_date=timezone.now())
q.save()
```

### 데이터 조회(Read)

`Question.objects.all()`로 전체 조회를 할 수 있고, `filter()`는 조건에 맞는 여러 건을 `QuerySet` 형태로 반환하며, `get()`은 조건에 맞는 단일 객체를 반환한다. `get()`은 조건에 맞는 데이터가 없거나 여러 개면 오류가 날 수 있지만, `filter()`는 조건에 맞는 데이터가 없으면 빈 `QuerySet`을 돌려준다. 문자열 일부 검색은 `subject__contains='장고'`처럼 수행한다. 이는 CRUD 중 **Read**에 해당한다. 

또한 `__str__` 메서드를 모델에 추가하면 조회 시 `<Question object (1)>` 대신 제목이 보이도록 만들 수 있다. 

### 데이터 수정(Update)

조회한 객체의 속성을 바꾸고 `save()`를 다시 호출해야 실제 DB에 반영된다. `save()`를 하지 않으면 수정 사항은 저장되지 않는다. 이는 CRUD 중 **Update**에 해당한다. 

### 데이터 삭제(Delete)

객체에 `delete()`를 호출하면 즉시 DB에서 삭제된다. 이는 CRUD 중 **Delete**에 해당한다.

## 5. 연결된 데이터 조회

`Answer`는 `Question`을 외래키로 참조하므로, 답변 객체에서 연결된 질문을 조회할 수 있다. 반대로 질문 객체에서는 `answer_set`을 통해 그 질문에 달린 답변들을 조회할 수 있다. 자료는 질문 하나에 답변 여러 개가 달릴 수 있으므로 `q.answer_set`처럼 세트 단위로 접근해야 한다고 설명한다.

## 6. WebSocket 보충 설명

[HTTP VS Web Socket](HTTP%20VS%20Web%20Socket.md)

후반부에는 생성형 AI 활용 샘플로 WebSocket 개념이 포함되어 있다. 자료는 일반 HTTP가 요청-응답-종료 구조라 실시간 통신에 불리한 반면, WebSocket은 한 번 연결하면 계속 열린 상태로 양방향 통신이 가능하다고 설명한다. 실시간 채팅, 실시간 대시보드, 온라인 게임, 알림 시스템 같은 경우 WebSocket이 적합한 예로 제시된다. 반대로 일반 웹 페이지, 로그인, 게시판, 단순 데이터 조회에는 굳이 WebSocket이 필요하지 않다고 한다. Django에서 WebSocket을 사용하려면 `Channels`, `Daphne`, `Redis` 등을 사용한다고 소개한다.

또한 자료는 WebSocket 도입 시 주의할 점도 함께 설명한다. 연결 유지로 인한 상태 관리 비용, 서버가 여러 대일 때의 메시지 브로커 필요성, 재접속/유실/중복/순서 꼬임 문제, 상태 동기화, `message id`, `ack`, `retry` 같은 추가 로직이 필요하다고 정리한다. 즉, “실시간성”이 정말 필요한지 먼저 판단한 뒤 기술을 선택해야 한다는 비판적 관점을 강조한다.
