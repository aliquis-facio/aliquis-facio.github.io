# 10주차

## 1. 전체 흐름

이번 주차의 큰 목표는 기존 Pybo 게시판을 단순한 질문·답변 게시판에서 더 실제 서비스에 가까운 형태로 확장하는 것이다.

주요 학습 내용은 다음과 같다.

| 구분        | 핵심 내용                                        |
| --------- | -------------------------------------------- |
| 게시판 기능    | 글쓴이 표시, 질문/답변 수정, 질문/답변 삭제, 수정일시 표시          |
| 권한 처리     | 로그인 사용자와 작성자가 같을 때만 수정·삭제 허용                 |
| 배포        | PythonAnywhere를 이용한 Django 프로젝트 무료 배포        |
| GitHub 연동 | 로컬 수정 → GitHub push → 서버 pull → Reload       |
| 디버깅       | Chrome DevTools로 HTML, CSS, Network, 상태코드 확인 |
| HTTP 흐름   | POST 후 redirect하는 PRG 패턴 이해                  |
| 데이터 생성    | Django shell로 더미 질문 데이터 대량 생성                |

## 2. 글쓴이 표시 기능

### 2.1 질문 목록에 글쓴이 표시

기존 질문 목록에는 번호, 제목, 작성일시 정도만 표시되었다. 10주차에서는 여기에 **글쓴이 칼럼**을 추가한다.

질문 목록 템플릿에서 테이블 헤더에 `글쓴이`를 추가하고, 각 질문 행에 다음 값을 출력한다.

```django
{{ question.author.username }}
```

즉, `Question` 모델의 `author` 필드를 통해 Django 기본 `User` 모델의 `username`을 출력한다.

### 2.2 질문 상세 화면에 글쓴이 표시

질문 상세 페이지에서는 질문 본문 아래에 작성자와 작성일시를 함께 표시한다.

```django
<div class="mb-2">{{ question.author.username }}</div>
<div>{{ question.create_date }}</div>
```

강의자료에서는 Bootstrap의 `badge`와 `d-flex justify-content-end`를 사용해 작성자 정보를 오른쪽에 배치했다. 이렇게 하면 본문과 메타 정보가 분리되어 가독성이 좋아진다.

### 2.3 답변에도 글쓴이 표시

답변 영역도 질문과 같은 방식으로 처리한다.

```django
<div class="mb-2">{{ answer.author.username }}</div>
<div>{{ answer.create_date }}</div>
```

정리하면 질문과 답변 모두 다음 구조를 갖게 된다.

```text
본문 내용
오른쪽 하단: 작성자 / 작성일시
```

## 3. 수정 기능 구현

### 3.1 modify_date 필드 추가

질문과 답변을 수정할 수 있도록 `Question`, `Answer` 모델에 `modify_date` 필드를 추가한다.

```python
modify_date = models.DateTimeField(null=True, blank=True)
```

각 옵션의 의미는 다음과 같다.

| 옵션           | 의미             |
| ------------ | -------------- |
| `null=True`  | DB에 NULL 저장 허용 |
| `blank=True` | 폼 검증에서 빈 값 허용  |

처음 작성된 글은 수정된 적이 없으므로 `modify_date`가 비어 있을 수 있다. 따라서 두 옵션이 필요하다.

모델을 수정한 후에는 반드시 마이그레이션을 수행한다.

```bash
python manage.py makemigrations
python manage.py migrate
```

## 4. 질문 수정 기능

### 4.1 수정 버튼 표시 조건

질문 상세 화면에 수정 버튼을 추가하되, 아무 사용자에게나 보여주면 안 된다.

수정 버튼은 **현재 로그인한 사용자와 질문 작성자가 같은 경우에만** 보여야 한다.

```django
{% if request.user == question.author %}
<a href="{% url 'pybo:question_modify' question.id %}"
   class="btn btn-sm btn-outline-secondary">수정</a>
{% endif %}
```

이 조건은 템플릿에서 버튼을 숨기는 역할이다. 하지만 이것만으로는 보안이 충분하지 않다. URL을 직접 입력해서 접근할 수 있기 때문에 뷰 함수에서도 반드시 한 번 더 권한 검사를 해야 한다.

### 4.2 질문 수정 URL 매핑

`pybo/urls.py`에 질문 수정 URL을 추가한다.

```python
path(
    'question/modify/<int:question_id>/',
    views.question_modify,
    name='question_modify'
)
```

예를 들어 2번 질문을 수정하면 다음과 같은 URL이 호출된다.

```text
/pybo/question/modify/2/
```

### 4.3 질문 수정 뷰 함수

질문 수정은 GET과 POST 요청을 나누어 처리한다.

| 요청 방식 | 역할                    |
| ----- | --------------------- |
| GET   | 기존 질문 데이터를 폼에 채워서 보여줌 |
| POST  | 수정된 데이터를 DB에 저장       |

핵심 코드는 다음과 같다.

```python
@login_required(login_url='common:login')
def question_modify(request, question_id):
    question = get_object_or_404(Question, pk=question_id)

    if request.user != question.author:
        messages.error(request, '수정권한이 없습니다')
        return redirect('pybo:detail', question_id=question.id)

    if request.method == "POST":
        form = QuestionForm(request.POST, instance=question)
        if form.is_valid():
            question = form.save(commit=False)
            question.author = request.user
            question.modify_date = timezone.now()
            question.save()
            return redirect('pybo:detail', question_id=question.id)
    else:
        form = QuestionForm(instance=question)

    context = {'form': form}
    return render(request, 'pybo/question_form.html', context)
```

여기서 가장 중요한 코드는 `instance=question`이다.

```python
QuestionForm(instance=question)
```

이 코드는 기존 질문 데이터를 폼에 채워 넣는다.

```python
QuestionForm(request.POST, instance=question)
```

이 코드는 새 질문을 만드는 것이 아니라 **기존 질문 객체를 수정**한다.

`instance=question`을 빼면 기존 글 수정이 아니라 새 글 생성처럼 동작할 수 있으므로 주의해야 한다.

## 5. 질문 삭제 기능

### 5.1 삭제 버튼 추가

삭제 버튼도 작성자 본인에게만 보여야 한다.

```django
{% if request.user == question.author %}
<a href="#"
   class="delete btn btn-sm btn-outline-secondary"
   data-uri="{% url 'pybo:question_delete' question.id %}">
   삭제
</a>
{% endif %}
```

여기서 중요한 부분은 `href="#"`와 `data-uri`다.

| 속성               | 역할                            |
| ---------------- | ----------------------------- |
| `href="#"`       | 클릭해도 바로 이동하지 않음               |
| `data-uri`       | 실제 삭제 URL 저장                  |
| `class="delete"` | JavaScript에서 삭제 버튼을 찾기 위한 클래스 |

삭제는 위험한 작업이므로 사용자가 클릭하자마자 바로 삭제하지 않고, 확인 창을 띄운다.

### 5.2 삭제 확인 JavaScript

강의자료에서는 jQuery 코드와 vanilla JS 코드가 함께 언급되었다. 음성 녹음에서는 교재 코드가 환경에 따라 작동하지 않을 수 있으므로, 스크립트 로딩 위치와 jQuery 로드 여부가 중요하다고 설명했다.

안정적으로 사용할 수 있는 vanilla JS 방식은 다음과 같다.

```javascript
const delete_elements = document.getElementsByClassName("delete");

Array.from(delete_elements).forEach(function(element) {
    element.addEventListener('click', function() {
        if(confirm("정말로 삭제하시겠습니까?")) {
            location.href = this.dataset.uri;
        }
    });
});
```

동작 흐름은 다음과 같다.

```text
삭제 버튼 클릭
→ confirm 창 표시
→ 확인 클릭
→ data-uri에 저장된 URL로 이동
→ 서버에서 삭제 처리
→ 목록 페이지로 redirect
```

### 5.3 질문 삭제 URL과 뷰 함수

URL 매핑은 다음과 같다.

```python
path(
    'question/delete/<int:question_id>/',
    views.question_delete,
    name='question_delete'
)
```

뷰 함수는 다음과 같다.

```python
@login_required(login_url='common:login')
def question_delete(request, question_id):
    question = get_object_or_404(Question, pk=question_id)

    if request.user != question.author:
        messages.error(request, '삭제권한이 없습니다')
        return redirect('pybo:detail', question_id=question.id)

    question.delete()
    return redirect('pybo:index')
```

삭제 후에는 기존 상세 페이지가 더 이상 존재하지 않는다. 따라서 `render`가 아니라 질문 목록으로 `redirect`해야 한다.

## 6. 답변 수정 및 삭제 기능

답변 수정·삭제도 질문 수정·삭제와 거의 같은 방식이다.

### 6.1 답변 수정 URL

```python
path(
    'answer/modify/<int:answer_id>/',
    views.answer_modify,
    name='answer_modify'
)
```

### 6.2 답변 삭제 URL

```python
path(
    'answer/delete/<int:answer_id>/',
    views.answer_delete,
    name='answer_delete'
)
```

### 6.3 답변 수정 뷰 핵심

답변 수정에서도 `instance=answer`가 중요하다.

```python
form = AnswerForm(request.POST, instance=answer)
```

수정 후에는 해당 답변이 속한 질문 상세 페이지로 이동한다.

```python
return redirect('pybo:detail', question_id=answer.question.id)
```

### 6.4 답변 삭제 뷰 핵심

답변 삭제도 작성자 본인만 가능해야 한다.

```python
if request.user != answer.author:
    messages.error(request, '삭제권한이 없습니다')
else:
    answer.delete()

return redirect('pybo:detail', question_id=answer.question.id)
```

답변은 삭제되어도 질문은 남아 있으므로, 삭제 후에는 질문 상세 페이지로 redirect한다.

## 7. 수정일시 표시

질문이나 답변을 수정하면 `modify_date`에 현재 시간이 저장된다.

```python
question.modify_date = timezone.now()
answer.modify_date = timezone.now()
```

템플릿에서는 `modify_date`가 있을 때만 표시한다.

```django
{% if question.modify_date %}
<div class="badge badge-light p-2 text-left mx-3">
    <div class="mb-2">modified at</div>
    <div>{{ question.modify_date }}</div>
</div>
{% endif %}
```

답변도 같은 구조로 처리한다.

```django
{% if answer.modify_date %}
<div class="badge badge-light p-2 text-left mx-3">
    <div class="mb-2">modified at</div>
    <div>{{ answer.modify_date }}</div>
</div>
{% endif %}
```

즉, 처음 작성된 글은 작성일시만 보이고, 수정된 글은 수정일시가 추가로 표시된다.

## 8. render와 redirect의 차이

음성 강의에서 매우 강조된 부분이다.

### 8.1 render

`render`는 템플릿과 데이터를 조합해서 HTML 응답을 직접 만들어 반환한다.

주로 GET 요청에서 화면을 보여줄 때 사용한다.

```python
return render(request, 'pybo/question_form.html', context)
```

예를 들어 질문 수정 버튼을 눌렀을 때 기존 질문 내용을 폼에 채워 보여주는 경우에는 `render`를 사용한다.

### 8.2 redirect

`redirect`는 브라우저에게 다른 URL로 이동하라고 응답한다.

주로 POST 처리 후 사용한다.

```python
return redirect('pybo:detail', question_id=question.id)
```

즉, 데이터를 저장하거나 삭제한 후에는 `redirect`로 페이지를 이동시킨다.

### 8.3 핵심 비교

| 구분      | render          | redirect |
| ------- | --------------- | -------- |
| 역할      | HTML을 직접 만들어 응답 |          |
| 역할      | 다른 URL로 이동시킴    |          |
| 주 사용    | GET 요청 화면 표시    |          |
| 주 사용    | POST 처리 후 이동    |          |
| URL 변화  | 보통 그대로          |          |
| URL 변화  | 이동한 URL로 바뀜     |          |
| 새로고침 위험 | POST 중복 실행 가능   |          |
| 새로고침 위험 | GET만 반복되어 안전    |          |

정리하면 다음과 같다.

```text
GET 요청 → render
POST 요청 → redirect
```

## 9. POST-Redirect-GET, PRG 패턴

음성 녹음에서 가장 중요하게 설명된 개념이다.

게시글 작성, 답변 등록, 수정, 삭제는 모두 데이터베이스를 변경하는 작업이다. 이런 작업은 보통 POST 요청으로 처리한다.

그런데 POST 요청 후 바로 `render`를 하면 문제가 생길 수 있다.

사용자가 F5 새로고침을 누르면 마지막 요청이 다시 실행되기 때문이다.

예를 들어 다음 문제가 생긴다.

| 작업    | render 사용 후 새로고침 시 문제 |
| ----- | --------------------- |
| 답변 등록 | 같은 답변이 두 번 등록됨        |
| 질문 수정 | 같은 수정 요청이 반복됨         |
| 삭제    | 삭제 요청이 다시 실행될 수 있음    |
| 송금 예시 | 같은 송금이 두 번 실행될 수 있음   |

그래서 POST 요청을 처리한 뒤에는 바로 화면을 렌더링하지 않고, redirect를 보낸다.

흐름은 다음과 같다.

```text
POST 요청
→ DB 생성/수정/삭제
→ 302 Redirect 응답
→ 브라우저가 GET 요청
→ 200 OK로 화면 표시
```

이 방식이 **POST-Redirect-GET 패턴**이다.

핵심 목적은 다음과 같다.

```text
F5 새로고침으로 인한 중복 생성·수정·삭제 방지
```

Chrome DevTools Network 패널에서 보면 POST 이후 `302`, 그 다음 GET 요청에서 `200`이 발생하는 것을 확인할 수 있다.

## 10. PythonAnywhere 배포

### 10.1 PythonAnywhere를 선택한 이유

강의자료와 음성 녹음 모두에서 PythonAnywhere가 수업용 배포 환경으로 적합하다고 설명했다.

이유는 다음과 같다.

| 항목                | 설명                    |
| ----------------- | --------------------- |
| 무료 플랜             | Beginner 플랜 사용 가능     |
| 카드 등록 불필요         | 과금 사고 위험이 적음          |
| Python/Django 친화적 | Django 프로젝트 배포에 적합    |
| 브라우저 IDE 제공       | 별도 SSH 도구 없이 콘솔 사용 가능 |
| 수업 환경 적합          | 학생들이 동일한 방식으로 실습 가능   |

AWS Lightsail도 배포 환경으로 사용할 수 있지만, 카드 등록이 필요하고 설정이 복잡하며 비용 발생 가능성이 있다. 그래서 수업 실습에서는 PythonAnywhere를 더 안전한 선택으로 제시했다.

### 10.2 가입 시 주의점

PythonAnywhere 가입 시에는 반드시 **Beginner Free** 플랜을 선택해야 한다.

유료 플랜을 선택하면 카드 정보 입력 화면이 나오므로 잘못 들어간 것이다.

또한 username은 곧 사이트 주소가 된다.

```text
https://username.pythonanywhere.com
```

따라서 username은 신중하게 정해야 한다.

### 10.3 settings.py 배포 설정

배포 전에는 `settings.py`를 수정해야 한다.

```python
import os

DEBUG = False

ALLOWED_HOSTS = [
    'username.pythonanywhere.com',
    'localhost',
    '127.0.0.1',
]

STATIC_URL = 'static/'
STATIC_ROOT = BASE_DIR / 'staticfiles'

SECRET_KEY = os.environ.get(
    'DJANGO_SECRET_KEY',
    'fallback-key-for-dev-only',
)
```

핵심 설정은 다음과 같다.

| 설정              | 의미                       |
| --------------- | ------------------------ |
| `DEBUG = False` | 운영 환경에서 디버그 정보 숨김        |
| `ALLOWED_HOSTS` | 접속 가능한 도메인 지정            |
| `STATIC_ROOT`   | collectstatic 결과물이 모일 위치 |
| `SECRET_KEY`    | 환경변수로 분리 권장              |

음성 강의에서는 이 배포 설정이 시험 범위에 포함된다고 언급했다.

## 11. GitHub를 이용한 배포 반영 흐름

음성 강의에서는 실제로 수정 버튼 색상을 바꾸는 예시를 통해 배포 흐름을 설명했다.

### 11.1 로컬에서 코드 수정

예시는 다음과 같다.

기존 버튼:

```html
class="btn btn-sm btn-outline-secondary"
```

수정 후:

```html
class="btn btn-sm btn-outline-primary"
```

즉, 회색 버튼을 파란색 버튼으로 바꾸는 수정이다.

### 11.2 로컬에서 GitHub로 push

```bash
git status
git add .
git commit -m "change secondary button to primary button"
git push origin main
```

각 명령의 의미는 다음과 같다.

| 명령어                    | 역할            |
| ---------------------- | ------------- |
| `git status`           | 변경 파일 확인      |
| `git add .`            | 변경 파일 staging |
| `git commit -m`        | 커밋 생성         |
| `git push origin main` | GitHub에 업로드   |

GitHub 인증 시에는 일반 비밀번호가 아니라 Personal Access Token을 사용해야 한다.

### 11.3 PythonAnywhere에서 pull

PythonAnywhere Bash Console에서 GitHub의 변경 사항을 가져온다.

```bash
git pull
```

이후 필요한 경우 다음 명령을 실행한다.

```bash
python manage.py migrate
python manage.py collectstatic
```

마지막으로 PythonAnywhere Web 탭에서 **Reload**를 눌러야 한다.

전체 흐름은 다음과 같다.

```text
로컬 코드 수정
→ git add
→ git commit
→ git push
→ PythonAnywhere Bash Console
→ git pull
→ migrate
→ collectstatic
→ Reload
→ 배포 사이트에서 확인
```

가장 흔한 실수는 `push`, `pull`까지 해놓고 **Reload를 누르지 않는 것**이다. Reload를 하지 않으면 새 코드가 실행되지 않는다.

## 12. PythonAnywhere Web 설정

PythonAnywhere에서 Web 앱을 등록할 때는 반드시 **Manual configuration**을 선택해야 한다.

`Django` 자동 옵션을 누르면 PythonAnywhere가 빈 Django 프로젝트를 새로 생성하기 때문에, GitHub에서 가져온 기존 프로젝트와 충돌할 수 있다.

설정해야 할 주요 영역은 다음과 같다.

| 영역                     | 예시                                      |
| ---------------------- | --------------------------------------- |
| Source code            | `/home/username/ws_2026`                |
| Working directory      | `/home/username/ws_2026`                |
| Virtualenv             | `/home/username/.virtualenvs/pybo-venv` |
| Static files URL       | `/static/`                              |
| Static files Directory | `/home/username/ws_2026/staticfiles`    |

### 12.1 WSGI 파일 설정

WSGI 파일은 배포에서 가장 중요한 설정이다.

```python
import os
import sys

path = '/home/username/ws_2026'
if path not in sys.path:
    sys.path.insert(0, path)

os.environ['DJANGO_SETTINGS_MODULE'] = 'config.settings'

from django.core.wsgi import get_wsgi_application
application = get_wsgi_application()
```

여기서 바꿔야 하는 값은 다음과 같다.

| 값                 | 의미                    |
| ----------------- | --------------------- |
| `username`        | PythonAnywhere 사용자명   |
| `ws_2026`         | 프로젝트 폴더명              |
| `config.settings` | Django settings.py 경로 |

### 12.2 흔한 배포 오류

| 오류                            | 원인                            | 해결                          |
| ----------------------------- | ----------------------------- | --------------------------- |
| `DisallowedHost`              | `ALLOWED_HOSTS`에 도메인 누락       | 도메인 추가 후 Reload             |
| `ModuleNotFoundError: django` | Virtualenv 경로 오류              | Web 탭에서 가상환경 경로 확인          |
| `ModuleNotFoundError: config` | WSGI path 오류                  | `manage.py`가 있는 폴더로 path 설정 |
| CSS/JS 적용 안 됨                 | collectstatic 또는 static 설정 누락 | `collectstatic` 재실행         |
| 500 Internal Server Error     | 내부 서버 오류                      | Error log 확인                |

## 13. requirements.txt 주의점

강의자료에서는 `conda` 환경에서 `pip freeze`를 사용할 때 생기는 문제도 설명했다.

`conda` 환경에서 `pip freeze`를 하면 다음처럼 로컬 경로가 들어갈 수 있다.

```text
absl-py @ file:///home/task_xxx/...
numpy @ file:///opt/conda/...
```

이런 경로는 PythonAnywhere 서버에 존재하지 않기 때문에 설치 오류가 발생한다.

따라서 웹 프로젝트에서는 필요한 패키지만 깔끔하게 작성하는 것이 좋다.

```text
Django==4.2.7
markdown
Pillow
```

또는 `conda` 대신 프로젝트별 `venv` 환경을 사용하는 것이 더 적합하다.

## 14. Chrome DevTools

### 14.1 DevTools의 역할

Chrome DevTools는 브라우저에 내장된 개발자 도구다.

브라우저가 단순히 결과 화면만 보여주는 것이 아니라, 페이지가 만들어지는 과정 전체를 확인할 수 있게 해준다.

확인 가능한 항목은 다음과 같다.

| 영역          | 확인 내용          |
| ----------- | -------------- |
| HTML        | DOM 구조         |
| CSS         | 적용된 스타일        |
| JavaScript  | 실행 로그, 에러      |
| Network     | HTTP 요청/응답     |
| Application | 쿠키, 세션, 로컬스토리지 |
| Performance | 성능 문제          |

### 14.2 실행 방법

| 방법                 | 설명            |
| ------------------ | ------------- |
| `F12`              | 가장 빠른 실행      |
| `Ctrl + Shift + I` | Windows/Linux |
| `Cmd + Option + I` | macOS         |
| 우클릭 → 검사           | 특정 요소 바로 검사   |
| `Ctrl + Shift + C` | 요소 직접 선택      |

### 14.3 Elements 패널

Elements 패널에서는 HTML DOM 구조와 CSS를 확인할 수 있다.

예를 들어 Pybo 페이지에서 다음을 확인할 수 있다.

```html
<head>
<body>
<nav>
<div class="container">
```

또한 Bootstrap 클래스가 어떻게 적용되는지도 볼 수 있다.

예를 들어 다음과 같은 스타일 요소를 확인할 수 있다.

```text
navbar
container
btn
badge
flex
padding
margin
border
```

음성 강의에서는 DevTools를 통해 “이 부분이 navigation bar구나”, “이 부분이 body구나”, “이 CSS가 Bootstrap에서 왔구나”를 확인할 수 있다고 설명했다.

### 14.4 Network 패널

Network 패널은 HTTP 요청과 응답을 확인하는 데 사용한다.

로그인, 로그아웃, 답변 등록, 질문 수정 등을 실행하면 Network 패널에 요청이 기록된다.

확인할 수 있는 값은 다음과 같다.

| 항목       | 의미                                |
| -------- | --------------------------------- |
| URL      | 요청 주소                             |
| Method   | GET, POST 등 요청 방식                 |
| Status   | 200, 302, 404, 500 등 상태코드         |
| Type     | document, stylesheet, script, xhr |
| Response | 서버 응답 내용                          |
| Headers  | 요청/응답 헤더                          |

특히 Bootstrap CSS, JS가 정상적으로 불러와졌는지 확인할 때도 Network 패널을 사용한다.

### 14.5 HTTP 상태코드

수업에서는 Network 패널을 통해 상태코드를 확인했다.

| 상태코드  | 의미          |
| ----- | ----------- |
| 200   | 요청 성공       |
| 302   | Redirect    |
| 300번대 | 다른 URL로 이동  |
| 400번대 | 클라이언트 요청 오류 |
| 500번대 | 서버 내부 오류    |

답변 등록이나 수정 후에는 보통 다음 흐름이 나타난다.

```text
POST 요청 → 302 Redirect → GET 요청 → 200 OK
```

이 흐름이 바로 PRG 패턴과 연결된다.

## 15. Application 패널과 쿠키 확인

Application 패널에서는 브라우저 저장소를 확인한다.

Django 개발에서 특히 중요한 것은 쿠키다.

| 쿠키          | 의미          |
| ----------- | ----------- |
| `sessionid` | 로그인 세션 확인   |
| `csrftoken` | CSRF 검증에 사용 |

로그인 문제가 발생하면 `sessionid`가 있는지 확인한다.

POST 요청에서 403 Forbidden이 발생하면 `csrftoken`이 있는지, 그리고 요청 헤더에 CSRF 토큰이 포함되어 있는지 확인한다.

## 16. Django shell을 이용한 더미 데이터 생성

음성 녹음 후반부에서는 Django shell을 이용해 질문 데이터를 대량 생성하는 방법도 다뤘다.

먼저 shell을 실행한다.

```bash
python manage.py shell
```

필요한 객체를 import한다.

```python
from django.contrib.auth.models import User
from django.utils import timezone
from pybo.models import Question
```

작성자로 사용할 사용자를 가져온다.

```python
user = User.objects.get(id=1)
```

질문을 여러 개 생성한다.

```python
for i in range(400):
    q = Question(
        subject='test:[%03d]' % i,
        content='content',
        create_date=timezone.now(),
        author=user,
    )
    q.save()
```

이 실습의 목적은 페이징이나 검색 기능을 테스트하기 위해 충분한 게시글 데이터를 만드는 것이다.

중요한 점은 `author=user`를 반드시 넣어야 한다는 것이다. `Question` 모델의 `author` 필드가 필수라면 이 값을 넣지 않았을 때 DB 제약 조건 오류가 발생할 수 있다.

## 17. 권한 검사

10주차 전체에서 반복되는 핵심은 **권한 검사**다.

수정·삭제 기능은 버튼을 숨기는 것만으로는 충분하지 않다.

템플릿에서는 다음처럼 버튼을 숨긴다.

```django
{% if request.user == question.author %}
```

하지만 사용자가 URL을 직접 입력할 수도 있으므로, 뷰 함수에서도 다시 검사해야 한다.

```python
if request.user != question.author:
    messages.error(request, '수정권한이 없습니다')
    return redirect('pybo:detail', question_id=question.id)
```

정리하면 다음과 같다.

```text
템플릿 권한 검사: 버튼 표시 여부 제어
뷰 함수 권한 검사: 실제 서버 보안 제어
```

수정·삭제 권한은 반드시 서버 측에서 최종 검증해야 한다.

## 20. 최종 요약

10주차의 핵심은 **게시판 기능 구현과 웹 서비스 운영 흐름을 연결해서 이해하는 것**이다.

Pybo 내부에서는 질문과 답변에 대해 글쓴이 표시, 수정, 삭제, 수정일시 표시를 구현한다. 이때 `instance=obj`를 사용해 기존 객체를 수정하고, `request.user != obj.author` 조건으로 권한을 검사한다.

서버 동작 관점에서는 POST 요청 후 바로 `render`하지 않고 `redirect`해야 한다. 그래야 F5 새로고침으로 같은 등록, 수정, 삭제 작업이 반복되는 문제를 막을 수 있다. 이 흐름이 PRG 패턴이며, Network 패널에서는 `POST → 302 → GET → 200` 흐름으로 확인할 수 있다.

배포 관점에서는 로컬에서 수정한 코드를 GitHub에 push하고, PythonAnywhere에서 pull한 뒤 `migrate`, `collectstatic`, `Reload` 순서로 반영한다. 특히 PythonAnywhere에서는 Reload를 누르지 않으면 새 코드가 적용되지 않는다.

디버깅 관점에서는 Chrome DevTools를 사용해 HTML 구조, CSS 적용 여부, Bootstrap 정적 파일 로딩, HTTP 상태코드, 쿠키와 CSRF 토큰을 확인한다.

결국 이번 주차에서 반드시 기억해야 할 문장은 다음이다.

> **수정·삭제는 작성자 본인만 가능해야 하며, POST 요청 처리 후에는 중복 실행을 막기 위해 반드시 redirect해야 한다.**
