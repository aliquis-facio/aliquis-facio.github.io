# 1. SSE 한 줄 정의

**SSE는 HTTP 연결을 유지한 채, 서버가 클라이언트에게 실시간 이벤트를 계속 push하는 단방향 통신 방식**이다.

---

# 2. 정확한 정의

SSE(Server-Sent Events)는  
브라우저와 서버가 **하나의 HTTP 연결을 길게 유지**하면서,  
서버가 새로운 데이터가 생길 때마다 클라이언트에게 **이벤트 스트림 형태로 전달**하는 기술이다.

핵심은 두 가지야.

- **서버 → 클라이언트 방향만 가능**
    
- **기반은 여전히 HTTP**
    

즉 WebSocket처럼 완전히 별도 프로토콜로 업그레이드하는 게 아니라,  
HTTP 응답을 바로 끝내지 않고 **계속 열어 둔 채 데이터 조각을 순서대로 흘려보내는 방식**이야.

---

# 3. 왜 필요한가?

HTTP 기본 구조는 요청-응답이야.  
클라이언트가 요청하면 서버가 응답하고 끝난다.

그런데 어떤 서비스는 서버가 새 정보를 발견할 때마다 바로 알려주고 싶어.

예를 들면:

- 실시간 알림
    
- 뉴스 속보
    
- 주식 가격 변동
    
- 작업 진행률 표시
    
- 로그 스트리밍
    
- AI 답변 스트리밍
    
- 서버 상태 모니터링
    

이걸 순수 HTTP만으로 처리하면 보통 polling을 써야 한다.

예:

- 1초마다 `/notifications` 요청
    
- 새 알림이 있는지 반복 확인
    

이건 비효율적이야.

- 새 데이터가 없어도 계속 요청함
    
- 서버와 네트워크 부담이 커짐
    
- 실시간성도 완벽하지 않음
    

그래서  
“클라이언트가 한 번 요청하면, 서버가 연결을 유지하면서 새 데이터가 생길 때마다 바로 보내자”  
라는 방식이 필요했고, 그게 SSE야.

---

# 4. SSE의 핵심 특징

## 4-1. 단방향 통신

SSE는 **서버 → 클라이언트** 만 된다.

즉:

- 서버는 브라우저에게 계속 보낼 수 있음
    
- 브라우저는 같은 연결로 역방향 전송 불가
    

클라이언트가 서버에 무언가 보내려면 별도의 HTTP 요청을 해야 해.

---

## 4-2. HTTP 기반

SSE는 기존 HTTP 위에서 동작한다.

즉:

- 별도 소켓 프로토콜을 직접 다룰 필요가 적고
    
- 방화벽/프록시 환경에서 비교적 친화적이며
    
- 브라우저 API도 단순하다
    

---

## 4-3. 지속 연결

요청 하나로 응답을 끝내지 않고,  
서버가 응답 스트림을 계속 열어 둔다.

그 위에서 이벤트를 한 줄씩, 여러 번 보낸다.

---

## 4-4. 텍스트 기반

SSE는 기본적으로 **텍스트 스트림**이다.  
보통 JSON 문자열을 실어 보내는 식으로 많이 쓴다.

---

## 4-5. 자동 재연결 지원

브라우저의 `EventSource` 는 연결이 끊기면 자동으로 재연결을 시도한다.  
이게 SSE의 꽤 큰 장점이다.

---

# 5. 동작 원리

SSE 동작은 생각보다 단순해.

## 1단계: 클라이언트가 HTTP 요청

브라우저가 서버에 SSE 엔드포인트를 요청한다.

예:

```http
GET /events HTTP/1.1
Host: example.com
Accept: text/event-stream
```

클라이언트는  
“일반 HTML이나 JSON 한 번 받고 끝내는 게 아니라, 이벤트 스트림을 받고 싶다”  
는 의미로 연결을 연다.

---

## 2단계: 서버가 응답을 열어 둠

서버는 응답 헤더를 이렇게 설정한다.

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

여기서 가장 중요한 건:

- `Content-Type: text/event-stream`
    

이건  
“이 응답은 일반 텍스트가 아니라 SSE 이벤트 스트림이다”  
라는 뜻이다.

---

## 3단계: 서버가 이벤트를 계속 전송

서버는 응답을 닫지 않고, 아래처럼 이벤트를 계속 보낸다.

```text
data: hello

data: second message

data: third message

```

각 이벤트는 **빈 줄**로 구분된다.  
즉 `\n\n` 이 하나의 이벤트 경계를 만든다.

---

## 4단계: 클라이언트가 이벤트 수신

브라우저는 열린 연결에서 계속 들어오는 이벤트를 실시간으로 처리한다.

---

# 6. SSE 메시지 형식

SSE는 단순 문자열을 보내는 게 아니라, 정해진 필드 형식이 있어.

대표 필드는 이거야.

- `data:`
    
- `event:`
    
- `id:`
    
- `retry:`
    

---

## 6-1. data

실제 데이터를 담는 필드

```text
data: hello world

```

클라이언트는 이 데이터를 받아서 처리한다.

여러 줄 데이터도 가능하다.

```text
data: line 1
data: line 2

```

이 경우 클라이언트는 이를 하나의 데이터 블록으로 합쳐서 받는다.

---

## 6-2. event

이벤트 이름을 지정할 수 있다.

```text
event: notification
data: 새 알림이 도착했습니다

```

클라이언트는 특정 이벤트 이름 기준으로 핸들러를 분리할 수 있다.

---

## 6-3. id

이벤트 ID다.

```text
id: 101
data: 메시지 내용

```

브라우저는 마지막으로 받은 ID를 기억할 수 있고,  
재연결 시 이어받기 로직에 활용할 수 있다.

---

## 6-4. retry

재연결 간격을 서버가 지정할 수 있다.

```text
retry: 5000
data: reconnect delay set

```

이건 연결이 끊어졌을 때 브라우저가 몇 ms 뒤 재시도할지 알려준다.

---

# 7. 브라우저에서 SSE 사용법

브라우저에서는 `EventSource` 객체를 사용한다.

```javascript
const source = new EventSource("/events");

source.onmessage = (event) => {
  console.log("받은 데이터:", event.data);
};

source.onerror = (error) => {
  console.error("SSE 오류:", error);
};
```

이 정도만으로도 기본 수신이 가능하다.

이벤트 이름을 따로 쓰면 이렇게 된다.

```javascript
const source = new EventSource("/events");

source.addEventListener("notification", (event) => {
  console.log("알림:", event.data);
});
```

# 8. 서버 예시

## 8-1. 쉬운 개념 예시

서버가 1초마다 숫자를 하나씩 보낸다고 하자.

```text
data: 1

data: 2

data: 3

```

클라이언트는 실시간 카운터처럼 볼 수 있다.

---

## 8-2. 전공/실무 예시

예를 들어 AI 서버가 답변을 한 번에 다 만들지 않고 토큰 단위로 조금씩 만든다고 하자.

서버는 이렇게 스트리밍 가능하다.

```text
data: 안녕
data: 하세요
data: . 오늘

```

클라이언트는 이걸 이어 붙여서  
답변이 한 글자씩 생성되는 것처럼 보여줄 수 있다.

또는 로그 스트리밍이라면:

```text
event: log
data: [INFO] worker started

event: log
data: [INFO] processing request

event: log
data: [DONE] completed

```

# 9. HTTP와 SSE의 관계

SSE는 HTTP와 완전히 다른 게 아니라, HTTP를 확장적으로 사용하는 방식이다.

## 공통점

- 둘 다 HTTP 기반
    
- URL로 접근
    
- 헤더 사용
    
- 프록시/웹 인프라와 친화적
    

## 차이점

- 일반 HTTP: 응답 한 번 주고 끝
    
- SSE: 응답을 끝내지 않고 계속 유지
    

즉 SSE는  
**“끝나지 않는 HTTP 응답”**  
처럼 이해해도 꽤 정확하다.

---

# 10. WebSocket과 SSE 차이

이 부분이 시험이나 면접에서 정말 자주 나온다.

## SSE

- 서버 → 클라이언트 단방향
    
- HTTP 기반
    
- 텍스트 중심
    
- 브라우저 기본 지원 단순
    
- 자동 재연결 편리
    

## WebSocket

- 양방향
    
- 업그레이드 핸드셰이크 필요
    
- 텍스트/바이너리 모두 강함
    
- 더 범용적
    
- 채팅/게임 같은 상호작용에 적합
    

---

# 11. 언제 SSE를 쓰고, 언제 WebSocket을 쓰나?

## SSE가 적합한 경우

- 알림 전송
    
- 로그 스트리밍
    
- 진행률 업데이트
    
- AI 응답 스트리밍
    
- 실시간 대시보드 값 갱신
    
- 뉴스/주식/상태 변화 feed
    

공통점은 보통  
**서버가 계속 알려주기만 하면 되는 경우**다.

---

## WebSocket이 더 적합한 경우

- 채팅
    
- 멀티플레이 게임
    
- 공동 편집
    
- 화상회의 signaling
    
- 실시간 양방향 협업
    

공통점은  
**클라이언트와 서버가 계속 서로 말해야 하는 경우**다.

---

# 12. SSE의 장점

## 12-1. 구현이 단순하다

WebSocket보다 훨씬 단순한 편이다.

- 클라이언트는 `EventSource`
    
- 서버는 `text/event-stream` 응답만 잘 흘려주면 됨
    

---

## 12-2. HTTP 인프라 활용이 쉽다

기존 웹 서버, 프록시, 인증 구조와 잘 어울린다.

---

## 12-3. 자동 재연결

브라우저가 끊기면 알아서 재시도한다.

---

## 12-4. 서버 push 용도에 딱 맞다

알림, 상태 변경 통지, 스트리밍 출력 등에는 과하지 않고 적절하다.

---

## 12-5. 오버헤드가 적은 편

WebSocket만큼 범용적이진 않지만,  
서버→클라이언트 알림 목적에는 충분히 효율적이다.

---

# 13. SSE의 단점

## 13-1. 양방향이 아니다

클라이언트가 같은 연결로 서버에 메시지를 못 보낸다.

그래서 상호작용이 많으면 결국 HTTP 요청을 따로 섞어야 한다.

---

## 13-2. 기본적으로 텍스트 중심

바이너리 전송에는 적합하지 않다.

---

## 13-3. 연결 수 관리가 필요

HTTP라고 해서 공짜가 아니다.  
연결을 오래 유지하므로 동시 접속이 많으면 서버 자원을 신경 써야 한다.

---

## 13-4. 일부 프록시/버퍼링 이슈

중간 프록시나 웹서버가 응답을 버퍼링하면 실시간 전송이 깨질 수 있다.  
예를 들어 Nginx 설정에서 버퍼링을 끄는 조치가 필요할 수 있다.

---

# 14. SSE 응답 헤더에서 중요한 것

## Content-Type

```http
Content-Type: text/event-stream
```

필수다.

## Cache-Control

```http
Cache-Control: no-cache
```

중간 캐시에 잡히지 않게 하는 데 중요하다.

## Connection

```http
Connection: keep-alive
```

지속 연결 의도를 명확히 한다.

실무에서는 여기에 더해 프록시 버퍼링 비활성화 설정도 자주 필요하다.

---

# 15. 재연결과 Last-Event-ID

SSE의 좋은 기능 중 하나가 재연결이다.

서버가 각 이벤트에 ID를 붙여 보낼 수 있다.

```text
id: 1
data: first

id: 2
data: second

```

연결이 끊겼다가 다시 붙으면,  
브라우저는 마지막으로 받은 이벤트 ID를 기준으로 이어받기를 시도할 수 있다.

이건 실시간 알림이나 로그 스트림에서 꽤 유용하다.

---

# 16. 심장박동(heartbeat)

SSE 연결도 오래 유지되기 때문에  
중간 장비가 “이 연결 죽은 거 아니야?” 하고 끊어버릴 수 있다.

그래서 서버는 가끔 주석 라인이나 빈 이벤트를 보낸다.

예:

```text
: keepalive

```

앞의 `:` 는 코멘트 라인이다.  
클라이언트 애플리케이션 로직에는 의미 없지만, 연결 유지에는 도움이 된다.

---

# 17. SSE와 Polling 비교

## Polling

- 클라이언트가 주기적으로 계속 요청
    
- 구현은 쉬움
    
- 비효율적
    

## SSE

- 한 번 연결 후 서버가 필요할 때만 전송
    
- 더 실시간적
    
- 서버 push에 적합
    

즉 알림/스트리밍 용도에서는 대개 SSE가 polling보다 낫다.

---

# 18. SSE와 Long Polling 비교

## Long Polling

- 요청을 오래 잡고 있다가 데이터 생기면 응답
    
- 응답 후 다시 요청해야 함
    

## SSE

- 한 번 연결 후 계속 이벤트를 여러 개 보냄
    
- 훨씬 자연스러운 스트리밍 구조
    

---

# 19. Django/FastAPI 관점에서 SSE

## Django

전통적인 Django에서는 일반 응답이 아니라 **StreamingHttpResponse** 같은 방식으로 접근할 수 있다.

개념적으로는:

- generator로 데이터를 조금씩 yield
    
- content type을 `text/event-stream` 으로 지정
    

---

## FastAPI

FastAPI도 SSE 구현이 잘 맞는 편이다.  
비동기 generator를 사용해 이벤트를 계속 흘려보낼 수 있다.

이런 구조가 많다.

```python
async def event_generator():
    while True:
        yield "data: hello\n\n"
```

즉 핵심은 프레임워크가 아니라  
**응답을 끊지 않고 이벤트 형식으로 계속 보낸다**는 점이다.

---

# 20. 시험에 자주 나오는 포인트

## 1. SSE는 HTTP 기반이다

WebSocket처럼 101 업그레이드가 아니다.

## 2. 단방향이다

서버 → 클라이언트만 된다.

## 3. MIME 타입은 `text/event-stream`

이거 자주 나온다.

## 4. 이벤트는 빈 줄로 구분된다

`data: ...` 뒤에 최종적으로 빈 줄.

## 5. 브라우저에서 `EventSource` 사용

대표 API 이름 기억.

## 6. 자동 재연결 지원

WebSocket과 비교 포인트로 자주 나온다.

## 7. 채팅에는 보통 부적합

양방향이 아니기 때문.

---

# 21. 자주 틀리는 부분

## “SSE는 서버가 클라이언트에게만 보내니 HTTP보다 완전히 다른 프로토콜이다”

아니야.  
SSE는 **HTTP 위에서 동작하는 스트리밍 방식**이다.

## “SSE는 WebSocket의 하위호환이다”

아니야.  
둘은 목적이 다르다.

## “SSE로 채팅을 구현하면 된다”

부분적으로는 가능하지만 불편하다.  
클라이언트→서버는 별도 POST 요청 등을 써야 해서 보통 WebSocket이 더 적합하다.

## “SSE는 그냥 큰 문자열 한 번 보내는 것이다”

아니야.  
이벤트를 **순차적으로 끊어서 지속적으로 보내는 스트림**이다.

# 22. 암기 포인트

딱 이렇게 외우면 된다.

**SSE = HTTP 기반 + 서버 푸시 + 단방향 + text/event-stream + EventSource**

조금 더 풀면:

- HTTP 연결을 유지한다
    
- 서버가 클라이언트에게 계속 보낸다
    
- 클라이언트는 같은 연결로 못 보낸다
    
- 브라우저는 `EventSource` 로 받는다
    
- 알림/로그/스트리밍 출력에 좋다
    

---

# 23. 쉬운 예시

## 쉬운 예시

학교 공지판 앱을 생각해보자.

- 학생 앱이 서버에 한 번 연결
    
- 교수님이 새 공지를 올림
    
- 서버가 연결된 학생들에게 바로 “새 공지” 이벤트 전송
    

학생이 같은 연결로 서버에 답장을 보내는 건 아니야.  
그냥 공지를 **받기만** 하면 된다.  
이럴 때 SSE가 잘 맞는다.

---

# 24. 전공 예시

## 전공 예시 1: 서버 로그 스트리밍

운영자가 관리자 페이지를 열어 두면

- 서버에서 새 로그가 생길 때마다
    
- 실시간으로 브라우저에 출력
    

이건 서버→클라이언트만 필요하므로 SSE가 적합하다.

## 전공 예시 2: AI 답변 스트리밍

LLM이 답변을 한 번에 다 만든 뒤 보내지 않고

- 토큰 생성
    
- 토큰 생성
    
- 토큰 생성
    

순서대로 흘려보내면,  
사용자는 답변이 점점 완성되는 걸 본다.  
이것도 SSE가 자주 쓰이는 영역이다.

# 25. 한 문장으로 다시 정리

SSE는  
**HTTP 연결을 길게 유지하면서 서버가 클라이언트에게 실시간 이벤트를 계속 전달하는 단방향 스트리밍 기술**이다.
