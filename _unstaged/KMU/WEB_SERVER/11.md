# 1. Pybo 댓글 기능 구현

11주차 초반에는 Pybo 게시판에 **댓글 기능**을 추가했다. 기존 Pybo에는 질문과 답변 기능이 있었고, 여기에 질문 댓글과 답변 댓글을 모두 지원하는 구조를 추가한다.

핵심 모델은 `Comment`이다.

```python
class Comment(models.Model):
    author = models.ForeignKey(User, on_delete=models.CASCADE)
    content = models.TextField()
    create_date = models.DateTimeField()
    modify_date = models.DateTimeField(null=True, blank=True)
    question = models.ForeignKey(
        Question,
        null=True,
        blank=True,
        on_delete=models.CASCADE
    )
    answer = models.ForeignKey(
        Answer,
        null=True,
        blank=True,
        on_delete=models.CASCADE
    )
```

`Comment` 모델은 질문에도 달릴 수 있고 답변에도 달릴 수 있어야 하므로 `question`, `answer` 필드를 모두 가진다. 다만 하나의 댓글은 질문 또는 답변 중 하나에만 연결되므로 두 필드 모두 `null=True`, `blank=True`가 필요하다. 질문에 달린 댓글이면 `question`에 값이 들어가고, 답변에 달린 댓글이면 `answer`에 값이 들어간다. 

`on_delete=models.CASCADE`는 연결된 질문이나 답변이 삭제되면 그 댓글도 함께 삭제되도록 하는 설정이다.

---

## 1-1. 질문 댓글 구현

질문 상세 화면에서는 `question.comment_set.all`을 사용해 해당 질문에 달린 댓글을 출력한다.

```django
{% for comment in question.comment_set.all %}
<div class="comment py-2 text-muted">
    <span style="white-space: pre-line;">{{ comment.content }}</span>
    <span>
        - {{ comment.author }}, {{ comment.create_date }}
        {% if comment.modify_date %}
        (수정:{{ comment.modify_date }})
        {% endif %}
    </span>
</div>
{% endfor %}
```

댓글을 본문보다 작게 보이도록 CSS도 추가한다.

```css
.comment {
    border-top: dotted 1px #ddd;
    font-size: 0.7em;
}
```

댓글 URL은 등록, 수정, 삭제로 나뉜다.

```python
path(
    'comment/create/question/<int:question_id>/',
    views.comment_create_question,
    name='comment_create_question'
)

path(
    'comment/modify/question/<int:comment_id>/',
    views.comment_modify_question,
    name='comment_modify_question'
)

path(
    'comment/delete/question/<int:comment_id>/',
    views.comment_delete_question,
    name='comment_delete_question'
)
```

등록할 때는 어느 질문에 댓글을 달 것인지 알아야 하므로 `question_id`가 필요하다. 반면 수정과 삭제는 이미 존재하는 댓글을 대상으로 하기 때문에 `comment_id`가 필요하다.

---

## 1-2. CommentForm

댓글 폼은 매우 단순하다.

```python
class CommentForm(forms.ModelForm):
    class Meta:
        model = Comment
        fields = ['content']
        labels = {
            'content': '댓글내용',
        }
```

사용자가 직접 입력하는 것은 댓글 내용인 `content`뿐이다. `author`, `create_date`, `question`, `answer`는 view 함수에서 자동으로 채운다.

---

## 1-3. 댓글 등록 함수

댓글 등록에서는 `commit=False`가 중요하다.

```python
@login_required(login_url='common:login')
def comment_create_question(request, question_id):
    question = get_object_or_404(Question, pk=question_id)

    if request.method == "POST":
        form = CommentForm(request.POST)

        if form.is_valid():
            comment = form.save(commit=False)
            comment.author = request.user
            comment.create_date = timezone.now()
            comment.question = question
            comment.save()
            return redirect('pybo:detail', question_id=question.id)
    else:
        form = CommentForm()

    context = {'form': form}
    return render(request, 'pybo/comment_form.html', context)
```

`form.save(commit=False)`는 DB에 바로 저장하지 않고 임시 객체만 만든다. 이후 작성자, 작성일, 연결 질문을 채운 뒤 `comment.save()`로 저장한다.

---

## 1-4. 댓글 수정과 삭제

댓글 수정과 삭제에서는 반드시 **작성자 권한 검사**가 필요하다.

```python
if request.user != comment.author:
    messages.error(request, '댓글수정권한이 없습니다')
    return redirect('pybo:detail', question_id=comment.question.id)
```

즉, 현재 로그인한 사용자와 댓글 작성자가 같을 때만 수정/삭제가 가능하다.

이 부분은 이후 DRF의 `permission_classes`와 연결되는 중요한 개념이다. Django FBV에서는 직접 `if request.user != comment.author`를 작성하지만, DRF에서는 권한 클래스로 분리할 수 있다. 

---

## 1-5. 답변 댓글 구현

답변 댓글은 질문 댓글과 거의 같다. 차이는 연결 대상이 `Question`이 아니라 `Answer`라는 점이다.

| 구분          | 질문 댓글                         | 답변 댓글                        |
| ----------- | ----------------------------- | ---------------------------- |
| 댓글 목록       | `question.comment_set`        | `answer.comment_set`         |
| 등록 URL 파라미터 | `question_id`                 | `answer_id`                  |
| 연결 필드       | `comment.question = question` | `comment.answer = answer`    |
| 리다이렉트       | `comment.question.id`         | `comment.answer.question.id` |

답변 댓글 등록 후에는 답변 자체가 아니라 **해당 답변이 달린 질문 상세 화면**으로 돌아가야 한다.

```python
return redirect(
    'pybo:detail',
    question_id=comment.answer.question.id
)
```

정리하면 댓글 기능은 다음 구조로 이해할 수 있다.

```text
User
 └── Comment
       ├── question FK → Question
       └── answer FK   → Answer
```

`Comment` 모델 하나로 질문 댓글과 답변 댓글을 모두 처리하고, `CommentForm`과 `comment_form.html`은 재사용한다.

# 2. 인증, 세션, 권한

음성 녹음에서는 질문 등록, 수정, 삭제 시연을 통해 **Authentication, Session, Permission** 개념이 강조되었다. 

## Authentication

사용자가 누구인지 확인하는 과정이다. 로그인/로그아웃이 여기에 해당한다.

## Session

로그인 상태를 유지하기 위한 구조이다. 사용자가 로그인하면 서버는 세션을 만들고, 브라우저는 쿠키를 통해 세션 정보를 유지한다.

## Permission

사용자가 어떤 작업을 할 수 있는지 판단하는 과정이다.

예를 들어 다음 조건이 Permission이다.

```django
{% if request.user == question.author %}
    <a href="...">수정</a>
    <a href="...">삭제</a>
{% endif %}
```

또는 view에서 다음처럼 검사할 수 있다.

```python
if request.user != question.author:
    messages.error(request, '수정권한이 없습니다')
    return redirect(...)
```

즉, 11주차에서 다룬 댓글 수정/삭제 권한 검사는 이후 DRF의 권한 처리로 확장되는 기초 개념이다.

# 3. 데코레이터, 클래스 메서드, 리턴 타입 힌트

음성 녹음에서는 과제와 관련해 **데코레이터**, **클래스 메서드**, **리턴 타입 힌트**도 설명되었다. 

## 3-1. 데코레이터

데코레이터는 함수나 클래스 위에 붙어 기존 동작을 감싸거나 확장하는 역할을 한다.

예시는 다음과 같다.

```python
@login_required(login_url='common:login')
def question_create(request):
    ...
```

여기서 `@login_required`는 질문 등록 함수 실행 전에 로그인 여부를 검사한다.

DRF에서도 데코레이터가 등장한다.

```python
@api_view(['GET', 'POST'])
def snippet_list(request):
    ...
```

또는 ViewSet에서는 다음처럼 커스텀 액션을 만들 때 사용한다.

```python
@action(detail=True, methods=['post'])
def toggle(self, request, pk=None):
    ...
```

## 3-2. 클래스 메서드

`@classmethod`도 데코레이터의 일종이다. 인스턴스가 아니라 클래스 자체를 기준으로 동작한다.

```python
class Question(models.Model):

    @classmethod
    def get_all_questions(cls):
        return cls.objects.all()
```

## 3-3. 리턴 타입 힌트

함수가 어떤 값을 반환하는지 명확히 표현할 수 있다.

```python
from django.db.models import QuerySet

@classmethod
def get_all_questions(cls) -> QuerySet:
    return cls.objects.all()
```

리턴 타입 힌트를 사용하면 함수의 입출력 구조가 명확해지고, IDE나 LLM의 도움을 받을 때도 유리하다.

# 4. Serializer 개념

11주차의 가장 중요한 개념은 **Serializer**이다.

기존 Django는 서버에서 Python 객체를 가져와 HTML 템플릿에 넣고 `render()`로 응답했다. 이 방식은 웹 브라우저에는 적합하지만, 모바일 앱, 챗봇, 외부 서버, SPA 프론트엔드에는 적합하지 않다. 이런 클라이언트들은 HTML이 아니라 **JSON 데이터**를 필요로 한다.  

따라서 Django 객체를 JSON으로 바꾸고, 반대로 클라이언트가 보낸 JSON을 Django 객체로 바꾸는 계층이 필요하다. 이것이 Serializer이다.

> Serializer는 Django 객체와 JSON 사이의 양방향 번역기이다.

---

## 4-1. 직렬화 Serialization

직렬화는 서버의 Python 객체를 JSON으로 보낼 수 있는 데이터 형태로 바꾸는 과정이다.

```python
question = Question.objects.get(pk=1)
serializer = QuestionSerializer(question)
serializer.data
```

여러 객체를 직렬화할 때는 반드시 `many=True`를 사용한다.

```python
qs = Question.objects.all()
serializer = QuestionSerializer(qs, many=True)
return Response(serializer.data)
```

`many=True`를 빼면 QuerySet을 단일 객체처럼 처리하려 하기 때문에 오류가 발생할 수 있다.

---

## 4-2. 역직렬화 Deserialization + 검증

역직렬화는 클라이언트가 보낸 JSON 데이터를 Python 객체로 바꾸고, 검증한 뒤 저장하는 과정이다.

```python
serializer = QuestionSerializer(data=request.data)

if serializer.is_valid():
    serializer.save(author=request.user)
    return Response(serializer.data, status=201)

return Response(serializer.errors, status=400)
```

흐름은 다음과 같다.

```text
request.data
→ Serializer(data=...)
→ is_valid()
→ validated_data
→ save()
→ DB 저장
```

즉, Serializer는 단순 변환기가 아니라 **입력 검증기**이자 **저장 로직의 일부**이다.

---

# 5. Form과 Serializer의 관계

Serializer는 Django의 Form과 매우 유사하다.

| Django HTML 방식      | DRF JSON 방식                 |
| ------------------- | --------------------------- |
| `QuestionForm`      | `QuestionSerializer`        |
| `form.is_valid()`   | `serializer.is_valid()`     |
| `form.cleaned_data` | `serializer.validated_data` |
| `form.save()`       | `serializer.save()`         |
| Template: 객체 → HTML | Serializer: 객체 → JSON       |

정리하면 다음과 같다.

```text
Serializer = Form + Template의 API 버전
```

Form은 HTML 폼 데이터를 처리하고, Serializer는 JSON API 데이터를 처리한다. 

---

# 6. Serializer와 ModelSerializer

Serializer에는 크게 두 가지 방식이 있다.

## 6-1. `serializers.Serializer`

필드를 직접 선언하는 수동 방식이다.

```python
class SnippetSerializer(serializers.Serializer):
    id = serializers.IntegerField(read_only=True)
    title = serializers.CharField(required=False, allow_blank=True, max_length=100)
    code = serializers.CharField(style={'base_template': 'textarea.html'})
    linenos = serializers.BooleanField(required=False)
    language = serializers.ChoiceField(choices=LANGUAGE_CHOICES, default='python')
    style = serializers.ChoiceField(choices=STYLE_CHOICES, default='friendly')

    def create(self, validated_data):
        return Snippet.objects.create(**validated_data)

    def update(self, instance, validated_data):
        instance.title = validated_data.get('title', instance.title)
        instance.code = validated_data.get('code', instance.code)
        instance.linenos = validated_data.get('linenos', instance.linenos)
        instance.language = validated_data.get('language', instance.language)
        instance.style = validated_data.get('style', instance.style)
        instance.save()
        return instance
```

이 방식은 세밀하게 제어할 수 있지만 코드가 길다. 특히 `create()`와 `update()`를 직접 작성해야 한다.

---

## 6-2. `serializers.ModelSerializer`

모델 기반으로 필드, 검증, 생성, 수정 로직을 자동으로 만들어주는 방식이다.

```python
class SnippetSerializer(serializers.ModelSerializer):
    class Meta:
        model = Snippet
        fields = ['id', 'title', 'code', 'linenos', 'language', 'style']
```

`ModelSerializer`는 Django의 `ModelForm`과 대응된다.

```text
Form → ModelForm
Serializer → ModelSerializer
```

실무적인 CRUD API에서는 대부분 `ModelSerializer`를 우선 사용한다.

# 7. JSONRenderer와 JSONParser

0514 실습에서는 Django shell에서 `JSONRenderer`와 `JSONParser`를 직접 사용해 직렬화와 역직렬화 흐름을 확인했다. 

## 7-1. JSONRenderer

`JSONRenderer`는 Python dict 형태의 데이터를 JSON bytes로 렌더링한다.

```python
serializer = SnippetSerializer(snippet)
content = JSONRenderer().render(serializer.data)
```

흐름은 다음과 같다.

```text
Python 객체
→ Serializer
→ Python dict
→ JSONRenderer
→ JSON bytes
```

서버에서 클라이언트로 데이터를 보낼 때의 방향이다.

## 7-2. JSONParser

`JSONParser`는 JSON bytes를 Python dict로 파싱한다.

```python
stream = io.BytesIO(content)
data = JSONParser().parse(stream)
```

흐름은 다음과 같다.

```text
JSON bytes
→ JSONParser
→ Python dict
→ Serializer(data=...)
→ is_valid()
→ save()
```

클라이언트가 서버로 데이터를 보낼 때의 방향이다.

---

## 7-3. Python과 JSON 표현 차이

Python과 JSON은 비슷하게 key-value 구조를 가지지만 표현 규칙이 다르다.

| Python      | JSON        |
| ----------- | ----------- |
| `True`      | `true`      |
| `False`     | `false`     |
| `None`      | `null`      |
| Python dict | JSON object |

따라서 API 테스트에서 JSON을 작성할 때는 Python 문법이 아니라 JSON 문법을 사용해야 한다.

# 8. 일반 Django view로 JSON API 만들기

DRF의 Generic View로 가기 전, 먼저 일반 Django FBV로 JSON API를 만들었다.  

## 8-1. 목록 조회와 생성

```python
@csrf_exempt
def snippet_list(request):
    if request.method == 'GET':
        snippets = Snippet.objects.all()
        serializer = SnippetSerializer(snippets, many=True)
        return JsonResponse(serializer.data, safe=False)

    elif request.method == 'POST':
        data = JSONParser().parse(request)
        serializer = SnippetSerializer(data=data)

        if serializer.is_valid():
            serializer.save()
            return JsonResponse(serializer.data, status=201)

        return JsonResponse(serializer.errors, status=400)
```

| HTTP Method | 기능      | 방향        |
| ----------- | ------- | --------- |
| GET         | 목록 조회   | 직렬화       |
| POST        | 새 객체 생성 | 역직렬화 + 저장 |

여기서 `safe=False`는 리스트 형태의 JSON 응답을 허용하기 위해 필요하다.

---

## 8-2. 단건 조회, 수정, 삭제

```python
@csrf_exempt
def snippet_detail(request, pk):
    try:
        snippet = Snippet.objects.get(pk=pk)
    except Snippet.DoesNotExist:
        return HttpResponse(status=404)

    if request.method == 'GET':
        serializer = SnippetSerializer(snippet)
        return JsonResponse(serializer.data)

    elif request.method == 'PUT':
        data = JSONParser().parse(request)
        serializer = SnippetSerializer(snippet, data=data)

        if serializer.is_valid():
            serializer.save()
            return JsonResponse(serializer.data)

        return JsonResponse(serializer.errors, status=400)

    elif request.method == 'DELETE':
        snippet.delete()
        return HttpResponse(status=204)
```

| HTTP Method | 기능    |
| ----------- | ----- |
| GET         | 단건 조회 |
| PUT         | 전체 수정 |
| PATCH       | 부분 수정 |
| DELETE      | 삭제    |

이 단계에서는 API가 동작하기는 하지만, 반복 코드가 많다.

문제점은 다음과 같다.

* `@csrf_exempt` 필요
* `JSONParser().parse(request)` 반복
* `JsonResponse` 직접 사용
* `safe=False` 직접 지정
* `201`, `400`, `404`, `204` 같은 숫자 상태 코드 직접 사용
* `if request.method == ...` 반복

이 반복을 줄이기 위해 DRF의 `Request`, `Response`, `APIView`, `GenericAPIView`, `ViewSet`으로 발전한다.

---

# 9. Request와 Response

DRF는 Django의 기본 `HttpRequest`, `HttpResponse` 대신 API 개발에 적합한 `Request`, `Response`를 제공한다. 

## 9-1. Request

DRF의 `Request`에서 핵심은 `request.data`이다.

```python
request.data
```

기존 Django의 `request.POST`와 비교하면 다음과 같다.

| 항목             | 설명                                              |
| -------------- | ----------------------------------------------- |
| `request.POST` | form 데이터 중심, 주로 POST에서 사용                       |
| `request.data` | JSON, form 등 다양한 데이터 처리, POST/PUT/PATCH에서 사용 가능 |

API에서는 `request.POST`보다 `request.data`가 훨씬 적합하다.

## 9-2. Response

DRF의 `Response`는 클라이언트가 원하는 형식에 맞춰 응답을 렌더링한다.

```python
return Response(serializer.data)
```

브라우저에서 요청하면 Browsable API 화면을 보여줄 수 있고, JSON을 요청하면 JSON으로 응답한다. 이것을 **Content Negotiation**, 즉 콘텐츠 협상이라고 한다.

## 9-3. status 모듈

숫자 상태 코드 대신 명시적인 이름을 사용한다.

```python
status.HTTP_201_CREATED
status.HTTP_400_BAD_REQUEST
status.HTTP_404_NOT_FOUND
status.HTTP_204_NO_CONTENT
```

이렇게 하면 코드의 의미가 더 명확하다.

# 10. DRF `@api_view`

함수형 view에 DRF 기능을 적용하려면 `@api_view`를 사용할 수 있다.

```python
@api_view(['GET', 'POST'])
def snippet_list(request):
    if request.method == 'GET':
        snippets = Snippet.objects.all()
        serializer = SnippetSerializer(snippets, many=True)
        return Response(serializer.data)

    elif request.method == 'POST':
        serializer = SnippetSerializer(data=request.data)

        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=status.HTTP_201_CREATED)

        return Response(serializer.errors, status=status.HTTP_400_BAD_REQUEST)
```

기존 일반 Django view와 비교하면 다음이 사라진다.

| 일반 Django view                | DRF `@api_view`           |
| ----------------------------- | ------------------------- |
| `JSONParser().parse(request)` | `request.data`            |
| `JsonResponse`                | `Response`                |
| `safe=False`                  | 불필요                       |
| `status=201`                  | `status.HTTP_201_CREATED` |
| 일부 오류 직접 처리                   | ParseError, 405 등 자동 처리   |

# 11. Django CBV

11주차에서는 Django의 클래스 기반 view, 즉 **CBV**도 다뤘다.  

기존 FBV는 다음처럼 GET/POST 분기를 직접 처리한다.

```python
@login_required(login_url='common:login')
def question_create(request):
    if request.method == 'POST':
        form = QuestionForm(request.POST)

        if form.is_valid():
            q = form.save(commit=False)
            q.author = request.user
            q.create_date = timezone.now()
            q.save()
            return redirect('pybo:index')
    else:
        form = QuestionForm()

    return render(request, 'pybo/question_form.html', {'form': form})
```

이를 CBV로 바꾸면 다음처럼 된다.

```python
class QuestionCreateView(LoginRequiredMixin, CreateView):
    model = Question
    form_class = QuestionForm
    template_name = 'pybo/question_form.html'
    login_url = 'common:login'

    def form_valid(self, form):
        form.instance.author = self.request.user
        form.instance.create_date = timezone.now()
        return super().form_valid(form)
```

## FBV와 CBV 대응

| FBV                           | CBV                            |
| ----------------------------- | ------------------------------ |
| 함수                            | 클래스                            |
| `@login_required`             | `LoginRequiredMixin`           |
| `if request.method == 'POST'` | `dispatch()`                   |
| `QuestionForm(request.POST)`  | `form_class = QuestionForm`    |
| 저장 전 값 주입                     | `form_valid()`                 |
| URL에 함수 연결                    | `QuestionCreateView.as_view()` |

CBV는 반복되는 GET/POST 분기, 폼 처리, 객체 조회, 404 처리 등을 클래스 내부로 숨겨 코드량을 줄인다.

PRG를 지키면 새로고침, 뒤로가기, 중복 등록 문제가 줄어든다.

# 14. Mixin 개념

Mixin은 클래스에 특정 기능을 “섞어 넣는” 방식이다.

예를 들어 `LoginRequiredMixin`은 로그인 검사 기능을 제공한다.

```python
class QuestionCreateView(LoginRequiredMixin, CreateView):
    ...
```

Django CBV의 대표 Mixin은 다음과 같다.

| Mixin                          | 제공 기능                  |
| ------------------------------ | ---------------------- |
| `LoginRequiredMixin`           | 인증 검사                  |
| `ContextMixin`                 | `get_context_data()`   |
| `TemplateResponseMixin`        | `render_to_response()` |
| `SingleObjectMixin`            | pk 기반 단일 객체 조회         |
| `MultipleObjectMixin`          | queryset, 페이징          |
| `FormMixin` / `ModelFormMixin` | form 생성, 검증, 저장        |

DRF에서도 CRUD 동작이 Mixin으로 분리된다.

| HTTP         | DRF Mixin            | 메서드           |
| ------------ | -------------------- | ------------- |
| GET 목록       | `ListModelMixin`     | `.list()`     |
| POST 생성      | `CreateModelMixin`   | `.create()`   |
| GET 단건       | `RetrieveModelMixin` | `.retrieve()` |
| PUT/PATCH 수정 | `UpdateModelMixin`   | `.update()`   |
| DELETE 삭제    | `DestroyModelMixin`  | `.destroy()`  |

# 15. DRF API 작성 방식 3가지

DRF에서는 API를 작성하는 방식이 크게 세 단계로 발전한다.

```text
APIView → GenericAPIView → ModelViewSet
```

오른쪽으로 갈수록 자동화 수준이 높아지고, 왼쪽으로 갈수록 자유도가 높다.

---

## 15-1. APIView

APIView는 가장 기본적인 DRF 클래스 기반 view이다.

```python
class HelloAPIView(APIView):
    def get(self, request):
        return Response({"message": "GET hello"})

    def post(self, request):
        return Response({"message": "POST hello"})
```

특징은 다음과 같다.

| 항목     | 설명                      |
| ------ | ----------------------- |
| 자유도    | 높음                      |
| 자동화    | 낮음                      |
| 메서드    | `get()`, `post()` 직접 작성 |
| URL    | 직접 등록                   |
| 적합한 경우 | 커스텀 API, 복잡한 비즈니스 로직    |

---

## 15-2. GenericAPIView

GenericAPIView는 `queryset`과 `serializer_class`를 지정하면 CRUD 기능을 상당 부분 자동화한다.

```python
class BookListCreateView(generics.ListCreateAPIView):
    queryset = Book.objects.all()
    serializer_class = BookSerializer

class BookDetailView(generics.RetrieveUpdateDestroyAPIView):
    queryset = Book.objects.all()
    serializer_class = BookSerializer
```

대표 Generic View는 다음과 같다.

| 클래스                            | 지원 메서드                  |
| ------------------------------ | ----------------------- |
| `ListCreateAPIView`            | GET, POST               |
| `RetrieveAPIView`              | GET                     |
| `UpdateAPIView`                | PUT, PATCH              |
| `DestroyAPIView`               | DELETE                  |
| `RetrieveUpdateDestroyAPIView` | GET, PUT, PATCH, DELETE |

`ListCreateAPIView`는 내부적으로 다음과 같은 구조이다.

```python
class ListCreateAPIView(
    mixins.ListModelMixin,
    mixins.CreateModelMixin,
    GenericAPIView
):
    def get(self, request, *args, **kwargs):
        return self.list(request, *args, **kwargs)

    def post(self, request, *args, **kwargs):
        return self.create(request, *args, **kwargs)
```

즉, Generic View는 자주 쓰는 Mixin 조합을 미리 만들어 둔 클래스이다.

---

## 15-3. ModelViewSet + Router

ModelViewSet은 전체 CRUD를 가장 빠르게 만들 수 있는 방식이다.

```python
class BookViewSet(viewsets.ModelViewSet):
    queryset = Book.objects.all()
    serializer_class = BookSerializer
```

Router를 사용하면 URL도 자동 생성된다.

```python
router = DefaultRouter()
router.register('books', BookViewSet)

urlpatterns = [
    path('api/', include(router.urls))
]
```

자동 생성되는 URL은 다음과 같다.

| URL                | HTTP   | ViewSet 액션         |
| ------------------ | ------ | ------------------ |
| `/api/books/`      | GET    | `list()`           |
| `/api/books/`      | POST   | `create()`         |
| `/api/books/{pk}/` | GET    | `retrieve()`       |
| `/api/books/{pk}/` | PUT    | `update()`         |
| `/api/books/{pk}/` | PATCH  | `partial_update()` |
| `/api/books/{pk}/` | DELETE | `destroy()`        |

# 16. `@action` 커스텀 액션

ViewSet에서는 기본 CRUD 외의 동작을 `@action`으로 추가할 수 있다.

```python
class BookViewSet(viewsets.ModelViewSet):
    queryset = Book.objects.all()
    serializer_class = BookSerializer

    @action(detail=True, methods=['post'])
    def toggle(self, request, pk=None):
        book = self.get_object()
        book.is_available = not book.is_available
        book.save()
        return Response({'is_available': book.is_available})
```

이 경우 다음 URL이 자동 생성된다.

```text
POST /api/books/{pk}/toggle/
```

`@action`의 핵심 규칙은 다음과 같다.

| 옵션             | 의미                       |
| -------------- | ------------------------ |
| `detail=True`  | 특정 객체 대상, `/{pk}/` 뒤에 매핑 |
| `detail=False` | 컬렉션 전체 대상                |
| 함수명            | URL 끝부분                  |
| `methods`      | 허용 HTTP 메서드              |

예를 들어 추천, 좋아요, 토글, 최근 목록 같은 기능은 `@action`으로 구현하기 좋다.

# 17. 코드량 비교

같은 CRUD API를 구현하더라도 방식에 따라 코드량이 크게 달라진다. 

| 방식             |   코드량 | 특징                |
| -------------- | ----: | ----------------- |
| FBV            | 약 50줄 | 모든 흐름 직접 작성       |

| GenericAPIView |  약 9줄 | CRUD 조합 클래스 사용    |

코드가 줄어드는 이유는 기능이 사라져서가 아니라, 내부 클래스와 Mixin 안으로 숨어 들어가기 때문이다.

* 어떤 Mixin이 포함되어 있는가?
* 어떤 Serializer가 사용되는가?
* 어떤 URL이 Router로 생성되는가?
* 권한은 어디서 검사되는가?

# 18. Django CBV와 DRF Generic View의 대응

Django CBV와 DRF Generic View는 매우 비슷한 구조를 가진다. 차이는 응답 형식이다.

| Django CBV            | DRF Generic View                        |
| --------------------- | --------------------------------------- |
| `ListView`            | `ListAPIView` / `ListCreateAPIView`     |

```text
Django CBV = HTML 응답
DRF Generic View = JSON 응답
```

# 21. JSON parse error 주의점

API 테스트에서 자주 발생하는 오류는 JSON 문법 오류이다.

## 객체 2개를 연속 입력

잘못된 예:

```json
{"title":"A"} {"author":"B"}
```

올바른 예:

```json
{"title":"A", "author":"B"}
```

## trailing comma

잘못된 예:

```json
{"title":"A", "author":"B",}
```

올바른 예:

```json
{"title":"A", "author":"B"}
```

## Python 불리언 사용

잘못된 예:

```json
{"is_available": True}
```

올바른 예:

```json
{"is_available": true}
```

## form 형식 입력

잘못된 예:

```text
title=DRF&author=홍길동
```

올바른 예:

```json
{"title":"DRF", "author":"홍길동"}
```

디버깅 순서는 다음과 같다.

```text
중괄호 짝
→ 큰따옴표
→ trailing comma
→ true/false 소문자
→ 객체 1개만 입력했는지 확인
```

# 24. 핵심 결론

11주차의 핵심은 **반복 코드를 줄이고, HTML 중심 Django view를 JSON 중심 API 구조로 확장하는 과정**이다.

처음에는 Pybo 게시판에서 댓글 기능을 직접 구현하면서 모델, 폼, URL, view, 템플릿의 연결 구조를 확인했다. 이후 Serializer를 통해 Django 객체를 JSON으로 바꾸고, JSON 요청을 다시 Django 객체로 저장하는 API 흐름을 배웠다. 마지막으로 FBV, CBV, APIView, GenericAPIView, ModelViewSet을 비교하면서 Django와 DRF가 제공하는 추상화의 사다리를 학습했다.

최종적으로 다음 문장으로 정리할 수 있다.

> Django는 HTML 화면을 만드는 프레임워크로 시작하지만, DRF를 사용하면 같은 모델과 로직을 기반으로 JSON API 서버로 확장할 수 있다.
> 이때 Serializer는 객체와 JSON 사이의 계약이고, Generic View와 ViewSet은 반복되는 CRUD 코드를 표준화하는 도구이다.
