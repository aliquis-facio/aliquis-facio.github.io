# 1. 내비게이션바 추가

## 1-1. 내비게이션바의 필요성

초기 버전에서는 메인 페이지로 돌아가는 장치가 없어서, 사용자 입장에서 화면 이동이 불편했다. 이를 해결하기 위해 **모든 페이지에서 공통으로 보이는 내비게이션바를 `base.html`에 추가**한다. 

## 1-2. 부트스트랩 기반 내비게이션 적용

부트스트랩에서 제공하는 내비게이션 디자인을 활용해 UI를 구성한다. 왼쪽에는 항상 `pybo:index`로 이동하는 **Pybo 로고**, 오른쪽에는 **로그인 링크**를 배치한다. 이 시점에서는 로그인 기능 자체는 아직 구현되지 않았고, 링크만 준비한다. 

## 1-3. 반응형 웹 지원

[나무위키: 햄버거 메뉴](https://namu.wiki/w/%ED%96%84%EB%B2%84%EA%B1%B0%20%EB%A9%94%EB%89%B4)
부트스트랩은 작은 화면에서도 동작하도록 설계되어 있기 때문에, 모바일 환경에서는 햄버거 메뉴를 통해 링크가 접히고 펼쳐지는 **반응형 웹 UI**를 자연스럽게 구현할 수 있다. 

## 1-4. 정적 파일 추가

내비게이션바의 동작을 위해 `bootstrap.min.js`와 `jquery-3.4.1.min.js` 파일을 `static` 디렉터리에 두고, `base.html`의 `</body>` 직전에 스크립트 태그로 추가한다. 이렇게 해야 햄버거 버튼 클릭 시 숨겨진 메뉴가 정상적으로 표시된다. 

## 1-5. include로 분리

내비게이션 관련 HTML을 `base.html`에 직접 두지 않고, 별도의 `templates/navbar.html`로 분리한 다음 `base.html`에서 `{% include 'navbar.html' %}`로 삽입한다.
이 방식은 중복 사용 여부와 관계없이 **유지보수성을 높이는 템플릿 분리 방식**이라는 점에서 의미가 있다.

# 2. 게시판 페이징 기능 추가

## 2-1. 테스트 데이터 생성

페이징 기능을 확인하기 위해 장고 셸에서 질문 데이터를 약 300개 생성한다. 데이터가 많아지면 한 페이지에 모두 출력하는 방식은 비효율적이므로, 페이징의 필요성이 분명해진다. 

## 2-2. Paginator 사용

장고에서 페이징은 `Paginator` 클래스로 처리한다.
`request.GET.get('page', '1')`로 현재 페이지 번호를 받고, 페이지 번호가 없으면 기본값 1을 사용한다. 이후 `Paginator(question_list, 10)`처럼 전체 질문 목록과 페이지당 게시물 수를 전달해 페이징 객체를 만든다. 

여기서 핵심은 `page_obj = paginator.get_page(page)` 형태로 요청된 페이지에 맞는 객체를 생성하면, 내부적으로 전체 데이터를 전부 화면에 뿌리는 것이 아니라 **해당 페이지에 필요한 데이터만 조회하도록 쿼리가 조정된다는 점**이다. 

## 2-3. 템플릿에 페이징 UI 적용

질문 목록 템플릿에 이전/다음 링크와 페이지 번호 목록을 추가한다.
이전 페이지가 있으면 활성화하고, 없으면 비활성화한다. 다음 페이지도 같은 방식으로 처리한다. 또한 현재 페이지와 같은 번호에는 `active` 클래스를 적용해 강조 표시한다. UI는 부트스트랩의 `pagination` 컴포넌트를 사용한다. 

## 2-4. 페이지 번호 표시 제한

모든 페이지 번호를 한 번에 보여주면 화면이 복잡해진다. 이를 해결하기 위해 **현재 페이지 기준 좌우 5개씩만 보이도록 제한**한다.
즉, `question_list.number`보다 5 작거나 5 큰 범위의 페이지 번호만 출력한다. 이렇게 하면 페이지 수가 많아도 깔끔한 페이징 UI를 유지할 수 있다.

# 3. 템플릿 필터 직접 만들기

## 3-1. 게시물 번호 문제

페이징 이후에도 각 페이지의 게시물 번호가 항상 1부터 시작하는 문제가 있다. 이를 해결하려면 **현재 페이지의 시작 인덱스와 페이지 내부 인덱스를 고려한 역순 번호 계산 공식**이 필요하다. 

## 3-2. 사용자 정의 템플릿 필터

`pybo/templatetags` 디렉터리를 만들고, 그 안에 템플릿 필터 파일을 작성한다.
예를 들어 `sub` 같은 함수를 만들고 `@register.filter`를 붙이면, 템플릿 안에서 해당 함수를 필터처럼 사용할 수 있다. 

## 3-3. 템플릿에서 사용

직접 만든 필터를 사용하려면 템플릿 상단에서 `{% load pybo_filter %}`로 필터 파일을 로드해야 한다. 이를 통해 게시물 번호를 페이지 전체 기준의 연속 번호처럼 보이게 만들 수 있다. 

---

# 4. 질문에 달린 답변 개수 표시

질문 목록에서 각 질문에 답변이 몇 개 달렸는지 표시할 수 있다.
템플릿에서 `{% if question.answer_set.count > 0 %}`로 답변 존재 여부를 확인하고, `{{ question.answer_set.count }}`로 개수를 출력한다.
즉, **질문 목록 화면만 봐도 활성도가 높은 글을 바로 파악할 수 있게 하는 기능**이다.

# 5. 로그인 / 로그아웃 구현

## 5-1. common 앱 생성 및 등록

로그인/로그아웃 같은 공통 기능을 담당하기 위해 `common` 앱을 새로 만든다. 그리고 설정 파일에 `common` 앱을 등록한다. 

## 5-2. 로그인 화면 변화

기존에는 내비게이션의 로그인 버튼이 `#` 링크로 연결되어 있어서 아무 기능이 없었다. 이후 `/common/login/` URL로 이동하도록 바꾸면서 실제 로그인 폼 화면이 나타나게 된다. 이후 회원가입 버튼까지 추가되면서 `/common/signup/` 화면으로도 이동할 수 있게 된다. 

## 5-3. URL 매핑과 인증 뷰 사용

`common/urls.py`를 만들고, `path('login/', auth_views.LoginView.as_view(template_name='common/login.html'), name='login')` 같은 식으로 로그인 URL을 연결한다.
여기서 장고 기본 인증 시스템인 `django.contrib.auth`를 사용하기 때문에, 별도의 로그인 처리 로직을 직접 작성하지 않아도 된다. 또한 하드코딩된 URL 대신 URL 이름 별칭을 사용한다. 

## 5-4. 로그인 성공/로그아웃 성공 후 이동 경로

로그인 성공 후 기본적으로 `/accounts/profile/`로 이동하려 하는데, 이 URL이 준비되어 있지 않으면 오류가 난다. 그래서 `settings.py`에 `LOGIN_REDIRECT_URL = '/'`를 설정해 루트 페이지로 이동하게 만든다.
로그아웃도 마찬가지로 `LOGOUT_REDIRECT_URL = '/'`를 지정해 메인 화면으로 돌아가게 한다.  

## 5-5. 로그인 템플릿과 에러 처리

로그인 화면은 `common/login.html`로 작성한다. 이때 `username`, `password`는 `django.contrib.auth`에서 요구하는 필수 입력 항목이다.
또한 `form_errors.html`을 별도로 두어, 로그인 실패 시 입력 누락이나 잘못된 인증 정보 같은 오류 원인을 사용자에게 보여준다. 

## 5-6. 로그아웃 구현

내비게이션바에서 로그인 상태에 따라 로그아웃 링크를 보여주고, `common/urls.py`에 로그아웃 URL 매핑을 추가한다.
결과적으로 인증 기능이 내비게이션과 연결되면서, 사용자는 어느 화면에서든 로그인/로그아웃을 수행할 수 있게 된다.

# 6. 웹 폼 데이터 전송 방식 이해

강의 후반부에서는 로그인과 회원가입 같은 기능을 구현하기 위해, **HTML form의 데이터 전송 방식(GET/POST)** 을 설명한다. 

## 6-1. GET 방식

GET은 입력 데이터를 URL 쿼리스트링에 붙여서 전송한다.
예를 들어 검색창에 `django`를 입력하면 `/search/?q=django` 같은 형태가 된다.
검색, 필터링, 정렬처럼 **URL 공유와 북마크가 필요한 기능**에 적합하다. 

## 6-2. POST 방식

POST는 데이터를 HTTP 본문(body)에 담아 전송하므로 URL에 값이 드러나지 않는다.
로그인, 회원가입, 글쓰기처럼 **민감하거나 상태를 변경하는 작업**에 적합하다.
장고에서는 POST 요청 시 `{% csrf_token %}`이 반드시 필요하며, 없으면 403 Forbidden 오류가 발생한다. 

## 6-3. GET과 POST 비교

강의 자료에서는 GET/POST를 다음 기준으로 비교한다.

* GET: URL 노출, 공유 가능, 검색/필터/정렬에 적합
* POST: 본문 전송, URL 비노출, 로그인/회원가입/글쓰기/업로드에 적합

장고에서는 각각 `request.GET["key"]`, `request.POST["key"]`로 접근한다. 

## 6-4. 뷰에서 처리 흐름

하나의 URL에서도 `request.method`를 기준으로 GET과 POST를 구분하여 처리할 수 있다.
예를 들어 GET이면 빈 로그인 폼을 보여주고, POST면 전달받은 `username`, `password`를 읽어 로그인 로직을 수행한다.
즉, **같은 URL이더라도 요청 방식에 따라 “화면 표시”와 “데이터 처리” 역할을 나누는 구조**다.

# 전체 핵심 정리

## 이번 주차의 핵심 흐름

5주차는 단순 CRUD 게시판에서 벗어나, **실제 서비스형 게시판으로 확장하는 과정**이다.

1. 내비게이션바를 추가해 공통 이동 구조를 만든다.
2. 대량 데이터에 대응하기 위해 페이징을 적용한다.
3. 템플릿 필터로 화면 표현을 더 정교하게 다듬는다.
4. 답변 개수 표시로 게시글의 상태를 시각적으로 강화한다.
5. 로그인/로그아웃 기능을 붙여 사용자 인증을 도입한다.
6. GET/POST 요청 방식 차이를 이해해 폼 기반 기능 구현의 기반을 다진다.

## 실무적으로 중요한 포인트

* `base.html`과 `include`를 활용한 **공통 UI 재사용**
* `Paginator`를 활용한 **서버 사이드 페이징**
* `templatetags`를 통한 **표현 로직 분리**
* `django.contrib.auth`를 이용한 **기본 인증 시스템 활용**
* `GET`과 `POST`의 역할 구분을 통한 **HTTP 요청 처리 이해**
