## 2. 파이보 검색 기능

### 2.1 검색 기능의 목적

검색 기능은 사용자가 질문 목록에서 원하는 게시글을 빠르게 찾을 수 있도록 하는 기능입니다.

검색 대상은 단순히 질문 제목만이 아니라 다음과 같이 여러 필드를 포함할 수 있습니다.

| 검색 대상  | 설명                         |
| ------ | -------------------------- |
| 질문 제목  | `subject`                  |
| 질문 내용  | `content`                  |
| 질문 작성자 | `author__username`         |
| 답변 작성자 | `answer__author__username` |

이를 위해 Django의 `Q` 객체를 사용합니다.

```python
from django.db.models import Q

question_list = question_list.filter(
    Q(subject__icontains=kw) |
    Q(content__icontains=kw) |
    Q(author__username__icontains=kw) |
    Q(answer__author__username__icontains=kw)
).distinct()
```

여기서 `icontains`는 대소문자를 구분하지 않고 특정 문자열이 포함되어 있는지 검사하는 조건입니다.

### 2.2 Q 객체

`Q` 객체는 여러 검색 조건을 OR 조건으로 연결할 때 사용합니다.

일반적인 `filter()`는 조건을 여러 개 넣으면 AND 조건으로 처리됩니다.

```python
Question.objects.filter(subject__icontains=kw, content__icontains=kw)
```

위 코드는 제목에도 검색어가 있고, 내용에도 검색어가 있어야 합니다.

하지만 검색 기능에서는 제목, 내용, 작성자, 답변 작성자 중 하나라도 검색어가 포함되면 결과에 나와야 합니다. 그래서 `Q` 객체를 사용합니다.

```python
Q(subject__icontains=kw) | Q(content__icontains=kw)
```

`|` 연산자는 OR 조건을 의미합니다.

### 2.3 distinct 함수

검색 조건에 답변 작성자가 포함되면 `Question`과 `Answer`가 조인됩니다.

하나의 질문에 여러 답변이 있고, 그 답변들이 검색 조건에 걸리면 같은 질문이 중복으로 조회될 수 있습니다. 이때 `distinct()`를 사용하여 중복된 질문을 제거합니다.

```python
.distinct()
```

따라서 검색 쿼리의 마지막에 `distinct()`를 붙이는 이유는 **조인으로 인한 중복 질문 제거**입니다.

### 2.4 검색어 전달 방식: GET

검색어 `kw`는 POST가 아니라 GET 방식으로 전달합니다.

```html
<form id="searchForm" method="get" action="{% url 'index' %}">
    <input type="hidden" id="kw" name="kw" value="{{ kw|default_if_none:'' }}">
    <input type="hidden" id="page" name="page" value="{{ page }}">
</form>
```

GET 방식을 사용하는 이유는 다음과 같습니다.

| 이유               | 설명                         |
| ---------------- | -------------------------- |
| URL에 검색 상태가 남음   | 검색 결과 페이지를 공유하거나 새로고침하기 좋음 |
| 페이징과 함께 쓰기 쉬움    | `?kw=검색어&page=2` 형태로 관리 가능 |
| POST 중복 제출 문제 방지 | 브라우저의 재전송 경고를 피할 수 있음      |
| 검색은 데이터 변경이 아님   | 조회 요청이므로 GET이 의미상 적합       |

검색은 서버의 데이터를 변경하는 작업이 아니라 조회 작업이기 때문에 GET 방식이 적절합니다.

## 3. 검색과 페이징 연동

검색 기능을 추가하면 페이징 처리도 함께 수정해야 합니다.

기존에는 페이지 링크에 직접 URL을 넣는 방식이었습니다.

```html
<a class="page-link" href="?page={{ page_number }}">{{ page_number }}</a>
```

하지만 검색어와 정렬 조건까지 함께 유지하려면 링크에 직접 URL을 작성하는 방식은 불편합니다.

그래서 페이지 번호를 `data-page` 속성에 넣습니다.

```html
<a class="page-link" data-page="{{ page_number }}" href="javascript:void(0)">
    {{ page_number }}
</a>
```

그리고 JavaScript에서 클릭된 페이지 번호를 hidden form에 넣고 제출합니다.

```javascript
$(".page-link").on('click', function() {
    $("#page").val($(this).data("page"));
    $("#searchForm").submit();
});
```

이 방식의 장점은 검색어 `kw`, 페이지 번호 `page`, 정렬 기준 `so`를 하나의 form으로 통합 관리할 수 있다는 점입니다.

## 4. 검색 버튼 처리

질문 목록 화면에는 검색창과 검색 버튼을 추가합니다.

```html
<input type="text" id="search_kw" class="form-control" value="{{ kw|default_if_none:'' }}">
<button class="btn btn-outline-secondary" type="button" id="btn_search">찾기</button>
```

검색 버튼을 누르면 입력한 검색어를 hidden input에 넣고, 페이지는 1페이지로 초기화합니다.

```javascript
$("#btn_search").on('click', function() {
    $("#kw").val($("#search_kw").val());
    $("#page").val(1);
    $("#searchForm").submit();
});
```

검색 버튼을 누를 때 `page`를 1로 설정하는 이유는, 새로운 검색어로 검색하면 기존 페이지 번호가 의미 없어질 수 있기 때문입니다.

예를 들어 기존에 7페이지를 보고 있었는데 새로운 검색어의 결과가 1페이지만 있다면 오류 또는 빈 결과가 나올 수 있습니다. 따라서 검색 시에는 항상 1페이지부터 다시 조회합니다.

## 5. 정렬 기능

### 5.1 정렬 기준

13주차에서는 질문 목록에 정렬 기능을 추가합니다.

대표 정렬 기준은 다음과 같습니다.

| 정렬 기준 | 값           | 설명             |
| ----- | ----------- | -------------- |
| 최신순   | `recent`    | 최근 작성된 질문 우선   |
| 추천순   | `recommend` | 추천 수가 많은 질문 우선 |
| 인기순   | `popular`   | 답변 수가 많은 질문 우선 |

템플릿에는 `select` 태그를 사용합니다.

```html
<select class="form-control so">
    <option value="recent" {% if so == 'recent' %}selected{% endif %}>최신순</option>
    <option value="recommend" {% if so == 'recommend' %}selected{% endif %}>추천순</option>
    <option value="popular" {% if so == 'popular' %}selected{% endif %}>인기순</option>
</select>
```

### 5.2 정렬 조건 전달

정렬 조건도 검색어, 페이지 번호와 함께 hidden form으로 전달합니다.

```html
<form id="searchForm" method="get" action="{% url 'index' %}">
    <input type="hidden" id="kw" name="kw" value="{{ kw|default_if_none:'' }}">
    <input type="hidden" id="page" name="page" value="{{ page }}">
    <input type="hidden" id="so" name="so" value="{{ so }}">
</form>
```

정렬 조건이 바뀌면 JavaScript에서 `so` 값을 설정하고 form을 제출합니다.

```javascript
$(".so").on('change', function() {
    $("#so").val($(this).val());
    $("#page").val(1);
    $("#searchForm").submit();
});
```

정렬 조건이 바뀔 때도 페이지를 1로 초기화합니다.

### 5.3 index 함수의 정렬 처리

`views.py`의 `index` 함수에서는 GET 파라미터를 받습니다.

```python
page = request.GET.get('page', '1')
kw = request.GET.get('kw', '')
so = request.GET.get('so', 'recent')
```

정렬 기준에 따라 QuerySet을 다르게 구성합니다.

```python
from django.db.models import Q, Count

if so == 'recommend':
    question_list = Question.objects.annotate(
        num_voter=Count('voter')
    ).order_by('-num_voter', '-create_date')
elif so == 'popular':
    question_list = Question.objects.annotate(
        num_answer=Count('answer')
    ).order_by('-num_answer', '-create_date')
else:
    question_list = Question.objects.order_by('-create_date')
```

추천순에서는 `voter` 수를 계산하고, 인기순에서는 `answer` 수를 계산합니다.

### 5.4 annotate 함수

`annotate()`는 QuerySet의 각 객체에 계산된 필드를 임시로 추가하는 기능입니다.

예를 들어 다음 코드는 질문마다 추천 수를 `num_voter`라는 이름으로 붙입니다.

```python
Question.objects.annotate(num_voter=Count('voter'))
```

이렇게 하면 실제 데이터베이스 모델에 `num_voter`라는 필드가 없어도 정렬에 사용할 수 있습니다.

```python
.order_by('-num_voter')
```

즉, `annotate()`는 **정렬이나 필터링을 위해 계산 필드를 추가하는 기능**입니다.

## 6. 추천 기능과 모델 관계

0526 녹음에서는 추천 기능을 설명하면서 Django 모델 관계가 강조되었습니다.

파이보의 질문에는 사용자와 관련된 관계가 두 가지 있습니다.

| 관계  | 의미          | Django 필드         |
| --- | ----------- | ----------------- |
| 작성자 | 질문을 작성한 사람  | `ForeignKey`      |
| 추천자 | 질문을 추천한 사람들 | `ManyToManyField` |

### 6.1 작성자 관계: One-to-Many

한 명의 사용자는 여러 질문을 작성할 수 있습니다.
하지만 하나의 질문은 한 명의 작성자만 가질 수 있습니다.

```python
author = models.ForeignKey(User, on_delete=models.CASCADE)
```

관계는 다음과 같습니다.

```text
User 1명 → Question 여러 개 작성 가능
Question 1개 → author 1명
```

따라서 작성자 관계는 1:N 관계입니다.

### 6.2 추천자 관계: Many-to-Many

추천 관계는 작성자 관계와 다릅니다.

한 명의 사용자는 여러 질문을 추천할 수 있습니다.
그리고 하나의 질문은 여러 사용자에게 추천받을 수 있습니다.

```python
voter = models.ManyToManyField(User, related_name='voter_question')
```

관계는 다음과 같습니다.

```text
User 1명 → 여러 Question 추천 가능
Question 1개 → 여러 User에게 추천받을 수 있음
```

따라서 추천자 관계는 M:N 관계입니다.

### 6.3 related_name이 필요한 이유

`Question` 모델이 `User` 모델을 두 번 참조하는 경우가 있습니다.

```python
author = models.ForeignKey(User, on_delete=models.CASCADE, related_name='author_question')
voter = models.ManyToManyField(User, related_name='voter_question')
```

이때 `User` 입장에서 Question으로 역참조할 때, 어떤 관계를 통해 접근하는지 구분해야 합니다.

```python
user.author_question.all()  ## 내가 작성한 질문
user.voter_question.all()   ## 내가 추천한 질문
```

즉, `related_name`은 같은 모델을 여러 역할로 참조할 때 역참조 이름을 명확히 구분하기 위해 필요합니다.

## 7. 인증과 권한

13주차 음성 녹음에서는 **Authentication**과 **Permission**의 차이가 여러 번 강조되었습니다.

### 7.1 Authentication: 인증

인증은 사용자가 누구인지 확인하는 과정입니다.

예를 들어 로그인할 때 아이디와 비밀번호를 확인합니다.

```text
username/password 확인
→ 로그인 성공
→ session 또는 token 발급
```

인증의 질문은 다음과 같습니다.

```text
이 사용자는 누구인가?
```

### 7.2 Permission: 권한

권한은 인증된 사용자가 특정 작업을 수행할 수 있는지 확인하는 과정입니다.

예를 들어 사용자가 로그인했다고 해서 모든 질문을 수정할 수 있는 것은 아닙니다.
자신이 작성한 질문만 수정할 수 있어야 합니다.

```python
if request.user != question.author:
    ## 수정 불가
```

권한의 질문은 다음과 같습니다.

```text
이 사용자가 이 작업을 해도 되는가?
```

### 7.3 로그인만으로는 부족한 이유

로그인은 인증입니다.
하지만 게시글 수정, 삭제, 추천 같은 기능에서는 권한 확인이 추가로 필요합니다.

| 상황    | 필요한 검사              |
| ----- | ------------------- |
| 질문 작성 | 로그인 여부              |
| 질문 수정 | 로그인 여부 + 작성자 여부     |
| 질문 삭제 | 로그인 여부 + 작성자 여부     |
| 추천    | 로그인 여부 + 자기 글 추천 방지 |
| 답변 수정 | 로그인 여부 + 답변 작성자 여부  |

따라서 실제 서비스에서는 인증과 권한을 분리해서 생각해야 합니다.

## 8. Function-Based View

13주차 후반의 핵심 주제는 View 구조의 발전입니다.

처음에는 Function-Based View, 즉 함수 기반 뷰로 기능을 구현합니다.

```python
@login_required
def question_create(request):
    if request.method == 'POST':
        form = QuestionForm(request.POST)
        if form.is_valid():
            question = form.save(commit=False)
            question.author = request.user
            question.save()
            return redirect('pybo:index')
    else:
        form = QuestionForm()
    return render(request, 'pybo/question_form.html', {'form': form})
```

Function-Based View의 장점은 동작 흐름이 명확하다는 것입니다.

| 장점       | 설명                       |
| -------- | ------------------------ |
| 흐름이 잘 보임 | GET/POST 분기가 직접 보임       |
| 학습에 적합   | HTTP 요청 처리 과정을 이해하기 좋음   |
| 디버깅이 쉬움  | 어느 부분에서 문제가 생겼는지 추적하기 좋음 |

하지만 기능이 많아질수록 반복이 많아집니다.

## 9. Function-Based View의 한계

질문 생성, 질문 수정, 답변 생성, 답변 수정, 댓글 생성 등 여러 기능을 만들면 다음 코드가 계속 반복됩니다.

| 반복 요소    | 예시                                |
| -------- | --------------------------------- |
| 로그인 검사   | `@login_required`                 |
| 요청 방식 분기 | `if request.method == 'POST'`     |
| Form 생성  | `QuestionForm(request.POST)`      |
| Form 검증  | `form.is_valid()`                 |
| 저장 처리    | `form.save()`                     |
| 권한 검사    | `request.user != question.author` |
| 템플릿 렌더링  | `render()`                        |
| 리다이렉트    | `redirect()`                      |

이처럼 Function-Based View는 **working code**, 즉 일단 동작하는 코드로는 좋지만, 코드가 길어지고 중복이 많아지는 문제가 있습니다.

## 10. Class-Based View

반복되는 로직을 줄이기 위해 Class-Based View로 넘어갑니다.

예를 들어 질문 생성 기능은 다음처럼 표현할 수 있습니다.

```python
class QuestionCreateView(LoginRequiredMixin, CreateView):
    model = Question
    form_class = QuestionForm
    template_name = 'pybo/question_form.html'
```

Class-Based View에서는 GET/POST 처리, form 검증, 저장 흐름 일부가 클래스 내부에 들어갑니다.

개발자는 필요한 설정을 클래스 속성으로 지정합니다.

| 속성              | 의미           |
| --------------- | ------------ |
| `model`         | 사용할 모델       |
| `form_class`    | 사용할 Form     |
| `template_name` | 사용할 템플릿      |
| `success_url`   | 성공 후 이동할 URL |

## 11. Mixin

Mixin은 단독으로 동작하는 클래스라기보다는 다른 클래스와 결합되어 특정 기능을 추가하는 클래스입니다.

대표 예시는 `LoginRequiredMixin`입니다.

```python
class QuestionCreateView(LoginRequiredMixin, CreateView):
    ...
```

`LoginRequiredMixin`은 로그인 여부를 검사하는 기능을 제공합니다.
`CreateView`는 객체 생성 기능을 제공합니다.

두 클래스를 결합하면 “로그인이 필요한 생성 뷰”가 됩니다.

```text
LoginRequiredMixin + CreateView
= 로그인 검사가 포함된 생성 기능
```

## 12. Class-Based View에서도 남는 중복

Class-Based View를 사용해도 여전히 중복이 남습니다.

예를 들어 질문 생성과 질문 수정은 모두 다음 속성을 사용할 수 있습니다.

```python
model = Question
form_class = QuestionForm
template_name = 'pybo/question_form.html'
```

이런 중복을 줄이기 위해 공통 설정을 별도 Base Mixin으로 분리할 수 있습니다.

```python
class QuestionBaseMixin:
    model = Question
    form_class = QuestionForm
    template_name = 'pybo/question_form.html'
```

그리고 각 뷰에서 재사용합니다.

```python
class QuestionCreateView(LoginRequiredMixin, QuestionBaseMixin, CreateView):
    pass

class QuestionUpdateView(LoginRequiredMixin, QuestionBaseMixin, UpdateView):
    pass
```

이 구조는 반복을 줄이고 코드의 유지보수성을 높입니다.

## 13. Working Code → Suboptimal Code → Optimal Code

교수님은 코드 발전 흐름을 다음처럼 설명했습니다.

```text
Working Code → Suboptimal Code → Optimal Code
```

| 단계              | 의미              | 예시                                |
| --------------- | --------------- | --------------------------------- |
| Working Code    | 일단 동작하는 코드      | Function-Based View               |
| Suboptimal Code | 동작하지만 중복이 남은 코드 | 단순 Class-Based View               |
| Optimal Code    | 중복을 줄이고 구조화한 코드 | Mixin, Generic View, ModelViewSet |

중요한 점은 처음부터 최적화된 코드를 쓰는 것이 항상 좋은 것은 아니라는 점입니다.

처음에는 Function-Based View로 요청 처리 흐름을 이해하고, 이후 Class-Based View와 Generic View로 리팩터링하는 것이 학습에 적합합니다.

## 14. DRF와 Serializer

DRF에서는 Django 모델 데이터를 JSON으로 변환해야 합니다.

이때 사용하는 것이 Serializer입니다.

```text
Python 객체 / QuerySet
→ Serializer
→ JSON
```

예를 들어 `Question` 객체를 React나 모바일 앱에서 사용할 수 있도록 JSON으로 바꾸는 역할을 합니다.

기존 Django 템플릿 방식은 서버가 HTML을 반환합니다.

```text
Django View → Template → HTML 응답
```

DRF 방식은 서버가 JSON을 반환합니다.

```text
Django View / APIView → Serializer → JSON 응답
```

이 차이가 DRF의 핵심입니다.

## 15. DRF 구조 발전

DRF에서도 View 구조는 단계적으로 발전합니다.

```text
APIView
→ GenericAPIView + Mixins
→ Generic Views
→ ViewSet / ModelViewSet
```

### 15.1 APIView

`APIView`는 DRF의 기본 클래스입니다.
GET, POST, PUT, DELETE 등을 직접 메서드로 구현합니다.

```python
class QuestionList(APIView):
    def get(self, request):
        ...

    def post(self, request):
        ...
```

장점은 동작이 명확하다는 것입니다.
단점은 코드가 길고 반복이 많다는 것입니다.

### 15.2 GenericAPIView와 Mixins

GenericAPIView는 `queryset`, `serializer_class` 같은 공통 설정을 제공합니다.

Mixin은 list, create, retrieve, update, destroy 같은 동작을 조합할 수 있게 해줍니다.

예를 들어 목록 조회와 생성을 조합할 수 있습니다.

```python
class QuestionList(
    mixins.ListModelMixin,
    mixins.CreateModelMixin,
    generics.GenericAPIView
):
    queryset = Question.objects.all()
    serializer_class = QuestionSerializer
```

### 15.3 Generic Views

Generic Views는 자주 사용하는 조합을 미리 클래스로 제공한 것입니다.

```python
class QuestionListCreateView(generics.ListCreateAPIView):
    queryset = Question.objects.all()
    serializer_class = QuestionSerializer
```

여기서 `ListCreateAPIView`는 목록 조회와 생성을 모두 처리합니다.

| Generic View                   | 기능              |
| ------------------------------ | --------------- |
| `ListAPIView`                  | 목록 조회           |
| `CreateAPIView`                | 생성              |
| `ListCreateAPIView`            | 목록 조회 + 생성      |
| `RetrieveAPIView`              | 상세 조회           |
| `UpdateAPIView`                | 수정              |
| `DestroyAPIView`               | 삭제              |
| `RetrieveUpdateDestroyAPIView` | 상세 조회 + 수정 + 삭제 |

### 15.4 ModelViewSet

ModelViewSet은 DRF 자동화의 더 높은 단계입니다.

CRUD 기능뿐 아니라 Router와 결합하여 URL 패턴까지 줄일 수 있습니다.

```python
class QuestionViewSet(ModelViewSet):
    queryset = Question.objects.all()
    serializer_class = QuestionSerializer
```

Router를 사용하면 URL을 직접 하나하나 작성하지 않아도 됩니다.

```python
router.register('questions', QuestionViewSet)
```

즉, ModelViewSet은 **View 코드 반복 + URL 코드 반복**을 함께 줄이는 구조입니다.

## 16. 서비스 레이어

0528 녹음에서는 서비스 레이어가 중요한 주제로 다뤄졌습니다.

기존에는 View가 ORM에 직접 접근했습니다.

```python
question = get_object_or_404(Question, pk=question_id)
```

서비스 레이어를 도입하면 View가 직접 ORM을 호출하지 않고, Service를 통해 데이터를 가져옵니다.

```python
question = QuestionService.get_question(question_id)
```

구조는 다음과 같습니다.

```text
View
→ Service Layer
→ ORM / Model
```

### 16.1 서비스 레이어가 필요한 이유

서비스 레이어는 View가 지나치게 복잡해지는 것을 막기 위해 사용합니다.

예를 들어 도서 대출 서비스를 생각하면 단순 조회 외에도 다음 로직이 필요합니다.

```text
책이 대출 가능한지 확인
대출 이력 저장
누가 빌렸는지 기록
반납 여부 확인
연체 여부 계산
```

이런 비즈니스 로직을 View에 모두 작성하면 View가 너무 길어집니다.

따라서 비즈니스 로직은 Service Layer에 분리합니다.

### 16.2 서비스 레이어 예시

```python
class BookService:
    @staticmethod
    def get_all_books():
        return Book.objects.all()

    @staticmethod
    def get_book(book_id):
        return get_object_or_404(Book, id=book_id)

    @staticmethod
    def get_book_history(book):
        return BookHistory.objects.filter(book=book)
```

View에서는 다음처럼 Service만 호출합니다.

```python
def book_list(request):
    books = BookService.get_all_books()
    return render(request, 'book_list.html', {'books': books})
```

이렇게 하면 View는 요청과 응답에 집중하고, 비즈니스 로직은 Service가 담당합니다.

### 16.3 서비스 레이어의 장점

| 장점         | 설명                               |
| ---------- | -------------------------------- |
| View 간결화   | View가 요청/응답 처리에 집중               |
| 비즈니스 로직 분리 | 대출, 추천, 권한 등 복잡한 로직을 Service에 배치 |
| 재사용성 증가    | 여러 View에서 같은 Service 메서드 사용 가능   |
| 테스트 용이성    | Service 단위로 로직 테스트 가능            |
| 유지보수성 향상   | 로직 변경 시 View 전체를 수정하지 않아도 됨      |

교수님은 View를 “브레인”에 비유했습니다.
브레인이 모든 일을 직접 하면 복잡해지므로, 각각의 일을 서비스 레이어로 분배해야 한다는 설명입니다.

## 17. DRF + React 통합

13.pdf 후반부에서는 DRF와 React를 연결하는 실습이 나옵니다.

핵심 문장은 다음과 같습니다.

> FBV는 HTML을 보내고, DRF + React는 데이터를 보낸다.

기존 Django 방식에서는 서버가 HTML을 완성해서 브라우저에 보냅니다.
반면 DRF + React 방식에서는 서버가 JSON 데이터만 보내고, React가 화면을 구성합니다.

### 17.1 기존 Django FBV 방식

```text
Browser
→ Django View
→ Template 렌더링
→ HTML 응답
```

이 방식에서는 서버가 화면까지 책임집니다.

| 항목       | 설명                    |
| -------- | --------------------- |
| 서버 역할    | HTML 생성               |
| 클라이언트 역할 | HTML 표시               |
| 장점       | 구조가 단순함               |
| 단점       | 모바일 앱, 외부 서비스 재사용 어려움 |

### 17.2 DRF + React 방식

```text
React
→ DRF API 요청
→ JSON 응답
→ React가 화면 렌더링
```

이 방식에서는 서버와 클라이언트의 책임이 분리됩니다.

| 항목 | 서버           | 클라이언트     |
| -- | ------------ | --------- |
| 역할 | 데이터 제공       | 화면 구성     |
| 기술 | Django + DRF | React     |
| 응답 | JSON         | UI 렌더링    |
| 장점 | API 재사용 가능   | 화면 자유도 높음 |

## 20. CORS 설정

React 서버에서 Django 서버로 요청하려면 `django-cors-headers`를 설치합니다.

```bash
pip install django-cors-headers
```

`settings.py`에 앱을 추가합니다.

```python
INSTALLED_APPS = [
    ...
    'rest_framework',
    'corsheaders',
]
```

Middleware에 추가합니다.

```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'corsheaders.middleware.CorsMiddleware',
    'django.middleware.common.CommonMiddleware',
    ...
]
```

중요한 점은 `CorsMiddleware`가 `CommonMiddleware`보다 위에 있어야 한다는 것입니다.

허용할 origin도 등록합니다.

```python
CORS_ALLOWED_ORIGINS = [
    "http://localhost:5173",
    "http://127.0.0.1:5173",
]
```

## 21. React에서 API 호출

React에서는 `useState`, `useEffect`, `fetch()`를 사용해 DRF API를 호출합니다.

```jsx
import { useEffect, useState } from "react";

const API_URL = "http://127.0.0.1:8010/api/v1/questions/";

function App() {
  const [questions, setQuestions] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch(API_URL)
      .then(res => res.ok ? res.json() : Promise.reject(res.status))
      .then(data => {
        setQuestions(data);
        setLoading(false);
      })
      .catch(err => {
        setError(String(err));
        setLoading(false);
      });
  }, []);

  if (loading) return <p>로딩 중...</p>;
  if (error) return <p>오류: {error}</p>;

  return (
    <ul>
      {questions.map(q => (
        <li key={q.id}>
          <h3>{q.subject}</h3>
          <p>{q.content}</p>
        </li>
      ))}
    </ul>
  );
}

export default App;
```

여기서 중요한 흐름은 다음과 같습니다.

```text
컴포넌트 최초 렌더링
→ useEffect 실행
→ fetch로 API 요청
→ JSON 응답 수신
→ state 변경
→ 화면 렌더링
```

## 22. Vite Proxy

CORS 설정 대신 Vite Proxy를 사용할 수도 있습니다.

`vite.config.js`에서 `/api` 요청을 Django 서버로 넘깁니다.

```javascript
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  server: {
    proxy: {
      '/api': {
        target: 'http://127.0.0.1:8010',
        changeOrigin: true,
      },
    },
  },
})
```

React에서는 API URL을 상대 경로로 바꿉니다.

```javascript
const API_URL = "/api/v1/questions/";
```

이 방식은 브라우저 입장에서 같은 origin으로 요청하는 것처럼 보이기 때문에 CORS 문제가 발생하지 않습니다.

단, Vite Proxy는 개발 환경용입니다. 운영 배포에서는 Nginx, API Gateway, 백엔드 CORS 설정 등 별도 구성이 필요합니다.

## 23. Postman API 테스트

0528 녹음에서는 Postman을 이용한 API 테스트도 설명되었습니다.

브라우저는 GET 요청을 확인하는 데 적합하지만, POST, PUT, DELETE, Token 인증 테스트에는 한계가 있습니다.

Postman은 다음 기능을 지원합니다.

| 기능                   | 설명                |
| -------------------- | ----------------- |
| GET 요청               | 목록/상세 조회          |
| POST 요청              | 생성, 로그인, 토큰 발급    |
| PUT/PATCH 요청         | 수정                |
| DELETE 요청            | 삭제                |
| Body 설정              | JSON 데이터 전송       |
| Authorization Header | Token 인증          |
| Response 확인          | 상태 코드, 응답 Body 확인 |

### 23.1 질문 목록 조회

```text
GET http://127.0.0.1:8010/api/v1/questions/
```

정상 응답이면 `200 OK`와 함께 질문 목록 JSON이 반환됩니다.

### 23.2 Token 발급

토큰을 발급받으려면 username과 password를 Body에 넣고 POST 요청을 보냅니다.

```text
POST http://127.0.0.1:8010/api/v1/token/
```

Body 예시는 다음과 같습니다.

```json
{
  "username": "사용자이름",
  "password": "비밀번호"
}
```

인증에 성공하면 서버가 token을 반환합니다.

### 23.3 Token을 이용한 수정 요청

토큰을 받은 뒤에는 Authorization Header에 토큰을 넣고 요청합니다.

```text
PUT http://127.0.0.1:8010/api/v1/questions/9/
```

Body에는 수정할 데이터를 넣습니다.

```json
{
  "subject": "제목 수정",
  "content": "내용 수정"
}
```

단, 토큰이 있다고 해서 모든 질문을 수정할 수 있는 것은 아닙니다.
현재 로그인한 사용자가 해당 질문의 작성자여야 합니다.

즉, 여기서도 인증과 권한이 모두 필요합니다.

```text
Token 인증 → 누구인지 확인
Permission 검사 → 수정 가능한 사용자인지 확인
```

## 27. 최종 요약

13주차는 단순히 기능을 추가하는 주차가 아니라, Django 프로젝트가 실제 서비스 구조로 발전하는 과정을 보여주는 주차입니다.

처음에는 파이보 게시판에 검색과 정렬을 추가합니다. 검색은 `Q` 객체와 `distinct()`를 사용하고, 정렬은 `annotate()`와 `Count()`를 사용합니다. 검색어, 페이지 번호, 정렬 기준은 GET 방식으로 유지합니다.

그다음 추천 기능을 통해 `ForeignKey`, `ManyToManyField`, `related_name`의 차이를 이해합니다. 작성자는 1:N 관계이고, 추천자는 M:N 관계입니다.

이후에는 인증과 권한의 차이를 배웁니다. 인증은 사용자가 누구인지 확인하는 것이고, 권한은 그 사용자가 해당 작업을 해도 되는지 확인하는 것입니다.

후반부에서는 View 구조가 발전합니다. Function-Based View는 동작 원리를 이해하기 좋지만 반복이 많습니다. Class-Based View는 반복을 클래스로 정리하고, Mixin과 Generic View는 공통 CRUD 패턴을 더 줄입니다. ModelViewSet은 URL 라우팅까지 자동화합니다.

또한 서비스 레이어를 통해 View와 ORM 사이에 비즈니스 로직 계층을 두는 이유를 배웁니다. View가 모든 로직을 담당하면 복잡해지므로, Service Layer로 분리하는 것이 유지보수에 유리합니다.

마지막으로 DRF + React 구조에서는 서버가 HTML을 반환하지 않고 JSON 데이터를 제공하며, React가 UI를 담당합니다. 이 구조에서는 CORS 설정, Vite Proxy, Postman 테스트, Token 인증이 중요합니다.

결국 13주차의 핵심은 다음과 같습니다.

> Django 게시판 기능을 확장하고, 반복되는 서버 코드를 구조화하며, DRF API와 React 클라이언트로 서버와 화면의 책임을 분리하는 방법을 이해하는 것.
