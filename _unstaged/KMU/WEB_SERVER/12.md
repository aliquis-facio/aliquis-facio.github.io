# 12주차

## 1. 12주차 전체 흐름

12주차는 기존 **Django Pybo 게시판**을 더 구조화하고, 이후 **DRF 기반 API 서버**로 확장하는 흐름으로 진행되었다.

크게 보면 다음 순서다.

| 구분              | 핵심 내용                                                                   |
| --------------- | ----------------------------------------------------------------------- |
| Pybo 게시판 기능 고도화 | `views.py` 분리, 추천 기능, 스크롤 앵커, 마크다운                                      |
| DRF 기본 구조 이해    | Serializer, API View, JSON 응답                                           |
| DRF View 추상화 단계 | FBV → `@api_view` → `APIView` → GenericAPIView/Mixin → generics/ViewSet |
| 인증과 권한          | Authentication, Permission, `IsOwnerOrReadOnly`                         |
| 서비스 레이어 분리      | View-Service-Model 책임 분리, 커스텀 예외, pytest                                |
| 프론트엔드 연계 준비     | Django가 HTML이 아니라 JSON을 제공하고, React가 이를 받아 렌더링                          |

즉, 12주차는 단순히 게시판 기능을 추가하는 주차가 아니라, **Django 웹 앱에서 DRF API 서버로 넘어가는 전환점**이라고 볼 수 있다.

## 2. `views.py` 파일 분리

### 2.1 분리 이유

기존에는 `pybo/views.py` 하나에 질문, 답변, 댓글, 수정, 삭제 등의 함수가 모두 들어 있었다.

하지만 기능이 많아지면 다음 문제가 생긴다.

| 문제       | 설명               |
| -------- | ---------------- |
| 가독성 저하   | 파일 하나가 너무 길어짐    |
| 유지보수 어려움 | 특정 기능을 찾기 어려움    |
| 협업 충돌 증가 | 여러 사람이 같은 파일을 수정 |
| 책임 불명확   | 질문/답변/댓글 로직이 섞임  |

그래서 `views.py`를 기능별로 나누었다.

```text
pybo/
└── views/
    ├── base_views.py
    ├── question_views.py
    ├── answer_views.py
    ├── comment_views.py
    └── vote_views.py
```

각 파일의 역할은 다음과 같다.

| 파일                  | 역할                  |
| ------------------- | ------------------- |
| `base_views.py`     | 질문 목록, 질문 상세        |
| `question_views.py` | 질문 등록, 수정, 삭제       |
| `answer_views.py`   | 답변 등록, 수정, 삭제       |
| `comment_views.py`  | 질문/답변 댓글 등록, 수정, 삭제 |
| `vote_views.py`     | 질문/답변 추천            |

### 2.2 상대 import 주의

`views` 디렉터리 안으로 파일이 들어가면 `models.py`, `forms.py`가 상위 디렉터리에 있으므로 상대 경로를 사용해야 한다.

```python
from ..models import Question
from ..forms import QuestionForm
```

### 2.3 `urls.py`에서 명시적으로 연결

초기에는 `views/__init__.py`에서 모든 함수를 import할 수도 있지만, 유지보수 관점에서는 `urls.py`에서 모듈명을 명시하는 방식이 더 명확하다.

```python
from .views import base_views, question_views, answer_views, comment_views, vote_views

urlpatterns = [
    path('', base_views.index, name='index'),
    path('<int:question_id>/', base_views.detail, name='detail'),

    path('question/create/', question_views.question_create, name='question_create'),
    path('question/modify/<int:question_id>/', question_views.question_modify, name='question_modify'),
    path('question/delete/<int:question_id>/', question_views.question_delete, name='question_delete'),

    path('answer/create/<int:question_id>/', answer_views.answer_create, name='answer_create'),
    path('answer/modify/<int:answer_id>/', answer_views.answer_modify, name='answer_modify'),

    path('vote/question/<int:question_id>/', vote_views.vote_question, name='vote_question'),
    path('vote/answer/<int:answer_id>/', vote_views.vote_answer, name='vote_answer'),
]
```

이렇게 하면 URL만 봐도 어느 모듈의 함수가 실행되는지 알 수 있다.

## 3. 추천 기능 추가

### 3.1 추천 기능의 관계 구조

추천 기능은 사용자와 게시글 사이의 관계다.

한 사용자는 여러 질문을 추천할 수 있고, 하나의 질문은 여러 사용자에게 추천받을 수 있다.

따라서 관계는 **다대다 관계**다.

```text
User N : M Question
User N : M Answer
```

Django에서는 이를 `ManyToManyField`로 표현한다.

### 3.2 Question 모델 수정

```python
class Question(models.Model):
    author = models.ForeignKey(
        User,
        on_delete=models.CASCADE,
        related_name='author_question'
    )
    subject = models.CharField(max_length=200)
    content = models.TextField()
    create_date = models.DateTimeField(auto_now_add=True)
    modify_date = models.DateTimeField(null=True, blank=True)
    voter = models.ManyToManyField(
        User,
        related_name='voter_question',
        blank=True
    )
```

### 3.3 Answer 모델 수정

```python
class Answer(models.Model):
    author = models.ForeignKey(
        User,
        on_delete=models.CASCADE,
        related_name='author_answer'
    )
    question = models.ForeignKey(Question, on_delete=models.CASCADE)
    content = models.TextField()
    create_date = models.DateTimeField(auto_now_add=True)
    modify_date = models.DateTimeField(null=True, blank=True)
    voter = models.ManyToManyField(
        User,
        related_name='voter_answer',
        blank=True
    )
```

### 3.4 `related_name`이 필요한 이유

`author`도 `User`를 참조하고, `voter`도 `User`를 참조한다.

따라서 `related_name`을 지정하지 않으면 Django가 역참조 이름을 만들 때 충돌이 발생한다.

예를 들어 다음과 같이 구분한다.

```python
author = models.ForeignKey(User, related_name='author_question')
voter = models.ManyToManyField(User, related_name='voter_question')
```

이렇게 하면 다음처럼 사용할 수 있다.

```python
some_user.author_question.all()
some_user.voter_question.all()
```

### 3.5 질문 추천 처리

추천 버튼을 누르면 `vote_question()` 함수가 실행된다.

```python
@login_required(login_url='common:login')
def vote_question(request, question_id):
    question = get_object_or_404(Question, pk=question_id)

    if request.user == question.author:
        messages.error(request, '본인이 작성한 글은 추천할 수 없습니다.')
    else:
        question.voter.add(request.user)

    return redirect('pybo:detail', question_id=question.id)
```

핵심은 다음이다.

| 코드                                 | 의미              |
| ---------------------------------- | --------------- |
| `@login_required`                  | 로그인한 사용자만 추천 가능 |
| `get_object_or_404()`              | 추천 대상 질문 조회     |
| `request.user == question.author`  | 자기 글 추천 방지      |
| `question.voter.add(request.user)` | 추천인 추가          |
| `redirect()`                       | 상세 페이지로 이동      |

### 3.6 답변 추천 처리

답변 추천도 같은 구조다.

```python
@login_required(login_url='common:login')
def vote_answer(request, answer_id):
    answer = get_object_or_404(Answer, pk=answer_id)

    if request.user == answer.author:
        messages.error(request, '본인이 작성한 글은 추천할 수 없습니다.')
    else:
        answer.voter.add(request.user)

    return redirect('pybo:detail', question_id=answer.question.id)
```

### 3.7 추천 수 표시

질문 상세 화면에서는 다음처럼 추천 수를 표시한다.

```django
{{ question.voter.count }}
```

질문 목록에서는 추천 수가 0보다 클 때만 badge로 표시한다.

```django
{% if question.voter.all.count > 0 %}
<span class="badge bg-warning px-2 py-1">
    {{ question.voter.all.count }}
</span>
{% endif %}
```

## 4. 스크롤 초기화 문제 해결

### 4.1 문제 상황

답변이나 댓글을 등록/수정하면 페이지가 다시 로드되면서 스크롤이 맨 위로 이동한다.

답변이 많은 게시글에서는 사용자가 방금 작성한 답변이나 댓글 위치를 다시 찾아야 하므로 불편하다.

### 4.2 앵커 기능 사용

HTML에서는 URL 뒤에 `#anchor`를 붙이면 특정 위치로 이동할 수 있다.

답변마다 고유한 앵커를 만든다.

```django
<a name="answer_{{ answer.id }}"></a>
```

댓글도 마찬가지다.

```django
<a name="comment_{{ comment.id }}"></a>
```

### 4.3 redirect URL에 앵커 추가

답변 등록 후 다음처럼 해당 답변 위치로 이동한다.

```python
return redirect('{}#answer_{}'.format(
    resolve_url('pybo:detail', question_id=question.id),
    answer.id
))
```

댓글 등록/수정 후에는 다음처럼 이동한다.

```python
return redirect('{}#comment_{}'.format(
    resolve_url('pybo:detail', question_id=question.id),
    comment.id
))
```

삭제의 경우에는 삭제된 객체 위치로 이동할 필요가 없으므로 앵커를 사용하지 않는다.

## 5. 마크다운 기능 적용

### 5.1 마크다운이 필요한 이유

게시글 본문에 단순 텍스트만 표시하면 코드, 목록, 링크, 인용 등을 표현하기 어렵다.

마크다운을 적용하면 사용자가 다음과 같은 문법을 사용할 수 있다.

| 기능       | 문법           |
| -------- | ------------ |
| 순서 없는 목록 | `* 항목`       |
| 순서 있는 목록 | `1. 항목`      |
| 굵게       | `**강조**`     |
| 링크       | `[텍스트](URL)` |
| 코드 블록    | 백틱 3개        |
| 인용       | `> 인용문`      |

### 5.2 markdown 설치

```bash

```

### 5.3 커스텀 템플릿 필터 작성

```python
register = template.Library()

@register.filter

    extensions = ["nl2br", "fenced_code"]
    return mark_safe(markdown.markdown(value, extensions=extensions))
```

## 6. 기존 Django 방식과 DRF 방식의 차이

### 6.1 기존 Django 방식

기존 Pybo는 서버가 HTML을 만들어 브라우저에 전달했다.

```text
DB 조회
→ View에서 context 구성
→ Template 렌더링
→ HTML 응답
→ 브라우저 출력
```

즉, Django가 화면까지 만들어준다.

### 6.2 DRF 방식

DRF에서는 서버가 HTML이 아니라 JSON을 반환한다.

```text
DB 조회
→ Serializer로 JSON 변환
→ JSON 응답
→ React/Vue/모바일 앱이 화면 구성
```

즉, 서버는 데이터를 제공하고, 프론트엔드는 그 데이터를 받아 UI를 만든다.

### 6.3 왜 JSON이 필요한가?

React, Vue, Flutter, 모바일 앱 등은 Django 템플릿을 직접 사용하지 않는다.

이들은 서버로부터 JSON 데이터를 받아 자신만의 방식으로 화면을 구성한다.

따라서 Django가 다양한 클라이언트와 연결되려면 API 서버 역할을 해야 한다.

## 7. Serializer 개념

### 7.1 Serializer의 역할

Serializer는 Python 객체와 JSON 사이를 변환한다.

```text
Python 객체 → JSON
JSON → Python 객체
```

| 방향   | 의미                         |
| ---- | -------------------------- |
| 직렬화  | Django 모델 객체를 JSON으로 변환    |
| 역직렬화 | JSON 요청 데이터를 Python 객체로 변환 |

### 7.2 ModelForm과 ModelSerializer 비교

| 구분                | 역할                    |
| ----------------- | --------------------- |
| `ModelForm`       | 모델 데이터를 HTML form과 연결 |
| `ModelSerializer` | 모델 데이터를 JSON API와 연결  |

기존 Pybo에서 `QuestionForm`이 질문 등록/수정 폼을 담당했다면, DRF에서는 `QuestionSerializer`가 API 입력/출력을 담당한다.

### 7.3 QuestionSerializer 예시

```python
from rest_framework import serializers
from pybo.models import Question

class QuestionSerializer(serializers.ModelSerializer):
    author = serializers.ReadOnlyField(source='author.username')
    vote_count = serializers.SerializerMethodField()

    class Meta:
        model = Question
        fields = [
            'id',
            'subject',
            'content',
            'author',
            'create_date',
            'modify_date',
            'vote_count',
        ]
        read_only_fields = ['create_date', 'modify_date']

    def get_vote_count(self, obj):
        return obj.voter.count()
```

여기서 핵심은 다음이다.

| 요소                      | 설명                 |
| ----------------------- | ------------------ |
| `model = Question`      | 어떤 모델을 직렬화할지 지정    |
| `fields`                | JSON에 포함할 필드 지정    |
| `ReadOnlyField`         | 작성자 이름을 읽기 전용으로 표시 |
| `SerializerMethodField` | 추천 수처럼 계산된 값을 반환   |

## 9. DRF API 구현 단계

12주차에서는 같은 Question CRUD를 여러 방식으로 구현하면서 DRF의 추상화 과정을 배웠다.

### 9.1 전체 진화 단계

```text
Stage 1. csrf_exempt + 순수 Django FBV
Stage 2. @api_view
Stage 3. APIView
Stage 4. mixins + GenericAPIView
Stage 5. generics.ListCreateAPIView / RetrieveUpdateDestroyAPIView
Stage 6. ModelViewSet + Router
```

핵심은 단계가 올라갈수록 코드가 짧아지지만, 내부 동작을 이해해야 디버깅할 수 있다는 점이다.

### 9.2 Stage 1 — 순수 Django 함수형 API

```python
@csrf_exempt
def question_list(request):
    if request.method == 'GET':
        questions = Question.objects.all().order_by('-create_date')
        serializer = QuestionSerializer(questions, many=True)
        return JsonResponse(serializer.data, safe=False)

    elif request.method == 'POST':
        data = JSONParser().parse(request)
        serializer = QuestionSerializer(data=data)

        if serializer.is_valid():
            serializer.save(author=User.objects.first())
            return JsonResponse(serializer.data, status=201)

        return JsonResponse(serializer.errors, status=400)
```

Detail API:

```python
@csrf_exempt
def question_detail(request, pk):
    question = get_object_or_404(Question, pk=pk)

    if request.method == 'GET':
        return JsonResponse(QuestionSerializer(question).data)

    elif request.method == 'PUT':
        data = JSONParser().parse(request)
        serializer = QuestionSerializer(question, data=data)

        if serializer.is_valid():
            serializer.save()
            return JsonResponse(serializer.data)

        return JsonResponse(serializer.errors, status=400)

    elif request.method == 'DELETE':
        question.delete()
        return JsonResponse({}, status=204)
```

Stage 1의 반복 코드:

| 반복 코드                           | 문제             |
| ------------------------------- | -------------- |
| `@csrf_exempt`                  | CSRF 직접 우회     |
| `JSONParser().parse(request)`   | 요청 본문 직접 파싱    |
| `JsonResponse(..., safe=False)` | 리스트 응답 처리 필요   |
| `if request.method == ...`      | HTTP 메서드 직접 분기 |
| `status=201`, `status=400`      | 상태 코드 숫자 직접 작성 |

### 9.3 Stage 2 — `@api_view`

```python
@api_view(['GET', 'POST'])
def question_list(request):
    if request.method == 'GET':
        questions = Question.objects.all()
        serializer = QuestionSerializer(questions, many=True)
        return Response(serializer.data)

    elif request.method == 'POST':
        serializer = QuestionSerializer(data=request.data)

        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=status.HTTP_201_CREATED)

        return Response(serializer.errors, status=status.HTTP_400_BAD_REQUEST)
```

Stage 2에서 사라지는 것:

```text
csrf_exempt
JSONParser().parse()
JsonResponse
safe=False
매직 넘버 상태 코드
```

또한 DRF의 Browsable API 화면을 사용할 수 있어 개발 중 테스트가 편해진다.

### 9.4 Stage 3 — APIView

```python
class QuestionList(APIView):
    def get(self, request):
        questions = Question.objects.all()
        serializer = QuestionSerializer(questions, many=True)
        return Response(serializer.data)

    def post(self, request):
        serializer = QuestionSerializer(data=request.data)

        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=status.HTTP_201_CREATED)

        return Response(serializer.errors, status=status.HTTP_400_BAD_REQUEST)
```

URL에서는 반드시 `as_view()`를 사용한다.

```python
path('questions/', QuestionList.as_view())
```

이 단계에서는 기존의 `if request.method == 'GET'` 분기가 `get()` 메서드로, `POST` 분기가 `post()` 메서드로 분리된다.

### 9.5 Stage 4 — GenericAPIView + Mixin

```python
class QuestionList(
    mixins.ListModelMixin,
    mixins.CreateModelMixin,
    generics.GenericAPIView
):
    queryset = Question.objects.all()
    serializer_class = QuestionSerializer

    def get(self, request, *args, **kwargs):
        return self.list(request, *args, **kwargs)

    def post(self, request, *args, **kwargs):
        return self.create(request, *args, **kwargs)
```

이 단계부터는 `queryset`과 `serializer_class`를 선언하고, 나머지 CRUD 흐름은 Mixin이 담당한다.

### 9.6 Stage 5 — generics 클래스

```python
class QuestionList(generics.ListCreateAPIView):
    queryset = Question.objects.all()
    serializer_class = QuestionSerializer


class QuestionDetail(generics.RetrieveUpdateDestroyAPIView):
    queryset = Question.objects.all()
    serializer_class = QuestionSerializer
```

대부분의 CRUD API는 다음 두 클래스로 끝낼 수 있다.

| View                           | 역할              |
| ------------------------------ | --------------- |
| `ListCreateAPIView`            | 목록 조회 + 생성      |
| `RetrieveUpdateDestroyAPIView` | 단건 조회 + 수정 + 삭제 |

### 9.7 Stage 6 — ModelViewSet + Router

```python
class QuestionViewSet(viewsets.ModelViewSet):
    queryset = Question.objects.all()
    serializer_class = QuestionSerializer
```

Router:

```python
router = DefaultRouter()
router.register(r'questions', QuestionViewSet, basename='question')

urlpatterns = [
    path('', include(router.urls)),
]
```

Router는 다음 URL을 자동 생성한다.

| URL                | Method | Action             |
| ------------------ | ------ | ------------------ |
| `/questions/`      | GET    | `list()`           |
| `/questions/`      | POST   | `create()`         |
| `/questions/{pk}/` | GET    | `retrieve()`       |

## 10. GenericAPIView와 Mixin 책임 분리

### 10.1 전체 계층 구조

```text
Concrete View
    ↓
Mixin
    ↓
GenericAPIView
    ↓
APIView
```

| 계층               | 역할                                         |
| ---------------- | ------------------------------------------ |
| `APIView`        | 인증, 권한, 요청 처리 기반                           |
| `GenericAPIView` | queryset, serializer, pagination, filter   |
| `Mixin`          | list, create, retrieve, update, destroy 액션 |
| `Concrete View`  | HTTP 메서드와 액션 연결                            |

### 10.2 GenericAPIView의 역할

GenericAPIView는 데이터를 다루는 “공구함”이다.

| 속성                 | 역할             |
| ------------------ | -------------- |
| `queryset`         | 조회 대상 QuerySet |
| `serializer_class` | 사용할 Serializer |
| `lookup_field`     | 단일 객체 조회 키     |
| `pagination_class` | 페이지네이션 정책      |
| `filter_backends`  | 필터링 백엔드        |

핵심 메서드:

| 메서드                        | 역할                 |
| -------------------------- | ------------------ |
| `get_queryset()`           | QuerySet 반환        |
| `get_object()`             | 단일 객체 조회 + 권한 검사   |
| `get_serializer()`         | Serializer 인스턴스 생성 |
| `filter_queryset()`        | 필터 적용              |
| `paginate_queryset()`      | 페이지네이션             |
| `get_paginated_response()` | 페이지네이션 응답          |

### 10.3 Mixin의 역할

Mixin은 단순히 GenericAPIView를 호출만 하는 것이 아니라, CRUD 흐름과 HTTP 응답 의미를 가진다.

| 책임         | 설명                                          |
| ---------- | ------------------------------------------- |
| 워크플로우 제어   | 조회 → 검증 → 저장 → 응답                           |
| 유효성 검사     | `serializer.is_valid(raise_exception=True)` |
| HTTP 응답 생성 | 201 Created, 204 No Content 등               |
| 확장 훅 제공    | `perform_create()`, `perform_update()`      |

예시:

```python
class CreateModelMixin:
    def create(self, request, *args, **kwargs):
        serializer = self.get_serializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        self.perform_create(serializer)
        headers = self.get_success_headers(serializer.data)
        return Response(
            serializer.data,
            status=status.HTTP_201_CREATED,
            headers=headers
        )

    def perform_create(self, serializer):
        serializer.save()
```

### 10.4 `perform_create()` 오버라이드

현재 로그인한 사용자를 작성자로 자동 저장할 때 사용한다.

```python
def perform_create(self, serializer):
    serializer.save(author=self.request.user)
```

또는 Snippet 예제에서는 다음처럼 owner를 저장했다.

```python
def perform_create(self, serializer):
    serializer.save(owner=self.request.user)
```

## 11. 인증과 권한

### 11.1 Authentication과 Permission의 차이

| 개념             | 의미                        |
| -------------- | ------------------------- |
| Authentication | 사용자가 누구인지 확인              |
| Login          | 인증된 사용자가 세션 또는 토큰을 가지는 상태 |
| Permission     | 사용자가 특정 행동을 해도 되는지 확인     |

예를 들어 다음처럼 구분한다.

| 기능    | 허용 대상      |
| ----- | ---------- |
| 목록 조회 | 누구나        |
| 상세 조회 | 누구나        |
| 생성    | 로그인 사용자    |
| 수정/삭제 | 작성자 또는 소유자 |

### 11.2 DRF 기본 권한 클래스

| 클래스                         | 의미                 |
| --------------------------- | ------------------ |
| `AllowAny`                  | 모두 허용              |
| `IsAuthenticated`           | 인증 사용자만 허용         |
| `IsAdminUser`               | 관리자만 허용            |
| `IsAuthenticatedOrReadOnly` | 읽기는 모두, 쓰기는 인증 사용자 |
| `DjangoModelPermissions`    | Django 모델 권한 연동    |

### 11.3 권한 검사 2단계

DRF 권한 검사는 두 단계다.

```text
has_permission()
→ 뷰 진입 여부 검사

has_object_permission()
→ 객체 단위 권한 검사
```

`has_permission()`은 요청 자체를 허용할지 판단한다.

`has_object_permission()`은 특정 객체에 대해 수정/삭제 권한이 있는지 판단한다.

## 12. `IsOwnerOrReadOnly`

### 12.1 필요한 이유

`IsAuthenticatedOrReadOnly`만 사용하면 인증된 사용자는 쓰기 요청이 가능하다.

하지만 이 권한만으로는 다음 상황을 막을 수 없다.

```text
user11이 admin이 작성한 게시글을 수정/삭제
```

따라서 객체 소유자인지 확인하는 커스텀 권한이 필요하다.

### 12.2 구현

```python
from rest_framework import permissions

class IsOwnerOrReadOnly(permissions.BasePermission):
    """
    객체의 소유자만 수정/삭제 가능
    """

    def has_object_permission(self, request, view, obj):
        if request.method in permissions.SAFE_METHODS:
            return True

        return obj.owner == request.user
```

Pybo에 적용한다면 `obj.author == request.user` 형태가 될 수 있다.

```python
return obj.author == request.user
```

핵심은 다음이다.

| 요소             | 의미                 |
| -------------- | ------------------ |
| `SAFE_METHODS` | GET, HEAD, OPTIONS |
| 읽기 요청          | 누구나 허용             |
| 쓰기 요청          | 소유자만 허용            |
| `False` 반환     | 403 Forbidden      |

### 12.3 주의점

메서드 이름을 반드시 정확히 작성해야 한다.

```python
has_object_permission
```

이를 `has_permission`으로 잘못 쓰면 객체 단위 검사가 호출되지 않아 보안 버그가 생길 수 있다.

## 13. List View와 Detail View의 권한 전략

### 13.1 ListCreateAPIView

```python
class SnippetList(generics.ListCreateAPIView):
    queryset = Snippet.objects.all()

    permission_classes = [
```

## 15. 서비스 레이어 분리와 예외 처리

12주차 후반 자료에서는 Django 코드 구조를 더 유지보수하기 쉽게 만들기 위해 **서비스 레이어 분리**를 다루었다.

### 15.1 기존 View의 문제

기존 View가 ORM 조회, 예외 처리, 비즈니스 로직, 템플릿 렌더링을 모두 담당하면 코드가 복잡해진다.

```python
def book_history(request, book_id):
    try:
        book = Book.objects.get(id=book_id)
    except Book.DoesNotExist:
        return HttpResponseNotFound(...)

    histories = book.borrow_history.all()

    if not histories.exists():
        return render(request, 'no_history.html')

    histories = histories.order_by('-borrowed_at')
    return render(request, 'book_history.html', {...})
```

### 15.2 3계층 구조

```text
View → Service → Model
```

| 계층      | 역할                           |
| ------- | ---------------------------- |
| View    | 요청/응답 처리, Service 호출, render |
| Service | 비즈니스 로직, ORM 호출, 예외 변환       |
| Model   | 데이터 구조, DB 매핑                |

### 15.3 Service 함수

```python
def get_all_books():
    return Book.objects.all()

def get_book_by_id(book_id: int) -> Book:
    return Book.objects.get(id=book_id)

def get_borrow_history_for_book(book: Book):
    return book.borrow_history.order_by('-borrowed_at')
```

### 15.4 커스텀 예외

ORM 예외를 그대로 View에 노출하지 않고 도메인 예외로 변환한다.

```python
class BookNotFound(Exception):
    pass

class BookHasNoBorrowHistory(Exception):
    pass
```

Service에서 변환:

```python
def get_book_by_id(book_id: int) -> Book:
    try:
        return Book.objects.get(id=book_id)
    except Book.DoesNotExist:
        raise BookNotFound(f'ID {book_id}에 해당하는 책이 없습니다.')
```

대출 이력이 없을 때:

```python
def get_borrow_history_for_book(book: Book):
    histories = book.borrow_history.order_by('-borrowed_at')

    if not histories.exists():
        raise BookHasNoBorrowHistory(
            f"'{book.title}' 도서에는 대출 이력이 없습니다."
        )

    return histories
```

### 15.5 View는 표현만 담당

```python
def book_history(request, book_id):
    try:
        book = book_service.get_book_by_id(book_id)
        histories = book_service.get_borrow_history_for_book(book)

    except BookNotFound as e:
        return HttpResponseNotFound(str(e))

    except BookHasNoBorrowHistory as e:
        return render(request, 'library/no_history.html', {
            'message': str(e),
            'book': book,
        })

    return render(request, 'library/book_history.html', {
        'book': book,
        'histories': histories,
    })
```

### 15.6 pytest 단위 테스트

서비스 레이어를 분리하면 HTTP 요청 없이 서비스 함수만 직접 테스트할 수 있다.

```python
@pytest.mark.django_db
def test_get_book_by_id_success():
    book = Book.objects.create(
        title='Test Book',
        author='Tester',
        isbn='1234567890123'
    )

    result = get_book_by_id(book.id)

    assert result == book
```

예외 테스트:

```python
@pytest.mark.django_db
def test_get_book_by_id_not_found():
    with pytest.raises(BookNotFound):
        get_book_by_id(9999)
```
