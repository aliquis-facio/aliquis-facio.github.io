# 1. HTTP란 무엇인가

HTTP는 **HyperText Transfer Protocol**이다.
웹 브라우저와 웹 서버가 데이터를 주고받을 때 가장 기본적으로 사용하는 프로토콜이다.

우리가 평소 하는 거의 모든 웹 동작이 HTTP 기반이다.

예를 들면:

* 브라우저에서 웹페이지 열기
* 로그인 요청 보내기
* 게시글 목록 조회
* 검색 결과 가져오기
* 폼 제출
* 이미지, CSS, JS 파일 요청

즉, HTTP는 웹의 기본 통신 규약이다.

---

# 2. HTTP의 동작 방식

HTTP는 기본적으로 **request-response 구조**다.

흐름은 이렇게 된다.

1. 클라이언트가 서버에 요청(Request)을 보낸다.
2. 서버가 요청을 처리한다.
3. 서버가 응답(Response)을 돌려준다.
4. 한 번의 요청-응답이 끝나면 통신도 사실상 종료된다.

예를 들어 게시글 목록 조회는 이런 식이다.

```http
GET /posts HTTP/1.1
Host: example.com
```

서버는 이런 응답을 돌려준다.

```http
HTTP/1.1 200 OK
Content-Type: application/json

[
  {"id": 1, "title": "첫 글"},
  {"id": 2, "title": "둘째 글"}
]
```

여기서 중요한 점은,
**클라이언트가 먼저 요청해야 서버가 응답할 수 있다**는 것이다.

서버가 먼저 “갑자기” 브라우저에 데이터를 밀어 넣는 구조가 아니다.

---

# 3. HTTP의 특징

## 3-1. 비연결성 중심 구조

HTTP는 전통적으로 **필요할 때 연결하고, 응답 후 끊는** 성격이 강하다.
그래서 매 요청이 독립적이다.

이 말은 곧:

* 구현이 단순하고
* 확장성이 좋고
* 서버가 많은 사용자를 다루기 쉽다는 뜻이다

웹 서비스 대부분이 HTTP 위에서 잘 동작하는 이유가 여기 있다.

---

## 3-2. 무상태성(Stateless)

HTTP는 기본적으로 **상태를 기억하지 않는 프로토콜**이다.

예를 들어:

* 첫 번째 요청에서 로그인했다고 해도
* 두 번째 요청에서 서버는 그 사실을 자동으로 알지 못한다

그래서 로그인 상태 유지를 위해 보통 이런 것들을 쓴다.

* 쿠키
* 세션
* JWT
* Authorization 헤더

즉, HTTP 자체는 상태를 기억하지 않고,
애플리케이션 레벨에서 상태를 보완한다.

---

## 3-3. 요청이 있어야 응답이 가능

HTTP는 항상 **클라이언트 주도형**이다.

예를 들어 실시간 채팅방에서 새 메시지가 왔는지 알고 싶다면 HTTP만으로는 이런 문제가 생긴다.

* 클라이언트가 1초마다 서버에 “새 메시지 있나요?” 요청
* 서버가 “없음” 응답
* 또 1초 뒤 요청
* 또 없음

이런 방식을 **Polling**이라고 한다.
실시간 흉내는 낼 수 있지만 비효율적이다.

---

# 4. HTTP가 잘 맞는 경우

HTTP는 다음 같은 작업에 아주 적합하다.

## 조회 중심 기능

* 게시글 목록 조회
* 상품 상세 조회
* 사용자 프로필 조회

## 명령형 요청

* 회원가입
* 로그인
* 글 작성
* 수정/삭제

## 문서와 리소스 전달

* HTML 페이지
* CSS
* JavaScript
* 이미지
* 파일 다운로드

즉, **한 번 요청하고 결과 받으면 끝나는 작업**에 매우 강하다.

---

# 5. HTTP의 한계

HTTP는 강력하지만, 실시간성이 중요한 상황에서는 한계가 있다.

예를 들면:

* 실시간 채팅
* 주식 시세 스트리밍
* 실시간 게임 상태 동기화
* 알림 시스템
* 협업 문서 편집
* 실시간 대시보드

이런 경우에는 서버 쪽에서 새 이벤트가 생길 때마다 곧바로 클라이언트에 알려줘야 한다.
그런데 HTTP는 원칙적으로 클라이언트 요청이 먼저여야 하므로 비효율이 생긴다.

이 문제를 해결하기 위한 대표적인 방식이 **WebSocket**이다.

---

# 6. WebSocket이란 무엇인가

WebSocket은 **클라이언트와 서버가 한 번 연결을 맺은 뒤, 그 연결을 유지하면서 양방향으로 데이터를 주고받는 프로토콜**이다.

HTTP처럼 요청하고 끝나는 것이 아니라,
연결이 살아 있는 동안 서로 자유롭게 메시지를 보낼 수 있다.

즉:

* 클라이언트 → 서버 전송 가능
* 서버 → 클라이언트 전송 가능
* 연결 유지
* 실시간 통신에 유리

---

# 7. WebSocket의 동작 방식

WebSocket은 처음 연결할 때는 보통 HTTP를 이용해 시작한다.
이 과정을 **핸드셰이크(handshake)** 라고 한다.

대략 이런 흐름이다.

1. 클라이언트가 HTTP 요청으로 WebSocket 연결 업그레이드를 요청
2. 서버가 이를 승인
3. 그 뒤부터는 HTTP가 아니라 WebSocket 프로토콜로 통신
4. 연결이 유지되는 동안 양방향 메시지 송수신

즉, 시작은 HTTP이지만 연결이 성립되면 성격이 완전히 달라진다.

---

# 8. WebSocket의 핵심 특징

## 8-1. 연결 유지(Persistent Connection)

한 번 연결되면 계속 살아 있다.
그래서 매번 요청마다 새 연결을 만들 필요가 없다.

이 특성 때문에:

* 지연이 줄고
* 이벤트 전달이 빠르고
* 반복 요청 오버헤드가 줄어든다

---

## 8-2. 양방향 통신(Full Duplex)

HTTP는 보통 요청-응답 구조라 흐름이 분명히 나뉜다.
반면 WebSocket은 연결만 유지되면 양쪽이 아무 때나 메시지를 보낼 수 있다.

예를 들어 채팅에서는:

* 사용자가 메시지 보냄
* 서버가 저장
* 서버가 같은 방의 다른 사용자들에게 즉시 push

이게 매우 자연스럽다.

---

## 8-3. 서버 푸시가 자연스러움

WebSocket에서는 서버가 이벤트 발생 시 즉시 클라이언트로 보낼 수 있다.

예:

* “새 채팅이 도착했습니다”
* “주가가 변경되었습니다”
* “상대 플레이어가 이동했습니다”
* “주문 상태가 변경되었습니다”

HTTP에서는 이런 걸 하려면 polling, long polling, SSE 같은 우회 전략이 필요할 수 있다.

---

# 9. WebSocket이 잘 맞는 경우

WebSocket은 다음 상황에 특히 적합하다.

## 실시간 채팅

메시지가 도착하는 즉시 상대에게 보여야 한다.

## 실시간 알림

관리자 알림, 주문 상태 변경, 시스템 이벤트 등을 즉시 전달해야 한다.

## 온라인 게임

플레이어 위치, 행동, 상태 변화가 매우 자주 바뀐다.

## 실시간 시세 / 대시보드

주식, 코인, 센서 데이터, 모니터링 지표가 계속 바뀐다.

## 협업 편집

여러 사용자의 변경 사항을 빠르게 동기화해야 한다.

---

# 10. WebSocket의 단점과 운영상 어려움

WebSocket이 무조건 더 좋은 것은 아니다.
실시간성은 강하지만 운영 복잡도가 올라간다.

## 10-1. 연결 관리가 필요함

HTTP는 요청이 끝나면 정리되지만, WebSocket은 연결을 유지해야 한다.
그래서 서버는 많은 연결 상태를 계속 관리해야 한다.

고려할 것이 많다.

* 누가 접속 중인지
* 어느 채팅방에 들어가 있는지
* 끊긴 연결 처리
* 재접속 처리
* heartbeat/ping-pong
* idle timeout 관리

---

## 10-2. 서버 확장 시 복잡해짐

서버가 1대일 때는 비교적 단순하다.
하지만 여러 대로 확장하면 문제가 생긴다.

예를 들어 사용자 A는 서버 1에 붙어 있고, 사용자 B는 서버 2에 붙어 있을 수 있다.
이때 A의 메시지를 B에게 전달하려면 서버들 사이에서 메시지를 공유해야 한다.

그래서 보통 이런 것이 필요하다.

* Redis Pub/Sub
* Kafka
* RabbitMQ
* 별도의 message broker

즉, 수평 확장 환경에서는 아키텍처 복잡도가 높아진다.

---

## 10-3. 장애 처리 고려가 많음

실시간 시스템에서는 단순히 “연결됨”만으로 끝나지 않는다.

추가로 생각해야 한다.

* 메시지 유실 가능성
* 순서 보장 문제
* 중복 수신 문제
* 재전송 정책
* ACK 처리
* 클라이언트 재연결 후 동기화

채팅이나 게임처럼 상태가 중요한 시스템은 이 설계가 핵심이다.

---

# 11. HTTP와 WebSocket 비교

## 구조 비교

### HTTP

* 요청-응답
* 요청마다 독립적
* 클라이언트가 먼저 요청
* 일반 웹 기능에 적합

### WebSocket

* 연결 유지
* 양방향 통신
* 서버도 먼저 전송 가능
* 실시간 기능에 적합

---

## 비유로 이해하기

### HTTP

택배 주문과 비슷하다.

* 내가 요청
* 상대가 답변
* 한 건 처리 끝

즉, 필요할 때마다 새로 요청하는 방식이다.

### WebSocket

전화 통화와 비슷하다.

* 한 번 연결
* 연결 유지
* 서로 언제든 말 가능

그래서 즉시성이 필요한 상황에 유리하다.

---

# 12. 실무에서 둘 중 하나만 쓰는가?

아니다.
대부분 서비스는 **HTTP와 WebSocket을 함께 쓴다.**

예를 들어 채팅 서비스도 보통 이렇게 나뉜다.

## HTTP 담당

* 로그인
* 채팅방 목록 조회
* 이전 대화 내역 조회
* 프로필 조회
* 파일 업로드

## WebSocket 담당

* 새 메시지 실시간 수신
* 접속 상태 표시
* typing indicator
* 읽음 상태 실시간 반영

즉, WebSocket이 HTTP를 대체하는 게 아니라
**실시간이 필요한 일부 구간만 WebSocket으로 보완**하는 경우가 많다.

---

# 13. Django 관점에서 보면

장고 기본 구조는 HTTP 요청 처리에 매우 강하다.

* URL 라우팅
* View
* Template
* ORM
* Form 처리
* REST API

이런 부분은 HTTP 기반이다.

그런데 WebSocket을 쓰려면 기본 Django WSGI만으로는 부족하고, 보통 **Django Channels** 같은 구성이 필요하다.
실시간 연결 처리, consumer, channel layer, Redis 같은 요소가 들어온다.

즉, 장고에서 WebSocket은 “기본 게시판 기능”보다 “실시간 기능 확장”에 가깝다.

---

# 14. 어떤 상황에서 무엇을 선택해야 하나

판단 기준은 단순하다.

## HTTP를 선택하면 좋은 경우

* 한 번 요청하고 결과 받으면 충분함
* CRUD 중심 기능
* 일반적인 REST API
* 서버 구현 단순성이 중요함
* 실시간성이 필수는 아님

## WebSocket을 선택하면 좋은 경우

* 즉시 반응해야 함
* 서버가 먼저 클라이언트에 알려줘야 함
* 연결을 계속 유지할 가치가 있음
* 채팅, 게임, 라이브 대시보드 같은 기능임
