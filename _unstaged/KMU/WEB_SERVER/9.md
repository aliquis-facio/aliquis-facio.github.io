# 9주차

## 1. 9주차 전체 목표

9주차는 파이보 게시판을 단순 Q&A 게시판에서 **실제 서비스형 게시판**으로 확장하는 단계이다.

강의자료에서는 파이보 서비스 개발의 주요 기능으로 다음을 제시한다.

* 내비게이션 기능
* 게시판 페이징
* 템플릿 필터
* 답변 개수 표시
* 로그인 / 로그아웃
* 회원가입
* 글쓴이 추가 및 표시
* 게시물 수정 / 삭제
* 댓글
* 추천
* 스크롤 초기화
* 마크다운
* 검색 / 정렬

이 중 9주차에서 실제로 중요하게 다룬 내용은 다음 네 가지이다.

| 핵심 주제 | 설명 |
| --- | --- |
| 회원가입 구현 | `UserCreationForm`을 상속해 사용자 등록 기능 구현 |
| 로그인 / 로그아웃 | Django `auth` 시스템과 `LoginView`, `LogoutView` 활용 |
| 글쓴이 추가 | `Question`, `Answer` 모델에 `author` 필드 추가 |
| 세션과 쿠키 | HTTP의 stateless 문제를 해결하는 로그인 유지 방식 이해 |

## 2. 로그인은 인증과 세션으로 구성된다

> 로그인은 단순히 아이디와 비밀번호를 확인하는 작업만이 아니라, 인증 이후 세션을 만들어 로그인 상태를 유지하는 과정이다.

로그인은 크게 두 단계로 구성된다.

1. 인증(Authentication): 사용자가 입력한 username/password가 DB의 사용자 정보와 일치하는지 확인
2. 세션(Session): 인증된 사용자가 일정 기간 서비스를 이용할 수 있도록 상태를 유지

즉, 로그인은 다음 흐름이다.

```text
username/password 입력
→ DB의 User 정보와 비교
→ 인증 성공
→ 세션 생성
→ 세션 키를 쿠키로 클라이언트에게 전달
→ 이후 요청마다 쿠키를 통해 사용자 식별
```

## 3. HTTP는 stateless이기 때문에 세션이 필요하다

HTTP는 기본적으로 **stateless protocol**이다. 즉, 한 번 요청하고 응답하면 서버는 그 요청 상태를 기억하지 않는다.

```text
Request → Response → 종료
```

따라서 사용자가 한 번 로그인했다고 해서, 다음 요청에서 서버가 자동으로 그 사용자를 기억하지 않는다. 이를 보완하기 위해 세션과 쿠키를 사용한다.

| 개념 | 저장 위치 | 역할 |
| --- | --- | --- |
| 세션 | 서버 | 로그인한 사용자 상태 관리 |
| 쿠키 | 클라이언트 브라우저 | 세션 키를 저장하고 요청마다 서버에 전달 |

교수자는 이를 **학생증 비유**로 설명했다.

```text
처음 학교에 들어갈 때 신분 확인 = 인증
학생증을 발급받음 = 세션 생성
이후 건물마다 학생증을 보여줌 = 쿠키 전달
```

즉, 사용자는 매 요청마다 쿠키에 담긴 세션 키를 서버에 보내고, 서버는 그 세션 키를 통해 “이 사용자가 누구인지” 확인한다.

## 4. Django auth 시스템의 역할

Django는 사용자 인증을 직접 구현하지 않아도 되도록 `django.contrib.auth`를 제공한다.

이 시스템은 다음 기능을 포함한다.

* 사용자 모델 `User`
* 로그인
* 로그아웃
* 비밀번호 해싱
* 권한 관리
* 세션 인증
* 관리자 권한
* 마지막 로그인 시각 기록

직접 구현하려면 개발자가 다음 작업을 모두 해야 한다.

```text
사용자 테이블 생성
아이디 중복 검사
비밀번호 해싱
로그인 시 사용자 검색
비밀번호 비교
세션 생성
로그아웃 처리
권한 확인
```

하지만 Django는 이 기능들을 이미 제공한다. 그래서 개발자는 `django.contrib.auth`를 활용해서 빠르게 인증 기능을 붙일 수 있다.

이것이 프레임워크의 **생산성**이다.

## 5. Django User 모델

Django의 기본 사용자 모델은 `django.contrib.auth.models.User`이다.

```python
from django.contrib.auth.models import User
```

User 모델에는 다음과 같은 필드가 있다.

| 필드 | 설명 |
| ----- | ------- |
| `id` | 사용자 고유 번호 |
| `username` | 사용자 이름 |
| `password` | 해시 처리된 비밀번호 |
| `email` | 이메일 |
| `first_name` | 이름 |
| `last_name` | 성 |
| `is_staff` | 관리자 사이트 접근 가능 여부 |
| `is_active` | 활성 사용자 여부 |
| `is_superuser` | 슈퍼유저 여부 |
| `last_login` | 마지막 로그인 시각 |
| `date_joined` | 가입 일시 |

중요한 점은 비밀번호가 원문으로 저장되지 않는다는 것이다.

예를 들어 사용자가 `1234`를 입력해도 DB에는 `1234`가 그대로 저장되지 않고, 해시 문자열로 저장된다.

```text
사용자 입력 비밀번호: 1234
DB 저장 값: 해시 처리된 긴 문자열
```

## 6. 회원가입 구현

### 6.1 회원가입 URL 추가

회원가입 기능은 `common` 앱에서 처리한다.

```python
## common/urls.py

from django.urls import path
from . import views

app_name = 'common'

urlpatterns = [
    path('signup/', views.signup, name='signup'),
]
```

로그인 화면에는 회원가입 링크를 추가한다.

```html
<a href="{% url 'common:signup' %}">계정을 만드세요.</a>
```

이 링크를 누르면 `views.signup` 함수가 실행된다.

### 6.2 UserCreationForm 상속

Django는 기본 회원가입 폼인 `UserCreationForm`을 제공한다.

기본 필드는 다음과 같다.

| 필드 | 설명 |
| --- | --- |
| `username` | 사용자 이름 |
| `password1` | 비밀번호 |
| `password2` | 비밀번호 확인 |

강의에서는 여기에 `email` 필드를 추가해 `UserForm`을 만든다.

```python
## common/forms.py

from django import forms
from django.contrib.auth.forms import UserCreationForm
from django.contrib.auth.models import User

class UserForm(UserCreationForm):
    email = forms.EmailField(label="이메일")

    class Meta:
        model = User
        fields = ("username", "email")
```

여기서 `Meta` 클래스는 이 폼이 어떤 모델과 연결되는지, 어떤 필드를 사용할지 지정한다.

| 구성 | 의미 |
| --- | --- |
| `model = User` | Django User 모델과 연결 |
| `fields = ("username", "email")` | 폼에서 사용할 필드 지정      |

### 6.3 signup 함수

회원가입 요청은 GET과 POST로 나뉜다.

| 요청 방식 | 동작              |
| ----- | --------------- |
| GET   | 빈 회원가입 폼 출력     |
| POST  | 입력값 검증 후 사용자 생성 |

```python
## common/views.py

from django.contrib.auth import authenticate, login
from django.shortcuts import render, redirect
from common.forms import UserForm

def signup(request):
    if request.method == "POST":
        form = UserForm(request.POST)

        if form.is_valid():
            form.save()

            username = form.cleaned_data.get('username')
            raw_password = form.cleaned_data.get('password1')

            user = authenticate(username=username, password=raw_password)
            login(request, user)

            return redirect('index')
    else:
        form = UserForm()

    return render(request, 'common/signup.html', {'form': form})
```

핵심은 회원가입 성공 후 자동 로그인까지 처리한다는 점이다.

```python
user = authenticate(username=username, password=raw_password)
login(request, user)
```

## 7. common 앱의 역할

> `common` 앱은 여러 서비스가 공통으로 사용하는 인증 관련 기능을 담당한다.

프로젝트에는 여러 앱이 있을 수 있다.

```text
project/
├── pybo/
├── hotel/
└── common/
```

각 앱의 역할은 다음처럼 나눌 수 있다.

| 앱        | 역할                       |
| -------- | ------------------------ |
| `pybo`   | Q&A 게시판 서비스              |
| `hotel`  | 호텔 예약 서비스                |
| `common` | 로그인, 로그아웃, 회원가입 같은 공통 기능 |

로그인 기능은 게시판에서도 필요하고, 호텔 예약에서도 필요할 수 있다.

따라서 로그인 기능을 `pybo` 안에 넣지 않고 `common` 앱으로 분리한다.

## 8. 로그인 / 로그아웃 URL

로그인과 로그아웃은 Django가 제공하는 클래스 기반 뷰를 사용한다.

```python
## common/urls.py

from django.urls import path
from django.contrib.auth import views as auth_views
from . import views

app_name = 'common'

urlpatterns = [
    path(
        'login/',
        auth_views.LoginView.as_view(template_name='common/login.html'),
        name='login'
    ),
    path(
        'logout/',
        auth_views.LogoutView.as_view(),
        name='logout'
    ),
    path('signup/', views.signup, name='signup'),
]
```

여기서 중요한 차이는 다음이다.

| 기능   | 구현 방식              |
| ---- | ------------------ |
| 로그인  | Django 제공 클래스 기반 뷰 |
| 로그아웃 | Django 제공 클래스 기반 뷰 |
| 회원가입 | 직접 만든 함수 기반 뷰      |

즉, 로그인과 로그아웃은 표준화된 기능이므로 Django가 미리 제공하는 `LoginView`, `LogoutView`를 사용한다.

반면 회원가입은 프로젝트 요구에 맞게 직접 `signup` 함수를 만든다.

## 9. Function-Based View와 Class-Based View

강의에서는 로그인 구현을 통해 함수 기반 뷰와 클래스 기반 뷰의 차이를 설명했다.

### 9.1 함수 기반 뷰

직접 함수를 작성해 요청을 처리한다.

```python
def signup(request):
    ...
```

특징은 다음과 같다.

* 직접 흐름을 작성함
* GET/POST 분기 직접 처리
* 커스터마이징이 쉬움
* 코드가 길어질 수 있음

### 9.2 클래스 기반 뷰

Django가 미리 만들어둔 클래스를 사용한다.

```python
auth_views.LoginView.as_view()
```

특징은 다음과 같다.

* 반복적인 로직을 줄일 수 있음
* 표준 기능 구현에 적합
* 내부 동작을 이해해야 커스터마이징 가능

## 10. 글쓴이 author 필드 추가

로그인 기능이 들어오면 이제 게시글을 작성한 사용자를 알 수 있다.

따라서 `Question` 모델에 작성자 필드를 추가한다.

기존 `Question` 모델은 대략 다음과 같았다.

```python
class Question(models.Model):
    subject = models.CharField(max_length=200)
    content = models.TextField()
    create_date = models.DateTimeField()
```

DB 기준으로는 다음 컬럼이 있었다.

| 컬럼            | 의미    |
| ------------- | ----- |
| `id`          | 질문 번호 |
| `subject`     | 질문 제목 |
| `content`     | 질문 내용 |
| `create_date` | 작성일   |

여기에 `author`를 추가한다.

```python
from django.contrib.auth.models import User
from django.db import models

class Question(models.Model):
    author = models.ForeignKey(User, on_delete=models.CASCADE)
    subject = models.CharField(max_length=200)
    content = models.TextField()
    create_date = models.DateTimeField()
```

## 11. ForeignKey와 CASCADE

핵심 코드는 다음이다.

```python
author = models.ForeignKey(User, on_delete=models.CASCADE)
```

이 코드는 `Question`과 `User`를 연결한다.

| 코드                         | 의미                        |
| -------------------------- | ------------------------- |
| `ForeignKey(User)`         | 질문이 User 모델을 참조           |
| `author`                   | 질문 작성자                    |
| `on_delete=models.CASCADE` | User 삭제 시 해당 User의 질문도 삭제 |

예를 들어 `홍길동` 사용자가 질문을 5개 작성했다고 하자.

```text
홍길동 삭제
→ 홍길동이 작성한 질문 5개도 함께 삭제
```

이것이 `CASCADE`이다.

## 12. Answer 모델에도 author 추가

질문뿐 아니라 답변에도 작성자가 필요하다.

따라서 `Answer` 모델에도 `author` 필드를 추가한다.

```python
class Answer(models.Model):
    author = models.ForeignKey(User, on_delete=models.CASCADE)
    question = models.ForeignKey(Question, on_delete=models.CASCADE)
    content = models.TextField()
    create_date = models.DateTimeField()
```

질문과 답변 모두 작성자 정보가 있어야 이후에 다음 기능을 구현할 수 있다.

* 작성자 표시
* 본인 글만 수정
* 본인 글만 삭제
* 작성자별 게시글 조회
* 사용자 활동 추적

## 13. 마이그레이션 이해

모델을 수정하면 DB 테이블 구조도 바뀌어야 한다.

Django에서는 이 작업을 두 단계로 처리한다.

```bash
python manage.py makemigrations
python manage.py migrate
```

### 13.1 makemigrations

`makemigrations`는 모델 변경 내용을 바탕으로 마이그레이션 파일을 만든다.

```bash
python manage.py makemigrations
```

강의에서는 이것을 **청사진을 만드는 작업**이라고 설명했다.

즉, 실제 DB를 바로 바꾸는 것이 아니라, DB를 어떻게 바꿀지 규칙 파일을 만든다.

예를 들어 다음 파일이 생성될 수 있다.

```text
0002_question_author.py
0003_answer_author.py
```

### 13.2 migrate

`migrate`는 생성된 마이그레이션 파일을 실제 DB에 적용한다.

```bash
python manage.py migrate
```

이 명령을 실행해야 실제 DB 테이블에 `author_id` 컬럼이 생긴다.

### 13.3 차이 정리

| 명령어              | 역할            | 실제 DB 변경  |
| ---------------- | ------------- | --------- |
| `makemigrations` | 변경 계획 파일 생성   | 아직 변경 안 됨 |
| `migrate`        | 변경 계획을 DB에 적용 | 실제 변경됨    |

전체 흐름은 다음과 같다.

```text
models.py 수정
→ makemigrations
→ migration 파일 생성
→ migrate
→ DB 테이블 변경
```

## 14. 기존 데이터가 있을 때 생기는 문제

이미 질문 데이터가 있는 상태에서 `author` 필드를 추가하면 문제가 생긴다.

기존 질문에는 작성자 값이 없기 때문이다.

예를 들어 기존 질문이 300개 있다고 하자.

```text
질문 1번: author 없음
질문 2번: author 없음
...
질문 300번: author 없음
```

그런데 새 모델에서는 `author`가 필수 필드이다.

즉, DB 입장에서는 모든 질문 row에 `author_id` 값이 있어야 한다.

이때 Django는 다음과 같은 의미의 메시지를 낸다.

```text
non-nullable field 'author'를 추가하려면 기존 데이터에 들어갈 기본값이 필요하다.
```

## 15. 기존 데이터 author 기본값 지정

해결 방법은 크게 두 가지이다.

| 방법             | 설명                         |
| -------------- | -------------------------- |
| `null=True` 설정 | 작성자 값이 없어도 허용              |
| 기본 사용자 지정      | 기존 게시글의 작성자를 특정 사용자로 강제 지정 |

강의에서는 두 번째 방법을 사용했다.

`makemigrations` 중 기본값을 묻는 프롬프트가 나오면 `1`을 입력한다.

```text
Select an option: 1
Please enter the default value now: 1
```

여기서 `1`은 User 테이블의 id 값이다.

대개 첫 번째 슈퍼유저의 id가 `1`이므로, 기존 게시글의 작성자가 모두 1번 사용자로 설정된다.

```text
기존 질문 1번 author_id = 1
기존 질문 2번 author_id = 1
...
기존 질문 300번 author_id = 1
```

## 16. author와 author_id의 차이

Django 모델에서는 다음처럼 작성한다.

```python
author = models.ForeignKey(User, on_delete=models.CASCADE)
```

하지만 실제 DB에는 보통 다음 컬럼으로 저장된다.

```text
author_id
```

즉, Django 코드에서는 `author`라고 접근하지만, DB에는 연결된 User의 id가 저장된다.

예를 들어 다음과 같다.

| pybo_question.id | subject | content | author_id |
| ---------------- | ------- | ------- | --------- |
| 1                | 질문 제목 1 | 내용 1    | 1         |
| 2                | 질문 제목 2 | 내용 2    | 1         |
| 3                | 질문 제목 3 | 내용 3    | 2         |

여기서 `author_id = 1`이면 User 테이블의 id가 1인 사용자가 작성자라는 뜻이다.

Django 코드에서는 다음처럼 접근할 수 있다.

```python
question.author
question.author.username
```

## 17. 질문 등록 시 author 저장

모델에 `author` 필드를 추가한 뒤에는 질문 등록 로직도 수정해야 한다.

기존 코드가 다음과 같았다면,

```python
question = form.save(commit=False)
question.create_date = timezone.now()
question.save()
```

이제는 다음처럼 `author`를 넣어야 한다.

```python
question = form.save(commit=False)
question.author = request.user
question.create_date = timezone.now()
question.save()
```

여기서 `request.user`는 현재 로그인한 사용자 객체이다.

즉, 로그인한 사용자가 질문을 등록하면 그 사용자가 자동으로 작성자로 저장된다.

## 18. 답변 등록 시 author 저장

답변도 마찬가지이다.

```python
answer = form.save(commit=False)
answer.author = request.user
answer.create_date = timezone.now()
answer.question = question
answer.save()
```

질문과 답변 모두 현재 로그인한 사용자를 `author`에 저장해야 한다.

## 19. AnonymousUser 오류

문제는 로그아웃 상태에서 질문이나 답변을 등록할 때 발생한다.

로그인하지 않은 상태에서 `request.user`는 실제 User 객체가 아니다.

이때 `request.user`는 다음 객체이다.

```text
AnonymousUser
```

그런데 `Question.author`에는 반드시 `User` 객체가 들어가야 한다.

따라서 로그아웃 상태에서 질문 등록을 시도하면 다음 의미의 오류가 발생한다.

```text
Question.author에는 User 인스턴스가 들어가야 하는데 AnonymousUser가 들어왔다.
```

즉, 작성자 필드를 추가한 뒤에는 질문/답변 등록을 로그인 사용자에게만 허용해야 한다.

## 20. @login_required 적용

Django는 로그인 여부를 검사하는 데코레이터를 제공한다.

```python
from django.contrib.auth.decorators import login_required
```

질문 등록 함수에 적용한다.

```python
@login_required(login_url='common:login')
def question_create(request):
    ...
```

답변 등록 함수에도 적용한다.

```python
@login_required(login_url='common:login')
def answer_create(request, question_id):
    ...
```

동작 방식은 다음과 같다.

| 사용자 상태  | 동작          |
| ------- | ----------- |
| 로그인 상태  | 함수 실행       |
| 로그아웃 상태 | 로그인 페이지로 이동 |

## 21. 데코레이터의 의미

강의에서는 데코레이터를 “함수 위에서 도와주는 장식자”라고 설명했다.

```python
@login_required(login_url='common:login')
def question_create(request):
    ...
```

이 코드는 `question_create` 함수가 실행되기 전에 로그인 여부를 먼저 검사한다.

즉, 기존 함수에 로그인 검사 기능을 덧붙이는 것이다.

## 22. next 파라미터

로그인이 필요한 페이지에 로그아웃 상태로 접근하면 Django는 로그인 페이지로 이동시키면서 `next` 파라미터를 붙인다.

예를 들어 질문 등록 페이지에 접근했다면 다음처럼 이동할 수 있다.

```text
/common/login/?next=/pybo/question/create/
```

이 뜻은 다음과 같다.

```text
로그인 성공 후 원래 가려던 /pybo/question/create/ 로 돌아가라
```

이를 위해 로그인 템플릿에 hidden input을 추가한다.

```html
<input type="hidden" name="next" value="{{ next }}">
```

## 23. 로그아웃 상태에서 입력창 비활성화

로그아웃 상태에서는 답변 입력창 자체를 비활성화할 수도 있다.

```html
<textarea
    {% if not user.is_authenticated %}
    disabled
    {% endif %}
></textarea>
```

이렇게 하면 로그인하지 않은 사용자는 답변을 입력할 수 없다.

## 24. 작성자 화면 표시

DB에 작성자 정보가 저장되면 템플릿에서 작성자를 출력할 수 있다.

예를 들어 질문 목록에서 작성자를 보여주려면 다음처럼 쓴다.

```html
{{ question.author.username }}
```

답변 작성자도 마찬가지이다.

```html
{{ answer.author.username }}
```

작성자를 표시하려면 모델만 수정해서는 안 되고, 템플릿도 수정해야 한다.

## 26. 설치와 등록의 차이

| 구분 | 의미                                          |
| -- | ------------------------------------------- |
| 설치 | `pip install django`로 코드가 디스크에 깔리는 것        |

## 27. Django 앱이란?

Django 앱은 단순 폴더가 아니라 Django가 인식해야 하는 기능 단위이다.

앱은 다음 중 하나 이상을 포함할 수 있다.

* `models.py`: DB 테이블을 만들어야 함
* `migrations/`: 스키마 마이그레이션 추적
* `admin.py`: 관리자 사이트에 등록
* `signals`: 시그널 핸들러 연결
* `templates/`,  `static/`: 템플릿, 정적 파일 딜렉토리
* `AppConfig.ready()`: 부팅 시 실행 코드

예를 들어 `pybo`, `common`은 직접 만든 앱이므로 `INSTALLED_APPS`에 등록해야 한다.

```python
INSTALLED_APPS = [
    'pybo',
    'common.apps.CommonConfig',
]
```

### 31.2 signup

```python
from django.contrib.auth import authenticate, login
from django.shortcuts import render, redirect
from common.forms import UserForm

def signup(request):
    if request.method == "POST":
        form = UserForm(request.POST)

        if form.is_valid():
            form.save()

            username = form.cleaned_data.get('username')
            raw_password = form.cleaned_data.get('password1')

            user = authenticate(username=username, password=raw_password)
            login(request, user)

            return redirect('index')
    else:
        form = UserForm()

    return render(request, 'common/signup.html', {'form': form})
```

| 3 | Django의 `User` 모델은 `django.contrib.auth.models`에 있다 |
| 4 | 회원가입은 `UserCreationForm`을 상속해 구현한다 |
| 5 | 로그인/로그아웃은 `LoginView`, `LogoutView`를 사용할 수 있다 |
| 6 | 공통 인증 기능은 `common` 앱으로 분리한다 |
| 7 | 게시글 작성자를 저장하려면 `ForeignKey(User)`를 사용한다 |
| 8 | `on_delete=models.CASCADE`는 사용자 삭제 시 관련 글도 삭제한다 |
| 9 | 모델 수정 후 `makemigrations`, `migrate`가 필요하다 |
| 10 | 기존 데이터가 있으면 non-null 필드 추가 시 기본값 지정이 필요하다 |
| 11 | DB에는 `author`가 아니라 `author_id` 컬럼으로 저장된다 |
| 12 | 새 글 등록 시 `question.author = request.user`가 필요하다 |
| 13 | 로그아웃 상태의 `request.user`는 `AnonymousUser`이다 |
| 14 | `@login_required`로 로그인 사용자만 글을 작성하게 해야 한다 |
| 15 | 작성자 출력은 `question.author.username`으로 한다 |
| 16 | Core 모듈은 import해서 쓰고, contrib 앱은 `INSTALLED_APPS` 등록이 필요하다 |
