## 2. Django Admin

### 2-1. 슈퍼유저 생성

Django Admin을 사용하려면 먼저 슈퍼유저를 생성해야 한다. 

```bash
python manage.py createsuperuser
python manage.py runserver
```

이후 `/admin` 경로로 접속하여 관리자 페이지에 로그인할 수 있다.

### 2-2. Admin 접속 및 기능

Admin에서는 다음 작업이 가능하다. 

* 등록된 사용자와 그룹 조회
* 사용자 정보 수정
* 로그인 오류 메시지 확인

### 2-3. 모델 등록

`Question`, `Answer` 같은 모델을 Admin에 등록하면 웹에서 쉽게 관리할 수 있다. 

```python
from django.contrib import admin
from .models import Question

admin.site.register(Question)
```

### 2-4. 검색 기능 추가

Admin에 검색 기능을 추가하려면 `ModelAdmin`을 사용한다. 

```python
from django.contrib import admin
from .models import Question

class QuestionAdmin(admin.ModelAdmin):
    search_fields = ['subject']

admin.site.register(Question, QuestionAdmin)
```

이렇게 하면 질문 제목(`subject`) 기준 검색이 가능해진다.

## 3. 질문 목록 기능 구현

### 3-1. 질문 조회

질문 목록은 작성일시 역순으로 조회한다. 

```python
question_list = Question.objects.order_by('-create_date')
```

* `order_by('-create_date')`는 최신순 정렬을 의미한다.

### 3-2. 화면 출력

조회한 데이터를 템플릿에 전달해 HTML로 렌더링한다. 

```python
return render(request, 'pybo/question_list.html', context)
```

### 3-3. 템플릿 디렉터리 설정

템플릿 파일을 저장할 디렉터리를 만들고 `settings.py`에 등록해야 한다. 

* 프로젝트 루트 아래 `templates` 디렉터리 생성
* `config/settings.py`의 `DIRS`에 템플릿 경로 등록

### 3-4. 질문 목록 템플릿 작성

`question_list.html`에서 Django 템플릿 문법을 사용해 질문 목록을 출력한다. 

* `{% if %}`
* `{% for %}`
* `{{ question.subject }}`

### 3-5. 주의 사항

템플릿 디렉터리를 새로 만든 뒤에는 서버를 재시작해야 정상적으로 인식된다.

## 4. 질문 상세 기능 구현

### 4-1. URL 매핑 추가

질문 상세 페이지를 위해 URL 패턴을 추가한다. 

```python
path('<int:question_id>/', views.detail, name='detail')
```

* `/pybo/2/` 요청 시 `question_id=2`가 되어 `detail` 함수가 호출된다.

### 4-2. 상세 뷰 작성

전달받은 `question_id`로 해당 질문을 조회한다. 

```python
def detail(request, question_id):
    question = Question.objects.get(id=question_id)
    context = {'question': question}
    return render(request, 'pybo/question_detail.html', context)
```

### 4-3. 상세 템플릿 작성

상세 페이지에서는 질문 제목과 내용을 출력한다. 

```html
<h1>{{ question.subject }}</h1>
<div>
    {{ question.content }}
</div>
```

## 5. 오류 처리와 404 페이지

### 5-1. 기존 방식의 문제

존재하지 않는 질문 ID를 조회하면 예외가 발생한다. 

```python
question = Question.objects.get(id=question_id)
```

### 5-2. `get_object_or_404` 사용

이 문제를 해결하려면 `get_object_or_404`를 사용한다. 

```python
from django.shortcuts import render, get_object_or_404

question = get_object_or_404(Question, pk=question_id)
```

* 객체가 없으면 자동으로 404 페이지를 반환한다.

## 6. URL 하드코딩 문제와 URL 별칭

### 6-1. 하드코딩의 문제

템플릿에 실제 URL을 직접 작성하면 유지보수가 어렵다. 

```html
<a href="/pybo/{{ question.id }}/">
```

URL 구조가 바뀌면 모든 템플릿을 수정해야 한다.

### 6-2. URL 별칭 사용

`urls.py`에 `name`을 부여하고 템플릿에서는 별칭을 사용한다. 

```python
path('', views.index, name='index')
path('<int:question_id>/', views.detail, name='detail')
```

```django
{% url 'detail' question.id %}
```

이 방식은 URL 구조가 바뀌어도 템플릿 수정 부담을 줄여준다.

## 7. URL 네임스페이스

### 7-1. 필요성

여러 앱에서 같은 URL 별칭을 사용하면 충돌이 발생할 수 있다. 

### 7-2. 설정 방법

`pybo/urls.py`에 `app_name`을 지정한다. 

```python
app_name = 'pybo'
```

### 7-3. 사용 방법

템플릿에서는 네임스페이스를 포함해 URL을 참조한다. 

```django
{% url 'pybo:detail' question.id %}
```

## 8. 답변 등록 기능

### 8-1. 답변 표시

질문 상세 화면에서 답변 수와 답변 목록을 출력한다. 

```html
<h4>{{ question.answer_set.count }}개의 답변이 있습니다</h4>
<ul>
{% for answer in question.answer_set.all %}
    <li>{{ answer.content }}</li>
{% endfor %}
</ul>
```

### 8-2. 답변 입력 폼

상세 화면에 답변 작성 폼을 추가한다. 

```html
<form action="{% url 'pybo:answer_create' question.id %}" method="post">
    {% csrf_token %}
    <textarea name="content" id="content" rows="15"></textarea>
    <input type="submit" value="답변등록">
</form>
```

* `{% csrf_token %}`은 위조 요청 방지를 위한 보안 장치다.

### 8-3. URL 매핑

답변 등록을 위한 URL을 추가한다. 

```python
path('answer/create/<int:question_id>/', views.answer_create, name='answer_create')
```

### 8-4. 답변 저장 뷰

폼에서 전달된 답변 내용을 DB에 저장한 뒤 상세 페이지로 이동한다. 

```python
def answer_create(request, question_id):
    question = get_object_or_404(Question, pk=question_id)
    question.answer_set.create(
        content=request.POST.get('content'),
        create_date=timezone.now()
    )
    return redirect('pybo:detail', question_id=question.id)
```

핵심 흐름은 다음과 같다.

1. 질문 객체 조회
2. POST 데이터 읽기
3. 답변 생성
4. 상세 페이지로 리다이렉트

## 9. Celery 기반 Django MQ 오케스트레이션

후반부에서는 Django의 요청 처리와 무거운 후속 작업을 분리하는 구조를 소개한다. 

### 9-1. 개념

* **데이터 오케스트레이션**: 여러 데이터 처리 작업을 순서와 의존성에 맞게 자동 실행
* **MQ(Message Queue)**: 시스템 간 작업 메시지를 큐에 저장하고 비동기 전달

예시 기술:

* RabbitMQ
* Redis
* Kafka
* ActiveMQ
* Amazon SQS

### 9-2. Django + Celery 구조

구성은 다음과 같다. 

* **Producer**: Django API
* **Broker**: RabbitMQ 또는 Redis
* **Worker**: Celery Worker

동작 흐름:

1. 사용자가 주문 생성
2. Django가 DB에 저장
3. 브로커에 작업 메시지 등록
4. Worker가 재고 차감, 이메일 발송, 로그 적재 수행

### 9-3. 작업 메시지 예시

Celery 작업은 JSON 메시지 형태로 큐에 들어간다. 

```json
{
  "task": "tasks.send_email",
  "args": ["user@example.com", "환영합니다", "가입 축하합니다!"],
  "kwargs": {},
  "id": "고유한-태스크-ID"
}
```

## 10. 비판적 사고: 병렬 처리의 문제

### 10-1. 문제 상황

예를 들어 아래 작업을 동시에 실행한다고 가정한다.

* 재고 차감
* 주문 완료 이메일 발송
* 로그 적재

이때 재고 차감이 실패해도 이메일 발송 작업이 이미 큐에 들어갔다면 잘못된 이메일이 발송될 수 있다.

### 10-2. 해결: `chain`

Celery의 `chain`을 사용하면 앞 단계가 성공해야 다음 단계가 실행된다. 

* 재고 차감 성공 후 이메일 발송
* 재고 부족 시 잘못된 이메일 발송 방지
* 대신 전체 처리 시간은 늘어날 수 있음

### 10-3. 추가 패턴: `group`, `chain + group`

강의에서는 다른 패턴도 소개한다. 

#### `group`

* 여러 작업을 병렬 실행
* 서로 의존성이 없을 때 적합
* 하나가 실패해도 다른 작업은 이미 진행될 수 있음

#### `chain + group`

* `deduct_stock` 먼저 실행
* 성공 시 `send_order_email`, `log_analytics`를 병렬 실행
* 실패 시 뒤 작업은 실행되지 않음

이 구조는 **선행 조건 보장 + 후속 작업 병렬 처리**를 동시에 만족시킨다.

## 11. 핵심 요약

### Django 기본 기능

* URL과 View를 연결해 요청을 처리한다.
* ORM으로 데이터를 조회하고 정렬한다.
* 템플릿으로 목록/상세 화면을 출력한다.
* Admin을 통해 모델 데이터를 관리한다.

### 유지보수 관점

* URL 하드코딩은 지양해야 한다.
* URL 별칭과 네임스페이스를 사용하면 유지보수성이 좋아진다.

### 데이터 입력 기능

* 폼, POST, CSRF, redirect를 사용해 답변 등록 기능을 구현할 수 있다.

### 확장 개념

* Celery와 MQ를 활용하면 Django의 요청 처리와 비동기 후속 작업을 분리할 수 있다.
* 병렬 처리만이 정답은 아니며, 작업 의존성을 고려한 `chain`, `group`, `chain+group` 설계가 중요하다.
