# 횡단보도·음향신호기 통합 설계안 수정

**외부 API → 정규화 → DB 적재 → 앱 응답 DTO 변환** 구조

이 방식의 장점은 다음과 같다.

- 외부 공공데이터 API 지연이나 장애의 영향을 줄일 수 있다.
- 앱 요청 시마다 외부 API를 직접 호출하지 않아도 된다.
- 좌표 변환, 매칭, 정제 로직을 미리 수행해둘 수 있다.
- 주기적 갱신 정책을 따로 운영할 수 있다.

서울 대로변 횡단보도 위치정보는 위치·공간 구조 중심이고, 전국 횡단보도 표준데이터는 보행자신호등, 보행자작동신호기, 음향신호기 설치 여부, 점자블록, 보도턱낮춤, 집중조명시설 같은 보행 편의 속성을 제공한다. 서울시 음향신호기 API는 개별 장치의 좌표, 방향, 상태를 제공하므로 횡단보도와 좌표 기반 매칭에 활용할 수 있다.

---

# 1. 전체 구조

```text
외부 API
- 서울 대로변 횡단보도 위치정보
- 전국 횡단보도 표준데이터
- 서울 음향신호기 정보
- 전국 신호등 표준데이터

        ↓ 수집
Batch / Scheduler

        ↓ 정규화
Raw DTO → 내부 통합 모델

        ↓ 저장
Crosswalk / AcousticSignal / CrosswalkSignalMapping

        ↓ 조회
Service

        ↓ 응답
CrosswalkNearbyResponse / CrosswalkDetailResponse
````

즉, 앱이 보는 DTO는 **DB에 저장된 정제 결과**를 바탕으로 만들어지고,
외부 API는 **주기적으로 수집하는 데이터 원천**으로 동작한다.

# 2. 저장 전략

## 2-1. 어떤 데이터를 DB에 저장할지

최소한 아래 3개 축으로 나누는 것이 좋다.

### 1) 횡단보도 기본 정보

횡단보도 자체의 위치와 보행 편의 속성

### 2) 음향신호기 정보

음향신호기 개별 장치 정보

### 3) 매핑 정보

어떤 음향신호기가 어떤 횡단보도와 연결되는지에 대한 결과

이렇게 나누면 나중에 음향신호기 API만 갱신되어도 전체를 다시 엎지 않아도 된다.

# 3. 주기적 DB 업데이트 방식

## 3-1. 기본 방향

외부 API를 앱 요청 시마다 부르지 말고,
`@Scheduled` 또는 배치 작업으로 **정해진 주기마다 동기화**하는 것이 좋다.

추천 방식은 다음과 같다.

### 서울시 데이터

* 하루 1회 새벽 동기화

### 전국 표준데이터

* 하루 1회 또는 주 1회
* 데이터 갱신 주기가 길다면 주 1회도 가능

### 장애 복구용 수동 동기화

* 관리자 API 또는 운영용 버튼으로 강제 재동기화 가능하게 만들기

---

## 3-2. 스케줄러 예시

```java id="x2anpo"
@Component
@RequiredArgsConstructor
public class CrosswalkDataSyncScheduler {

    private final CrosswalkDataSyncService crosswalkDataSyncService;

    @Scheduled(cron = "0 0 3 * * *")
    public void syncDaily() {
        crosswalkDataSyncService.syncAll();
    }
}
```

이 예시는 매일 새벽 3시에 전체 동기화를 수행한다.

---

## 3-3. 동기화 서비스 예시

```java id="d4r0j1"
@Service
@RequiredArgsConstructor
@Transactional
public class CrosswalkDataSyncService {

    private final SeoulCrosswalkCollector seoulCrosswalkCollector;
    private final NationalCrosswalkCollector nationalCrosswalkCollector;
    private final SeoulAcousticSignalCollector seoulAcousticSignalCollector;
    private final CrosswalkMergeService crosswalkMergeService;
    private final CrosswalkAcousticSignalMatchService matchService;

    public void syncAll() {
        seoulCrosswalkCollector.collectAndSave();
        nationalCrosswalkCollector.collectAndSave();
        seoulAcousticSignalCollector.collectAndSave();

        crosswalkMergeService.mergeCrosswalkData();
        matchService.matchCrosswalkAndSignals();
    }
}
```

핵심은 단순 수집과 통합/매칭을 분리하는 것이다.

* `collectAndSave()`: 원천 데이터 수집
* `mergeCrosswalkData()`: 횡단보도 정보 통합
* `matchCrosswalkAndSignals()`: 좌표 기반 매칭

---

## 3-4. 동기화 시 주의할 점

### 전체 삭제 후 재적재는 가급적 피하는 것이 좋다

데이터량이 늘어나면 부담이 커진다.

대신 아래 방식이 낫다.

* 고유 식별자로 upsert
* 갱신 시각 기록
* 이번 배치에 안 들어온 데이터는 비활성 처리 또는 별도 정리

### 원본 데이터 기준일도 저장해야 한다

외부 API에서 `referenceDate`가 오면 함께 저장하는 것이 좋다.
그래야 “최근 동기화는 오늘 했지만, 원본 데이터 기준일은 지난주” 같은 구분이 가능하다.

### 실패해도 앱 조회는 DB 기준으로 계속 가능해야 한다

이번 동기화가 실패하더라도 기존 DB 데이터를 계속 사용하면 된다.
이게 바로 네트워크 영향 최소화의 핵심이다.

# 5. 내부 판단값은 어떻게 다룰 것인지

아래 값들은 개념적으로는 유용하지만, 현재 단계에서는 **기본 응답 DTO에서 제외**하는 것이 더 적절하다.

## 제외 대상

- `blindPedestrianFriendly`
- `CrosswalkMatchDto`
- `match.method`
- `match.distanceMeters`
- `match.confidence`

## 이유

- 모두 원천 공공데이터의 직접 값이 아니라 내부 계산/매칭 결과다.
- 사용자에게 바로 노출할 핵심 정보라기보다 내부 로직, 디버깅, 관리자 확인용 성격이 강하다.
- 나중에 관리자 API나 내부 모니터링 DTO로 분리하는 것이 더 자연스럽다.

# 6. DTO에서 저장하는 값 설명

아래는 각 DTO가 **무슨 값을 담는지**를 밑에서 설명한 것이다.

---

### `CrosswalkNearbyResponse`

주변 횡단보도 목록에서 한 건을 나타내는 응답이다.

#### 저장 값

* `crosswalkId`
	  * 앱과 서버가 해당 횡단보도를 식별하기 위한 ID
	  * 보통 DB의 `crosswalkCode`를 사용
* `distanceMeters`
	  * 현재 사용자 위치와 횡단보도 사이 거리
	  * nearby 조회 시 계산값
* `location`
	  * 횡단보도의 좌표와 행정구역 정보
* `crosswalk`
	  * 목록에서 필요한 최소 횡단보도 속성
	  * 보행자신호등, 점자블록, 보도턱낮춤 여부
* `acousticSignal`
	  * 음향신호기 존재 여부 요약
* `guidance`
	  * 사용자에게 바로 읽어줄 요약 문장

---

### `CrosswalkLocationDto`

### `AcousticSignalDto`

음향신호기 전체 요약 정보를 담는다.

#### 저장 값

* `installed`
	  * 음향신호기 설치 여부
* `count`
	  * 이 횡단보도에 매칭된 음향신호기 개수
* `status`
	  * 전체 상태 요약
	  * 예: `NORMAL`, `PARTIAL`, `UNKNOWN`, `NONE`
* `devices`
	  * 실제 장치 목록

즉, “있다/없다”를 넘어서 몇 개가 있고 상태가 어떤지까지 내려주는 DTO다.

#### status

개별 장치 상태 하나가 아니라 **통합 결과 상태**

`NORMAL`

의미:

- 음향신호기 설치가 확인되었고
- 매칭된 장치 정보가 충분히 있고
- 매칭 신뢰도도 일정 수준 이상인 상태

예:

- 장치 2개 매칭됨
- 거리도 가깝고
- 방향도 대체로 맞음

즉,

> 이 횡단보도에 음향신호기가 있다고 비교적 안정적으로 판단 가능한 상태

---

`PARTIAL`

의미:

- 음향신호기 관련 정보는 일부 확인되지만
- 장치 수가 부족하거나
- 매칭 신뢰도가 애매하거나
- 원천 데이터와 개별 장치 매칭이 완전히 일치하지 않는 상태

예:

- 전국 횡단보도 데이터에는 `Y`인데 서울 음향신호기 장치가 근처에서 1개만 잡힘
- 장치는 잡혔지만 방향 정보가 애매함
- 설치 여부는 보이는데 세부 장치 정보는 부족함

즉,

> 음향신호기가 있을 가능성은 높지만, 정보가 완전하지 않은 상태

---

`UNKNOWN`

의미:

- 음향신호기 설치 여부를 확실히 판단하기 어려운 상태

예:

- 원천 데이터 없음
- 좌표 매칭 실패
- 설치 여부 필드도 없음
- 후보가 너무 많아 확정 불가

즉,

> 있다고도 없다고도 확정하기 어려운 상태

---

`NONE`

의미:

- 음향신호기 없음

---

### `CrosswalkGuidanceDto`

앱이 사용자에게 읽어줄 안내 요약을 담는다.

#### 저장 값

* `summary`
	  * 사용자에게 보여줄 또는 읽어줄 안내 문장

예:

* `음향신호기와 점자블록이 있는 횡단보도`
* `보행자 신호는 있으나 음향신호기 정보는 확인되지 않음`

#### blindPedestrianFriendly 판별 기준

가산 요소

- 음향신호기 있음: +3
- 보행자 신호등 있음: +2
- 점자블록 있음: +2
- 보도턱낮춤 있음: +1
- 매칭 신뢰도 높음: +1

감점 또는 보수 처리

- 핵심 정보 다수 없음: 0
- 음향신호기 정보 불명확: -1
- 보행자 신호등 없음: 크게 감점

최종 판정 예시

- 6점 이상: `true`
- 5점 이하: `false`

### `CrosswalkMatchDto`

횡단보도와 음향신호기 매칭 결과를 설명한다.

#### 저장 값

* `method`
	  * 어떤 방식으로 매칭했는지
	  * 예: `COORDINATE_NEAREST`
* `distanceMeters`
	  * 횡단보도와 음향신호기 간 거리
* `confidence`
	  * 매칭 신뢰도

즉, “이 음향신호기 정보가 얼마나 믿을 만한가”를 나타내는 값이다.

#### method 방식

- `COORDINATE_NEAREST`
	- 가장 가까운 좌표 기준으로 매칭
	- 예: 횡단보도 중심점 반경 10m 내 가장 가까운 음향신호기 선택
	- 가장 기본적인 방식
- `COORDINATE_DIRECTION`
	- 좌표 + 방향 정보까지 함께 고려한 매칭
	- 예: 가까우면서 방향도 일치
	- `COORDINATE_NEAREST`보다 신뢰도 높음
- `ADMINISTRATIVE_AREA`
	- 행정구역 단위 보조 매칭
	- 같은 구/동 단위로만 연결
	- 좌표가 부정확할 때 보조 수단
- `SOURCE_FLAG`
	- 원천 데이터에서 이미 음향신호기 설치 여부만 확인된 경우
	- 예: 전국 횡단보도 표준데이터의 `sondSgngnrYn = Y`
	- 실제 장치 개별 매칭은 못 했지만 “설치 여부”는 확인 가능
- `MANUAL_MAPPING`
	- 운영 중 수동 보정한 경우
	- 특정 교차로나 예외 케이스에서 사람이 직접 연결
- `UNKNOWN`
	- 매칭 방법을 특정하기 어려운 경우

추천

실제로는 초기에 아래 3개만 있어도 충분하다.

- `COORDINATE_NEAREST`
- `COORDINATE_DIRECTION`
- `SOURCE_FLAG`

#### confidence 기준

1) 거리

가장 중요하다.

- 0~5m: 매우 높음
- 5~10m: 높음
- 10~20m: 보통
- 20m 초과: 낮음

2) 행정구역 일치 여부

- 같은 시군구/동이면 가산
- 다르면 감점

3) 방향 정보 일치 여부

- 음향신호기 `direction`과 횡단보도 방향이 어느 정도 맞으면 가산
- 반대 방향이면 감점

4) 매칭 후보 개수

- 근처 후보가 1개뿐이면 신뢰도 상승
- 여러 개가 몰려 있으면 신뢰도 하락

해석 기준 예시

- `0.85 이상`: 매우 높음
- `0.70 이상`: 높음
- `0.50 이상`: 보통
- `0.50 미만`: 낮음

# 7. 추천 구현 원칙

## 조회 API는 DB만 보게 하는 것이 좋다

실제 앱 요청 흐름은 이렇게 가져가는 것이 좋다.

```text
앱 요청
→ CrosswalkQueryService
→ DB 조회
→ DTO 변환
→ 응답
```

외부 API는 이 흐름에서 완전히 분리하는 것이 안정적이다.

---

## 동기화 API와 조회 API를 분리해야 한다

* 조회 API: 앱이 호출
* 동기화 API: 스케줄러 또는 관리자만 호출

이렇게 분리해야 운영이 깔끔하다.

---

## DTO는 응답 전용으로 유지하는 것이 좋다

외부 API 응답 DTO, DB 적재용 내부 DTO, 앱 응답 DTO를 섞지 않는 편이 좋다.

추천 구분은 이렇다.

```text
external.dto
- 외부 API 응답 파싱용

internal.dto
- 통합/매칭/정규화용

response.dto
- 앱 응답용
```

---

# 8. 가장 추천하는 최종 방향

정리하면 가장 좋은 구조는 다음과 같다.

## 저장

* `Crosswalk`
* `AcousticSignal`
* `CrosswalkAcousticSignalMapping`

## 주기적 업데이트

* `@Scheduled` 기반 일 배치
* 필요 시 수동 재동기화 API 추가

## 응답 DTO

* 목록: `CrosswalkNearbyResponse`
* 상세: `CrosswalkDetailResponse`

## 핵심 원칙

* 앱 요청 시 외부 API 직접 호출 금지
* 외부 API 데이터는 먼저 DB에 적재
* 매칭 결과까지 DB에 저장
* DTO는 DB 기반 응답만 담당

이 방향이면 네트워크 불안정성의 영향을 크게 줄이면서도, 주기적으로 최신 데이터로 갱신할 수 있다.
