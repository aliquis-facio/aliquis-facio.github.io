# 전국 횡단보도 표준데이터 조회 API가 TMAP 보행자 경로 안내 API와 겹치는 부분을 제외하고 개발해야 할 내용 정리

TMAP 보행자 경로 안내 API는 기본적으로 **출발지에서 목적지까지의 보행 경로**, **이동 방향**, **회전 시점**, **경로 재탐색** 같은 기능을 담당한다.
반면 전국 횡단보도 표준데이터 조회 API는 **횡단보도 자체의 정적 시설 정보**를 제공한다.

따라서 두 API가 겹치는 부분을 빼고 보면,
전국 횡단보도 표준데이터 조회 API를 붙이면서 추가로 개발해야 할 내용은 결국 다음과 같다.

* **경로 생성이 아니라 횡단보도 시설 해석**
* **경로 안내가 아니라 횡단 안전성 판단**
* **길찾기가 아니라 횡단보도 선택/추천**
* **단순 위치 표시가 아니라 시각장애인용 안내 문장 생성**

즉, 이 API는 TMAP의 길안내를 대체하는 것이 아니라,
**TMAP이 알려준 경로 위의 횡단 구간을 더 안전하게 해석하기 위해 붙이는 보조 데이터 API**로 보는 것이 맞다.

---

# 1. 겹치는 부분과 겹치지 않는 부분 구분

## TMAP 보행자 경로 안내 API가 이미 담당하는 부분

이 부분은 전국 횡단보도 표준데이터 API로 새로 개발할 필요가 없다.

* 출발지~목적지 보행 경로 계산
* 직진 / 좌회전 / 우회전 등 이동 방향 안내
* 전체 거리 / 예상 시간 계산
* 경로 선형 표시
* 경유지 기반 우회 경로 구성
* 경로 이탈 시 재탐색

즉, **“어디로 갈 것인가”** 는 이미 TMAP 쪽 역할이다.

## 전국 횡단보도 표준데이터 API로 추가 개발해야 하는 부분

이 부분이 실제 개발 포인트다.

* 경로 위 또는 경로 근처 횡단보도 후보 추출
* 여러 횡단보도 중 어떤 횡단보도가 더 안전한지 판단
* 음향신호기, 점자블록, 보도턱낮춤 같은 보행약자 친화 요소 반영
* 차로수, 횡단 길이, 교통섬 여부를 반영한 위험도 계산
* ~~신호등 유무에 따라 안내 문장 분기~~
* ~~시각장애인 관점의 음성/진동 안내 생성~~

즉, **“어떤 횡단 지점을 택할 것인가 / 어떻게 설명할 것인가”** 가 새로 개발해야 할 핵심이다.

---

# 2. 개발해야 할 핵심 기능

## 2-1. 경로 위 횡단보도 매칭 기능

TMAP이 내려준 경로에는 이동선이 있고,
횡단보도 API에는 횡단보도 좌표가 있다.

여기서 추가 개발해야 하는 것은:

* 경로 선형 근처의 횡단보도 데이터 수집
* 현재 사용자가 지나야 할 횡단 구간과 실제 횡단보도 데이터 연결
* 같은 교차로 근처 횡단보도 여러 개 중 현재 진행 방향에 맞는 후보 선택

즉, 단순히 “근처 횡단보도 목록”을 보여주는 것이 아니라
**TMAP 경로의 특정 횡단 단계와 횡단보도 시설 데이터를 연결하는 매칭 로직**이 필요하다.

### 개발 포인트

* 경로 좌표와 횡단보도 좌표 간 거리 계산
* 현재 진행 방향과 횡단보도 위치 방향 비교
* 너무 먼 횡단보도 제거
* 후보가 여러 개면 우선순위 점수 계산

---

## 2-2. 횡단보도 안전도 평가 기능

TMAP은 “건너라”를 알려줄 수는 있어도
그 횡단이 **얼마나 안전한가**까지는 직접 평가하지 않는다.

전국 횡단보도 표준데이터 API를 활용하면 아래를 개발할 수 있다.

* 차로수가 많으면 위험도 증가
* 횡단 길이가 길면 위험도 증가
* 보행자신호등이 없으면 위험도 증가
* 음향신호기가 있으면 시각장애인 친화도 증가
* 점자블록이 있으면 기준선 보행 친화도 증가
* 보도턱낮춤이 있으면 보행약자 편의성 증가
* 교통섬이 있으면 긴 횡단에서 중간 대기 가능성 반영

### 개발 포인트

* 횡단보도별 위험도 점수 모델
* 시각장애인 친화도 점수 모델
* “최단 횡단”보다 “안전 횡단”을 우선하는 선택 기준
* 상황에 따라 추천 기준 분리

  * 전맹 사용자 우선 기준
  * 저시력 사용자 우선 기준
  * 휠체어/유모차 등 보행약자 우선 기준

---

# 3. 백엔드에서 개발해야 할 내용

## 3-1. 데이터 수집 및 정규화 계층

전국 횡단보도 표준데이터 API는 정적 공공데이터 성격이 강하므로
매번 앱에서 직접 원본을 두드리기보다 백엔드에서 정규화하는 것이 좋다.

### 개발 내용

* 공공데이터 API 호출 모듈
* 페이지네이션 처리
* JSON/XML 파싱
* 필드명 내부 표준화
* `Y/N`, 공란값, 숫자형 변환 처리
* 잘못된 좌표 / 누락 데이터 예외 처리
* 기준일자 기준 캐시 관리

---

## 3-2. 내부 횡단보도 모델 설계

앱에서 직접 원본 필드를 다 쓸 필요는 없다.
백엔드에서 내부 모델로 바꿔 주는 것이 좋다.

### 예시 내부 모델

* 횡단보도 ID
* 위치 좌표
* 횡단보도 종류
* 차로수
* 횡단 길이
* 신호등 유무
* 음향신호기 유무
* 점자블록 유무
* 보도턱낮춤 유무
* 교통섬 유무
* 위험도 점수
* 시각장애인 친화도 점수

즉, 원본 API를 그대로 노출하지 말고
**서비스 목적에 맞는 가공 모델**을 만들어야 한다.

---

## 3-3. TMAP 경로와 결합하는 후처리 계층

이게 가장 중요하다.

### 개발 내용

* TMAP 경로 단계 중 횡단 관련 구간 추출
* 앱에 최종 안내 단계로 전달

즉:

* TMAP = 길의 뼈대
* 횡단보도 API = 횡단 구간 속성
* 백엔드 = 둘을 붙여서 의미 있는 안내로 재구성

---

## 3-4. 위험도/추천 엔진

### 개발 내용

* 정적 위험 점수 계산식
* 친화도 점수 계산식
* 횡단 추천 우선순위 로직
* 사용성 테스트 기반 가중치 조정
* 향후 AI 결과 반영 가능한 구조 설계

예시 기준:

* 차로수 많음 → 위험 가중치 증가
* 횡단 길이 김 → 위험 증가
* 신호등 없음 → 위험 크게 증가
* 음향신호기 있음 → 친화도 증가
* 점자블록 있음 → 친화도 증가
* 보도턱낮춤 있음 → 편의성 증가
* 교통섬 있음 → 긴 횡단 시 위험 일부 완화

---

# 4. 앱에서 개발해야 할 내용

## 4-1. 횡단보도 상세 안내 UI/UX

TMAP 기본 경로 안내만으로는 부족하다.
앱에서는 횡단 구간에서 별도 안내가 필요하다.

### 개발 내용

* 횡단보도 접근 시 별도 음성 안내
* 횡단 시작 전 주의 안내
* 음향신호기 유무 안내
* 점자블록 진입부 안내
* 긴 횡단 / 무신호 횡단 경고
* 교통섬 존재 시 중간 대기 안내

---

## 4-2. 진동 패턴 분기

횡단 관련 정보는 음성뿐 아니라 진동으로도 구분하면 좋다.

### 개발 내용

* 횡단보도 접근 진동
* 주의 횡단보도 진동
* 신호등 없는 횡단 경고 진동
* 안전한 횡단 추천 진동
* 재탐색 필요 시 진동

---

## 4-3. 사용자 설정 기능

사용자마다 우선하는 기준이 다를 수 있다.

### 개발 내용

* 음향신호기 우선
* 점자블록 우선
* 짧은 횡단 우선
* 교통섬 있는 횡단 우선
* 무신호 횡단 회피
* 야간 안전 우선

이 설정이 있으면 TMAP 기본 경로 위에서도
횡단 구간에서 더 개인화된 안내가 가능해진다.

---

# 6. 실제로 개발 목록만 압축하면

## 백엔드

* 횡단보도 공공데이터 수집 모듈
* 페이지네이션 및 캐시 처리
* 내부 횡단보도 데이터 모델 설계
* TMAP 경로와 횡단보도 데이터 매칭 로직
* 위험도 점수 계산 로직
* 안전한 횡단보도 추천 로직
* 사용자 설정 기반 우선순위 반영 로직

## 앱

* 횡단보도 접근 음성 안내
* 횡단 위험도별 경고 UX
* 점자블록/음향신호기 안내 UX
* 진동 패턴 설계
* 사용자 선호 설정 화면
* 재안내 및 재탐색 흐름
