# 서울시 대로변 횡단보도 위치정보 정리

<font color="#ff0000">횡단보도를 공간정보 객체로 본다 → GIS/위치 분석 정보 중심이다 → 사용 O</font>
좌표계: WGS84

## 1. API 개요

이 API는 **서울시내 대로변 횡단보도 위치 공간정보**를 제공하는 Open API이다.
서비스명은 **`tbTraficCrsng`** 이며, 서울 열린데이터 광장에서 발급받은 인증키를 사용해 호출한다. 

즉, 이 API는 일반적인 횡단보도 시설 상세 정보보다는 **횡단보도의 위치와 공간 구조 정보**를 제공하는 데 초점이 있다.
특히 노드와 링크 기반의 공간 데이터가 포함되어 있어, 지도 표시나 경로 분석용 데이터로 활용하기 적합하다. 

---

## 2. 호출 형식

샘플 URL은 다음과 같이 제시되어 있다. 

```text
http://openapi.seoul.go.kr:8088/(인증키)/xml/tbTraficCrsng/1/5/
```

일반화하면 호출 구조는 다음과 같다.

```text
http://openapi.seoul.go.kr:8088/{KEY}/{TYPE}/{SERVICE}/{START_INDEX}/{END_INDEX}/
```

여기서 `SERVICE` 값은 반드시 `tbTraficCrsng`를 사용해야 한다. 

---

## 3. 요청 인자

이 API의 요청 인자는 비교적 단순하다. 별도의 검색 조건 없이 범위 기반 조회를 수행하는 구조다. 

### 필수 요청값

* `KEY`
  서울 열린데이터 광장에서 발급받은 인증키

* `TYPE`
  응답 형식
  지원 형식:

  * `xml`
  * `xmlf`
  * `xls`
  * `json`

* `SERVICE`
  서비스명
  값: `tbTraficCrsng`

* `START_INDEX`
  조회 시작 위치

* `END_INDEX`
  조회 종료 위치

### 특징

이 API는 앞서 보았던 다른 횡단보도 API와 달리, 선택 조회용 파라미터가 따로 보이지 않는다.
즉, **페이지 범위를 지정해 전체 목록을 나눠 조회하는 방식**으로 사용하는 API라고 볼 수 있다. 

---

## 4. 응답 구조

응답은 공통 정보와 데이터 본문으로 구성된다. 

### 공통 응답값

* `list_total_count` : 총 데이터 건수
* `RESULT.CODE` : 요청 결과 코드
* `RESULT.MESSAGE` : 요청 결과 메시지

즉, 요청이 성공했는지, 실패했는지, 실패했다면 왜 실패했는지를 공통 필드에서 먼저 확인할 수 있다. 

---

## 5. 출력 항목 정리

이 API의 본문 데이터는 **노드-링크 기반 횡단보도 위치 정보**를 담고 있다. 주요 출력값은 다음과 같다. 

### 노드 정보

* `NODE_TYPE` : 노드링크 유형
* `NODE_WKT` : 노드 WKT
* `NODE_ID` : 노드 ID
* `NODE_TYPE_CD` : 노드 유형 코드

### 링크 정보

* `LNKG_WKT` : 링크 WKT
* `LNKG_ID` : 링크 ID
* `LNKG_TYPE_CD` : 링크 유형 코드
* `BGNG_LNKG_ID` : 시작노드 ID
* `END_LNKG_ID` : 종료노드 ID
* `LNKG_LEN` : 링크 길이

### 행정구역 정보

* `SGG_CD` : 시군구코드
* `SGG_NM` : 시군구명
* `EMD_CD` : 읍면동코드
* `EMD_NM` : 읍면동명

---

## 6. 각 항목의 의미

이 API는 일반 사용자가 보기에는 다소 GIS 중심 용어가 많다. 쉽게 풀면 다음과 같다.

### 노드

횡단보도 위치를 점 단위로 표현한 정보다.
`NODE_ID`, `NODE_WKT` 등을 통해 **특정 횡단보도의 기준 위치점**을 알 수 있다.

### 링크

횡단보도를 선 형태 또는 연결 구조로 표현한 정보다.
`LNKG_WKT`, `LNKG_LEN`, 시작노드/종료노드 정보를 통해 **횡단보도의 연결 방향과 길이 정보**를 파악할 수 있다.

### WKT

`NODE_WKT`, `LNKG_WKT`에 들어가는 WKT는 공간정보를 문자열로 표현하는 형식이다.
즉, 지도 시스템이나 GIS 도구에서 바로 읽을 수 있는 좌표 표현 방식이라고 보면 된다. 

### 행정구역 코드

시군구와 읍면동 코드 및 이름이 함께 제공되므로,
횡단보도를 **지역별로 필터링하거나 구 단위 통계**를 낼 때 활용할 수 있다. 

---

## 7. 이 API로 할 수 있는 것

이 데이터는 다음과 같은 용도로 쓰기 좋다.

### 7-1. 횡단보도 위치 시각화

`NODE_WKT`, `LNKG_WKT`를 이용해 지도 위에 횡단보도 위치를 표시할 수 있다. 

### 7-2. 경로 상 횡단보도 분석

보행 경로와 횡단보도 링크 데이터를 비교해,
경로 주변 또는 경로 상의 횡단보도를 찾는 데 활용할 수 있다.

### 7-3. 지역별 횡단보도 관리

`SGG_CD`, `SGG_NM`, `EMD_CD`, `EMD_NM`을 기준으로
구별·동별 횡단보도 위치를 분류할 수 있다. 

### 7-4. 보행 보조 서비스 연동

프로젝트 관점에서는 사용자의 이동 경로 주변에 어떤 횡단보도가 있는지 확인하고,
향후 신호등 정보나 음성신호기 정보와 결합하는 기초 데이터로 사용할 수 있다.

## 8. 오류 코드 및 메시지

이 API는 서울 열린데이터 광장 공통 형식의 오류 코드를 사용한다. 주요 항목은 다음과 같다. 

### 정상/데이터 없음

* `INFO-000` : 정상 처리
* `INFO-200` : 해당 데이터 없음

### 요청값 오류

* `ERROR-300` : 필수 값 누락
* `ERROR-301` : 파일 타입 값 누락 또는 유효하지 않음
* `ERROR-310` : 서비스명 오류
* `ERROR-331` : 요청시작위치 값 오류
* `ERROR-332` : 요청종료위치 값 오류
* `ERROR-333` : 요청 위치 값 타입 오류
* `ERROR-334` : 시작 위치가 종료 위치보다 큼

### 조회 범위 제한

* `ERROR-335` : 샘플키는 한 번에 최대 5건까지만 조회 가능
* `ERROR-336` : 실제 데이터 요청도 한 번에 최대 1000건까지만 가능

### 서버/DB 오류

* `ERROR-500` : 서버 오류
* `ERROR-600` : 데이터베이스 연결 오류
* `ERROR-601` : SQL 문장 오류

## 9. 구현 시 주의할 점

### 9-1. 반복 조회 필요

한 번에 최대 1000건까지만 받을 수 있으므로,
전체 데이터를 수집하려면 `START_INDEX`, `END_INDEX`를 바꿔가며 반복 호출해야 한다. 

### 9-2. 샘플 조회와 실제 조회 구분

샘플 URL은 `sample` 키 기준이며 최대 5건까지만 볼 수 있다.
실제 서비스에서는 반드시 발급받은 인증키를 사용해야 한다. 

### 9-3. GIS 처리 필요

핵심 데이터가 `WKT` 형식이므로,
지도 시각화나 좌표 계산을 하려면 이를 해석할 수 있는 로직이나 라이브러리가 필요하다.
즉, 단순 JSON 필드 처리만으로는 부족하고 공간 데이터 처리가 필요하다.

### 9-4. 좌표계 검증 필요

문서에는 WKT 필드가 보이지만 좌표계 종류 자체는 이 텍스트에 직접 나타나지 않는다.
따라서 실제 지도 서비스와 붙일 때는 응답 예시나 원본 데이터 설명에서 좌표계를 한 번 더 확인하는 것이 안전하다.

## 10. 한 줄 요약

이 API는 **서울시내 대로변 횡단보도의 위치를 노드와 링크 기반 공간정보 형태로 제공하는 Open API**이며, `tbTraficCrsng` 서비스명으로 호출한다. 요청은 인증키와 조회 범위만 지정하면 되고, 응답에는 횡단보도의 노드·링크 WKT, ID, 길이, 시군구·읍면동 정보가 포함된다. 따라서 지도 표시, 경로 분석, 보행 보조 서비스의 횡단보도 위치 인식용 기초 데이터로 활용하기에 적합하다.
