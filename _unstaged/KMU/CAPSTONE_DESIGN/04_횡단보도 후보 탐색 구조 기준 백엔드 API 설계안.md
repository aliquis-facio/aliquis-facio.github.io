# 스프링 기준으로 개발해야 할 내용 정리

지금 정리한 범위에서 **AI 파트**와 **프론트 파트**를 제외하면,
스프링 백엔드가 맡아야 하는 핵심은 아래 4가지다.

1. **횡단보도 공공데이터 수집 및 정규화**
2. **TMAP 경로와 횡단보도 데이터 매칭**
3. **횡단보도 위험도 / 추천 점수 계산**
4. **앱에 내려줄 백엔드 응답 구조 설계**

즉, 스프링은 단순히 외부 API를 중계하는 서버가 아니라,
**외부 API 데이터를 서비스 목적에 맞게 가공하고 판단하는 서버**로 설계하는 것이 맞다.

---

# 1. 스프링 백엔드의 역할

TMAP과 공공데이터 API를 그대로 앱에 붙이면 문제가 있다.

* 앱이 외부 API 구조에 직접 의존하게 됨
* 응답 형식이 제각각임
* 횡단보도와 경로를 연결하는 판단 로직이 앱에 퍼짐
* 위험도 계산 기준이 일관되지 않음
* 캐시, 재시도, 예외 처리, 데이터 정제가 어려움

그래서 스프링 백엔드는 아래 역할을 맡아야 한다.

* 외부 API 호출
* 원본 데이터 정리
* 내부 표준 모델로 변환
* TMAP 경로와 횡단보도 연결
* 위험도 및 추천 점수 계산
* 앱이 바로 쓸 수 있는 결과 형태로 응답

---

# 2. 개발 범위 정리

## 2-1. 횡단보도 공공데이터 수집 모듈

전국 횡단보도 표준데이터 조회 API는 정적 공공데이터 성격이 강하므로,
스프링에서 안정적으로 수집하고 내부 형식으로 바꿔야 한다.

### 개발할 내용

* 공공데이터 API 호출 클라이언트
* 인증키 관리
* 요청 파라미터 조합
* 페이지네이션 처리
* JSON/XML 파싱
* 응답 실패 시 재시도 또는 예외 처리
* 원본 필드 검증
* 누락값 처리
* 숫자형 변환
* `Y/N` 값을 boolean으로 변환
* 기준일자 기반 갱신 정책 관리

### 왜 필요한가

공공데이터 원본은 서비스에서 바로 쓰기에는 불안정할 수 있다.

예를 들면:

* 빈 값이 섞일 수 있음
* 같은 의미인데 표기 방식이 다를 수 있음
* 좌표가 비정상일 수 있음
* 문자열 숫자가 섞여 있을 수 있음

그래서 스프링에서 **정규화 계층**을 두는 것이 필수다.

---

## 2-2. 내부 횡단보도 모델 설계

외부 API 필드를 그대로 앱에 노출하지 말고,
서비스 목적에 맞게 내부 모델을 따로 만들어야 한다.

### 예시 필드

* 횡단보도 ID
* 위도
* 경도
* 횡단보도 종류
* 차로수
* 횡단 길이
* 보행자 신호등 유무
* 음향신호기 유무
* 점자블록 유무
* 보도턱낮춤 유무
* 교통섬 유무
* 위험도 점수
* 친화도 점수

### 스프링에서 해야 할 일

* DTO 설계
* Entity 설계 여부 결정
* 외부 응답 → 내부 모델 매핑
* null 허용 범위 정의
* 필수값 검증 규칙 정의

즉, 핵심은 **원본 API 중심 설계가 아니라 서비스 중심 설계**다.

---

## 2-3. TMAP 경로와 횡단보도 데이터 매칭 로직

이게 가장 중요한 백엔드 로직이다.

TMAP은 경로를 주고,
횡단보도 API는 횡단보도 위치와 속성을 준다.

스프링은 이 둘을 연결해야 한다.

### 개발할 내용

* TMAP 경로 응답 파싱
* 경로 좌표 목록 추출
* 횡단 관련 구간 식별
* 경로 주변 횡단보도 후보 검색
* 거리 기준 후보 필터링
* 진행 방향 기준 후보 정제
* 여러 후보 중 최적 횡단보도 선택

### 핵심 판단 로직

* 경로와 너무 먼 횡단보도 제거
* 현재 이동 흐름과 맞지 않는 횡단보도 제거
* 안전 점수가 더 높은 횡단보도 우선
* 같은 교차로 주변 후보 중 최종 1개 선택

즉, 스프링은 단순 조회가 아니라
**경로 단계와 횡단보도 후보를 연결하는 매칭 엔진** 역할을 해야 한다.

---

## 2-4. 위험도 계산 로직

TMAP은 “건너는 지점”은 줄 수 있어도
“그 횡단이 얼마나 안전한가”는 직접 계산하지 않는다.

이 부분은 스프링에서 점수화해야 한다.

### 반영할 요소

* 차로수
* 횡단 길이
* 보행자신호등 유무
* 음향신호기 유무
* 점자블록 유무
* 보도턱낮춤 유무
* 교통섬 유무

### 예시 방향

* 차로수 많음 → 위험도 증가
* 횡단 길이 김 → 위험도 증가
* 신호등 없음 → 위험도 크게 증가
* 음향신호기 있음 → 친화도 증가
* 점자블록 있음 → 친화도 증가
* 보도턱낮춤 있음 → 편의성 증가
* 교통섬 있음 → 긴 횡단 시 위험 일부 완화

### 스프링에서 해야 할 일

* 점수 계산 정책 클래스 분리
* 가중치 상수 관리
* 정책 변경이 쉬운 구조 설계
* 테스트 가능한 순수 계산 로직으로 분리

여기서 중요한 건,
점수 계산 로직을 Controller나 Service 한 곳에 뭉개지 말고
**정책 계층**으로 분리하는 것이다.

---

## 2-5. 추천 우선순위 로직

위험도 계산만으로는 부족하다.
후보가 여러 개일 때 어떤 횡단보도를 최종 선택할지 정해야 한다.

### 개발할 내용

* 위험도 + 친화도 + 거리 기반 종합 점수 계산
* 가장 가까운 횡단보도와 가장 안전한 횡단보도 중 우선순위 정의
* 사용자 설정값 반영 가능 구조 설계

예를 들면:

* 음향신호기 있는 횡단보도 우선
* 점자블록 있는 횡단보도 우선
* 무신호 횡단보도 후순위
* 긴 횡단보도 후순위

즉, 단순 정렬이 아니라
**서비스 목적에 맞는 추천 정책**이 필요하다.

---

## 2-6. 캐시 및 저장 전략

공공데이터 API를 요청마다 직접 호출하면 비효율적일 가능성이 크다.

그래서 스프링에서 캐시 또는 저장 전략이 필요하다.

### 선택지

## 메모리 캐시

* 간단함
* 개발 초기 빠름
* 서버 재시작 시 데이터 유실

## Redis 캐시

* 조회 속도 빠름
* TTL 관리 쉬움
* 운영 환경에서 유리함

## DB 저장

* 지역별 횡단보도 데이터 장기 저장 가능
* 기준일자 관리 가능
* 후처리 및 분석에 유리함

### 추천 방향

MVP라면:

* 우선 DB 저장 또는 Redis 캐시 중 하나 선택
* 최소한 지역 단위 캐시는 두는 것이 좋다

이유는 다음과 같다.

* 공공데이터는 자주 바뀌지 않음
* 같은 지역 재조회가 많을 수 있음
* 매 요청마다 외부 API 호출은 낭비가 큼

---

# 3. 스프링 계층별로 나누면

## Controller

역할은 단순해야 한다.

### 담당

* 요청 받기
* 입력 검증
* Service 호출
* 응답 반환

### 하지 말아야 할 것

* 외부 API 직접 호출
* 점수 계산
* 경로-횡단보도 매칭 로직 작성

---

## Service

핵심 비즈니스 로직이 들어간다.

### 담당

* 경로 조회 orchestration
* 횡단보도 조회 orchestration
* 매칭 로직 호출
* 추천 로직 호출
* 최종 응답 조립

예를 들면:

* `RouteCrosswalkService`
* `CrosswalkSelectionService`
* `CrosswalkRiskScoringService`

처럼 역할을 나누는 것이 좋다.

---

## External Client

외부 API 연동 전용 계층이다.

### 담당

* TMAP API 호출
* 공공데이터 API 호출
* 요청/응답 매핑
* 네트워크 예외 처리

### 권장 방식

* `WebClient` 또는 `RestClient`
* 타임아웃 설정
* 재시도 정책
* 로깅
* 실패 응답 매핑

---

## Domain / Policy

추천, 위험도, 매칭 규칙은 여기로 분리하는 것이 좋다.

### 담당

* 위험도 계산
* 친화도 계산
* 후보 필터링
* 추천 정렬 정책

이 계층을 따로 두면 테스트가 쉬워진다.

---

## Repository

선택 사항이지만 사실상 필요할 가능성이 높다.

### 담당

* 캐시된 횡단보도 데이터 조회
* 저장된 횡단보도 데이터 관리
* 지역 단위 검색
* 기준일자 관리

# 5. API 관점에서 스프링이 제공해야 할 내부 엔드포인트

프론트를 제외하더라도, 백엔드가 어떤 형태의 내부 API를 제공할지는 정리해두는 것이 좋다.

## 5-1. 경로 기반 횡단보도 후보 조회

```text
POST /api/routes/crosswalks/analyze
```

### 역할

* 출발지/목적지 또는 TMAP 경로 결과를 받아
* 경로 위 횡단보도 후보를 분석해서 반환

### 응답 예시 항목

* 횡단보도 ID
* 위치
* 위험도
* 친화도
* 추천 여부
* 선택 이유

---

## 5-2. 특정 지역 횡단보도 동기화

```text
POST /api/admin/crosswalks/sync
```

### 역할

* 공공데이터 API에서 지역별 횡단보도 정보를 수집하여 저장

운영용 또는 관리자용으로 쓰기 좋다.

---

## 5-3. 횡단보도 상세 조회

```text
GET /api/crosswalks/{id}
```

### 역할

* 특정 횡단보도의 내부 가공 데이터 반환

---

# 6. 구현 우선순위

## 1단계

* 공공데이터 API 연동
* 내부 DTO/모델 설계
* 기본 파싱 및 정규화
* 위험도 계산 로직 구현

## 2단계

* TMAP 경로 연동
* 경로-횡단보도 매칭 구현
* 후보 선택 로직 구현

## 3단계

* 캐시 또는 저장소 도입
* 사용자 설정 기반 추천 분기 반영
* 운영 예외 처리 강화

---

# 7. 스프링으로 개발할 때 주의할 점

## 외부 API 응답을 믿고 바로 쓰면 안 됨

반드시 해야 한다.

* null 처리
* 숫자 변환 예외 처리
* 좌표 유효성 검사
* 문자열 정규화

---

## 정책 로직을 하드코딩하지 말아야 함

위험도 계산식과 추천 기준은 나중에 바뀔 가능성이 크다.
그래서 상수, 정책 객체, 설정값으로 분리해야 한다.

---

## 외부 API 호출과 비즈니스 로직을 섞지 말아야 함

예를 들어 Service 안에서 직접 `WebClient`로 호출하고 바로 점수 계산까지 하면 유지보수가 어려워진다.
연동 계층과 판단 계층을 분리해야 한다.

---

## 처음부터 너무 복잡하게 만들 필요는 없음

MVP 단계에서는 아래만 먼저 만들어도 충분하다.

* 횡단보도 수집
* 위험도 계산
* 경로와 횡단보도 매칭
* 추천 1건 반환

즉, 처음부터 완벽한 추천 엔진보다
**일관된 데이터 흐름**을 먼저 만드는 것이 더 중요하다.

---

# 8. 최종 정리

AI와 프론트를 제외하고 스프링이 맡아야 할 개발 내용은 아래로 정리할 수 있다.

## 반드시 개발할 것

* 전국 횡단보도 표준데이터 API 연동
* 응답 파싱 및 정규화
* 내부 횡단보도 모델 설계
* TMAP 경로와 횡단보도 매칭 로직
* 위험도 계산 로직
* 추천 우선순위 로직
* 앱에 내려줄 응답 DTO 설계

## 있으면 좋은 것

* 지역별 캐시
* DB 저장
* 관리자용 동기화 API
* 설정 기반 추천 정책 분리
* 외부 API 장애 대응 로직

## 한 줄로 요약하면

스프링 백엔드는
**“공공 횡단보도 데이터와 TMAP 경로 데이터를 수집하고, 이를 결합해 가장 적절한 횡단보도 후보를 계산해서 앱이 바로 사용할 수 있는 형태로 내려주는 판단 서버”**
로 설계하면 된다.

원하면 다음 단계로 이어서
**스프링 기준 클래스 설계**, **DTO 예시**, **Service 흐름도**, **DB 테이블 초안**까지 바로 정리해줄게.
