# 1. RAG란?

**RAG**는 **Retrieval-Augmented Generation**, 즉 **검색 증강 생성**이다.

쉽게 말하면:

> 사용자의 질문과 관련 있는 문서 조각을 먼저 찾고, 그 내용을 근거로 AI가 답변을 생성하는 방식

기존 챗봇은 모델이 이미 학습한 내용만 답할 수 있다. 반면 RAG는 외부 문서, 내부 자료, 최신 문서 등을 검색해서 답변에 반영할 수 있다.

## RAG의 3단계

|단계|의미|
|---|---|
|Retrieval|질문과 관련 있는 문서 조각을 검색|
|Augmented|검색한 내용을 프롬프트에 추가|
|Generation|모델이 문서 내용을 근거로 답변 생성|

핵심 문장으로 정리하면:

> RAG는 “질문과 비슷한 내 문서 조각을 찾아서, 그걸 근거로 모델이 답하게 하는 방식”이다.

# 2. 이 과정의 전체 흐름

이 과정은 **콘솔 체험 → 이론 이해 → 코드 구현** 순서로 진행된다.

|단계|목적|
|---|---|
|Lab 01~02|AWS 콘솔에서 관리형 RAG를 먼저 체험|
|Lab 03|RAG 내부 원리 이해|
|Lab 04~05|boto3와 Python으로 직접 구현|

처음부터 코드를 작성하지 않고 콘솔부터 시작하는 이유는, RAG의 구성 요소인 **청킹, 임베딩, 벡터 검색, 프롬프트 조립**을 먼저 눈으로 확인하기 위해서다.

결론적으로 이 과정에서 강조하는 핵심은 다음과 같다.

> RAG는 마법이 아니라 **임베딩 + 벡터 검색 + 프롬프트 조립의 합**이다.

# 3. Lab별 핵심 내용

## Lab 01: 콘솔 관리형 RAG

AWS 콘솔에서 **Bedrock Knowledge Base**를 만들고, 문서를 기반으로 답변하는 RAG 챗봇을 구성한다.

주요 작업은 다음과 같다.

|단계|내용|
|---|---|
|데이터 업로드|2019~2022 아마존 주주서한 PDF를 S3에 업로드|
|Knowledge Base 생성|S3 데이터를 연결하고 Fixed-size 청킹 설정|
|임베딩 생성|Titan Text Embeddings V2 사용|
|벡터 저장|Amazon S3 Vectors 사용|
|테스트|Retrieval only와 Retrieval + Generation 비교|

여기서 중요한 점은 콘솔이 자동으로 **문서 청킹, 임베딩 생성, 벡터 저장, 검색**을 처리한다는 것이다.

---

## Lab 02: Guardrail 적용

**Guardrail**은 AI 입력과 출력을 제어하는 안전장치다.

주요 기능은 다음과 같다.

|기능|설명|
|---|---|
|콘텐츠 필터|Hate, Insults, Sexual, Violence, Misconduct 등 유해 콘텐츠 차단|
|금지 주제 설정|예: 투자 조언 차단|
|PII 보호|개인정보 관련 내용 제어|
|환각 억제|문서에 없는 내용을 답하지 않도록 제한|

Guardrail 적용 전과 후를 비교하면서, 정상 질문은 통과하고 금지된 질문만 차단되는지 확인한다.

---

## Lab 03: RAG 이론

콘솔에서 자동으로 일어난 일을 이론적으로 분해한다.

|개념|설명|
|---|---|
|임베딩|텍스트를 의미를 담은 숫자 벡터로 변환|
|벡터 검색|질문 벡터와 문서 벡터의 유사도를 계산|
|코사인 유사도|두 벡터가 얼마나 비슷한지 측정하는 방식|
|청킹|긴 문서를 작은 조각으로 나누는 작업|
|메타데이터|출처, 날짜, 문서 정보 등을 검색 조건으로 활용|
|한계|검색된 청크가 부정확하면 답변도 부정확해짐|

특히 **검색이 틀리면 생성도 틀릴 수 있다**는 점이 중요하다.

## Lab 04: 관리형 KB 코드 호출

콘솔에서 테스트하던 기능을 코드로 호출한다.

여기서 중요한 구분은 다음과 같다.

|클라이언트|역할|
|---|---|
|bedrock-runtime|모델 직접 호출|
|bedrock-agent-runtime|Knowledge Base 호출|

주요 API는 두 가지다.

|API|설명|
|---|---|
|retrieve()|검색만 수행하고 관련 문서 청크 반환|
|retrieve_and_generate()|검색 후 답변 생성까지 수행|

즉, 콘솔의 Test 패널은 실제로는 `retrieve`와 `retrieve_and_generate` API를 시각적으로 제공한 GUI라고 볼 수 있다.

## Lab 05: from-scratch 미니 RAG

마지막 단계에서는 RAG를 Python으로 직접 구현한다.

|함수|역할|
|---|---|
|embed()|Titan 모델을 호출해 텍스트를 벡터로 변환|

|search()|질문과 가장 가까운 문서 청크 검색|
|rag_answer()|검색된 청크를 프롬프트에 넣고 답변 생성|

이 실습을 통해 관리형 서비스가 내부적으로 어떤 일을 하는지 직접 확인한다.

# 4. 사용하는 AWS 서비스

|서비스|용도|
|---|---|
|Amazon Bedrock Knowledge Bases|관리형 RAG 구성|
|Amazon S3|원본 문서 저장|

S3는 원본 문서를 저장하는 장소이고, Knowledge Base는 이 문서를 가져와 자동으로 청킹, 임베딩, 벡터 저장을 수행한다.

# 5. 콘솔과 코드의 대응 관계

이 과정의 핵심은 콘솔에서 자동으로 처리한 작업을 코드로 다시 구현해 보는 것이다.

|콘솔에서 자동 처리|코드에서 직접 구현|
|---|---|
|Fixed-size 청킹|문단 단위 청킹|
|Titan 임베딩|`embed()` 함수|
|S3 Vectors 검색|`INDEX`와 `cosine()`|
|RetrieveAndGenerate|boto3 API 호출|

즉, 콘솔에서 클릭 몇 번으로 완성한 RAG는 코드로 보면 다음 구조다.

```text
문서 업로드
→ 문서 청킹
→ 임베딩 생성
→ 벡터 저장
→ 질문 임베딩
→ 유사 문서 검색
→ 검색 결과를 프롬프트에 삽입
→ 모델이 답변 생성
```

# 6. 핵심 키워드 정리

|키워드|의미|
|---|---|
|RAG|검색한 문서를 근거로 답변을 생성하는 방식|
|Retrieval|관련 문서 조각 검색|

|Vector Search|벡터 간 유사도를 기반으로 검색|
|Cosine Similarity|벡터 유사도 계산 방식|
|Knowledge Base|AWS Bedrock의 관리형 RAG 기능|
|Guardrail|AI 입력/출력 안전장치|
|S3|원본 문서 저장소|
|S3 Vectors|벡터 저장소|
|Titan Text Embeddings V2|임베딩 모델|
|retrieve()|검색만 수행|
|retrieve_and_generate()|검색 후 답변 생성|
