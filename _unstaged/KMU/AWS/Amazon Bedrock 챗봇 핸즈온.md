## 1. 전체 주제

즉, 학습 흐름은 다음과 같다.

```text
invoke_model 단일 턴
→ 멀티턴 대화
→ 스트리밍
→ Converse
→ 세 API 비교
```

---

## 2. Amazon Bedrock이란?

Amazon Bedrock은 여러 회사의 생성형 AI 기반 모델, 즉 Foundation Model을 **하나의 AWS API로 호출할 수 있게 해 주는 완전관리형 서비스**다.

직접 GPU 서버를 구축하거나 모델 가중치를 내려받을 필요 없이, `boto3`로 Bedrock API를 호출하면 AWS가 모델 실행을 대신 처리한다.

중요한 호출 방식은 두 가지다.

|구분|설명|
|---|---|
|`invoke_model`|저수준 API. 모델별 요청 body를 직접 만들고 응답 JSON도 직접 파싱해야 한다.|
|`Converse`|고수준 API. 모델이 달라도 표준 형식으로 호출할 수 있다. 모델 교체가 쉽다.|

---

## 3. 실습 목표

이 실습의 목표는 “챗봇 UI 만들기”가 아니라 **Bedrock 모델 호출 로직을 이해하는 것**이다.

채팅 화면, 입력 처리, 출력 루프는 제공 모듈인 `chat_ui.py`가 담당한다. 학습자는 각 Lab 파일의 `respond()` 또는 `respond_stream()` 함수 안에서 Bedrock 호출과 응답 파싱 부분만 구현한다.

제공되는 핵심 모듈은 다음과 같다.

|파일|역할|
|---|---|
|`chat_ui.py`|콘솔 기반 챗봇 UI 담당|
|`config.py`|리전, 모델 ID, Bedrock 클라이언트 생성 담당|
|Lab 파일|학습자가 TODO 부분을 채워 Bedrock 호출 구현|

## 5. Lab 01 — `invoke_model` 단일 턴

가장 기본적인 Bedrock 호출 방식이다.

핵심 구조는 다음과 같다.

```text
요청 body(JSON)를 만든다
→ Bedrock invoke_model 호출
→ 응답 JSON을 파싱한다
→ 텍스트 답변을 출력한다
```

Anthropic Claude 호출 시 body에 직접 넣어야 하는 주요 요소는 다음과 같다.

|요소|의미|
|---|---|
|`anthropic_version`|Anthropic API 버전|
|`max_tokens`|최대 응답 토큰 수|
|`messages`|사용자 입력 메시지 배열|

단점은 **단일 턴이라 이전 대화를 기억하지 못한다는 것**이다. 예를 들어 “내 이름은 철수야”라고 말한 뒤 “내 이름이 뭐야?”라고 물어도, 이전 대화가 전달되지 않으면 모델은 알 수 없다.

---

## 6. Lab 02 — 멀티턴 대화

모델은 자체적으로 대화를 기억하지 않는다. 따라서 맥락을 유지하려면 **이전 질문과 답변을 매번 함께 보내야 한다**.

즉, `messages` 배열에 사용자 질문과 모델 답변을 계속 누적한다.

```text
사용자 질문 1
모델 답변 1
사용자 질문 2
모델 답변 2
...
```

이렇게 하면 챗봇이 이전 내용을 기억하는 것처럼 동작한다.

하지만 단점도 있다.

|장점|단점|
|---|---|
|대화 맥락 유지 가능|히스토리를 직접 관리해야 함|
|이름, 이전 질문 등 기억 가능|대화가 길어질수록 토큰 사용량과 비용 증가|

---

## 7. Lab 03 — 스트리밍

스트리밍은 응답 전체가 한 번에 도착하는 방식이 아니라, **작은 이벤트 청크가 순차적으로 도착하는 방식**이다.

주요 이벤트 타입은 다음과 같다.

|이벤트 타입|의미|
|---|---|
|`message_start`|메시지 시작|
|`content_block_delta`|실제 텍스트 조각 도착|
|`message_stop`|메시지 종료|

학습자는 각 청크의 `type`을 확인하고, 필요한 경우 `delta.text`만 골라 출력한다. 파이썬에서는 `yield`를 사용하는 제너레이터 방식으로 구현한다.

장점은 사용자가 답변을 더 빨리 받는 것처럼 느낀다는 점이다. 단점은 raw 청크를 직접 분기하고 파싱해야 해서 코드가 복잡해진다는 점이다.

## 8. Lab 04 — `Converse`

`Converse`는 Bedrock의 고수준 API다.

앞선 실습에서 경험한 불편함을 줄여 준다.

|기존 불편함|Converse에서 개선되는 점|
|---|---|
|모델마다 body 형식이 다름|표준 형식으로 통일|
|system prompt 처리 방식이 번거로움|`system=` 인자로 분리|
|모델별 응답 구조 파싱 필요|더 일관된 응답 구조 사용|
|모델 교체 시 코드 수정 필요|`modelId`만 바꿔도 교체 가능|

핵심 데모는 Haiku에서 Amazon Nova로 모델을 바꿔도 **코드 변경 없이 `modelId`만 바꿔 호출할 수 있다는 점**이다.

## 9. Lab 05 — 세 API 비교

마지막 실습에서는 같은 질문을 여러 방식으로 호출해 비교한다.

비교 기준은 다음과 같다.

|비교 항목|설명|
|---|---|
|응답 내용|같은 질문에 대해 답변 품질이 어떻게 다른지 확인|
|지연 시간|어떤 방식이 더 빠르게 느껴지는지 비교|
|구현 복잡도|직접 처리해야 하는 코드 양 비교|
|선택 기준|새 프로젝트에서 어떤 API를 기본으로 쓸지 판단|

정리하면, 새 프로젝트에서는 보통 **`Converse`를 기본값으로 사용**하고, 세밀한 제어나 모델별 특수 기능이 필요할 때만 저수준 API로 내려가는 방식이 적절하다.

## 10. 사용 모델 정책

이 과정에서는 Claude 3 계열 대신 최신 **Claude Haiku 4.5**를 주력으로 사용한다. 실습 비용과 속도를 고려해 Sonnet, Opus 계열은 사용하지 않는다. 모델 스왑 데모에서는 제공사가 다른 **Amazon Nova**를 함께 사용한다.

|용도|모델|
|---|---|
|주력 실습|Claude Haiku 4.5|
|모델 스왑 데모|Amazon Nova|
|참고용 임베딩|Amazon Titan Embed Text v2|

신형 모델은 `us.`로 시작하는 **인퍼런스 프로파일 ID**를 사용한다. 예시는 다음과 같다.

```text
us.anthropic.claude-haiku-4-5-20251001-v1:0
us.amazon.nova-lite-v1:0
amazon.titan-embed-text-v2:0
```

## 11. 실습 환경

실습은 브라우저에서 동작하는 **code-server, 즉 웹 VS Code 환경**에서 진행한다. 개인 노트북에 Python이나 AWS CLI를 따로 설치할 필요가 없다.

실습 환경은 미리 구성된 AMI로 제공되며, 다음이 사전 설치되어 있다.

|항목|설명|
|---|---|
|Python|실습 코드 실행|
|boto3|Python AWS SDK|
|AWS CLI|AWS 명령어 사용|

실습 코드는 AMI에 포함되어 있지 않기 때문에 Lab 01에서 Git 저장소를 직접 clone한다.

```bash
git clone https://github.com/nxtcloud-org/nxt-workshop-code.git
```

자격증명은 EC2 인스턴스에 부여된 **IAM Role**을 통해 자동 처리된다. 따라서 액세스 키를 코드에 직접 적지 않는다. 호출 오류가 발생하면 `us-east-1` 리전인지, IAM Role에 신형 모델 호출 권한이 있는지 확인해야 한다.

## 12. 핵심 암기 포인트

```text
Bedrock = 여러 Foundation Model을 하나의 AWS API로 호출하는 완전관리형 서비스

invoke_model = 저수준 API
→ body 직접 작성
→ 응답 직접 파싱
→ 자유롭지만 번거로움

멀티턴 = 모델이 기억하는 것이 아니라 messages 배열에 이전 대화를 다시 보내는 것

스트리밍 = 응답을 이벤트 청크 단위로 받아 delta.text를 골라 실시간 출력

Converse = 고수준 API
→ 표준 형식
→ system 분리
→ 모델 교체 쉬움
→ 새 프로젝트 기본 선택지

실습 구조 = UI는 제공 모듈, 학습자는 Bedrock 호출 함수만 구현

실습 모델 = Claude Haiku 4.5 중심, Nova는 모델 스왑 데모용
```

요약하면, 이 자료는 **Amazon Bedrock으로 챗봇을 만들면서 저수준 호출의 구조와 불편함을 이해하고, 최종적으로 Converse API의 장점을 체감하도록 설계된 실습 안내서**다.
