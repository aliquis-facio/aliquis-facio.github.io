## 1. AWS 인프라 구조

AWS 인프라는 크게 다음 구조로 이해하면 된다.

```text
AWS 서비스
 └─ 리전 Region
     └─ 가용영역 Availability Zone, AZ
         └─ 데이터 센터 Data Center
```

즉, **리전은 큰 지역 단위**, **가용영역은 리전 안의 독립된 데이터센터 묶음**, **엣지 로케이션은 사용자 가까이에서 콘텐츠를 빠르게 제공하는 캐시 거점**이다. 

## 2. 리전 Region

**리전은 AWS가 물리적으로 분리해 운영하는 지역 단위**이다. 쉽게 말하면 하나의 국가 또는 큰 지역 단위로 볼 수 있다.

예시:

```text
서울 리전
도쿄 리전
버지니아 북부 리전
오하이오 리전
```

리전은 여러 데이터 센터의 집합이며, AWS가 전 세계 여러 국가에서 서비스를 제공하기 위해 지역별로 나눈 단위다. 

### 리전을 사용하는 이유

첫째, **네트워크 지연 시간 감소**이다. 한국 사용자를 대상으로 하는 서비스라면 서울 리전에 배포하는 것이 해외 리전에 배포하는 것보다 응답 속도가 빠르다.

둘째, **고가용성 확보**이다. 특정 리전에 장애가 발생하더라도 다른 리전을 함께 운영하면 서비스 중단 위험을 줄일 수 있다.

셋째, **법률 및 규제 준수**이다. 어떤 국가는 개인정보나 데이터가 자국 내에 저장되어야 한다는 규정을 둘 수 있으므로, 해당 국가의 리전을 사용해야 할 수 있다. 

## 3. 가용영역 Availability Zone, AZ

**가용영역은 하나의 리전 안에서 독립적으로 운영되는 데이터 센터 단위**이다.

쉽게 비유하면 다음과 같다.

```text
리전 = 국가
가용영역 = 국가 안의 도시
데이터 센터 = 도시 안의 건물
```

하나의 리전에는 여러 개의 가용영역이 존재한다. 각 가용영역은 물리적으로 떨어져 있지만, 고속 네트워크로 연결되어 있다. 그래서 하나의 가용영역에 장애가 발생해도 다른 가용영역은 영향을 덜 받도록 설계된다. 

### AZ 이름이 계정마다 다를 수 있는 이유

AWS의 `ap-northeast-2a`, `ap-northeast-2b` 같은 AZ 이름은 모든 계정에서 실제 같은 위치를 의미하지 않을 수 있다.

예를 들어:

```text
A 계정의 ap-northeast-2a ≠ B 계정의 ap-northeast-2a
```

AWS는 계정별로 AZ 이름과 실제 데이터 센터를 다르게 매핑할 수 있다. 이렇게 하는 이유는 특정 AZ에 트래픽이나 리소스가 과도하게 몰리는 것을 막고, 보안상 특정 위치가 노출되는 문제를 줄이기 위해서다. 

## 4. 엣지 로케이션 Edge Location

**엣지 로케이션은 사용자에게 콘텐츠를 더 빠르게 제공하기 위한 캐싱 거점**이다.

주로 AWS CloudFront 같은 CDN 서비스에서 사용된다.

예를 들어 한국 서버에 있는 콘텐츠를 아프리카 사용자가 요청하면, 한국 서버까지 직접 접근해야 하므로 지연 시간이 커질 수 있다. 이때 아프리카 근처의 엣지 로케이션에 콘텐츠를 캐싱해두면, 사용자는 더 가까운 위치에서 데이터를 받아올 수 있다. 

정리하면:

```text
원본 서버 → 엣지 로케이션에 콘텐츠 캐싱 → 사용자에게 빠르게 응답
```

엣지 로케이션은 정적 파일, 이미지, 동영상, 웹 페이지 캐시 등에 자주 사용된다.

## 5. 글로벌 서비스와 리전 서비스

AWS 서비스는 제공 범위에 따라 **글로벌 서비스**와 **리전 서비스**로 나뉜다. 

| 구분      | 의미                     | 예시                                                 |
| ------- | ---------------------- | -------------------------------------------------- |
| 글로벌 서비스 | 모든 리전에서 공통적으로 사용되는 서비스 | IAM, Route 53, CloudFront, WAF, Global Accelerator |
| 리전 서비스  | 특정 리전에 종속되는 서비스        | EC2, S3, ECS, EKS, Lambda                          |

주의할 점은 **서비스마다 리전 선택 여부가 다르다**는 것이다.  
예를 들어 EC2 인스턴스를 만들 때는 리전을 선택해야 하지만, IAM 사용자나 역할은 전역적으로 관리된다.
