# 로그인이란?

로그인(login)이란 운영체제, 웹사이트, 애플리케이션 등의 서비스를 이용하기 위해 사용자가 자신의 계정으로 접속하는 과정이다.  
즉, 서비스를 제공받을 수 있는 사용자인지 확인하기 위해 **사용자를 식별하고 인증하는 절차**라고 할 수 있다.

위키백과에서는 로그인을 다음과 같이 설명한다.

> 로그인(login)과 로그아웃(logout)은 접근 허가 증명을 얻기 위해 사용자 인증으로 개인이 컴퓨터 시스템에 접근하는 작업을 말한다.

나무위키에서는 다음과 같이 설명한다.

> 운영체제나 웹사이트 등에서 자신의 계정을 이용하기 위해 사용자 인증을 하여 자신의 계정으로 접속하는 것

정리하면, 로그인은 단순히 ID와 비밀번호를 입력하는 행위가 아니라,  
서비스가 사용자를 구분하고 해당 사용자가 실제 계정 소유자인지 확인하는 과정이다.

---

# 식별과 인증

로그인 과정은 크게 **식별(Identification)** 과 **인증(Authentication)** 으로 나눌 수 있다.

## 1. 식별

식별이란 여러 사용자 중에서 특정 사용자가 누구인지 분별하는 과정이다.

예를 들어 로그인 화면에서 사용자가 ID나 이메일을 입력하면, 서버는 해당 정보를 바탕으로 어떤 사용자인지 찾는다.

```text
사용자 입력: user01
서버 판단: user01이라는 계정이 존재하는가?
````

이때 사용되는 정보가 **식별 정보**이다.

대표적인 식별 정보는 다음과 같다.

* ID
* 이메일
* 전화번호
* 사용자 번호
* 학번
* 사번

식별 정보는 사용자를 구분하기 위한 값이므로 일반적으로 **유일성**이 필요하다.
즉, 같은 시스템 안에서 동일한 ID를 여러 사용자가 동시에 사용할 수 없어야 한다.

데이터베이스 관점에서는 사용자를 구분하기 위해 `id` 값이나 `user_id` 같은 값을 사용하며, 이는 보통 기본키(PK, Primary Key) 또는 고유 제약 조건을 가진 값으로 관리된다.

```text
식별 정보 = 사용자를 구분하기 위한 정보
예: ID, 이메일, 사용자 PK
```

---

## 2. 인증

인증이란 식별된 사용자가 실제로 그 계정의 소유자인지 확인하는 과정이다.

예를 들어 사용자가 ID와 비밀번호를 입력하면, 서버는 먼저 ID로 사용자를 찾고, 입력된 비밀번호가 해당 사용자의 비밀번호와 일치하는지 확인한다.

```text
사용자 입력:
ID: user01
PW: 1234

서버 확인:
user01 계정의 비밀번호가 1234와 일치하는가?
```

이때 사용되는 정보가 **인증 정보**이다.

대표적인 인증 정보는 다음과 같다.

* 비밀번호
* OTP 번호
* 생체 정보
* 인증서
* 보안 토큰
* 이메일 인증 코드
* SMS 인증 코드

```text
인증 정보 = 사용자가 본인임을 증명하기 위한 정보
예: PW, OTP, 인증 코드
```

인증 정보는 유출될 경우 계정 탈취로 이어질 수 있으므로 매우 민감하게 관리해야 한다.
특히 비밀번호는 데이터베이스에 평문으로 저장하면 안 되며, 반드시 해시 처리하여 저장해야 한다.

---

# 로그인 과정 예시

일반적인 로그인 과정은 다음과 같다.

```text
1. 사용자가 ID와 PW를 입력한다.
2. 서버는 ID를 이용해 사용자를 찾는다.
3. 사용자가 존재하지 않으면 로그인에 실패한다.
4. 사용자가 존재하면 입력한 PW와 저장된 PW를 비교한다.
5. PW가 일치하면 인증에 성공한다.
6. 서버는 세션 또는 토큰을 발급한다.
7. 사용자는 발급받은 세션 또는 토큰을 이용해 서비스를 이용한다.
```

이를 식별과 인증으로 나누면 다음과 같다.

```text
ID 입력 → 식별
PW 입력 → 인증
로그인 성공 → 접근 권한 부여
```

# 식별 정보와 인증 정보의 차이

| 구분    | 설명                | 예시              |
| ----- | ----------------- | --------------- |
| 식별 정보 | 사용자가 누구인지 구분하는 정보 | ID, 이메일, 사용자 PK |
| 인증 정보 | 사용자가 본인임을 증명하는 정보 | PW, OTP, 인증 코드  |
| 목적    | 사용자 구분            | 사용자 검증          |
| 유출 위험 | 개인정보 노출, 계정 대상화   | 계정 탈취 가능        |
| 저장 방식 | 유일성 보장 필요         | 암호화 또는 해시 필요    |

예를 들어 `user01`이라는 ID는 사용자를 찾기 위한 식별 정보이고, `1234`라는 비밀번호는 해당 사용자가 실제 계정 소유자인지 확인하기 위한 인증 정보이다.

# 비밀번호 관리의 중요성

비밀번호는 대표적인 인증 정보이기 때문에 안전하게 관리해야 한다.

데이터베이스에 비밀번호를 그대로 저장하면, DB가 유출되었을 때 모든 사용자의 계정이 위험해진다.

따라서 비밀번호는 다음과 같이 관리해야 한다.

```text
나쁜 예:
PW = 1234

좋은 예:
PW = hash(1234 + salt)
```

비밀번호 보관 시 고려해야 할 사항은 다음과 같다.

* 평문 저장 금지
* 해시 함수 사용
* salt 적용
* 강력한 해시 알고리즘 사용
* 로그인 실패 횟수 제한
* 비밀번호 재사용 제한
* 비밀번호 찾기 기능 보안 강화

# JWT와 로그인

JWT(JSON Web Token)는 로그인 성공 후 사용자 인증 상태를 유지하기 위해 사용할 수 있는 토큰 방식이다.

사용자가 로그인을 성공하면 서버는 사용자 정보를 바탕으로 JWT를 발급한다.
이후 사용자는 요청을 보낼 때 JWT를 함께 전달하고, 서버는 JWT를 검증하여 사용자가 인증된 사용자인지 확인한다.

```text
1. 사용자가 ID/PW로 로그인한다.
2. 서버가 ID/PW를 검증한다.
3. 인증에 성공하면 JWT를 발급한다.
4. 클라이언트는 JWT를 저장한다.
5. 이후 요청마다 JWT를 함께 보낸다.
6. 서버는 JWT를 검증하고 요청을 처리한다.
```

JWT는 보통 다음과 같이 사용된다.

```http
Authorization: Bearer <access_token>
```

JWT에는 사용자 식별자, 권한, 만료 시간 등의 정보가 포함될 수 있다.

```json
{
  "userId": 1,
  "role": "USER",
  "exp": 1710000000
}
```

다만 JWT는 탈취되면 공격자가 해당 토큰을 이용해 인증된 사용자처럼 행동할 수 있다.
따라서 JWT 역시 안전하게 관리해야 한다.

JWT 보안에서 중요한 점은 다음과 같다.

* 짧은 만료 시간 설정
* HTTPS 사용
* Access Token과 Refresh Token 분리
* Refresh Token 저장소 관리
* 토큰 탈취 대응 로직 구현
* 로그아웃 시 토큰 무효화 전략 고려

# OTP 번호 인증 우회 취약점

OTP는 One-Time Password의 약자로, 한 번만 사용할 수 있는 인증 번호를 의미한다.
일반적으로 비밀번호 외에 추가 인증 수단으로 사용된다.

```text
ID/PW 인증 + OTP 인증 = 2단계 인증
```

하지만 OTP 인증 과정이 잘못 구현되면 공격자가 OTP 검증을 우회할 수 있다.

예를 들어 다음과 같은 경우 취약점이 발생할 수 있다.

## 1. OTP 검증 없이 인증 성공 처리

서버가 OTP 값을 실제로 검증하지 않고, 클라이언트에서 전달한 성공 여부만 믿는 경우이다.

```text
클라이언트 요청:
otpVerified=true

서버 처리:
otpVerified가 true이므로 로그인 성공
```

이 방식은 매우 위험하다.
공격자가 요청 값을 조작하여 `otpVerified=true`로 보내면 OTP를 몰라도 인증을 통과할 수 있기 때문이다.

---

## 2. OTP 검증을 클라이언트에서만 수행

OTP 검증이 프론트엔드에서만 이루어지고 서버에서는 검증하지 않는 경우도 위험하다.

```text
프론트엔드:
OTP가 맞으면 다음 페이지로 이동

서버:
OTP 검증 여부 확인 안 함
```

공격자는 브라우저 개발자 도구나 프록시 도구를 이용해 프론트엔드 검증 로직을 우회할 수 있다.

보안 검증은 반드시 서버에서 수행해야 한다.

---

## 3. OTP 재사용 가능

OTP는 한 번만 사용할 수 있어야 한다.
하지만 이미 사용한 OTP를 다시 사용할 수 있다면 공격자가 탈취한 OTP를 재사용할 수 있다.

```text
정상 사용자:
OTP 123456 사용

공격자:
동일한 OTP 123456 재사용
```

이를 막기 위해 OTP는 검증 성공 후 즉시 만료 처리해야 한다.

---

## 4. OTP 만료 시간이 너무 김

OTP의 유효 시간이 너무 길면 공격자가 OTP를 알아낼 가능성이 높아진다.

```text
취약한 예:
OTP 유효 시간 30분

권장 예:
OTP 유효 시간 3분 또는 5분
```

OTP는 짧은 시간 동안만 유효해야 한다.

---

## 5. 무차별 대입 공격 방어 부족

OTP는 보통 6자리 숫자로 구성된다.
경우의 수는 000000부터 999999까지 총 100만 개이다.

만약 서버가 OTP 입력 횟수를 제한하지 않는다면 공격자는 여러 번 요청을 보내 OTP를 맞힐 수 있다.

```text
000001
000002
000003
...
999999
```

따라서 OTP 검증에는 반드시 시도 횟수 제한이 필요하다.

# OTP 우회 취약점 방어 방법

OTP 인증 우회 취약점을 방어하기 위해서는 다음과 같은 보안 처리가 필요하다.

* OTP 검증은 반드시 서버에서 수행한다.
* 클라이언트가 보낸 인증 성공 여부를 신뢰하지 않는다.
* OTP는 한 번 사용되면 즉시 폐기한다.
* OTP에 짧은 만료 시간을 설정한다.
* OTP 입력 실패 횟수를 제한한다.
* 동일 사용자 또는 동일 IP의 반복 요청을 제한한다.
* 인증 단계별 상태를 서버에서 관리한다.
* 로그인 성공 처리는 모든 인증 절차가 끝난 후 수행한다.

# 정리

로그인은 서비스를 이용하려는 사용자가 누구인지 확인하고, 실제 계정 소유자가 맞는지 검증하는 과정이다.

로그인 과정에서 ID는 사용자를 구분하기 위한 **식별 정보**이고, PW는 사용자가 본인임을 증명하기 위한 **인증 정보**이다.

식별 정보는 유일해야 하며, 인증 정보는 안전하게 보호되어야 한다.

로그인 성공 후에는 세션이나 JWT 같은 인증 유지 수단을 사용하여 사용자의 로그인 상태를 관리한다.

또한 OTP와 같은 추가 인증 수단을 사용할 때에도 서버 측 검증, 재사용 방지, 만료 시간, 시도 횟수 제한 등을 제대로 구현하지 않으면 인증 우회 취약점이 발생할 수 있다.

따라서 로그인 기능은 단순한 입력 비교 기능이 아니라, 사용자 식별, 인증, 권한 부여, 인증 상태 유지, 인증 정보 보호까지 포함하는 중요한 보안 기능이다.
