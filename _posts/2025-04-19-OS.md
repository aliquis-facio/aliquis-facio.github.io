# 2장
## 2.1 Virtualizing the CPU
운영체제는 하나의 물리 CPU 위에서 여러 프로세스가 동시에 실행되는 것처럼 보이도록 스케줄링과 컨텍스트 스위칭을 활용합니다. 각 프로세스는 고유의 가상 CPU를 부여받아 실행되고, 타이머 인터럽트를 통해 OS가 주기적으로 실행 중인 프로세스를 중단하고 다른 프로세스로 전환합니다. 이러한 시간 분할(time‑sharing) 기법을 통해 사용자에게 무한에 가까운 가상 CPU를 제공하는 것이 CPU 가상화의 핵심입니다. 

## 2.2 Virtualizing Memory
각 프로세스는 독립된 **가상 주소 공간(address space)**을 가지며, OS는 하드웨어 MMU와 페이지 테이블을 통해 이 가상 주소를 물리 메모리와 매핑합니다. 그 결과, 모든 프로세스가 같은 가상 주소(예: 0x200000)를 사용하더라도 서로 간섭 없이 독립적으로 메모리를 사용할 수 있습니다. 이는 프로세스 격리(isolation)와 메모리 보호를 가능케 하는 핵심 메커니즘입니다. 

## 2.3 Concurrency
동시성 문제는 여러 프로세스나 스레드가 비원자적(non‑atomic) 연산(예: counter++)을 수행할 때 발생합니다. OS는 스레드 생성/관리, 문맥 전환, 그리고 동기화 프리미티브(mutex, semaphore 등)를 제공하여 이러한 경쟁 상태(race condition)를 제어하며, 올바른 병행 실행을 보장합니다. 

## 2.4 Persistence
데이터의 영속성을 위해 OS는 파일 시스템을 통해 디스크(I/O 장치)에 데이터를 저장·관리합니다. 프로세스는 open(), write(), close() 시스템 콜을 호출하며, 파일 시스템은 저널링(journaling) 또는 카피-온-라이트(copy‑on‑write) 같은 기술로 쓰기 순서를 관리하여 시스템 장애 후에도 일관된 상태로 복구할 수 있도록 합니다. 

## 2.5 Design Goals
운영체제 설계 시 주요 목표는 다음과 같습니다:

추상화(abstraction) 제공
성능 최적화(과도한 오버헤드 최소화)
보호(protection) 및 격리(isolation)
신뢰성(reliability)
에너지 효율, 보안(security), 이동성(mobility) 등
각 목표 간의 트레이드오프를 이해하고 균형을 맞추는 것이 핵심 설계 과제입니다. 

## 2.6 Some History
초기 운영체제는 단순히 라이브러리 형태로 제공되었고, 배치 처리(batch processing) 방식이 일반적이었습니다. 이후 시스템 콜 개념이 도입되면서 보호 메커니즘이 강화되었고, 시간 공유(time‑sharing), 다중 프로그래밍, 다단계 스케줄러 등이 발전하여 오늘날의 운영체제로 진화했습니다. 

## 2.7 Summary
이 장에서는 운영체제의 세 가지 핵심 기능인 가상화(virtualization), 동시성(concurrency), **영속성(persistence)**을 소개하고, 이를 구현하기 위한 메커니즘과 정책, 그리고 설계 목표와 역사적 맥락을 개괄적으로 살펴보았습니다. 이후 장에서는 이 개념들을 실제 예제와 더 깊은 이론으로 확장해 나갑니다.

# 3장
## 복숭아 비유(Peach Analogy)
교수는 하나의 물리적 복숭아(physical peach)를 여러 명의 먹는 이에게 각각의 가상 복숭아(virtual peach)로 나누어 제공한다고 말합니다. 여기서 ‘가상화’란 단일 자원을 시분할(time‑sharing)하여, 각 사용자에게는 마치 자신만의 자원이 있는 것처럼 보이게 만드는 기법을 가리킵니다. 

## CPU 가상화 설명
복숭아 비유를 컴퓨터 자원에 적용하여, 단일 물리 CPU를 컨텍스트 스위칭(context switching)과 타이머 인터럽트를 통해 여러 개의 가상 CPU로 보이도록 하는 방식을 설명합니다. 애플리케이션은 자신만의 CPU를 독립적으로 사용하는 것처럼 실행됩니다. 

## 가상화의 의의
이 장은 가상화가 운영체제의 핵심 추상화 계층임을 강조하며, 이후 메모리·I/O 등 다른 자원에 대한 가상화 기법을 다룰 준비를 합니다. 가상화는 자원 관리의 유연성과 확장성을 제공하여 현대 운영체제 설계의 토대를 이룹니다.

# 4장
## 4.1 The Abstraction: A Process
프로세스는 “실행 중인 프로그램”이라는 가장 기본적인 운영체제 추상화입니다. 이 추상화는 기계 상태(machine state)를 캡처하는데, 여기에는 프로세스가 접근하는 메모리(address space), CPU 레지스터(PC, SP 등), 그리고 I/O 상태(열린 파일 디스크립터 목록 등)가 포함됩니다. 이들을 묶어 ‘프로세스’를 정의함으로써, OS는 사용자에게 여러 독립 실행 환경을 제공할 수 있습니다 

## 4.2 Process API
운영체제는 프로세스를 제어하기 위해 다음과 같은 인터페이스를 제공합니다:
Create: 새 프로세스 생성 (예: 셸 명령 실행)
Destroy: 프로세스 강제 종료 (예: kill 명령)
Wait: 다른 프로세스가 종료될 때까지 대기
Suspend/Resume: 실행 일시 중단 및 재개
Status: 실행 시간, 현재 상태 등 프로세스 정보 조회 

## 4.3 Process Creation: A Little More Detail
OS가 프로그램을 프로세스로 만드는 주요 단계는 다음과 같습니다:
코드/정적 데이터 로딩: 디스크의 실행 파일을 메모리에 적재 (초기 OS는 eager, 최신 OS는 필요 시 lazy 로딩)
스택 할당 및 초기화: argc, argv 등을 채워 함수 호출 준비
힙 초기화: malloc() 호출을 위한 초기 힙 메모리 설정
I/O 초기화: 표준 입출력용 파일 디스크립터 할당
제어권 이전: main() 진입점으로 점프해 프로세스 실행 시작 

## 4.4 Process States
프로세스는 세 가지 주요 상태를 가집니다:
Running: CPU에서 실행 중
Ready: 실행 가능하지만 스케줄링 대기
Blocked: I/O 완성 등 이벤트 대기

상태 전이는 스케줄러의 scheduled/descheduled 이벤트나 I/O initiate/complete 이벤트로 발생하며, 상태 다이어그램(Fig. 4.2)과 트레이스 예제(Fig. 4.3, 4.4)를 통해 이해할 수 있습니다

fork(), exec(), wait() 주요 API

fork()는 부모 프로세스에 자식 PID를, 자식 프로세스에 0을 반환하며, 실패 시 –1을 반환하고 errno를 설정한다.

6.1 기본 기법: Limited Direct Execution
OS는 사용자 프로그램을 가능한 한 직접 CPU에서 실행하도록 하고, 시스템 콜이나 예외 발생 시에만 개입하는 방식.

부팅 시 커널이 trap table을 설정하여 시스템 콜·인터럽트 발생 시 어느 코드로 점프할지 하드웨어에 알려둠.

프로세스 시작 시 커널은 프로세스 리스트 항목 생성, 메모리 할당, 스택 초기화 등을 수행한 뒤 return-from-trap으로 사용자 모드로 전환해 main()을 실행. 

6.2 문제 #1: 제한된 연산(Restricted Operations)
사용자가 임의로 커널 코드를 호출하거나 임의 주소로 점프하는 것을 방지해야 함.

사용자 모드에서 privileged instruction(예: trap table 설정)을 실행하려 하면 하드웨어가 거부.

시스템 콜은 번호 기반 호출로 구현되어 사용자 코드는 단순히 번호만 지정하고, 커널이 유효성 검사 후 해당 서비스로 분기. 

6.3 문제 #2: 프로세스 전환(Switching Between Processes)
사용자 프로그램이 실행 중인 동안 OS는 CPU를 제어할 수 없음.

협력적 접근: 프로세스가 system call이나 illegal operation을 수행할 때만 CPU 반환.

비협력적 접근: 하드웨어 타이머 인터럽트를 설정해 주기적으로 강제 trap 발생 → OS가 컨텍스트 스위치 수행. 

6.4 동시성 걱정(Worried About Concurrency?)
인터럽트 중첩 문제: 시스템 콜 처리 도중 타이머 인터럽트가 오면 어떻게 되는가?

8장. Scheduling: The Multi‑Level Feedback Queue
상세 내용 요약
Multi‑Level Feedback Queue(MLFQ)는 프로세스의 동작 이력을 기반으로 동적 우선순위를 조정하는 스케줄링 기법입니다. 핵심 아이디어는 여러 개의 우선순위 큐를 두고, 새로운 작업은 최상위 큐에 배치한 뒤 시간이 지날수록 CPU 사용량에 따라 낮은 큐로 강등(demotion)시키며, 일정 주기마다 최고 큐로 다시 승격(boost)하는 방식입니다. 이를 통해 짧은 작업(interactive jobs)에 유리하면서도 긴 작업(CPU‑bound jobs)이 굶주리지 않도록 설계합니다. 

MLFQ의 주요 규칙은 다음과 같습니다:

우선순위가 높은 큐의 작업이 실행된다.

동일 우선순위 큐 내에서는 라운드 로빈(time‑slice)으로 실행된다.

새 작업은 항상 최상위 큐에 들어간다.

주어진 레벨에서 할당 시간을 모두 사용하면 한 단계 낮은 큐로 강등된다.

일정 시간 S마다 모든 작업을 최상위 큐로 승격시켜 starvation을 방지한다. 

파라미터(큐 수, 각 큐의 time‑slice, boost 간격 S 등)는 워크로드와 시스템 특성에 맞춰 튜닝해야 하며, 과도한 튜닝 상수(voo‑doo constants) 사용을 피하기 위한 자동화 연구도 진행 중입니다.

상세 내용 요약
이 장에서는 비례 공유(Proportional‑Share) 스케줄링 개념을 도입하고, 세 가지 주요 접근 방식을 다룹니다: 무작위 기반의 Lottery Scheduling, 결정론적 방식의 Stride Scheduling, 그리고 Linux의 Completely Fair Scheduler(CFS) 
.

Lottery Scheduling: 각 프로세스에 티켓을 할당하고, 스케줄러가 무작위로 티켓을 추첨하여 프로세스를 선택함으로써 보유 티켓 수에 비례하는 CPU 시간을 제공.

Stride Scheduling: 각 프로세스에 stride = 큰 상수/티켓 수를 할당하고, 매 스케줄링 시 최소 pass 값을 가진 프로세스를 선택한 뒤 해당 프로세스의 pass에 stride를 더해 deterministic한 비례 배분을 구현.

10장. Multiprocessor Scheduling (Advanced)
상세 내용 요약
다중 CPU 시스템에서는 Single‑Queue 대 Multiple‑Queue 접근이 존재합니다.

Single‑Queue Multiprocessor Scheduling (SQMS): 모든 실행 준비된 작업을 하나의 전역 큐에 유지하여 CPU가 작업을 가져감으로써 우수한 로드 밸런싱을 제공하지만, 큐 락(lock)과 캐시 지역성(cache affinity) 측면에서 확장성에 제약이 있습니다.

Multiple‑Queue Multiprocessor Scheduling (MQMS): 각 CPU에 별도 큐를 두어 캐시 지역성과 확장성을 확보하지만, 남은 작업 수에 따라 로드 불균형(load imbalance)이 발생할 수 있습니다 
.

이러한 불균형을 해소하기 위해 Work Stealing 기법을 사용합니다. 빈 큐(idle queue)가 다른 큐(target queue)를 주기적으로 확인해 더 많은 작업이 있는 경우 일부 작업을 훔쳐(steal) 와 균형을 맞춥니다 
.

Linux 커널에서는 SQMS와 MQMS를 혼합한 다양한 멀티프로세서 스케줄러가 존재합니다. 예를 들어, O(1) Scheduler와 CFS는 MQMS 접근을, **BFS(Earliest Eligible Virtual Deadline First)**는 SQMS 접근을 사용하며, 각기 다른 설계 목표와 성능 특성을 보입니다

16.1 기본 세그멘테이션 메커니즘
가상 주소 표현: (segment, offset) 쌍으로 구분.

오프셋 처리:

순방향 증가(code, heap): 오프셋은 0부터 최대 크기까지 양수

역방향 증가(stack): 오프셋은 음수로 표현하며, 물리 주소 계산 시 base + (음수 오프셋)으로 처리

물리 주소 계산:

하드웨어가 세그먼트 레지스터에서 base와 size를 읽음

offset의 절대값이 size 이하인지 검사 (bounds check)

유효하면 PA = base + offset 계산, 벗어나면 세그멘테이션 예외 발생

동작: 읽기 전용 코드 세그먼트를 공유하고, 쓰기 권한이 필요한 데이터는 별도 세그먼트로 분리 

16.3 세분화(Granularity): Coarse vs. Fine
Coarse-grained: 코드, 데이터, 스택 등 몇 개의 대형 세그먼트 사용

Fine-grained: Multics와 같은 시스템에서는 수천 개의 작은 세그먼트를 지원

하드웨어 지원을 위해 메모리에 세그먼트 테이블을 두고, 해시나 트리 구조로 빠르게 조회

Coarse: 단순·빠르나 유연성 낮음

Fine: 유연성 높으나 하드웨어·OS 복잡도 증가

17장 상세 내용 요약
17.1 힙 확장 (Heap Growth)
UNIX 계열 시스템에서는 사용자 프로그램이 sbrk() 등의 시스템 콜을 호출해 힙을 확장하며, OS는 물리 페이지를 찾아 프로세스 주소 공간에 매핑한 뒤 새 힙 끝 주소를 반환합니다. 

17.3 기본 할당 전략 (Basic Strategies)
Best Fit: 자유 리스트에서 요청 크기 이상인 블록 중 가장 작은 블록을 선택. 내부 단편화는 줄이지만, 전체 리스트를 탐색하므로 성능 비용이 크다. 

Worst Fit: 가장 큰 블록을 선택해 할당하고 남은 공간을 다시 자유 리스트에 남겨둠. 큰 블록을 남겨 두려 하지만, 여전히 전체 리스트 탐색 비용이 크고 실험상 단편화가 심해지는 경향이 있음. 

First Fit: 리스트를 앞에서부터 탐색해 첫 번째로 맞는 블록을 선택. 탐색 비용이 적으나 리스트 앞부분에 작은 조각들이 쌓일 수 있어, 주소 기준 정렬 등을 통해 보완할 수 있음. 

Next Fit: 마지막 탐색 위치부터 시작해 첫 번째로 맞는 블록을 선택. First Fit과 유사한 성능을 보이며, 리스트 앞부분만 몰려 사용하는 현상을 일부 완화함. 

17.4 고급 기법 (Other Approaches)
Segregated Lists: 특정 크기의 요청이 빈번할 때, 그 크기 전용 리스트를 유지해 빠른 할당/해제를 지원. Solaris의 슬랩(Slab) 할당자가 대표적 예시로, 고정 크기 오브젝트 캐시를 운영해 초기화 비용을 줄이고 성능을 높임. 

Buddy Allocation: 전체 공간을 2<sup>N</sup> 크기 블록으로 나눈 뒤, 요청 크기에 맞게 재귀적으로 분할하여 할당. 해제 시 “버디” 블록이 비어 있으면 병합(coalesce)해 빠른 합체가 가능. 블록 주소가 한 비트만 달라지기 때문에 버디 계산이 간단함. 

그 외 자료구조: 성능과 확장성을 위해 분할 리스트 대신 균형 이진 트리, 스플레이 트리, 부분 순서 트리 등을 사용하기도 함. 

17.5 요약 및 트레이드오프
속도 vs. 단편화: 빠른 검색(First/Next Fit)과 낮은 단편화(Best Fit)의 상충 관계

단순성 vs. 확장성: Buddy나 Segregated Lists는 단순 구현으로도 병합/공유 기능 제공, 그러나 멀티코어 환경에서는 동시성 문제와 자료구조 잠금 비용 고려 필요

실제 시스템: glibc, jemalloc, Hoard 같은 실전용 할당기들은 위 기법들을 조합·튜닝하여 다양한 워크로드를 지원 

19장: Paging: Faster Translations (TLBs)
19.1 TLB Basic Algorithm
가상 주소에서 VPN을 추출한 뒤, 하드웨어 TLB를 조회. 히트 시 보호 비트 검사를 거쳐 오프셋을 결합해 물리 주소를 반환하고, 미스 시 예외(trap)를 발생시켜 OS(또는 하드웨어)가 페이지 테이블을 조회해 TLB를 업데이트한 뒤 명령을 재시도한다. 

19.2 Example: Accessing an Array
배열을 순차 접근할 때 공간적·시간적 지역성(locality) 덕분에 같은 페이지 내 참조는 높은 TLB 히트율을 보이고, 페이지 경계 접근 시마다 미스가 발생함을 그래프로 시각화한다. 

19.3 Who Handles the TLB Miss?

하드웨어 관리형 TLB (예: x86): 미스 시 하드웨어가 페이지 테이블 워크를 수행해 TLB를 채운다.

소프트웨어 관리형 TLB (예: MIPS, SPARC): 미스 시 CPU가 예외를 트랩으로 전환하면, 커널 코드가 페이지 테이블을 조회·TLB 업데이트 후 명령을 재시도한다. 유연성은 높으나 트랩 오버헤드가 존재한다. 

19.4 TLB Contents: What’s In There?
완전 연관(fully associative) 32~128개 엔트리로 구성. 각 엔트리는 VPN, PFN, 보호 비트(protection bits), ASID, 더티 비트, 유효 비트 등 필드를 가지며, ASID를 통해 프로세스 구분이 가능하다. 

19.5 TLB Issue: Context Switches
프로세스 간 TLB 공유 시 동일 VPN에 대해 서로 다른 PFN이 매핑될 수 있어, 모든 엔트리를 플러시하거나(오버헤드 큼) ASID를 활용해 프로세스별 엔트리를 구분 관리할 수 있다. 

19.6 Issue: Replacement Policy
LRU, 랜덤(random) 등의 정책이 대표적. LRU는 공간·시간적 지역성을 활용하지만 반복 순환 접근 패턴에 취약하며, 랜덤은 구현이 단순하고 특정 코너 케이스에 안정적이다. 

19.7 A Real TLB Entry
MIPS R4000 예시: 19bit VPN, 24bit PFN, 글로벌 비트(G), 8bit ASID, 캐시 일관성 비트(C), 더티·유효 비트 등이 포함되며, OS용으로 예약된 ‘wired’ 엔트리도 지원한다. 

19.8 Summary
하드웨어 TLB를 통해 페이지 테이블 접근을 줄여 일반적인 메모리 참조를 가상화 이전 수준과 유사하게 유지한다. 그러나 TLB 커버리지를 넘으면 성능 저하가 두드러지며, 이를 완화하기 위해 슈퍼페이지나 가상 색인 캐시(virtually‑indexed cache) 같은 기법이 사용된다. 

20.1 슈퍼페이지(Large Pages)
목적: 연속된 물리 페이지를 하나의 대형 페이지로 묶어 TLB miss를 줄이고 주소 변환 성능을 높임

하드웨어 지원: x86 등에서 2 MiB, 1 GiB 페이지 크기 지원

OS 구현: 연속된 프리 리스트 탐색, 슈퍼페이지 할당 및 해제 시 단편화 관리

장단점:

장점: TLB miss 감소 → 빠른 주소 변환

단점: 내부 단편화(internal fragmentation) 증가 가능 

20.2 페이징 + 세그멘테이션(Paging + Segmentation)
개념: 가변 길이 세그먼트 안에서 페이징을 적용하여 두 기법의 장점을 모두 활용

예시: x86 아키텍처의 코드·데이터 세그먼트마다 독립 페이지 테이블 유지

설계 고려사항:

세그먼트 권한 검사 후, 해당 세그먼트의 페이지 테이블을 통한 변환

구조 복잡도 증가 

20.3 다단계 페이지 테이블(Multilevel Page Tables)
2단계 구조:

상위 디렉터리(Page Directory) → 하위 페이지 테이블 조각(Page Table Chunks)

가상 주소 상위 비트로 디렉터리, 중간 비트로 페이지 테이블, 하위 비트로 오프셋 분할

3단계 이상:

가상 주소 공간이 매우 크거나 페이지 크기가 작아 디렉터리가 한 페이지에 안 들어갈 때 추가 레벨 도입

각 레벨별 인덱스 비트 분할로 “모든 페이지 테이블 조각이 한 페이지 내에 들어가도록” 설계

제어 흐름: TLB → PDE(PD) 읽기 → PTE 읽기 → 물리 주소 계산 (Fig.20.6)

성능: TLB miss 시 메모리 참조 2회 추가 비용 발생 

20.4 역페이지 테이블(Inverted Page Tables)
구조: 프로세스별 페이지 테이블 대신 물리 페이지 수만큼의 엔트리 유지

엔트리: (프로세스 ID, VPN) 쌍

조회: 해시 테이블로 PFN→(PID, VPN) 매핑 찾기

특징: 메모리 절약 크지만 해시 조회 오버헤드 존재 

20.5 페이지 테이블 스와핑(Page Table Swapping)
문제: 표를 메모리에 모두 상주시키기 어렵다면 커널 가상 메모리에 두고 디스크로 스와핑

사용 예: VAX/VMS 같은 메모리 제약 큰 시스템에서 활용 

20.6 요약 및 설계 트레이드오프
시간 vs 공간:

큰 테이블 → 빠른 조회(적은 메모리 참조)지만 메모리 소비 큼

작은 테이블(역페이지, 다단계) → 메모리 절약하지만 조회 비용 증가

TLB 역할: 하드웨어 TLB 존재 시 대부분의 번거로운 페이지 테이블 접근 생략 가능

혁신 과제: 워크로드·시스템 제약에 맞춰 새로운 데이터 구조 고안 유도
